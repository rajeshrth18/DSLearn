Section 1 — What is a Hash Table?

A Hash Table (also called a hash map or dictionary) is a data structure that stores key-value pairs and provides — on average — O(1) time for insertion, deletion, and lookup. This is the fastest possible search time. It achieves this by converting any key into an array index via a mathematical function called a hash function.

🌍 Real-World Analogies:
  • Phone Book: You look up "Smith" → instantly know to flip to page 782 (the S section). You don't scan every name — you compute the page directly.
  • Locker Room: You're assigned locker #47. To retrieve your bag, you go directly to locker 47 — no searching needed.
  • Database Index: A SQL index on an email column is a hash table — finding a user by email is O(1), not O(n) full-table scan.
  • DNS Resolution: Your browser converts "google.com" → IP address instantly using a massive hash table cache (DNS cache).
HASH TABLE CONCEPT — Storing student grades Key (string) Hash Function Bucket Index Value ───────────────────────────────────────────────────────── "Alice" ──→ hash("Alice") = 3 ──→ [3]: 92 "Bob" ──→ hash("Bob") = 7 ──→ [7]: 85 "Charlie" ──→ hash("Charlie")= 1 ──→ [1]: 78 "Diana" ──→ hash("Diana") = 3 ──→ COLLISION! (same bucket as Alice) Internal array (buckets): Index: [0] [1] [2] [3] [4] [5] [6] [7] [ null | "Charlie" | null | "Alice","Diana" | null | null | null | "Bob" ] ↑ ↑ empty collision handled by chaining Lookup "Bob": hash("Bob") = 7 → go to index 7 → found! O(1) Lookup "Eve": hash("Eve") = 5 → go to index 5 → empty → not found! O(1)
📌 Key Insight — Why O(1)?
Array access by index is always O(1). A hash table converts ANY key (string, object, int) into an array index in O(1) time. So: compute index → access array slot = O(1) total. The hash function is the magic bridge between arbitrary keys and array indices.

Hash tables are used everywhere in software:

  • C# Dictionary<K,V> — key-value storage
  • C# HashSet<T> — unique membership testing
  • Compiler symbol tables — variable name → memory address
  • Caching layers (Redis, Memcached) — key → cached response
  • Counting frequencies — word → count in document analysis
  • Deduplication — "have I seen this item before?"
Section 2 — Hash Functions

A hash function takes a key of any type and converts it to a non-negative integer (the hash code), which is then mapped to a bucket index using modulo arithmetic: index = hashCode % bucketCount.

2.1 — Properties of a Good Hash Function

PropertyMeaningWhy It Matters
DeterministicSame key always → same hashYou must find what you stored
FastO(1) computationDefeats the purpose if slow
UniformDistributes keys evenly across bucketsFewer collisions = better performance
Avalanche EffectSmall input change → large hash changePrevents clustering of similar keys

2.2 — Hash Function Examples

INTEGER HASHING — Simple modulo Key: 42, Bucket count: 10 hashCode = 42 index = 42 % 10 = 2 → store in bucket [2] Key: 157, Bucket count: 10 index = 157 % 10 = 7 → store in bucket [7] Key: 52, Bucket count: 10 index = 52 % 10 = 2 → COLLISION with bucket [2]! Bad bucket count: 10 (even numbers cluster in even buckets) Good bucket count: prime numbers (11, 23, 47, 97...) distribute better
STRING HASHING — Polynomial rolling hash (used by .NET internally) hash("cat"): Start: h = 0 'c' (99): h = (h * 31 + 99) = 99 'a' (97): h = (h * 31 + 97) = 99 * 31 + 97 = 3166 't' (116): h = (h * 31 + 116) = 3166 * 31 + 116 = 98262 Each character multiplied by 31^position — prime multiplier avoids patterns Final: index = 98262 % bucketCount "cat" and "act" produce DIFFERENT hashes → good avalanche effect "cat" and "cats" produce very different hashes → good distribution
// How .NET computes GetHashCode() for your objects

// Built-in types have optimized hash codes:
int    n = 42;       Console.WriteLine(n.GetHashCode());        // 42
string s = "hello";  Console.WriteLine(s.GetHashCode());        // stable within process

// Custom class — override GetHashCode() for use in Dictionary/HashSet
public class Point
{
    public int X, Y;

    // Bad: don't override GetHashCode alone without Equals
    public override int GetHashCode()
        => HashCode.Combine(X, Y);  // .NET 8 — best practice

    public override bool Equals(object? obj)
        => obj is Point p && p.X == X && p.Y == Y;
}

// HashCode.Combine internally uses a fast hash algorithm (XXHash or similar)
// It combines multiple fields into one well-distributed hash

var map = new Dictionary<Point, string>();
map[new Point { X=1, Y=2 }] = "origin area";
Console.WriteLine(map[new Point { X=1, Y=2 }]); // works because Equals+GetHashCode match
🚨 Critical Rule — Always Override Both or Neither:
If two objects are equal (Equals returns true), they must have the same hash code. If you override Equals without overriding GetHashCode, your object will silently break when used as a Dictionary key. Always override both together using HashCode.Combine().
Section 3 — Collision Resolution

A collision occurs when two different keys hash to the same bucket index. Since any hash function with a finite number of buckets will eventually produce collisions (by the Pigeonhole Principle), every hash table implementation must handle them.

📌 Pigeonhole Principle: If you have 10 buckets and 11 keys, at least two keys must share a bucket — guaranteed. Even with a great hash function and 1000 buckets, with 731 keys, the probability of at least one collision exceeds 99.9% (Birthday Problem).

3.1 — Strategy 1: Separate Chaining

Each bucket holds a linked list (or other collection) of all key-value pairs that hash to that index. Colliding keys simply append to the list. The most common strategy — used by C# Dictionary<K,V> internally.

SEPARATE CHAINING — Multiple entries per bucket via linked list Bucket count: 5 Keys: "Alice"→92, "Bob"→85, "Eve"→91 (all hash to bucket 2 for illustration) Bucket [0]: null Bucket [1]: null Bucket [2]: ["Alice":92] → ["Bob":85] → ["Eve":91] → null Bucket [3]: null Bucket [4]: null Lookup "Bob": 1. hash("Bob") % 5 = 2 → go to bucket [2] 2. Walk the chain: "Alice"? No. "Bob"? YES → return 85 Worst case if ALL n keys in one bucket: O(n) — but avg is O(1) with good distribution Insertion "Charlie": 1. hash("Charlie") % 5 = 0 → go to bucket [0] 2. Chain is empty → insert directly O(1) average
// Conceptual representation of separate chaining
class ChainNode<TKey, TValue>
{
    public TKey   Key;
    public TValue Value;
    public ChainNode<TKey, TValue>? Next;
}

// Array of linked list heads
ChainNode<string, int>?[] buckets = new ChainNode<string, int>[16];

// To find key: hash → bucket → walk chain comparing keys
// To insert:   hash → bucket → prepend to chain (O(1))
// To delete:   hash → bucket → find and unlink node

3.2 — Strategy 2: Open Addressing

When a collision occurs, find another empty bucket using a probing sequence. All data lives directly in the array — no linked lists. Better cache performance but requires lower load factors.

OPEN ADDRESSING: LINEAR PROBING Bucket count: 7 Insert "Alice" → hash=0 → bucket[0] empty → place at [0] Insert "Bob" → hash=3 → bucket[3] empty → place at [3] Insert "Charlie"→hash=0 → bucket[0] TAKEN → probe [1] → empty → place at [1] Insert "Diana" → hash=0 → probe [1] TAKEN → probe [2] → empty → place at [2] Index: [0] [1] [2] [3] [4] [5] [6] [Alice] [Charlie] [Diana] [Bob] [ ] [ ] [ ] Problem: "Clustering" — filled slots form long runs, making future probes slow QUADRATIC PROBING — reduces clustering: probe sequence: 0, 1, 4, 9, 16... (h + 1², h + 2², h + 3²...) DOUBLE HASHING — best distribution: probe sequence: h1(key), h1+h2, h1+2*h2, h1+3*h2... (uses a second hash function for step size)

3.3 — Chaining vs Open Addressing

AspectSeparate ChainingOpen Addressing
MemoryExtra nodes (linked lists)All in-array — no overhead

DeletionSimple (unlink node)Complex (tombstone markers)

Section 4 — Load Factor & Rehashing

The load factor (α) measures how full the hash table is:

α = number of stored entries / number of buckets

As α grows, collisions increase and performance degrades. When α exceeds a threshold, the table must rehash — create a larger array and reinsert all entries.

LOAD FACTOR IMPACT — Separate Chaining n=3 entries, m=7 buckets → α = 3/7 ≈ 0.43 — sparse, fast n=7 entries, m=7 buckets → α = 1.0 — balanced, still fast n=14 entries,m=7 buckets → α = 2.0 — chains growing, slower .NET Dictionary default: rehash when α > 1.0 (doubles bucket count) Java HashMap default: rehash when α > 0.75 After rehash (double buckets from 7 to 13): All n entries must be re-hashed (because index = hashCode % bucketCount changes!) This is O(n) for the rehash — but amortized O(1) per insertion overall REHASHING SEQUENCE — Amortized Analysis: Insertions 1..7: O(1) each = 7 operations Rehash at 8: O(7) to copy = 7 operations Insertions 8..14: O(1) each = 7 operations Rehash at 15: O(14) to copy = 14 operations Total work per insertion: (7+7+7+14) / 15 ≈ 2.3 — still O(1) amortized!
// C# Dictionary automatically manages load factor and rehashing for you.
// The initial capacity affects initial performance — set it if you know the size.

// Default (starts small, rehashes as needed)
var dict1 = new Dictionary<string, int>();

// Pre-sized (avoids rehashing if you know approximate count)
var dict2 = new Dictionary<string, int>(1000); // hint: ~1000 entries

// Pre-sized HashSet
var set = new HashSet<string>(500);

// Why pre-size matters:
// Loading 1 million items without hint → ~20 rehash operations (doubles each time)
// Loading 1 million items WITH hint(1M) → 0 rehash operations → faster!
// Use when: large data loads, performance-critical code, batch insertions
Section 5 — Build Your Own Hash Table (Separate Chaining)

Building a hash table from scratch is the best way to truly understand how Dictionary<K,V> works. Here is a full generic implementation using separate chaining.

5.1 — Design

OUR HASHTABLE DESIGN HashTable<TKey, TValue> ├── Entry[] buckets — the backing array of chain heads ├── int count — total number of stored entries ├── const float MAX_LOAD = 0.9f │ ├── Put(key, value) — insert or update ├── Get(key) — retrieve value (throws if missing) ├── TryGet(key, out value) — safe retrieval ├── Remove(key) — delete entry ├── ContainsKey(key) — O(1) membership test └── Count property — number of entries Entry (linked list node): ├── TKey Key ├── TValue Value └── Entry? Next — chain pointer
public class HashTable<TKey, TValue> where TKey : notnull
{
    private class Entry
    {
        public TKey   Key;
        public TValue Value;
        public Entry? Next;
        public Entry(TKey key, TValue value) { Key = key; Value = value; }
    }

    private Entry?[] _buckets;
    private int _count;
    private const float MaxLoad = 0.9f;

    public HashTable(int capacity = 16)
    {
        _buckets = new Entry[NextPrime(capacity)];
    }

    public int Count => _count;

    private int GetBucketIndex(TKey key)
    {
        int hash = key.GetHashCode() & 0x7FFFFFFF; // strip sign bit
        return hash % _buckets.Length;
    }

    public void Put(TKey key, TValue value)
    {
        int idx = GetBucketIndex(key);
        for (var e = _buckets[idx]; e != null; e = e.Next)
        {
            if (e.Key.Equals(key))
            {
                e.Value = value; // Update existing
                return;
            }
        }
        // Not found — prepend new entry to chain
        var node = new Entry(key, value) { Next = _buckets[idx] };
        _buckets[idx] = node;
        _count++;

        if ((float)_count / _buckets.Length > MaxLoad)
            Rehash();
    }

    public TValue Get(TKey key)
    {
        if (TryGet(key, out var value)) return value!;
        throw new KeyNotFoundException($"Key '{key}' not found.");
    }

    public bool TryGet(TKey key, out TValue? value)
    {
        int idx = GetBucketIndex(key);
        for (var e = _buckets[idx]; e != null; e = e.Next)
        {
            if (e.Key.Equals(key)) { value = e.Value; return true; }
        }
        value = default;
        return false;
    }

    public bool Remove(TKey key)
    {
        int idx = GetBucketIndex(key);
        Entry? prev = null, curr = _buckets[idx];
        while (curr != null)
        {
            if (curr.Key.Equals(key))
            {
                if (prev == null) _buckets[idx] = curr.Next;
                else               prev.Next = curr.Next;
                _count--;
                return true;
            }
            prev = curr; curr = curr.Next;
        }
        return false;
    }

    public bool ContainsKey(TKey key) => TryGet(key, out _);

    private void Rehash()
    {
        int newSize = NextPrime(_buckets.Length * 2);
        var newBuckets = new Entry?[newSize];
        foreach (var head in _buckets)
        {
            var e = head;
            while (e != null)
            {
                var next = e.Next;
                int idx = (e.Key.GetHashCode() & 0x7FFFFFFF) % newSize;
                e.Next = newBuckets[idx];
                newBuckets[idx] = e;
                e = next;
            }
        }
        _buckets = newBuckets;
    }

    private static int NextPrime(int min)
    {
        if (min <= 2) return 2;
        for (int n = min | 1; ; n += 2)
        {
            bool prime = true;
            for (int i = 3; i * i <= n; i += 2)
                if (n % i == 0) { prime = false; break; }
            if (prime) return n;
        }
    }
}

5.2 — Using Our Hash Table

var ht = new HashTable<string, int>();

ht.Put("Alice", 92);
ht.Put("Bob",   85);
ht.Put("Charlie", 78);
ht.Put("Alice", 95);  // Update — replaces 92 with 95

Console.WriteLine(ht.Get("Alice"));      // 95
Console.WriteLine(ht.ContainsKey("Bob")); // True

if (ht.TryGet("Diana", out int score))
    Console.WriteLine($"Diana: {score}");
else

ht.Remove("Bob");
Console.WriteLine(ht.Count);              // 2
Section 6 — C# Dictionary<K,V> — Full API

C#'s Dictionary<TKey, TValue> is the production-ready hash table. It uses a highly optimised internal structure (open addressing with prime bucket counts in newer .NET versions) and is what you should always use in real code.

6.1 — Creation & Basic Operations

// ── Creation ──────────────────────────────────────────────
var scores = new Dictionary<string, int>();
var preSet = new Dictionary<string, int>
{
    { "Alice", 92 },
    { "Bob",   85 },
    { "Charlie", 78 }
};
var capacityHint = new Dictionary<string, int>(1000); // pre-size

// ── Add ───────────────────────────────────────────────────
scores.Add("Alice", 92);              // throws if key exists
scores["Bob"] = 85;                   // add OR update (safe, prefer this)
scores.TryAdd("Charlie", 78);         // add only if key absent, returns bool

// ── Read ──────────────────────────────────────────────────
int a = scores["Alice"];              // 92 — throws KeyNotFoundException if absent
if (scores.TryGetValue("Bob", out int b))
    Console.WriteLine($"Bob: {b}");   // 85 — safe, no exception

int diana = scores.GetValueOrDefault("Diana", 0); // 0 (not found, returns default)

// ── Membership ────────────────────────────────────────────
bool hasAlice = scores.ContainsKey("Alice");  // true  — O(1)
bool has90    = scores.ContainsValue(90);      // false — O(n)! linear scan

// ── Remove ────────────────────────────────────────────────
bool removed = scores.Remove("Bob");              // true — removes and returns bool
scores.Remove("Charlie", out int removedScore);    // also retrieves removed value

// ── Size ──────────────────────────────────────────────────
int count = scores.Count;             // O(1)
scores.Clear();                        // removes all entries

6.2 — Iteration

var city = new Dictionary<string, int>
{
    ["Tokyo"]   = 37_400_000,
    ["Delhi"]   = 31_100_000,
    ["Shanghai"] = 27_800_000
};

// Iterate key-value pairs (most common)
foreach (var kvp in city)
    Console.WriteLine($"{kvp.Key}: {kvp.Value:N0}");

// Iterate keys only
foreach (string name in city.Keys)
    Console.WriteLine(name);

// Iterate values only
foreach (int pop in city.Values)
    Console.WriteLine(pop);

// LINQ on dictionary
var megacities = city.Where(kvp => kvp.Value > 30_000_000)
                     .OrderByDescending(kvp => kvp.Value)
                     .ToDictionary(kvp => kvp.Key, kvp => kvp.Value);
// Result: { "Tokyo": 37400000, "Delhi": 31100000 }

// Deconstruct into tuple (C# 7+)
foreach (var (name, pop) in city)
    Console.WriteLine($"{name} has {pop:N0} people");

6.3 — Common Dictionary Patterns

// ── Pattern 1: Counting (Frequency Map) ───────────────────
string[] words = { "cat", "dog", "cat", "bird", "dog", "cat" };
var freq = new Dictionary<string, int>();
foreach (string w in words)
{
    if (!freq.TryGetValue(w, out int cnt)) cnt = 0;
    freq[w] = cnt + 1;
    // Or shorter: freq[w] = freq.GetValueOrDefault(w) + 1;
}
// freq: { "cat":3, "dog":2, "bird":1 }

// ── Pattern 2: Grouping ────────────────────────────────────
string[] names = { "Alice", "Bob", "Anna", "Beth", "Charlie" };
var byFirstLetter = new Dictionary<char, List<string>>();
foreach (string name in names)
{
    char key = name[0];
    if (!byFirstLetter.ContainsKey(key))
        byFirstLetter[key] = new List<string>();
    byFirstLetter[key].Add(name);
}
// byFirstLetter: { 'A':["Alice","Anna"], 'B':["Bob","Beth"], 'C':["Charlie"] }

// ── Pattern 3: Memoization (cache expensive computations) ──
var memo = new Dictionary<int, long>();
long Fib(int n)
{
    if (n <= 1) return n;
    if (memo.TryGetValue(n, out long cached)) return cached;
    return memo[n] = Fib(n - 1) + Fib(n - 2);  // cache result
}
Console.WriteLine(Fib(50)); // 12586269025 — instant! (was O(2ⁿ), now O(n))

// ── Pattern 4: Two-pass index building ────────────────────
int[] nums = { 2, 7, 11, 15 }; int target = 9;
var indexMap = nums.Select((v, i) => (v, i))
                   .ToDictionary(x => x.v, x => x.i);
foreach (var (v, i) in nums.Select((v, i) => (v, i)))
    if (indexMap.TryGetValue(target - v, out int j) && j != i)
        Console.WriteLine($"[{i}, {j}]"); // [0, 1] → nums[0]+nums[1]=9
Section 7 — C# HashSet<T>

A HashSet<T> is a Dictionary with only keys, no values. It stores unique elements and provides O(1) membership testing. Perfect for deduplication, set operations, and fast "have I seen this?" checks.

7.1 — Basic Operations

// ── Creation ──────────────────────────────────────────────
var set = new HashSet<int>();
var fromArray = new HashSet<int>(new int[] { 1, 2, 3, 2, 1 }); // {1, 2, 3} — auto-dedup

// ── Add / Remove / Contains ───────────────────────────────
set.Add(10);        // returns true  (added)
set.Add(20);        // returns true  (added)
set.Add(10);        // returns false (duplicate — silently ignored)
set.Remove(20);     // returns true  (removed)
set.Remove(99);     // returns false (not found — no exception)

bool has = set.Contains(10);  // true — O(1)
int count = set.Count;         // 1

// ── Deduplication pattern ────────────────────────────────
int[] withDupes = { 3, 1, 4, 1, 5, 9, 2, 6, 5, 3 };
int[] unique = withDupes.Distinct().ToArray();          // LINQ way
int[] unique2 = new HashSet<int>(withDupes).ToArray(); // HashSet way (faster)
// Result: { 3, 1, 4, 5, 9, 2, 6 } — order not guaranteed!

// ── Fast "seen before" check ─────────────────────────────
var seen = new HashSet<int>();
bool HasDuplicate(int[] arr)
{
    foreach (int x in arr)
        if (!seen.Add(x)) return true; // Add returns false if already present
    return false;
}
// O(n) time, O(n) space — vs O(n²) brute force nested loop

7.2 — Set Operations

SET OPERATIONS — Math sets with O(n) or O(min) complexity A = {1, 2, 3, 4, 5} B = {3, 4, 5, 6, 7} Union A ∪ B = {1, 2, 3, 4, 5, 6, 7} — all elements from both Intersect A ∩ B = {3, 4, 5} — only elements in BOTH Except A \ B = {1, 2} — elements in A but NOT in B SymDiff A △ B = {1, 2, 6, 7} — elements in one but NOT both
var A = new HashSet<int> { 1, 2, 3, 4, 5 };
var B = new HashSet<int> { 3, 4, 5, 6, 7 };

// NOTE: These methods MODIFY the calling set in-place!

// Union — A becomes A ∪ B
var union = new HashSet<int>(A);
union.UnionWith(B);
// {1, 2, 3, 4, 5, 6, 7}

// Intersection — keeps only elements in BOTH
var intersect = new HashSet<int>(A);
intersect.IntersectWith(B);
// {3, 4, 5}

// Difference — removes elements that are in B
var diff = new HashSet<int>(A);
diff.ExceptWith(B);
// {1, 2}

// Symmetric Difference — elements in one but not both
var symDiff = new HashSet<int>(A);
symDiff.SymmetricExceptWith(B);
// {1, 2, 6, 7}

// Subset / Superset testing (non-destructive)
var C = new HashSet<int> { 3, 4 };
Console.WriteLine(C.IsSubsetOf(A));    // true  — {3,4} ⊆ {1,2,3,4,5}
Console.WriteLine(A.IsSupersetOf(C));  // true  — {1,2,3,4,5} ⊇ {3,4}
Console.WriteLine(A.Overlaps(B));      // true  — they share elements
Console.WriteLine(A.SetEquals(B));     // false — not identical

💡 When to Reach for a Hash Table:
  • You need O(1) lookup by a non-integer key (strings, objects)
  • You need to count/group/categorize items
  • You need to check "have I seen this before?" efficiently
  • You want to cache (memoize) expensive computations
  • You need to eliminate duplicates
  • You're converting an O(n²) nested-loop solution to O(n)
🌍 Hash Tables in Production C# Systems:
  • ASP.NET Core routing: URL path → controller action mapping via dictionary
  • Dependency Injection: Type → implementation mapping in service container
  • EF Core change tracking: Entity primary key → tracked state dictionary
  • JWT validation: Claim type → claim value dictionary in token parsing
  • Memory caching (IMemoryCache): Cache key → cached value hash table

Key Takeaways:
• Hash tables achieve O(1) average by converting keys to array indices via hash functions
• Collisions are inevitable — chaining (linked list per bucket) is C#'s strategy
• Load factor α drives rehashing — amortized O(1) per insertion overall
• Dictionary<K,V> for key-value pairs; HashSet<T> for unique membership
