Section 1 — What Is a Tree?

A tree is a hierarchical data structure made of nodes connected by edges, with no cycles. Every tree has a root (the topmost node) and each node has zero or more children.

🌍 Real-World Trees Everywhere:
  • File system: C:\Users\Alice\Documents\report.txt — folder hierarchy is a tree
  • HTML DOM: <html> → <body> → <div> → <p>
  • Organization chart: CEO → VPs → Managers → Engineers
  • Database indexes: B+ Trees power every SQL query
  • Compiler AST: Abstract Syntax Tree represents program code
TREE ANATOMY [50] ← Root (level 0, depth 0) / \ [30] [70] ← Internal nodes (level 1) / \ / \ [20] [40] [60] [80] ← Leaf nodes (level 2, no children) Vocabulary: ┌──────────────────┬─────────────────────────────────────────┐ │ Root │ Topmost node (50) │ │ Leaf │ Node with no children (20,40,60,80) │ │ Internal node │ Has at least one child (30,50,70) │ │ Height of tree │ Longest path from root to leaf = 2 │ │ Depth of node │ Distance from root (depth of 60 = 2) │ │ Parent │ Direct ancestor (parent of 30 is 50) │ │ Child │ Direct descendant (children of 50: 30,70)│ │ Subtree │ Node + all its descendants │ │ Binary tree │ Each node has at most 2 children │ └──────────────────┴─────────────────────────────────────────┘

Section 2 — The BST Property

A Binary Search Tree enforces one critical invariant at every single node:

BST INVARIANT (must hold for EVERY node, not just root): For node N: ┌──────────────────────────────────────────────────────┐ │ ALL values in LEFT subtree < N.Value │ │ ALL values in RIGHT subtree > N.Value │ │ (equal values: convention varies — usually go right) │ └──────────────────────────────────────────────────────┘ VALID BST: INVALID BST: [8] [8] / \ / \ [3] [10] [3] [10] / \ \ / \ \ [1] [6] [14] [1] [6] [7] ← 7 is in RIGHT / \ subtree of 8 [4] [7] but 7 < 8 ✗ Check: For 6's right child (7): Is 7 < 6? No, 7 > 6 ✓ (locally OK) But is 7 in the right subtree of 8? Must also be > 8 — it's not! Invalid. → BST validation requires checking ranges, not just parent!
💡 Key Insight: The BST property is not just about a node vs its immediate parent. Every left-descendant must be less than every ancestor on the "right turn" path to it. This is why BST validation requires passing down min/max bounds — see Section 9.

The BST property allows binary search: at each node, you can discard half the tree because you know which side the target must be on. This gives O(log n) for balanced trees.

Section 4 — Insert

Inserting into a BST follows the BST property to navigate to the correct leaf position. At each node: if new value < current, go left; if >, go right; stop at null.

INSERT 5 into BST [8,3,10,1,6,4,7]: Start at root (8): 5 < 8 → go LEFT At 3: 5 > 3 → go RIGHT At 6: 5 < 6 → go LEFT At 4: 5 > 4 → go RIGHT 4.Right is null → INSERT HERE [8] / \ [3] [10] / \ [1] [6] / [4] \ [5] ← new node
// ── Recursive Insert ──────────────────────────────────────
public void Insert(int val) => _root = InsertRec(_root, val);

private TreeNode InsertRec(TreeNode? node, int val)
{
    if (node == null) return new TreeNode(val); // base: insert here
    if (val < node.Val)
        node.Left  = InsertRec(node.Left,  val);
    else if (val > node.Val)
        node.Right = InsertRec(node.Right, val);
    // val == node.Val: duplicate — ignore (or handle as needed)
    return node;
}

// ── Iterative Insert (no recursion overhead) ───────────────
public void InsertIterative(int val)
{
    var newNode = new TreeNode(val);
    if (_root == null) { _root = newNode; return; }
    TreeNode curr = _root;
    while (true)
    {
        if (val < curr.Val)
        {
            if (curr.Left == null) { curr.Left  = newNode; return; }
            curr = curr.Left;
        }
        else if (val > curr.Val)
        {
            if (curr.Right == null) { curr.Right = newNode; return; }
            curr = curr.Right;
        }
        else return; // duplicate
    }
}
// Time: O(h) — h is tree height. Space: O(h) recursive, O(1) iterative
Section 5 — Search
SEARCH 6 in BST [8,3,10,1,6,4,7]: Start at 8: 6 < 8 → go LEFT At 3: 6 > 3 → go RIGHT At 6: 6 == 6 → FOUND! ✓ Search 5 (not in tree): Start at 8: 5 < 8 → LEFT → 3 → 5 > 3 RIGHT → 6 → 5 < 6 LEFT → null → NOT FOUND
// Recursive Search
public bool Contains(int val) => SearchRec(_root, val);

private bool SearchRec(TreeNode? node, int val)
{
    if (node == null) return false;
    if (val == node.Val) return true;
    return val < node.Val
        ? SearchRec(node.Left,  val)
        : SearchRec(node.Right, val);
}

// Iterative Search — preferred for deep trees
public bool ContainsIterative(int val)
{
    var curr = _root;
    while (curr != null)
    {
        if (val == curr.Val) return true;
        curr = val < curr.Val ? curr.Left : curr.Right;
    }
    return false;
}

// Return the node itself (useful for finding min/max/successor)
public TreeNode? Find(int val)
{
    var curr = _root;
    while (curr != null)
    {
        if (val == curr.Val) return curr;
        curr = val < curr.Val ? curr.Left : curr.Right;
    }
    return null;
}

// Minimum value: go left as far as possible
public int Min()
{
    if (_root == null) throw new InvalidOperationException("Empty tree");
    var curr = _root;
    while (curr.Left != null) curr = curr.Left;
    return curr.Val;
}

// Maximum value: go right as far as possible
public int Max()
{
    if (_root == null) throw new InvalidOperationException("Empty tree");
    var curr = _root;
    while (curr.Right != null) curr = curr.Right;
    return curr.Val;
}
Section 6 — Delete

Deletion is the trickiest BST operation because we must maintain the BST property after removal. There are three cases depending on the node's children:

DELETE — 3 Cases CASE 1: Node is a LEAF (no children) ───────────────────────────────────── Delete 4 from: [3] → [4] → null Just set parent.Left = null Simple: set parent's pointer to null CASE 2: Node has ONE child ───────────────────────────────────── Delete 3 from: [5] [5] / / [3] → [4] \ [4] Replace node with its single child — bypass the node CASE 3: Node has TWO children ───────────────────────────────────── Delete 5 from: [5] [6] / \ / \ [3] [7] → [3] [7] / / [6] (removed) Strategy: Find INORDER SUCCESSOR (smallest value in RIGHT subtree) Copy its value into current node, then delete the successor (successor has at most one child — right child only) Why inorder successor? It's the next-largest value — keeps BST valid. Alternatively: use inorder PREDECESSOR (largest in left subtree)
// Delete — handles all 3 cases
public void Delete(int val) => _root = DeleteRec(_root, val);

private TreeNode? DeleteRec(TreeNode? node, int val)
{
    if (node == null) return null; // value not found

    if (val < node.Val)
        node.Left  = DeleteRec(node.Left,  val);
    else if (val > node.Val)
        node.Right = DeleteRec(node.Right, val);
    else // found the node to delete
    {
        // Case 1 & 2: 0 or 1 child — return the other child (or null)
        if (node.Left  == null) return node.Right;
        if (node.Right == null) return node.Left;

        // Case 3: 2 children — replace with inorder successor
        TreeNode successor = GetMin(node.Right);
        node.Val   = successor.Val;                         // copy successor value
        node.Right = DeleteRec(node.Right, successor.Val); // delete successor
    }
    return node;
}

// Helper: minimum node in a subtree (leftmost node)
private TreeNode GetMin(TreeNode node)
{
    while (node.Left != null) node = node.Left;
    return node;
}
// Time: O(h), Space: O(h) due to recursion
📝 Dry Run — Delete 5 from {3,5,7,6,8}:
Tree: 5→left=3, 5→right=7, 7→left=6, 7→right=8
1. Found node 5 — has two children (3 and 7)
2. GetMin(7's subtree) → go left from 7 → find 6 (no left child → 6 is min)
3. Copy 6 into node: node.Val = 6
4. Recursively delete 6 from right subtree
5. Result: 6→left=3, 6→right=7, 7→left=null, 7→right=8 ✓
Section 7 — In-Order Gives Sorted Output

In-order traversal (Left → Root → Right) of a BST always visits nodes in ascending order. This is not a coincidence — it follows directly from the BST property.

IN-ORDER TRAVERSAL OF BST = SORTED ARRAY BST: [8] / \ [3] [10] / \ \ [1] [6] [14] / \ [4] [7] In-order: LEFT → ROOT → RIGHT 1 → 3 → 4 → 6 → 7 → 8 → 10 → 14 ← sorted! ✓ Proof sketch: For any node N, all left-subtree nodes < N < all right-subtree nodes. In-order processes left subtree first (all smaller), then N, then right (all larger). Applied recursively → sorted output by induction.
// In-Order traversal — returns sorted list
public List<int> InOrder()
{
    var result = new List<int>();
    InOrderRec(_root, result);
    return result;
}
private void InOrderRec(TreeNode? node, List<int> result)
{
    if (node == null) return;
    InOrderRec(node.Left,  result);
    result.Add(node.Val);
    InOrderRec(node.Right, result);
}

// In-Order iterative (using explicit stack — avoids recursion limit)
public List<int> InOrderIterative()
{
    var result = new List<int>();
    var stack  = new Stack<TreeNode>();
    var curr   = _root;
    while (curr != null || stack.Count > 0)
    {
        while (curr != null) { stack.Push(curr); curr = curr.Left; } // go left
        curr = stack.Pop();
        result.Add(curr.Val);  // process
        curr = curr.Right;       // go right
    }
    return result;
}
// BST inorder → sorted array — O(n) time, O(h) space
Section 8 — Build BST from Sorted Array

If you insert a sorted array into a BST naively (left to right), you get a degenerate BST — essentially a linked list with O(n) operations. The solution: always pick the middle element as root to build a balanced BST.

BUILD BALANCED BST FROM [1,2,3,4,5,6,7] Mid = index 3 (value 4) → becomes root Left half: [1,2,3] → mid=1 (value 2) → left subtree root Right half: [5,6,7] → mid=1 (value 6) → right subtree root Recursion tree: [1,2,3,4,5,6,7] → root=4 [1,2,3] → root=2 [5,6,7] → root=6 [1]→1 [3]→3 [5]→5 [7]→7 Result (perfectly balanced, height = log₂7 ≈ 3): [4] / \ [2] [6] / \ / \ [1] [3][5] [7] All operations now O(log n), not O(n)!
// Build balanced BST from sorted array
public void BuildFromSorted(int[] sorted) => _root = BuildRec(sorted, 0, sorted.Length - 1);

private TreeNode? BuildRec(int[] arr, int lo, int hi)
{
    if (lo > hi) return null;
    int mid = lo + (hi - lo) / 2; // avoid overflow — same as (lo+hi)/2
    var node = new TreeNode(arr[mid]);
    node.Left  = BuildRec(arr, lo,      mid - 1);
    node.Right = BuildRec(arr, mid + 1, hi);
    return node;
}

// LeetCode 108: Convert Sorted Array to Binary Search Tree
// Same algorithm — return the root node directly
public TreeNode? SortedArrayToBST(int[] nums)
    => BuildRec(nums, 0, nums.Length - 1);
// Input:  [-10,-3,0,5,9]
// Output: balanced BST with root 0
// Time:   O(n), Space: O(log n) recursion stack
Section 9 — Validate a BST

A common mistake is to only compare each node with its direct parent. That's wrong. The correct approach passes valid value ranges (min, max) down the tree. Each node must be strictly within its inherited range.

VALIDATE BST — Range Propagation Invariant: every node N must satisfy min < N.Val < max Start: root has range (-∞, +∞) [5] range(-∞,∞) → 5 is in range ✓ / \ [3] [7] left: range(-∞,5) → 3 ✓ right: range(5,∞) → 7 ✓ / \ / \ [1] [4] [6] [8] 1:range(-∞,3) ✓ 4:range(3,5) ✓ 6:range(5,7) ✓ 8:range(7,∞) ✓ INVALID EXAMPLE: [5] / \ [3] [7] \ [6] ← 6 is in RIGHT subtree of 3 → range for this slot is (3,5) 6 is NOT in (3,5). BST violated! ✗
// Validate BST — range propagation (most reliable method)
public bool IsValidBST() => ValidateRec(_root, long.MinValue, long.MaxValue);

private bool ValidateRec(TreeNode? node, long min, long max)
{
    if (node == null) return true; // empty subtree is valid
    if (node.Val <= min || node.Val >= max) return false;
    return ValidateRec(node.Left,  min,       node.Val) &&
           ValidateRec(node.Right, node.Val, max);
}
// Use long to avoid overflow when int.MinValue/MaxValue are valid node values

// Alternative: validate via in-order traversal (must be strictly increasing)
public bool IsValidBSTInOrder()
{
    long prev = long.MinValue;
    return ValidateInOrder(_root, ref prev);
}
private bool ValidateInOrder(TreeNode? node, ref long prev)
{
    if (node == null) return true;
    if (!ValidateInOrder(node.Left, ref prev)) return false;
    if (node.Val <= prev) return false; // not strictly increasing
    prev = node.Val;
    return ValidateInOrder(node.Right, ref prev);
}
// Time: O(n), Space: O(h)

⚠️ Degenerate BST Warning: Inserting a sorted sequence into a plain BST creates a degenerate tree (essentially a linked list). All O(log n) guarantees disappear — operations become O(n). Solutions:

• Shuffle data before inserting into a plain BST
• Use a self-balancing BST: AVL Tree or Red-Black Tree (both O(log n) guaranteed)
• In C#: SortedDictionary<K,V> and SortedSet<T> are backed by Red-Black Trees
🌍 C# Standard Library BST Equivalents:
  • SortedDictionary<K,V> — Red-Black Tree, O(log n) insert/lookup, iterates in key order
  • SortedSet<T> — Red-Black Tree, O(log n), set operations (GetViewBetween, Min, Max)
  • SortedList<K,V> — Sorted array internally, O(log n) lookup, O(n) insert — good for read-heavy

Key Takeaways:
• BST invariant: all left descendants < node < all right descendants
• Insert/Search/Delete: O(h) — O(log n) balanced, O(n) degenerate
• Delete with 2 children: replace with inorder successor, then delete successor
• In-order traversal = sorted output (use for BST → sorted array)
• Build balanced BST: always pick the middle element as root
• Validate BST: pass min/max range bounds down — don't just compare with parent
