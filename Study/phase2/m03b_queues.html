Section 1 — What is a Queue? FIFO Explained

A Queue is the opposite of a stack in one key way: items are added to the back and removed from the front. This gives it the property: First In, First Out (FIFO) — the item that has waited longest is the first to be served.

🌍 Real-World Analogies:
  • Supermarket checkout line: The first customer in line is the first to be served. New customers join at the back.
  • Printer job queue: Documents print in the order they were submitted. Your document waits behind earlier ones.
  • CPU task scheduler: Operating systems use queues to manage which process gets CPU time next.
  • Breadth-First Search: In graph/tree traversal, a queue processes nodes level-by-level — the closest neighbors first.
QUEUE — FIFO Structure (First In, First Out) ENQUEUE 10,20,30 then DEQUEUE: FRONT BACK ↓ ↓ [ 10 | 20 | 30 ] Dequeue → returns 10 (first in = first out) [ 20 | 30 ] Enqueue(40): [ 20 | 30 | 40 ] Dequeue → returns 20 [ 30 | 40 ] FRONT = where items EXIT (oldest item) BACK = where items ENTER (newest item)
📌 Stack vs Queue:
  • Stack (LIFO): Most recent item is served. Good for "undo", DFS, call stack.
  • Queue (FIFO): Oldest item is served. Good for "fairness", BFS, task scheduling.
  • In an interview: If the problem needs processing in order of arrival, think Queue. If it needs reversing or the most recent context, think Stack.

Section 3 — Array-Based Queue & The Circular Array Problem

A naive array-based queue uses two indices: front and back. Enqueue increments back; Dequeue increments front. This sounds fine but has a critical problem: the array fills up at the right but has empty space at the left after dequeues.

NAIVE ARRAY QUEUE — The Problem Enqueue 10,20,30,40,50 (capacity=5): front=0 back=4 [ 10 | 20 | 30 | 40 | 50 ] ← FULL Dequeue 3 times: front=3 back=4 [ | | | 40 | 50 ] ← indices 0,1,2 are WASTED Enqueue(60): back would be 5 — OUT OF BOUNDS! But arr[0], arr[1], arr[2] are FREE! The naive approach would say "queue is full" even with 3 free slots.
⚠️ Naive Fix — Shift Elements: You could shift all elements left on every Dequeue. This makes Dequeue O(n). Never do this in production — use the circular array approach below.
Section 4 — Circular Array Queue

The solution: use modulo arithmetic to wrap the indices around. When back reaches the end of the array, it wraps back to index 0 — using the "freed" space from dequeued elements. The array acts as a ring.

CIRCULAR ARRAY QUEUE — Wrap-Around capacity = 6, indices 0-5 After Enqueue 10,20,30,40,50 then Dequeue 3: front=3, back=5, count=2 [ | | | 40 | 50 | ] Enqueue 60: back = (5+1) % 6 = 0 ← WRAPS to index 0! front=3, back=0, count=3 [ 60 | | | 40 | 50 | ] Enqueue 70: back = (0+1) % 6 = 1 front=3, back=1, count=4 [ 60 | 70 | | 40 | 50 | ] Dequeue: returns arr[front=3]=40, front = (3+1)%6 = 4 [ 60 | 70 | | | 50 | ] The circle keeps rotating — no wasted space!
public class CircularQueue<T>
{
    private T[] items;
    private int front = 0;
    private int back  = 0;   // Points to next empty slot
    private int capacity;
    public int Count { get; private set; }
    public bool IsEmpty => Count == 0;
    public bool IsFull  => Count == capacity;

    public CircularQueue(int capacity)
    {
        this.capacity = capacity;
        items = new T[capacity];
    }

    public void Enqueue(T item)
    {
        if (IsFull) Resize();
        items[back] = item;
        back = (back + 1) % capacity;  // Wrap around
        Count++;
    }

    public T Dequeue()
    {
        if (IsEmpty) throw new InvalidOperationException("Queue underflow");
        T value = items[front];
        items[front] = default!;            // Help GC
        front = (front + 1) % capacity;   // Wrap around
        Count--;
        return value;
    }

    public T Peek() => IsEmpty
        ? throw new InvalidOperationException("Queue is empty")
        : items[front];

    private void Resize()
    {
        // Copy from front in order to a new larger array
        T[] newArr = new T[capacity * 2];
        for (int i = 0; i < Count; i++)
            newArr[i] = items[(front + i) % capacity];
        front = 0;
        back = Count;
        capacity *= 2;
        items = newArr;
    }
}
Section 5 — Linked-List Queue Implementation

A queue backed by a linked list has no capacity limit. Enqueue adds to the tail; Dequeue removes from the head — both O(1) with tail and head references.

public class LinkedQueue<T>
{
    private class Node
    {
        public T Data; public Node? Next;
        public Node(T d) { Data = d; }
    }

    private Node? head;  // Front (dequeue here)
    private Node? tail;  // Back  (enqueue here)
    public int Count { get; private set; }
    public bool IsEmpty => Count == 0;

    public void Enqueue(T item)         // Add to TAIL — O(1)
    {
        Node n = new(item);
        if (IsEmpty) { head = tail = n; }
        else         { tail!.Next = n; tail = n; }
        Count++;
    }

    public T Dequeue()                  // Remove from HEAD — O(1)
    {
        if (IsEmpty) throw new InvalidOperationException("Queue empty");
        T val = head!.Data;
        head = head.Next;
        Count--;
        if (Count == 0) tail = null;
        return val;
    }

    public T Peek()
    {
        if (IsEmpty) throw new InvalidOperationException();
        return head!.Data;
    }
}
Section 6 — C#'s Built-in Queue<T>

C# provides System.Collections.Generic.Queue<T> — a circular-array based queue that auto-resizes. This is what you use in production.

Queue<string> printQueue = new Queue<string>();

// Enqueue — add to back
printQueue.Enqueue("Report.pdf");
printQueue.Enqueue("Photo.jpg");
printQueue.Enqueue("Slides.pptx");

// Peek — view front without removing
Console.WriteLine(printQueue.Peek());  // "Report.pdf"

// Dequeue — remove from front
Console.WriteLine(printQueue.Dequeue());  // "Report.pdf"
Console.WriteLine(printQueue.Count);      // 2

// TryDequeue / TryPeek — safe versions
while (printQueue.TryDequeue(out string? doc))
    Console.WriteLine($"Printing: {doc}");
// Printing: Photo.jpg
// Printing: Slides.pptx

// Contains — O(n) scan
var tasks = new Queue<int>(new[] { 1, 2, 3, 4, 5 });
Console.WriteLine(tasks.Contains(3));  // true
Section 7 — Application: BFS Level-Order Tree Traversal

The most important queue application in algorithms: Breadth-First Search (BFS). BFS visits nodes level by level — all nodes at depth 1 before depth 2, etc. A queue is the natural data structure because you process nodes in the order they were discovered (FIFO).

BFS LEVEL-ORDER TRAVERSAL Tree: 1 / \ 2 3 / \ / \ 4 5 6 7 Queue state during BFS: Start: enqueue root → [1] Process 1: dequeue 1, enqueue 2,3 → [2, 3] Level 0: 1 Process 2: dequeue 2, enqueue 4,5 → [3, 4, 5] Level 1: 2 Process 3: dequeue 3, enqueue 6,7 → [4, 5, 6, 7] Level 1: 3 Process 4: dequeue 4 → [5, 6, 7] Level 2: 4 Process 5: dequeue 5 → [6, 7] Level 2: 5 Process 6: dequeue 6 → [7] Level 2: 6 Process 7: dequeue 7 → [] Level 2: 7 BFS ORDER: 1 2 3 4 5 6 7 (level by level)
public class TreeNode
{
    public int Val;
    public TreeNode? Left, Right;
    public TreeNode(int val) { Val = val; }
}

// BFS — returns list of node values in level-order
public static List<List<int>> LevelOrder(TreeNode? root)
{
    var result = new List<List<int>>();
    if (root == null) return result;

    Queue<TreeNode> queue = new();
    queue.Enqueue(root);

    while (queue.Count > 0)
    {
        int levelSize = queue.Count;   // All nodes at current level
        var level = new List<int>();

        for (int i = 0; i < levelSize; i++)
        {
            TreeNode node = queue.Dequeue();
            level.Add(node.Val);

            if (node.Left  != null) queue.Enqueue(node.Left);
            if (node.Right != null) queue.Enqueue(node.Right);
        }
        result.Add(level);
    }
    return result;
}
// Returns [[1],[2,3],[4,5,6,7]] for the tree above
💡 BFS vs DFS Decision:
  • Use BFS (Queue) when: finding the shortest path, level-order processing, checking if a graph is bipartite
  • Use DFS (Stack/Recursion) when: exploring all paths, checking connectivity, topological sort
Section 8 — The Deque (Double-Ended Queue)

A Deque (pronounced "deck", short for Double-Ended Queue) supports adding and removing from both the front and back in O(1). It is the most flexible of the linear data structures — it can act as both a stack and a queue simultaneously.

DEQUE — Operations at BOTH ends FRONT BACK ↓ ↓ [ 1 | 2 | 3 | 4 | 5 ] AddFront(0) → [0 | 1 | 2 | 3 | 4 | 5 ] AddBack(6) → [0 | 1 | 2 | 3 | 4 | 5 | 6] RemoveFront → returns 0 RemoveBack → returns 6 PeekFront → returns 1 (no remove) PeekBack → returns 5 (no remove) Use as Stack: AddFront + RemoveFront (LIFO) Use as Queue: AddBack + RemoveFront (FIFO)
OperationDequeStackQueue
Add to front✅ O(1)✅ Push❌ O(n)
Add to back✅ O(1)❌ O(n)✅ Enqueue
Remove from front✅ O(1)❌ O(n)✅ Dequeue
Remove from back✅ O(1)✅ Pop❌ O(n)
Peek front/back✅ O(1)✅ Peek✅ Peek
🌍 Real-World Deque Uses:
  • Sliding window maximum — The most famous deque algorithm: find the max in every window of size k in O(n)
  • Browser history — Back uses stack behavior (remove from front); sometimes you also need to trim old history from the back
  • Work-stealing schedulers — Threads steal tasks from the back of other threads' deques (Java ForkJoinPool)
  • Palindrome checking — Add characters; compare front and back characters simultaneously
Section 9 — C#'s LinkedList<T> as Deque

C# does not have a dedicated Deque<T> class in the standard library. The standard approach is to use LinkedList<T> (a doubly linked list) as a deque, since it supports O(1) operations at both ends.

// LinkedList<T> as Deque
LinkedList<int> deque = new LinkedList<int>();

// Add to front (equivalent to Push / AddFirst)
deque.AddFirst(10);
deque.AddFirst(5);   // deque: 5 ⇄ 10

// Add to back (equivalent to Enqueue / AddLast)
deque.AddLast(20);
deque.AddLast(30);  // deque: 5 ⇄ 10 ⇄ 20 ⇄ 30

// Peek front and back
Console.WriteLine(deque.First!.Value);  // 5
Console.WriteLine(deque.Last!.Value);   // 30

// Remove from front (equivalent to Pop / Dequeue)
deque.RemoveFirst();  // removes 5

// Remove from back
deque.RemoveLast();   // removes 30

// deque: 10 ⇄ 20
Console.WriteLine(deque.Count);  // 2

// ── Sliding Window Maximum using Deque ──────────────────────
// For each window of size k, find the maximum element in O(n) total.
// Deque stores INDICES of useful elements (descending order of values).
// Front always holds the index of the current window's maximum.

public static int[] MaxSlidingWindow(int[] nums, int k)
{
    if (nums.Length == 0) return [];
    int[] result = new int[nums.Length - k + 1];
    LinkedList<int> dq = new();  // stores indices
    int ri = 0;

    for (int i = 0; i < nums.Length; i++)
    {
        // Remove indices outside the window
        while (dq.Count > 0 && dq.First!.Value < i - k + 1)
            dq.RemoveFirst();

        // Remove smaller elements from the back (they'll never be max)
        while (dq.Count > 0 && nums[dq.Last!.Value] < nums[i])
            dq.RemoveLast();

        dq.AddLast(i);

        // Window is full: record the max (front of deque)
        if (i >= k - 1)
            result[ri++] = nums[dq.First!.Value];
    }
    return result;
}

// nums=[1,3,-1,-3,5,3,6,7], k=3
// Output: [3,3,5,5,6,7]  — each window's max in O(n)
