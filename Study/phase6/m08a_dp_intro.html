1. What Is Dynamic Programming?

Dynamic Programming (DP) is an algorithm design technique that solves complex problems by breaking them into smaller overlapping subproblems, solving each subproblem once, and storing the result to avoid redundant computation.

DP applies when a problem has two key properties:

1. Overlapping Subproblems

The same subproblem is computed multiple times in a naive recursive approach. DP caches (memoizes) results so each is solved only once.

2. Optimal Substructure

The optimal solution to the whole problem can be constructed from optimal solutions to its subproblems — choices don't interfere with each other.

Note — DP vs Greedy: Greedy also uses optimal substructure, but makes a locally optimal choice without reconsidering. DP explores all possibilities and picks the global optimum via stored results.
DP = Recursion + Memoization OR DP = Iterative + Tabulation Problem: fib(5) Naive calls: fib(5) → fib(4) → fib(3) → fib(2) → fib(1) ↘ fib(2) [REPEATED] ↘ fib(3) [REPEATED] ↘ fib(2) [REPEATED] With DP: compute fib(0..5) once, store in table fib(0)=0, fib(1)=1, fib(2)=1, fib(3)=2, fib(4)=3, fib(5)=5

DP vs Other Techniques

TechniqueApproachWhen to Use
Brute ForceTry all possibilitiesOnly for tiny inputs
Divide & ConquerSplit into independent subproblemsSubproblems don't overlap (merge sort)
GreedyLocal optimal at each stepGreedy choice is always globally optimal
Dynamic ProgrammingOverlapping subproblems + memoizationOptimal substructure + overlapping subproblems
BacktrackingTry + undoConstraint satisfaction, no optimal substructure

2. Two Approaches: Top-Down vs Bottom-Up

Every DP problem can be solved in two ways. Understanding both is crucial for interviews.

Top-Down (Memoization)

  • Write the natural recursive solution
  • Add a cache/memo (dictionary or array)
  • Before computing, check cache first
  • Store result before returning
  • Intuitive — follows problem definition directly

Bottom-Up (Tabulation)

  • Identify the base cases
  • Fill a table iteratively in order
  • Each cell depends only on previously computed cells
  • No recursion → no call stack overhead
  • Often easier to optimize space
TOP-DOWN (Memo): BOTTOM-UP (Table): ───────────────────── ───────────────────── fib(5) dp[0] = 0 └─ fib(4) dp[1] = 1 └─ fib(3) dp[2] = dp[1]+dp[0] = 1 └─ fib(2) dp[3] = dp[2]+dp[1] = 2 └─ fib(1) = 1 dp[4] = dp[3]+dp[2] = 3 └─ fib(0) = 0 dp[5] = dp[4]+dp[3] = 5 ← fib(2) = 1 ← fib(3) = 2 No recursion! O(n) time, O(n) space ← fib(4) = 3 → Optimize to O(1) space with 2 vars ← fib(5) = 5

The 5-Step DP Framework

Apply these steps to any DP problem:

  1. Define the subproblem: What does dp[i] (or dp[i][j]) represent?
  2. Identify the recurrence: How does dp[i] relate to smaller subproblems?
  3. Determine base cases: What are the smallest inputs with known answers?
  4. Determine the computation order: Which subproblems must be solved first?
  5. Extract the answer: Which cell(s) of the table hold the final answer?
Tip: Always write the recurrence relation before coding. A clear recurrence makes the implementation almost mechanical. If you can't state the recurrence, you don't understand the problem yet.

3. Fibonacci — A Deep Dive

Fibonacci is the "Hello World" of DP. We'll evolve it through 4 stages to show every key concept.

Stage 1 — Naive Recursion: O(2ⁿ) time, O(n) space

public static int FibNaive(int n)
{
    // Base cases
    if (n <= 1) return n;
    // Exponential: computes fib(2) over 2^(n-2) times!
    return FibNaive(n - 1) + FibNaive(n - 2);
}
// fib(50) takes minutes — each call branches into 2 more calls
Call tree for fib(5): — 15 total calls, many duplicates fib(5) ┌──────────┐ fib(4) fib(3) ┌─────┐ ┌─────┐ fib(3) fib(2) fib(2) fib(1) ┌──┐ ┌──┐ ┌──┐ fib(2) fib(1) fib(1) fib(0) fib(1) fib(0) ┌──┐ fib(1) fib(0) fib(2) is computed 3×, fib(3) is computed 2× For fib(n): T(n) = T(n-1) + T(n-2) + O(1) → O(φⁿ) where φ ≈ 1.618

Stage 2 — Top-Down Memoization: O(n) time, O(n) space

public static int FibMemo(int n, Dictionary<int, int> memo = null)
{
    memo ??= new Dictionary<int, int>();

    if (n <= 1) return n;

    // 1. Check cache before computing
    if (memo.ContainsKey(n)) return memo[n];

    // 2. Compute and STORE result
    int result = FibMemo(n - 1, memo) + FibMemo(n - 2, memo);
    memo[n] = result;
    return result;
}
// Each fib(k) computed exactly once → O(n) time, O(n) space (cache + call stack)

// Alternatively: use an array if n is known upfront
public static int FibMemoArray(int n, int[] memo)
{
    if (n <= 1) return n;
    if (memo[n] != -1) return memo[n];
    memo[n] = FibMemoArray(n - 1, memo) + FibMemoArray(n - 2, memo);
    return memo[n];
}
// Initialize: int[] memo = Enumerable.Repeat(-1, n+1).ToArray();

Stage 3 — Bottom-Up Tabulation: O(n) time, O(n) space

public static int FibTab(int n)
{
    if (n <= 1) return n;

    // Recurrence: dp[i] = dp[i-1] + dp[i-2]
    int[] dp = new int[n + 1];
    dp[0] = 0;   // Base case 1
    dp[1] = 1;   // Base case 2

    for (int i = 2; i <= n; i++)
        dp[i] = dp[i - 1] + dp[i - 2]; // Fill in order

    return dp[n]; // Answer is dp[n]
}
// No recursion overhead. dp array: [0, 1, 1, 2, 3, 5, 8, 13, ...]

Stage 4 — Space-Optimized: O(n) time, O(1) space

public static int FibOptimal(int n)
{
    if (n <= 1) return n;

    // Key insight: dp[i] only depends on dp[i-1] and dp[i-2]
    // We only need 2 variables, not the whole array
    int prev2 = 0, prev1 = 1;

    for (int i = 2; i <= n; i++)
    {
        int curr = prev1 + prev2;
        prev2 = prev1;
        prev1 = curr;
    }
    return prev1;
}
// O(1) space — only 2 variables in memory regardless of n
Space Optimization Rule: If dp[i] only depends on a fixed number of previous states (like dp[i-1] and dp[i-2]), you can replace the full array with just those variables. This is a key interview technique.

4. Climbing Stairs

Problem: You can climb 1 or 2 steps at a time. How many distinct ways can you climb to step n?

n=1: [1] → 1 way n=2: [1,1] or [2] → 2 ways n=3: [1,1,1] [1,2] [2,1] → 3 ways n=4: [1,1,1,1][1,1,2][1,2,1][2,1,1][2,2] → 5 ways Pattern: ways(n) = ways(n-1) + ways(n-2) (take 1 step from n-1, OR take 2 steps from n-2) → FIBONACCI sequence! ways(1)=1, ways(2)=2

Step-by-Step Application of 5-Step Framework

  1. Subproblem: dp[i] = number of distinct ways to reach step i
  2. Recurrence: dp[i] = dp[i-1] + dp[i-2]
  3. Base cases: dp[1] = 1, dp[2] = 2
  4. Order: left to right (i = 3 to n)
  5. Answer: dp[n]
public int ClimbStairs(int n)
{
    if (n <= 2) return n;

    int prev2 = 1, prev1 = 2;

    for (int i = 3; i <= n; i++)
    {
        int curr = prev1 + prev2;
        prev2 = prev1;
        prev1 = curr;
    }
    return prev1;
}
// Time: O(n), Space: O(1)
// LeetCode 70 — Medium
Pattern Recognition: Climbing Stairs = Fibonacci with different base cases. Whenever you see "do X or Y at each step, count ways to reach end", think DP with a recurrence that sums from the choices available.

5. House Robber — Decision DP

Problem: Given an array of houses with amounts, rob the maximum money without robbing two adjacent houses. (LeetCode 198)

Input: [2, 7, 9, 3, 1] Choices at each house: ROB it (skip prev) OR SKIP it dp[0] = 2 (only house 0, must take it) dp[1] = 7 (max(2, 7) = 7, take house 1 alone) dp[2] = max(dp[1], dp[0]+9) = max(7, 11) = 11 dp[3] = max(dp[2], dp[1]+3) = max(11, 10) = 11 dp[4] = max(dp[3], dp[2]+1) = max(11, 12) = 12 ← answer Robbed: houses [0, 2, 4] → 2+9+1 = 12

Framework Application

  1. Subproblem: dp[i] = max money robbing from houses 0..i
  2. Recurrence: dp[i] = max(dp[i-1], dp[i-2] + nums[i])
    → either skip house i (keep dp[i-1]) or rob house i (add nums[i] to dp[i-2])
  3. Base cases: dp[0] = nums[0], dp[1] = max(nums[0], nums[1])
  4. Order: left to right
  5. Answer: dp[n-1]
public int Rob(int[] nums)
{
    int n = nums.Length;
    if (n == 1) return nums[0];

    // Space-optimized: only need prev2 and prev1
    int prev2 = nums[0];
    int prev1 = Math.Max(nums[0], nums[1]);

    for (int i = 2; i < n; i++)
    {
        int curr = Math.Max(prev1, prev2 + nums[i]);
        prev2 = prev1;
        prev1 = curr;
    }
    return prev1;
}
// Time: O(n), Space: O(1)

House Robber II — Circular Array (LeetCode 213)

Now the houses are in a circle — house 0 and house n-1 are adjacent. We can't rob both.

Key insight: Run House Robber I twice — once on nums[0..n-2] (exclude last) and once on nums[1..n-1] (exclude first). Take the max.

public int RobII(int[] nums)
{
    int n = nums.Length;
    if (n == 1) return nums[0];
    if (n == 2) return Math.Max(nums[0], nums[1]);

    // Run RobI on two sub-ranges, take max
    return Math.Max(
        RobRange(nums, 0, n - 2),  // Skip last house
        RobRange(nums, 1, n - 1)   // Skip first house
    );
}

private int RobRange(int[] nums, int start, int end)
{
    int prev2 = 0, prev1 = 0;
    for (int i = start; i <= end; i++)
    {
        int curr = Math.Max(prev1, prev2 + nums[i]);
        prev2 = prev1;
        prev1 = curr;
    }
    return prev1;
}
// Time: O(n), Space: O(1)

6. Min Cost Climbing Stairs

Problem: Each step has a cost. You can start at step 0 or 1. Climb 1 or 2 steps at a time. Find the minimum cost to reach the top (past the last step). (LeetCode 746)

Input: cost = [10, 15, 20] Steps: 0 1 2 3(top) dp[i] = min cost to reach step i dp[0] = cost[0] = 10 dp[1] = cost[1] = 15 dp[2] = cost[2] + min(dp[0], dp[1]) = 20 + 10 = 30 dp[3] = 0 + min(dp[1], dp[2]) = 0 + 15 = 15 ← answer Alternative: go 1→3 costs 15, total = 15 go 0→2→3 costs 10+20 = 30 (worse)
public int MinCostClimbingStairs(int[] cost)
{
    int n = cost.Length;
    // dp[i] = min cost to reach step i
    // Recurrence: dp[i] = cost[i] + min(dp[i-1], dp[i-2])
    // Answer: min(dp[n-1], dp[n-2]) — can reach top from last 2 steps

    int prev2 = cost[0];
    int prev1 = cost[1];

    for (int i = 2; i < n; i++)
    {
        int curr = cost[i] + Math.Min(prev1, prev2);
        prev2 = prev1;
        prev1 = curr;
    }
    // Can reach the top from either of the last 2 steps
    return Math.Min(prev1, prev2);
}
// Time: O(n), Space: O(1)

7. Maximum Subarray — Kadane's Algorithm

Problem: Find the contiguous subarray with the largest sum. (LeetCode 53)

Input: [-2, 1, -3, 4, -1, 2, 1, -5, 4] Answer: [4, -1, 2, 1] → sum = 6 Kadane's key insight: At each position i, either: - Extend the current subarray: prev + nums[i] - Start fresh at i: nums[i] Take whichever is larger. i: -2 1 -3 4 -1 2 1 -5 4 c: -2 1 -2 4 3 5 6 1 5 ↑ ↑ ↑ ↑ start extend restart maxSum=6

DP Formulation

  1. Subproblem: dp[i] = max subarray sum ending at index i
  2. Recurrence: dp[i] = max(nums[i], dp[i-1] + nums[i])
  3. Base case: dp[0] = nums[0]
  4. Answer: max(dp[0..n-1])
public int MaxSubArray(int[] nums)
{
    int currentSum = nums[0];
    int maxSum = nums[0];

    for (int i = 1; i < nums.Length; i++)
    {
        // Extend current or start fresh
        currentSum = Math.Max(nums[i], currentSum + nums[i]);
        maxSum = Math.Max(maxSum, currentSum);
    }
    return maxSum;
}
// Time: O(n), Space: O(1) — Kadane's Algorithm

// Extended: also return the start/end indices
public (int sum, int start, int end) MaxSubArrayWithIndices(int[] nums)
{
    int maxSum = nums[0], currSum = nums[0];
    int start = 0, end = 0, tempStart = 0;

    for (int i = 1; i < nums.Length; i++)
    {
        if (nums[i] > currSum + nums[i])
        {
            currSum = nums[i];
            tempStart = i; // Starting fresh
        }
        else
            currSum += nums[i];

        if (currSum > maxSum)
        {
            maxSum = currSum;
            start = tempStart;
            end = i;
        }
    }
    return (maxSum, start, end);
}

8. Word Break — String DP

Problem: Given a string s and a word dictionary, can s be segmented into space-separated dictionary words? (LeetCode 139)

s = "leetcode", dict = {"leet", "code"} dp[i] = can we form s[0..i-1] using dictionary words? dp[0] = true (empty string — base case) dp[1] = false (no word = "l") dp[2] = false (no word = "le") dp[3] = false (no word = "lee") dp[4] = true (dp[0]=true, s[0..3]="leet" ✓) dp[5] = false dp[6] = false dp[7] = false dp[8] = true (dp[4]=true, s[4..7]="code" ✓) Answer: dp[8] = true → "leet code"
public bool WordBreak(string s, IList<string> wordDict)
{
    int n = s.Length;
    var wordSet = new HashSet<string>(wordDict); // O(1) lookup
    bool[] dp = new bool[n + 1];
    dp[0] = true; // Empty string is always valid

    for (int i = 1; i <= n; i++)
    {
        // Try all possible last word endings
        for (int j = 0; j < i; j++)
        {
            // If s[0..j-1] can be formed AND s[j..i-1] is a word
            if (dp[j] && wordSet.Contains(s.Substring(j, i - j)))
            {
                dp[i] = true;
                break; // No need to check more j values
            }
        }
    }
    return dp[n];
}
// Time: O(n² * L) where L = avg word length, Space: O(n)
Note — Inner Loop Direction: The inner loop tries all split points. For each position i, we ask: "Is there any j where the left part s[0..j-1] was breakable AND the right part s[j..i-1] is in the dictionary?" This is a classic DP partition pattern.

9. Decode Ways — Counting DP

Problem: A message is encoded as digits 1-26 → A-Z. Given a digit string, how many ways to decode it? (LeetCode 91)

"226" → three ways: "2","2","6" → B,B,F "22","6" → V,F "2","26" → B,Z dp[i] = number of ways to decode s[0..i-1] At each position i, consider: - Take 1 digit: s[i-1] alone (valid if s[i-1] != '0') - Take 2 digits: s[i-2..i-1] (valid if 10 ≤ number ≤ 26) dp[0] = 1 (empty string → 1 way) dp[1] = 1 (single non-zero digit → 1 way; "0" → 0 ways) dp[2]: s="22" → 1-digit "2" + 2-digit "22" → 2 ways dp[3]: "6" alone (1) + "26" valid (dp[1]=1) → 1+1 = 2... but also 2-digit: "26" → dp[1]=1, 1-digit "6" → dp[2]=2, total=3
public int NumDecodings(string s)
{
    int n = s.Length;
    int[] dp = new int[n + 1];
    dp[0] = 1; // Base: empty prefix
    dp[1] = s[0] == '0' ? 0 : 1; // "0" can't be decoded

    for (int i = 2; i <= n; i++)
    {
        // Take single digit s[i-1]
        int oneDigit = s[i - 1] - '0';
        if (oneDigit != 0)           // '0' has no single-digit encoding
            dp[i] += dp[i - 1];

        // Take two digits s[i-2..i-1]
        int twoDigit = (s[i - 2] - '0') * 10 + oneDigit;
        if (twoDigit >= 10 && twoDigit <= 26) // valid: "10" to "26"
            dp[i] += dp[i - 2];
    }
    return dp[n];
}
// Time: O(n), Space: O(n)  → can optimize to O(1) with 2 vars

10. Jump Game — Reachability DP

Problem: Given an array where nums[i] = max jump from index i, can you reach the last index? (LeetCode 55)

Jump Game II: Find the minimum number of jumps to reach the last index. (LeetCode 45)

Jump Game I — Greedy (O(n))

public bool CanJump(int[] nums)
{
    int maxReach = 0; // Furthest index reachable so far

    for (int i = 0; i < nums.Length; i++)
    {
        if (i > maxReach) return false; // Can't reach i
        maxReach = Math.Max(maxReach, i + nums[i]);
    }
    return true;
}
// Greedy: track max reachable index. If we ever get stuck, return false.

Jump Game II — Greedy BFS Levels (O(n))

public int Jump(int[] nums)
{
    int jumps = 0, currEnd = 0, farthest = 0;

    for (int i = 0; i < nums.Length - 1; i++)
    {
        farthest = Math.Max(farthest, i + nums[i]);

        if (i == currEnd) // Reached end of current jump range
        {
            jumps++;          // Must make another jump
            currEnd = farthest; // Extend range to farthest reachable
        }
    }
    return jumps;
}
// Think of it as BFS levels: each "level" = one jump
nums = [2, 3, 1, 1, 4] Levels: Level 0 (start): [0] → can reach indices 1,2 Level 1 (jump 1): [1, 2] → can reach indices 2,3,4 → index 4 = last! Level 2 (jump 2): not needed Answer: 2 jumps

11. Coin Change — Unbounded Knapsack Pattern

Problem: Given coins of certain denominations, find the fewest coins needed to make amount. Infinite coins of each type. (LeetCode 322)

coins = [1, 5, 6, 9], amount = 11 dp[i] = min coins to make amount i dp[0] = 0 (0 coins for amount 0) dp[1] = 1 (one "1") dp[2] = 2 (two "1"s) ... dp[5] = 1 (one "5") dp[6] = 1 (one "6") dp[7] = 2 (6+1) dp[8] = 2 (6+1+1 or 5+1+1+1? min is still...) dp[9] = 1 (one "9") dp[10] = 2 (9+1 or 5+5) dp[11] = 2 (9+1+1? or 6+5? → 6+5=2 coins!) ← answer = 2

Framework Application

  1. Subproblem: dp[i] = min coins to make amount i
  2. Recurrence: dp[i] = 1 + min(dp[i - coin]) for each coin ≤ i
  3. Base case: dp[0] = 0
  4. Order: amount 1 to target, for each coin
  5. Answer: dp[amount] (or -1 if still infinity)
public int CoinChange(int[] coins, int amount)
{
    int[] dp = new int[amount + 1];
    Array.Fill(dp, amount + 1); // Initialize to "infinity"
    dp[0] = 0;

    for (int i = 1; i <= amount; i++)
    {
        foreach (int coin in coins)
        {
            if (coin <= i)
                dp[i] = Math.Min(dp[i], 1 + dp[i - coin]);
        }
    }
    return dp[amount] > amount ? -1 : dp[amount];
}
// Time: O(amount * coins.Length), Space: O(amount)

Coin Change II — Count Ways (LeetCode 518)

How many combinations of coins make amount exactly? (Order doesn't matter.)

public int Change(int amount, int[] coins)
{
    int[] dp = new int[amount + 1];
    dp[0] = 1; // One way to make amount 0 (take nothing)

    // CRITICAL: loop coins OUTSIDE, amount INSIDE
    // This ensures each coin denomination is used at most once per combination
    foreach (int coin in coins)
    {
        for (int i = coin; i <= amount; i++)
            dp[i] += dp[i - coin];
    }
    return dp[amount];
}
// If loops were swapped (amount outside, coins inside), you'd count PERMUTATIONS
Warning — Loop Order Matters! For counting combinations (no duplicates like {1,5} and {5,1} counted separately), iterate coins in outer loop. For counting permutations (order matters), put amount in outer loop. This is a very common interview trap.

Interview Tip — How to Identify DP:
  • The problem asks for a count, minimum, maximum, or yes/no
  • There are choices at each step that affect future choices
  • A brute-force recursion would revisit the same inputs
  • Keywords: "ways", "minimum cost", "can you reach", "maximum profit", "fewest"
