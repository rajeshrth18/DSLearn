1. GPS Navigation — Google Maps / Waze

🌎

Find the Fastest Route Through a City

Google Maps · Waze · Apple Maps · HERE Technologies
Dijkstra's Algorithm A* Search Bidirectional BFS Phase 5 — Graphs Phase 9 — Advanced Graphs Scale: 10⁹ road segments

Google Maps has over 1 billion users requesting routes every day. The road network of a country contains hundreds of millions of intersections (nodes) and roads (edges). A naive Dijkstra would be too slow — real systems use a hierarchy of optimizations.

Road Network Graph: ┌─────────────────────────────────────────┐ │ Node = Intersection (lat/lng) │ │ Edge = Road segment (direction, speed) │ │ Weight = travel time (seconds) │ └─────────────────────────────────────────┘ Challenge: London → Edinburgh (700km, millions of nodes in-between) Naive Dijkstra: explores ~10 million nodes → too slow for real-time A* with heuristic: focuses search toward destination → 100x faster Contraction Hierarchies: pre-contract road graph → 1000x faster

Core Algorithm: A* (Dijkstra + Heuristic)

A* extends Dijkstra by adding a heuristic function — an estimate of the remaining distance to the goal (usually straight-line distance). This focuses the search toward the destination and drastically reduces nodes explored.

public class AStarRouter
{
    // Node = intersection, heuristic = straight-line distance to target
    public List<int> FindRoute(int src, int dst,
        List<(int to, double time)>[] adj, double[] lat, double[] lng)
    {
        int n = adj.Length;
        double[] gScore = Enumerable.Repeat(double.MaxValue, n).ToArray();
        int[] prev = Enumerable.Repeat(-1, n).ToArray();
        gScore[src] = 0;

        // Priority queue sorted by f = g (actual) + h (heuristic)
        var pq = new PriorityQueue<int, double>();
        pq.Enqueue(src, Heuristic(src, dst, lat, lng));

        while (pq.Count > 0)
        {
            pq.TryDequeue(out int u, out _);
            if (u == dst) return ReconstructPath(prev, src, dst);

            foreach (var (v, w) in adj[u])
            {
                double tentative = gScore[u] + w;
                if (tentative < gScore[v])
                {
                    gScore[v] = tentative;
                    prev[v] = u;
                    // f = actual cost + straight-line estimate to destination
                    pq.Enqueue(v, tentative + Heuristic(v, dst, lat, lng));
                }
            }
        }
        return new List<int>(); // No path
    }

    private double Heuristic(int a, int b, double[] lat, double[] lng)
    {
        // Haversine distance (straight-line on Earth's surface)
        double dlat = lat[b] - lat[a], dlng = lng[b] - lng[a];
        return Math.Sqrt(dlat*dlat + dlng*dlng) * 111000; // rough meters
    }
}
// Real Google Maps uses Contraction Hierarchies (CH) on top of A*
// CH pre-computes shortcuts: contracts low-importance nodes, creating a hierarchy
// Result: London→Edinburgh in milliseconds on a billion-node graph
Real-World Scale: Google Maps' road network has ~300 million nodes. Standard Dijkstra explores all of them. Contraction Hierarchies reduce this to ~10,000 nodes per query. Live traffic updates modify edge weights — the system re-runs routing every few seconds for active navigations.

2. Social Network — LinkedIn / Facebook

👥

"People You May Know" — Connection Recommendations

LinkedIn · Facebook · Twitter · Instagram
BFS (Degree of Separation) Union-Find Hash Maps Phase 5 — Graphs Phase 3 — Hash Tables Scale: LinkedIn has 950M users

LinkedIn's "People You May Know" finds users who share mutual connections. This is a classic graph problem: find all nodes within 2 hops from the current user. At LinkedIn's scale, the social graph has billions of edges.

// Find 2nd-degree connections (friends of friends)
public List<int> GetSuggestions(int userId, HashSet<int>[] graph)
{
    var direct = graph[userId];   // 1st-degree connections
    var counts = new Dictionary<int, int>(); // candidate → mutual count

    foreach (int friend in direct)
    foreach (int fof in graph[friend])  // Friend-of-friend
    {
        if (fof == userId || direct.Contains(fof)) continue; // Already connected
        counts[fof] = counts.GetValueOrDefault(fof) + 1;
    }
    // Sort by mutual connection count (most mutuals → best suggestion)
    return counts.OrderByDescending(kv => kv.Value)
                  .Take(10).Select(kv => kv.Key).ToList();
}

// Six Degrees of Separation: BFS to find shortest path between any two users
public int DegreesOfSeparation(int a, int b, HashSet<int>[] graph)
{
    if (a == b) return 0;
    var visited = new HashSet<int> { a };
    var q = new Queue<int>();
    q.Enqueue(a);
    int degree = 0;
    while (q.Count > 0)
    {
        degree++;
        int sz = q.Count;
        while (sz-- > 0)
        {
            int u = q.Dequeue();
            foreach (int v in graph[u])
            {
                if (v == b) return degree;
                if (visited.Add(v)) q.Enqueue(v);
            }
        }
    }
    return -1; // Not connected
}
// LinkedIn famously showed that average degrees of separation = 4.7 (not 6)

3. Package Manager — npm / NuGet / Maven

📦

Dependency Resolution — Install Packages in Correct Order

npm · NuGet · Maven · Cargo · pip
Topological Sort Cycle Detection DAG Phase 5 — Graph Algorithms Phase 9 — Advanced Graphs npm registry: 2.5M packages

When you run dotnet add package Newtonsoft.Json, the package manager must install all dependencies in the correct order — a package must be installed after all its dependencies. This is exactly topological sort. If a circular dependency exists, installation must fail with an error.

Your App ├── EntityFramework 8.0 │ ├── Microsoft.Data.SqlClient 5.0 │ │ └── Microsoft.Identity.Client 4.0 │ └── System.Text.Json 8.0 └── Serilog 3.0 └── System.Text.Json 8.0 ← shared dependency Install order (topological): Microsoft.Identity.Client → Microsoft.Data.SqlClient → System.Text.Json → EntityFramework → Serilog → App Circular dependency example (ILLEGAL): PackageA depends on PackageB PackageB depends on PackageA ← cycle detected → error!
public class PackageInstaller
{
    // Returns install order or throws if circular dependency exists
    public List<string> ResolveOrder(Dictionary<string, List<string>> deps)
    {
        var inDegree = new Dictionary<string, int>();
        foreach (var pkg in deps.Keys) inDegree[pkg] = 0;
        foreach (var (_, depList) in deps)
            foreach (string dep in depList)
                inDegree[dep] = inDegree.GetValueOrDefault(dep) + 1;

        // Kahn's Algorithm — packages with no remaining dependencies install first
        var ready = new Queue<string>(inDegree.Where(kv => kv.Value == 0).Select(kv => kv.Key));
        var order = new List<string>();

        while (ready.Count > 0)
        {
            string pkg = ready.Dequeue();
            order.Add(pkg);
            foreach (string dependant in deps.GetValueOrDefault(pkg, new()))
                if (--inDegree[dependant] == 0) ready.Enqueue(dependant);
        }

        if (order.Count != deps.Count)
            throw new InvalidOperationException("Circular dependency detected!");
        return order;
    }
}
// Time: O(V+E), Space: O(V)
// This is literally what `npm install` and `dotnet restore` do internally

Google's search box has three algorithmic layers working together: Trie for instant prefix autocomplete, Edit Distance DP for "Did you mean X?" spell correction, and Inverted Index (hash map from word → documents) for finding matching pages.

// Layer 1: Autocomplete via Trie (shows suggestions as you type)
public class SearchTrie
{
    private readonly TrieNode _root = new();

    public void Insert(string query, int frequency)
    {
        var node = _root;
        foreach (char c in query)
            node = node.Children.GetOrAdd(c, _ => new TrieNode());
        node.Frequency = frequency;
    }

    public List<string> TopSuggestions(string prefix, int k)
    {
        var node = _root;
        foreach (char c in prefix)
            if (!node.Children.TryGetValue(c, out node)) return new();

        // BFS / DFS to collect top-k frequent completions
        var results = new List<(int freq, string word)>();
        CollectAll(node, new StringBuilder(prefix), results);
        return results.OrderByDescending(x => x.freq).Take(k).Select(x => x.word).ToList();
    }
}

// Layer 2: "Did you mean?" via Edit Distance DP
public int EditDistance(string s, string t)
{
    int m = s.Length, n = t.Length;
    int[] prev = Enumerable.Range(0, n + 1).ToArray();
    for (int i = 1; i <= m; i++)
    {
        int[] curr = new int[n + 1];
        curr[0] = i;
        for (int j = 1; j <= n; j++)
            curr[j] = s[i-1] == t[j-1] ? prev[j-1]
                    : 1 + Math.Min(prev[j-1], Math.Min(prev[j], curr[j-1]));
        prev = curr;
    }
    return prev[n];
}

5. Compiler / IDE — Visual Studio / Roslyn

🛠

Expression Evaluation, Syntax Checking & Build Ordering

Roslyn (.NET) · JVM · Clang · GCC · IntelliJ
Stack (Expression Parsing) Trie (Symbol Table) Topological Sort (Build) Interval DP (Regex) Phase 2 — Stacks Phase 4 — Trie Phase 9 — Graphs Roslyn: compiles millions of lines/sec

When you write int x = (3 + 4) * 2 - 1;, the C# compiler uses a stack to parse and evaluate that expression. The Roslyn compiler also uses a Trie-based symbol table for name resolution and topological sort for build project ordering.

// Evaluate arithmetic expression: "3 + 4 * 2 - 1" → 10
// Uses two stacks: one for numbers, one for operators
public int Evaluate(string expr)
{
    var nums = new Stack<int>();
    var ops  = new Stack<char>();
    var prec = new Dictionary<char, int> { {'+',1},{'-',1},{'*',2},{'/',2} };

    void Apply() {
        int b = nums.Pop(), a = nums.Pop();
        nums.Push(ops.Pop() switch { '+' => a+b, '-' => a-b, '*' => a*b, _ => a/b });
    }

    int i = 0;
    while (i < expr.Length)
    {
        char c = expr[i];
        if (c == ' ') { i++; continue; }
        if (char.IsDigit(c)) {
            int num = 0;
            while (i < expr.Length && char.IsDigit(expr[i])) num = num*10 + expr[i++]-'0';
            nums.Push(num); continue;
        }
        if (c == '(') { ops.Push(c); }
        else if (c == ')') { while (ops.Peek() != '(') Apply(); ops.Pop(); }
        else if (prec.ContainsKey(c)) {
            while (ops.Count > 0 && ops.Peek() != '(' && prec.GetValueOrDefault(ops.Peek()) >= prec[c])
                Apply();
            ops.Push(c);
        }
        i++;
    }
    while (ops.Count > 0) Apply();
    return nums.Pop();
}
// This is the Shunting-Yard algorithm — how every compiler/interpreter works

6. Stock Exchange — NASDAQ / NYSE Order Book

📈

Real-Time Order Book — Match Buy/Sell Orders

NASDAQ · NYSE · Binance · Interactive Brokers
SortedDictionary (Price Levels) Priority Queue Monotonic Stack (Next Greater Price) Phase 2 — Monotonic Stack Phase 4 — Heaps NASDAQ: 30 billion orders/day

An exchange's order book matches buyers and sellers. Buy orders are sorted by descending price (highest bid wins), sell orders by ascending price (lowest ask wins). When the best bid ≥ best ask, a trade executes. This uses two priority queues (max-heap for bids, min-heap for asks).

public class OrderBook
{
    // Bids: max-heap (highest price first)
    private SortedDictionary<int, Queue<Order>> _bids = new(Comparer<int>.Create((a,b) => b-a));
    // Asks: min-heap (lowest price first)
    private SortedDictionary<int, Queue<Order>> _asks = new();

    public List<Trade> AddOrder(Order order)
    {
        var trades = new List<Trade>();
        if (order.IsBuy)
        {
            // Try to match against asks (sell orders at or below our bid price)
            while (order.Qty > 0 && _asks.Count > 0 && _asks.First().Key <= order.Price)
            {
                var ask = _asks.First().Value.Peek();
                int filled = Math.Min(order.Qty, ask.Qty);
                trades.Add(new Trade { Price = ask.Price, Qty = filled });
                ask.Qty -= filled; order.Qty -= filled;
                if (ask.Qty == 0) { _asks.First().Value.Dequeue(); CleanEmpty(_asks); }
            }
            if (order.Qty > 0) AddToBook(_bids, order); // Rest goes to book
        }
        // Symmetric for sell orders...
        return trades;
    }
}
// Real exchanges use lock-free data structures and FPGA hardware
// Latency target: match an order in under 1 microsecond

7. Warehouse Robotics — Amazon Fulfillment Centers

🏗

Optimal Item Pick Path — Traveling Robot Problem

Amazon Robotics · Ocado · DHL · FedEx
Bitmask DP (TSP variant) Dijkstra (grid navigation) BFS (obstacle avoidance) Phase 5 — Graphs Phase 10 — Bitmask DP Amazon: 750,000 robots deployed

An order might contain 7 items scattered across a warehouse. A robot must pick all items and return to the packing station in the minimum total distance. With 7 items, there are 7! = 5040 possible orders — manageable with bitmask DP (n ≤ 20 is the sweet spot).

// Minimum distance to pick all items and return to start
// Distances pre-computed via BFS/Dijkstra on the warehouse grid
public int OptimalPickPath(int[] items, int[,] dist, int start)
{
    int n = items.Length;
    // dp[mask][i] = min cost to pick items in mask, currently at item i
    int[,] dp = new int[1 << n, n];
    for (int i = 0; i < (1<<n); i++)
        for (int j = 0; j < n; j++) dp[i, j] = int.MaxValue / 2;

    // Initialize: cost to travel from start to each item
    for (int i = 0; i < n; i++) dp[1 << i, i] = dist[start, items[i]];

    for (int mask = 1; mask < (1<<n); mask++)
    for (int u = 0; u < n; u++)
    {
        if ((mask & (1<<u)) == 0 || dp[mask, u] == int.MaxValue/2) continue;
        for (int v = 0; v < n; v++)
            if ((mask & (1<<v)) == 0)
                dp[mask|(1<<v), v] = Math.Min(dp[mask|(1<<v), v],
                                              dp[mask, u] + dist[items[u], items[v]]);
    }

    int full = (1<<n) - 1, ans = int.MaxValue;
    for (int i = 0; i < n; i++)  // Return to packing station
        ans = Math.Min(ans, dp[full, i] + dist[items[i], start]);
    return ans;
}
// O(n² * 2^n) — Amazon limits order batches to ~15 items per robot trip

8. Version Control — Git

🌸

Diff, Merge & Commit Graph

Git · GitHub · GitLab · Azure DevOps
LCS / Edit Distance DP DAG (Commit History) SHA Hash (Content Addressing) Topological Sort (rebase) Phase 6 — DP Phase 5 — Graphs GitHub: 420M repositories

When you run git diff, Git computes the Longest Common Subsequence (LCS) between two file versions to find the minimal set of line additions/deletions. The commit history is a Directed Acyclic Graph (DAG) where git rebase is a topological sort.

// git diff — Longest Common Subsequence shows what changed between versions
public List<string> DiffFiles(string[] oldLines, string[] newLines)
{
    int m = oldLines.Length, n = newLines.Length;
    int[,] dp = new int[m+1, n+1];

    for (int i = 1; i <= m; i++)
    for (int j = 1; j <= n; j++)
        dp[i,j] = oldLines[i-1] == newLines[j-1]
            ? dp[i-1,j-1] + 1
            : Math.Max(dp[i-1,j], dp[i,j-1]);

    // Reconstruct diff by backtracking through dp table
    var diff = new List<string>();
    int r = m, c = n;
    while (r > 0 || c > 0)
    {
        if (r > 0 && c > 0 && oldLines[r-1] == newLines[c-1])
            { diff.Add("  " + oldLines[--r]); c--; }
        else if (c > 0 && (r == 0 || dp[r,c-1] >= dp[r-1,c]))
            { diff.Add("+ " + newLines[--c]); }
        else
            { diff.Add("- " + oldLines[--r]); }
    }
    diff.Reverse();
    return diff;
}

// to split large files into blocks before diffing — reducing O(n²) work

9. Banking — Real-Time Fraud Detection

💰

Detect Fraudulent Transaction Clusters

Visa · Mastercard · PayPal · Stripe · HSBC
Sliding Window (velocity checks) Union-Find (fraud rings) Hash Map (deduplication) Graph Anomaly Detection Phase 3 — Hash Tables Phase 5 — Union-Find Phase 1 — Sliding Window Visa: 65,000 transactions/second

Fraud detection runs multiple checks simultaneously: a sliding window checks if too many transactions occurred in the last 60 seconds (velocity fraud), Union-Find clusters accounts that share devices/IPs to find fraud rings, and rolling hashes deduplicate retry attempts.

// Velocity check: flag if > k transactions in last t seconds (sliding window)
public bool IsVelocityFraud(Queue<long> txTimestamps, long nowMs, int maxTx, int windowSec)
{
    long windowMs = windowSec * 1000L;
    while (txTimestamps.Count > 0 && nowMs - txTimestamps.Peek() > windowMs)
        txTimestamps.Dequeue(); // Remove expired transactions
    txTimestamps.Enqueue(nowMs);
    return txTimestamps.Count > maxTx;
}

// Fraud ring detection: group accounts by shared fingerprints
// If account A and B share device_id → union them → same cluster = suspicious
public Dictionary<int, List<int>> FindFraudClusters(
    int[] accounts, Dictionary<string, List<int>> fingerprintToAccounts)
{
    int[] parent = Enumerable.Range(0, accounts.Length).ToArray();
    int Find(int x) => parent[x] == x ? x : (parent[x] = Find(parent[x]));

    // Accounts sharing any fingerprint (device, IP, card) get unioned
    foreach (var (_, accts) in fingerprintToAccounts)
        for (int i = 1; i < accts.Count; i++)
            parent[Find(accts[0])] = Find(accts[i]);

    // Group by root → clusters of 3+ accounts are fraud ring candidates
    return accounts.GroupBy(a => Find(a))
                   .Where(g => g.Count() >= 3)  // Suspicious if 3+ share fingerprint
                   .ToDictionary(g => g.Key, g => g.ToList());
}
// Visa processes these checks in under 100 milliseconds per transaction

10. Video Game AI — NPC Pathfinding

🎮

NPC Navigation in a Game World with Obstacles

Unreal Engine · Unity · Valve · CD Projekt · Ubisoft
A* Search BFS (visibility graph) Priority Queue Grid Graph Phase 5 — Graphs + BFS Unity: 5B+ installs

In every open-world game, NPCs (enemies, allies, vehicles) navigate around obstacles in real time. This is A* on a grid or navigation mesh. Games like The Witcher 3 run hundreds of pathfinding queries per frame (60fps), so optimization is critical.

// A* pathfinding on a 2D game grid (0=walkable, 1=wall)
public List<(int,int)> FindPath(int[,] grid, (int r, int c) src, (int r, int c) dst)
{
    int rows = grid.GetLength(0), cols = grid.GetLength(1);
    double[,] g = new double[rows, cols];
    for (int i = 0; i < rows; i++) for (int j = 0; j < cols; j++) g[i,j] = double.MaxValue;
    g[src.r, src.c] = 0;

    var prev = new (int,int)?[rows, cols];
    var pq   = new PriorityQueue<(int,int), double>();
    pq.Enqueue(src, H(src, dst));

    int[] dr = {-1,1,0,0,-1,-1,1,1}; // 8-directional movement
    int[] dc = {0,0,-1,1,-1,1,-1,1};
    double[] cost = {1,1,1,1, 1.414,1.414,1.414,1.414}; // √2 for diagonal

    while (pq.Count > 0)
    {
        pq.TryDequeue(out var u, out _);
        if (u == dst) return ReconstructPath(prev, src, dst);
        for (int d = 0; d < 8; d++) {
            int nr = u.Item1+dr[d], nc = u.Item2+dc[d];
            if (nr<0||nr>=rows||nc<0||nc>=cols||grid[nr,nc]==1) continue;
            double ng = g[u.Item1,u.Item2] + cost[d];
            if (ng < g[nr,nc]) {
                g[nr,nc] = ng; prev[nr,nc] = u;
                pq.Enqueue((nr,nc), ng + H((nr,nc), dst));
            }
        }
    }
    return new(); // No path
}

private double H((int r, int c) a, (int r, int c) b)
    => Math.Sqrt((a.r-b.r)*(a.r-b.r) + (a.c-b.c)*(a.c-b.c)); // Euclidean
// Games use navigation meshes (NavMesh) instead of raw grids for large worlds
// NavMesh = convex polygon graph → A* on much fewer nodes

11. Bioinformatics — DNA Sequence Alignment

🧬

Align Two DNA Sequences to Find Mutations

NCBI BLAST · 23andMe · Illumina · DeepMind AlphaFold
Edit Distance DP (Smith-Waterman) LCS (Needleman-Wunsch) Rolling Hash (k-mer indexing) KMP (pattern in genome) Phase 6 — DP (LCS) Phase 9 — String Algorithms Human genome: 3.2 billion base pairs

Comparing two DNA sequences to find genetic mutations is equivalent to computing edit distance. The Smith-Waterman algorithm (a DP variant) finds the best local alignment — the most similar substring between two sequences, ignoring mismatched ends.

// Smith-Waterman local alignment: find best matching region between two DNA strings
// Used to find where a gene mutation occurred
public (int score, string aligned) SmithWaterman(string seq1, string seq2,
    int matchScore = 2, int mismatch = -1, int gap = -1)
{
    int m = seq1.Length, n = seq2.Length;
    int[,] dp = new int[m+1, n+1];
    int maxScore = 0; (int,int) maxPos = (0,0);

    for (int i = 1; i <= m; i++)
    for (int j = 1; j <= n; j++)
    {
        int match = dp[i-1,j-1] + (seq1[i-1]==seq2[j-1] ? matchScore : mismatch);
        dp[i,j] = Math.Max(0, Math.Max(match, Math.Max(dp[i-1,j]+gap, dp[i,j-1]+gap)));
        if (dp[i,j] > maxScore) { maxScore = dp[i,j]; maxPos = (i,j); }
    }

    // Traceback from maxPos to find the aligned region
    var sb = new StringBuilder();
    var (r, c) = maxPos;
    while (dp[r, c] > 0)
    {
        if      (r > 0 && c > 0 && dp[r,c] == dp[r-1,c-1] + (seq1[r-1]==seq2[c-1] ? matchScore : mismatch))
            { sb.Insert(0, seq1[--r]); c--; }
        else if (r > 0 && dp[r,c] == dp[r-1,c] + gap)
            { sb.Insert(0, '-'); r--; }
        else    { sb.Insert(0, '-'); c--; }
    }
    return (maxScore, sb.ToString());
}
// NCBI BLAST uses this to align your DNA sample against 3 billion known sequences
// Hashing (k-mer seeds) narrows candidates before running full Smith-Waterman

12. Cloud CDN — Content Delivery & Load Balancing

🌐

Consistent Hashing — Distribute Requests Across Servers

Akamai · Cloudflare · AWS CloudFront · Netflix CDN
Consistent Hashing SortedDictionary (Ring) LRU Cache (Edge Caching) Phase 3 — Hash Tables Phase 2 — LRU Cache Cloudflare: 55M HTTP requests/sec

When you stream Netflix, your request is routed to one of hundreds of edge servers. Consistent hashing maps both requests and servers onto a virtual ring. A request goes to the nearest server clockwise on the ring — adding/removing servers only remaps a fraction of keys (vs. ~100% with plain modulo hashing).

public class ConsistentHashRing
{
    private readonly SortedDictionary<int, string> _ring = new();
    private readonly int _virtualNodes;

    public ConsistentHashRing(int virtualNodes = 150) => _virtualNodes = virtualNodes;

    public void AddServer(string server)
    {
        for (int i = 0; i < _virtualNodes; i++)
        {
            int hash = Hash($"{server}#{i}");
            _ring[hash] = server;
        }
    }

    public void RemoveServer(string server)
    {
        for (int i = 0; i < _virtualNodes; i++)
            _ring.Remove(Hash($"{server}#{i}"));
    }

    public string GetServer(string requestKey)
    {
        if (_ring.Count == 0) throw new InvalidOperationException("No servers");
        int hash = Hash(requestKey);
        // Find nearest server clockwise: first key ≥ hash
        foreach (var kv in _ring)
            if (kv.Key >= hash) return kv.Value;
        return _ring.First().Value; // Wrap around the ring
    }

    private int Hash(string key) => (int)(uint)key.GetHashCode(); // Always non-negative
}
// Adding 1 server redistributes only 1/n requests (not all)
// Virtual nodes ensure even distribution even with heterogeneous servers

13. Ride-Hailing — Uber / Lyft

🚗

Find the Nearest Available Driver in Real Time

Uber · Lyft · Grab · Bolt · DiDi
Geohash (Spatial Indexing) Min-Heap (Nearest K) Haversine Distance Phase 3 — Hash Tables Phase 4 — Heaps Uber: 14M location updates/sec

When you request a ride, Uber must find the nearest available driver among millions of moving vehicles — in under 100ms. Brute-force O(n) scan of every driver is too slow. Geohashing partitions the Earth into a grid; only drivers in nearby cells are evaluated, shrinking the candidate pool from millions to dozens.

Geohash Grid (each cell ≈ 1.2km × 0.6km at precision 6): ┌──────┬──────┬──────┬──────┐ │ A3 │ B3 │ C3 │ D3 │ ├──────┼──────┼──────┼──────┤ │ A2 │[B2] │ C2 │ D2 │ ← rider in cell B2 ├──────┼──────┼──────┼──────┤ search B2 + 8 neighbours = 9 cells │ A1 │ B1 │ C1 │ D1 │ ~50-200 drivers vs 5.4M total └──────┴──────┴──────┴──────┘ Key insight: nearby cells share a common geohash prefix "9q8yh" and "9q8yk" are adjacent — O(1) dictionary lookup per cell

Core Algorithm: Geohash Dictionary + Min-Heap Nearest K

Drivers send GPS updates ~every 4 seconds, re-registering in their current geohash cell. On a ride request, look up the rider's cell plus 8 neighbours, then use a min-heap to extract the k nearest drivers by actual Haversine distance.

public class DriverDispatch
{
    // Spatial index: geohash cell → list of available drivers in that cell
    private readonly Dictionary<string, List<Driver>> _grid = new();

    // Encode lat/lng to a grid cell string (simplified geohash)
    private string Encode(double lat, double lng, int bits = 6)
    {
        int latB = (int)((lat + 90) / 180.0 * (1 << bits));
        int lngB = (int)((lng + 180) / 360.0 * (1 << bits));
        return $"{latB}:{lngB}";
    }

    public void UpdateDriver(Driver d, double lat, double lng)
    {
        // Remove from old cell, register in new cell
        if (d.Cell != null && _grid.TryGetValue(d.Cell, out var old)) old.Remove(d);
        d.Cell = Encode(lat, lng);
        if (!_grid.TryGetValue(d.Cell, out var list)) _grid[d.Cell] = list = new();
        list.Add(d);
    }

    // Find k nearest drivers: geohash lookup → min-heap sort by real distance
    public List<Driver> FindNearest(double lat, double lng, int k)
    {
        var pq = new PriorityQueue<Driver, double>(); // min-heap: nearest first
        foreach (string cell in GetNeighborCells(Encode(lat, lng))) // cell + 8 neighbours
        {
            if (!_grid.TryGetValue(cell, out var drivers)) continue;
            foreach (var d in drivers)
                pq.Enqueue(d, HaversineKm(lat, lng, d.Lat, d.Lng));
        }
        var result = new List<Driver>();
        while (result.Count < k && pq.Count > 0) result.Add(pq.Dequeue());
        return result;
    }
    // GetNeighborCells: decode "latB:lngB", offset ±1 in each direction, re-encode
}
// Uber's real system uses Google S2 geometry: hierarchical spherical cells
// Redis GEOSEARCH / GEORADIUS implements the same pattern in O(1) cache lookup
// 5.4M drivers × 1 update/4sec = 1.35M location writes/sec → sharded Redis
Real-World Scale: Without geohashing, matching 1 rider against 5.4M drivers takes ~500ms. With geohash precision-6 cells (~1.2km wide), only ~50 drivers per cell are evaluated — <1ms. Uber's S2 library extends this to curved-Earth geometry with hierarchical cells from 0.44cm² to half the globe.

Top-K Personalised Recommendations at Scale

Spotify · Netflix · YouTube · Apple Music · TikTok
Min-Heap (Top-K Selection) Sliding Window (Trending) Hash Map (Frequency Count) Phase 4 — Heaps Phase 1 — Sliding Window Spotify: 600M users · 100M tracks

Spotify's "Discover Weekly" selects 30 personalised songs from ~50,000 candidates per user. Fully sorting 50,000 songs to take 30 wastes work. A min-heap of size K does it in O(n log K) — maintaining only the current Top-30, instantly evicting the weakest candidate as a better one arrives.

Top-K with min-heap of size K=3 (weakest score always on top for fast eviction): Candidates: [0.95, 0.82, 0.78, 0.91, 0.65, 0.88, 0.97, ...] Push 0.95 → heap: {0.95} min=0.95 Push 0.82 → heap: {0.82, 0.95} min=0.82 Push 0.78 → heap: {0.78, 0.95, 0.82} min=0.78 Push 0.91 → 0.91 > min(0.78) → pop 0.78, push 0.91 → {0.82, 0.95, 0.91} Push 0.65 → 0.65 < min(0.82) → skip (cannot improve top-3) Push 0.88 → 0.88 > min(0.82) → pop 0.82, push 0.88 → {0.88, 0.95, 0.91} Push 0.97 → 0.97 > min(0.88) → pop 0.88, push 0.97 → {0.91, 0.95, 0.97} Result: [0.91, 0.95, 0.97] — Top-3 in O(n log K) vs O(n log n) full sort

Core Algorithm: Min-Heap Top-K + Sliding Window Trending

// Top-K recommendations: O(n log K) — optimal for large n, small K
public List<Song> TopKRecommendations(IEnumerable<Song> candidates, int k)
{
    // Min-heap: smallest score at top, so we evict weak candidates cheaply
    var heap = new PriorityQueue<Song, double>();
    foreach (var song in candidates)
    {
        heap.Enqueue(song, song.Score);
        if (heap.Count > k) heap.Dequeue(); // evict lowest scorer — O(log K)
    }
    return heap.UnorderedItems.Select(x => x.Element)
                             .OrderByDescending(s => s.Score).ToList();
}

// Trending this week: sliding window over a stream of play events (Kafka-style)
public Dictionary<int, int> TrendingTracks(
    List<(long timestampMs, int trackId)> plays, long nowMs,
    long windowMs = 7L * 24 * 3600 * 1000) // 7-day rolling window
{
    var counts = new Dictionary<int, int>();
    int left = 0;
    for (int right = 0; right < plays.Count; right++)
    {
        counts[plays[right].trackId] = counts.GetValueOrDefault(plays[right].trackId) + 1;
        while (nowMs - plays[left].timestampMs > windowMs) // expire old plays
        {
            if (--counts[plays[left].trackId] == 0) counts.Remove(plays[left].trackId);
            left++;
        }
    }
    return counts; // trackId → play count in the last 7 days
}
// Final score = α × personalisation + β × trending + γ × freshness
// Then TopKRecommendations(scoredCandidates, k=30) → your Discover Weekly
// Runs as a Spark batch job every Monday for 600M users in parallel
Real-World Scale: Spotify generates Discover Weekly for 600M users every Monday. Per user: ~50K candidates × log(30) ≈ 750K heap operations. Across 600M users on a distributed Spark cluster, this completes over the weekend. The min-heap ensures each worker holds only 30 items in memory regardless of the candidate pool size — a crucial memory bound at scale.

B-Tree Index — Fast Range Queries on a Billion-Row Table

PostgreSQL · MySQL · SQL Server · Oracle · SQLite

When you run SELECT * FROM orders WHERE date BETWEEN '2024-01-01' AND '2024-12-31', without an index the database scans every row (O(n) — catastrophic on a billion rows). A B-Tree index jumps directly to the matching range in O(log n) disk-page reads. Every CREATE INDEX builds one.

B-Tree index on orders.date (m = 100 keys/page, each page = 8 KB): [Jan'23 | Jul'23 | Jan'24] ← root (1 disk read) / | | \ [..Dec'22] [Jan–Jun'23] [Jul–Dec'23] [Jan'24–Dec'24] ← internal | [Jan 1 → row#1093847] ← leaf pages [Jan 2 → row#1093901] (linked list [Jan 3 → row#1094002] for range scan) ... Query WHERE date BETWEEN Jan'24 AND Dec'24: → 3 disk reads (root → internal → first leaf) → stream leaf entries forward vs 1,000,000,000 row reads for a full table scan

Core Algorithm: Binary Search to Find Range Start → Stream Leaf Pages

// Simulate a B-Tree index using SortedList (uses binary search internally)
public class BTreeIndex<TKey, TRow> where TKey : IComparable<TKey>
{
    // SortedList keeps keys in order — mirrors B-Tree leaf pages in sorted sequence
    private readonly SortedList<TKey, List<TRow>> _pages = new();

    public void Insert(TKey key, TRow row)
    {
        if (!_pages.TryGetValue(key, out var rows)) _pages[key] = rows = new();
        rows.Add(row);
    }

    // Range scan: O(log n) to find start + O(k) to stream k matching rows
    public IEnumerable<TRow> RangeScan(TKey lo, TKey hi)
    {
        // Binary search for the first key ≥ lo (B-Tree internal node traversal)
        int left = 0, right = _pages.Count - 1, start = _pages.Count;
        while (left <= right)
        {
            int mid = (left + right) / 2;
            if (_pages.GetKeyAtIndex(mid).CompareTo(lo) >= 0)
                { start = mid; right = mid - 1; }
            else left = mid + 1;
        }
        // Stream forward through leaf pages while key ≤ hi (linked-list scan)
        for (int i = start; i < _pages.Count; i++)
        {
            if (_pages.GetKeyAtIndex(i).CompareTo(hi) > 0) yield break;
            foreach (var row in _pages.GetValueAtIndex(i)) yield return row;
        }
    }
}
// Same as: SELECT * FROM orders WHERE date BETWEEN lo AND hi
// Real PostgreSQL B-Tree: 8KB pages, ~100 keys/page
// On 1 billion rows: height = log₁₀₀(1B) ≈ 5 → max 5 disk reads to reach leaf
// CREATE INDEX is O(n log n) to build; every subsequent query is O(log n + k)
Real-World Scale: On a 1-billion-row table, a B-Tree date range query touches just 4–5 disk pages to reach the first matching row, then streams results sequentially. Without an index: 1 billion row reads. With index: 5 reads. Adding CREATE INDEX ON orders(date) can make a query 200,000,000× faster. This is why database indexing is the highest-ROI optimization in backend engineering.

16. Blockchain — Bitcoin Transaction Verification

🮽

Merkle Tree — Prove a Transaction Exists Without Downloading the Whole Block

Bitcoin · Ethereum · Solana · Hyperledger · AWS QLDB
Merkle Tree (Hash Tree) Divide & Conquer (Bottom-up) SHA-256 Cryptographic Hash Phase 3 — Trees / Hashing Phase 4 — Tree Traversal Bitcoin: 900M+ transactions

A Bitcoin block holds ~3000 transactions (~1.5MB). A mobile wallet needs to confirm one transaction without downloading the full block. A Merkle tree hashes pairs of transactions bottom-up into a single root hash. To prove TxA is in the block requires only O(log n) sibling hashes — just 12 hashes (384 bytes) for a 4096-transaction block.

Merkle Tree (4 transactions → height = 2): RootHash = H(H(AB) ‖ H(CD)) / \ H(TxA ‖ TxB) H(TxC ‖ TxD) / \ / \ H(TxA) H(TxB) H(TxC) H(TxD) Merkle Proof that TxA is in block (mobile wallet): Need: H(TxB) — sibling at leaf level H(TxC ‖ TxD) — sibling at next level Verify: H(H(H(TxA) ‖ H(TxB)) ‖ H(TxC ‖ TxD)) == RootHash? ✓ Only 2 hashes (64 bytes) to verify 1 of 4 transactions Only 12 hashes (384 bytes) to verify 1 of 4096 transactions

Core Algorithm: Build Merkle Tree + Generate and Verify Proof

public class MerkleTree
{
    private readonly List<byte[]> _leaves;

    public MerkleTree(IEnumerable<string> transactions)
        => _leaves = transactions.Select(Hash).ToList();

    // Build bottom-up: pair-hash each level until one root hash remains
    public byte[] Root()
    {
        var level = new List<byte[]>(_leaves);
        while (level.Count > 1)
        {
            if (level.Count % 2 != 0) level.Add(level[^1]); // duplicate last if odd count
            var next = new List<byte[]>();
            for (int i = 0; i < level.Count; i += 2)
                next.Add(HashPair(level[i], level[i + 1]));
            level = next;
        }
        return level[0];
    }

    // Merkle proof: O(log n) sibling hashes needed to prove leaf at index
    public List<(byte[] sibling, bool siblingOnRight)> GetProof(int idx)
    {
        var proof = new List<(byte[], bool)>();
        var level = new List<byte[]>(_leaves);
        while (level.Count > 1)
        {
            if (level.Count % 2 != 0) level.Add(level[^1]);
            bool isLeft = idx % 2 == 0;              // is current node a left child?
            proof.Add((level[isLeft ? idx + 1 : idx - 1], !isLeft)); // add sibling
            var next = new List<byte[]>();
            for (int i = 0; i < level.Count; i += 2)
                next.Add(HashPair(level[i], level[i + 1]));
            level = next; idx /= 2;
        }
        return proof;
    }

    // Verify: mobile wallet recomputes root from just O(log n) hashes
    public static bool Verify(string tx,
        List<(byte[] sibling, bool sibOnRight)> proof, byte[] expectedRoot)
    {
        byte[] cur = Hash(tx);
        foreach (var (sib, right) in proof)
            cur = right ? HashPair(cur, sib) : HashPair(sib, cur);
        return cur.SequenceEqual(expectedRoot);
    }

    private static byte[] Hash(string s)
        => SHA256.HashData(Encoding.UTF8.GetBytes(s));
    private static byte[] HashPair(byte[] a, byte[] b)
        => SHA256.HashData([..a, ..b]);
}
// Bitcoin SPV wallets: store only 80-byte block headers (not full blocks)
// To verify a payment: request Merkle proof → download ~384 bytes vs 1.5MB block
// Ethereum extends this to Merkle-Patricia Tries for state, transactions, receipts
Real-World Scale: Bitcoin SPV (Simplified Payment Verification) wallets — used by millions of mobile apps — never download the full 500GB+ blockchain. They store only 80-byte block headers and request Merkle proofs on demand. To verify 1 transaction in a 4096-transaction block: download 12 hashes (384 bytes) vs ~2MB of raw transactions. That's a 5000× bandwidth saving — all thanks to a binary hash tree.

Calendar apps constantly solve three interval problems: (1) merge overlapping meetings into consolidated busy blocks, (2) count rooms needed at peak (minimum conference rooms for a day), and (3) find free slots long enough for a new meeting. All three are solved with sort + sweep line.

Meetings: [9-11] [10-12] [13-15] [14-16] [16-17] Sorted by start: same order Merge overlapping (sweep line): [9-11] + [10-12] → overlap → extend to [9-12] [9-12] + [13-15] → gap → new block [13-15] [13-15] + [14-16] → overlap → extend to [13-16] [13-16] + [16-17] → touch → extend to [13-17] Result: [9-12] [13-17] → free slot: [12-13] = 1 hr gap Min-rooms (heap tracks active end times): 9am: add 11 → endTimes:{11} → 1 room 10am: add 12 → endTimes:{11,12} → 2 rooms 11am: add 15 → pop 11, add 15 → still 2 rooms (reuse freed room) 14am: add 16 → pop 12, add 16 → still 2 rooms

Core Algorithm: Interval Merge + Min-Heap Room Counter + Free Slot Finder

// Problem 1: Merge overlapping meetings — collapses conflicts into busy blocks
public int[][] MergeIntervals(int[][] meetings)
{
    Array.Sort(meetings, (a, b) => a[0] - b[0]); // sort by start time — O(n log n)
    var result = new List<int[]> { meetings[0] };
    foreach (var m in meetings.Skip(1))
    {
        var last = result[^1];
        if (m[0] <= last[1])              // overlaps or touches → extend
            last[1] = Math.Max(last[1], m[1]);
        else
            result.Add(m);               // gap → start new block
    }
    return result.ToArray();
}

// Problem 2: Minimum rooms needed = max simultaneous meetings
public int MinRoomsRequired(int[][] meetings)
{
    Array.Sort(meetings, (a, b) => a[0] - b[0]);
    var endTimes = new PriorityQueue<int, int>(); // min-heap of active meeting end times
    foreach (var m in meetings)
    {
        if (endTimes.Count > 0 && endTimes.Peek() <= m[0])
            endTimes.Dequeue();        // reuse room whose meeting just ended
        endTimes.Enqueue(m[1], m[1]); // assign room, track its end time
    }
    return endTimes.Count; // rooms still occupied = peak rooms needed
}

// Problem 3: Find free slots ≥ minDuration for a new meeting
public List<(int start, int end)> FindFreeSlots(
    int[][] busy, int dayStart, int dayEnd, int minDuration)
{
    var merged = MergeIntervals(busy);
    var free = new List<(int, int)>();
    int prev = dayStart;
    foreach (var m in merged)
    {
        if (m[0] - prev >= minDuration) free.Add((prev, m[0])); // gap before meeting
        prev = Math.Max(prev, m[1]);
    }
    if (dayEnd - prev >= minDuration) free.Add((prev, dayEnd)); // gap after last
    return free;
}
// Calendly runs FindFreeSlots across your calendar entries to show booking windows
// Google "Find a time": fetch all attendees' calendars → intersect free slots
// Intersection of two free-slot lists: O(m+n) two-pointer merge
Real-World Scale: Calendly processes millions of scheduling requests daily. For each booking link visit, it fetches the host's calendar (n busy slots), runs MergeIntervals + FindFreeSlots in O(n log n), and displays available windows in under 200ms. Google "Find a time" extends this to 10+ attendees by intersecting each person's free slots with a two-pointer O(m+n) merge — the same algorithm you learned in Phase 1.
