Section 1 — The Analogy: The Spam Filter

Imagine an email spam filter that maintains a list of known spam domains. For every incoming email, it checks whether the sender's domain is on the list. There are billions of domains — storing them all in a hash set would require gigabytes of RAM.

Instead, the filter keeps a compact 1 million-bit array (just 125 KB) and uses three hash functions. When a spam domain is added, three bits are set. When checking an incoming email, the filter checks those same three bit positions. If all three bits are set, the domain is probably spam. If any bit is 0, the domain is definitely not on the spam list.

This is a Bloom filter. It can have false positives (occasionally flags a legitimate domain as spam) but never false negatives (never clears a known spam domain). For spam filtering, this tradeoff is acceptable.

Section 2 — What is a Bloom Filter?

A Bloom filter is a space-efficient probabilistic data structure for set membership testing. It uses a bit array of size m and k independent hash functions.

PropertyBloom FilterHashSet
Memory usageO(m) bits — typically 10 bits/elementO(n) — stores actual elements
InsertO(k) — set k bitsO(1) amortized
Query "is X in set?"O(k)O(1)
False positivesPossible (controllable)Never
False negativesNeverNever
DeleteNot supported (standard)O(1)
Count elementsNot supportedO(1)
💡 The Golden Rule of Bloom Filters:
  • "Definitely NOT in set" → 100% certain (any zero bit means absent)
  • "Probably IN set" → might be a false positive
This asymmetry makes Bloom filters perfect for "fast reject" scenarios: quickly eliminate candidates that definitely aren't in the set, and only do expensive checks on the (rare) positives.

Section 4 — False Positives vs. False Negatives

False positives (saying "probably in set" for an absent element) happen when all k bit positions checked happen to have been set by other insertions. The probability increases as the filter fills up.

False negatives are impossible: inserting an element sets certain bits permanently. Querying that same element will always find those bits set. A bit, once set, is never cleared (in a standard Bloom filter).

📝 Why no deletion? Clearing a bit during deletion might remove bits set by other elements (sharing bits is the whole trick). A Counting Bloom Filter extends each bit to a counter, enabling deletion at the cost of more memory.
Section 5 — Optimal Parameters

Given n expected insertions and a target false positive probability p, the optimal parameters are:

  • Bit array size: m = −n · ln(p) / (ln 2)² ≈ −n · 1.44 · log₂(p)
  • Number of hash functions: k = (m / n) · ln 2 ≈ 0.693 · (m / n)
PARAMETER EXAMPLES — n = 1,000,000 elements Target FP rate Bits/element (m/n) # hash funcs (k) Total memory ───────────────────────────────────────────────────────────────────────── p = 1% 9.59 bits 7 1.2 MB p = 0.1% 14.4 bits 10 1.8 MB p = 0.01% 19.2 bits 13 2.4 MB p = 0.001% 24.0 bits 17 3.0 MB Compare: HashSet of 1M strings ≈ 50–100 MB (storing actual 10–20 char strings) → Bloom filter is 20–50× smaller for 1% false positive rate
Section 6 — Complete C# Implementation
C# 
using System;
using System.Collections;
using System.Text;

public class BloomFilter
{
    private readonly BitArray _bits;
    private readonly int _hashCount;
    private readonly int _size;
    private int _insertCount;

    /// <param name="capacity">Expected number of elements</param>
    /// <param name="errorRate">Desired false positive probability (e.g., 0.01)</param>
    public BloomFilter(int capacity, double errorRate = 0.01)
    {
        // Optimal m: -n * ln(p) / (ln2)^2
        _size = (int)(-capacity * Math.Log(errorRate) /
                      (Math.Log(2) * Math.Log(2)));
        // Optimal k: (m/n) * ln2
        _hashCount = (int)((double)_size / capacity * Math.Log(2));
        _bits = new BitArray(_size);
    }

    public void Add(string item)
    {
        foreach (var idx in GetBitIndices(item))
            _bits[idx] = true;
        _insertCount++;
    }

    public bool MightContain(string item)
    {
        foreach (var idx in GetBitIndices(item))
            if (!_bits[idx]) return false;  // definite miss
        return true;  // probable hit (may be false positive)
    }

    public double EstimatedFalsePositiveRate
        => Math.Pow(1 - Math.Exp(-(double)_hashCount * _insertCount / _size),
                    _hashCount);

    // Generate k independent hash values for an item
    private int[] GetBitIndices(string item)
    {
        var indices = new int[_hashCount];
        byte[] bytes = Encoding.UTF8.GetBytes(item);

        // Double hashing: h(i) = (h1 + i * h2) % m
        long h1 = FNV1a(bytes, seed: 0);
        long h2 = FNV1a(bytes, seed: 1597);
        for (int i = 0; i < _hashCount; i++)
            indices[i] = (int)((Math.Abs(h1 + (long)i * h2)) % _size);
        return indices;
    }

    private static long FNV1a(byte[] data, long seed)
    {
        const uint prime  = 16777619;
        uint hash = (uint)(2166136261 ^ seed);
        foreach (var b in data) { hash ^= b; hash *= prime; }
        return hash;
    }
}

class Program
{
    static void Main()
    {
        var bf = new BloomFilter(capacity: 1000, errorRate: 0.01);

        string[] spamDomains = { "evil.com", "spam99.net", "phishing.org" };
        foreach (var d in spamDomains) bf.Add(d);

        Console.WriteLine(bf.MightContain("evil.com"));      // True
        Console.WriteLine(bf.MightContain("legit.com"));     // False (probably)
        Console.WriteLine($"Est. FP rate: {bf.EstimatedFalsePositiveRate:P3}");

        /* ── Output ──
        True
        False
        Est. FP rate: 0.000%   (only 3 items in a 1000-capacity filter)
        */
    }
}
Section 7 — Complete Python Implementation
Python 
import math
import hashlib


class BloomFilter:
    """Space-efficient probabilistic set membership filter."""

    def __init__(self, capacity: int, error_rate: float = 0.01):
        # Optimal bit array size
        self._size = int(
            -capacity * math.log(error_rate) / (math.log(2) ** 2)
        )
        # Optimal number of hash functions
        self._k = int(self._size / capacity * math.log(2))
        self._bits = bytearray((self._size + 7) // 8)  # byte array
        self._count = 0

    def _get_indices(self, item: str):
        """Generate k bit indices using double hashing."""
        data = item.encode()
        h1 = int(hashlib.md5(data).hexdigest(), 16)
        h2 = int(hashlib.sha1(data).hexdigest(), 16)
        return [(abs(h1 + i * h2)) % self._size for i in range(self._k)]

    def _set_bit(self, i: int):
        self._bits[i // 8] |= (1 << (i % 8))

    def _get_bit(self, i: int) -> bool:
        return bool(self._bits[i // 8] & (1 << (i % 8)))

    def add(self, item: str):
        for idx in self._get_indices(item):
            self._set_bit(idx)
        self._count += 1

    def might_contain(self, item: str) -> bool:
        return all(self._get_bit(i) for i in self._get_indices(item))

    @property
    def estimated_fp_rate(self) -> float:
        return (1 - math.exp(-self._k * self._count / self._size)) ** self._k


if __name__ == "__main__":
    bf = BloomFilter(capacity=1000, error_rate=0.01)

    spam = ["evil.com", "spam99.net", "phishing.org"]
    for d in spam:
        bf.add(d)

    print(bf.might_contain("evil.com"))    # True
    print(bf.might_contain("legit.com"))   # False
    print(f"FP rate: {bf.estimated_fp_rate:.6f}")

    # Output:
    # True
    # False
    # FP rate: 0.000000
Section 8 — Count-Min Sketch

A Count-Min Sketch extends the Bloom filter idea to answer frequency queries rather than membership queries. Instead of a bit array, it uses a 2D array of counters (d rows × w columns), where d = number of hash functions and w = width.

COUNT-MIN SKETCH — d=3 hash functions, w=8 columns Query: "How many times was 'apple' inserted?" Insert "apple" 3 times: increment 3 cells per insertion ┌────────────────────────────────────────────────────────┐ │ Row 0 (h0): [ 0 3 0 0 0 0 0 0 ] ← h0(apple)=1 → col 1 → +=3 │ │ Row 1 (h1): [ 0 0 3 0 0 0 0 0 ] ← h1(apple)=2 → col 2 → +=3 │ │ Row 2 (h2): [ 0 0 0 0 3 0 0 0 ] ← h2(apple)=4 → col 4 → +=3 │ └────────────────────────────────────────────────────────┘ Query("apple"): min(row0[1], row1[2], row2[4]) = min(3,3,3) = 3 ✓ If other items collide and inflate some counters: Query("apple"): min(5, 3, 4) = 3 (still correct — other items can only ADD) GUARANTEE: answer >= true count (never underestimates) error bounded by: ε × total insertions with probability ≥ 1-δ
💡 Used in: Network traffic analysis (finding "heavy hitters" — IP addresses sending lots of packets), streaming databases (approximate aggregations), search engines (tracking trending queries), and ad-tech (counting ad impressions without storing every event).
Section 9 — Real-World Applications
🌍 Where Bloom Filters Run Right Now:
  • Google Chrome — uses a Bloom filter to check if a URL is a known malware/phishing site before doing a full lookup. The filter runs locally, saving a network round-trip for safe URLs.
  • Apache Cassandra — each SSTable (on-disk data file) has a Bloom filter. Before reading from disk, Cassandra checks if a key might be in that SSTable. False positives cause unnecessary disk reads; false negatives are impossible (we'd never miss data).
  • Redis — the RedisBloom module adds native Bloom filter commands. Used by e-commerce companies to filter already-seen recommendation items.
  • Akamai CDN — uses Bloom filters to avoid caching one-hit-wonder objects (pages visited only once). If a URL isn't in the filter, don't cache it; add it to the filter. On a second request, the filter says "probably seen before" and the CDN caches it.
  • Ethereum / Bitcoin — SPV (Simple Payment Verification) wallets use Bloom filters to request only relevant transactions from full nodes without revealing which addresses they own.

Section 10 — Check Your Understanding
Q1. A Bloom filter with n=1M insertions and m=10M bits (10 bits/element) has approximately what false positive rate?
Hint: FP rate ≈ (1 − e^(−kn/m))^k. Optimal k ≈ 7 for 10 bits/element.
With k=7 hash functions, n=1M elements, m=10M bits: FP ≈ (1 − e^(−7×1M/10M))^7 = (1 − e^(−0.7))^7 ≈ (1 − 0.4966)^7 ≈ 0.5034^7 ≈ 0.82%. Approximately 1% false positive rate — meaning one in every ~120 non-member queries incorrectly returns "probably present".
Q2. You insert element X into a Bloom filter, then later query for X and get "DEFINITELY NOT IN SET". Is this possible?
Hint: Think about the fundamental guarantee.
No, this is impossible in a standard (non-concurrent) Bloom filter. Inserting X sets k specific bits permanently. Querying X checks those same k bits — they are still set. A "definitely not present" result requires at least one bit to be 0, but all bits set during insertion remain 1 (standard Bloom filters have no deletion). This is the fundamental asymmetry: false negatives are impossible; only false positives can occur.
Q3. Cassandra's Bloom filter produces a false positive for key K. What actually happens in Cassandra?
Hint: A false positive means the filter says "probably here" but K isn't.
A false positive causes Cassandra to perform an unnecessary disk read from the SSTable — it opens the file and scans for key K, only to find it's not there. This wastes I/O time but does not produce incorrect results. The query still returns "not found" correctly. This is an acceptable cost: false positives waste time but don't cause data corruption or incorrect responses. The Bloom filter still eliminates the vast majority (99%+) of unnecessary reads.
Q4. How does a Count-Min Sketch differ from a Bloom filter, and what does it guarantee about its frequency estimates?
Hint: Think about what "false positive" means for frequency estimation.
A Bloom filter answers "is X in the set?" (yes/no). A Count-Min Sketch answers "how many times was X inserted?" (frequency). The CMS guarantee is: the estimate is always ≥ the true count (never underestimates). The error is bounded by ε × N (where N = total insertions) with probability ≥ 1−δ. Collisions can only inflate counts (other items accidentally increment the same counters), never deflate them. Taking the minimum across all d rows minimises — but cannot eliminate — this overcounting.
Q5. Why do we use double hashing (h1 + i×h2) rather than k fully independent hash functions in a Bloom filter implementation?
Hint: Think about implementation complexity and performance.
Computing k independent, well-distributed hash functions is complex and slow (each requires separate computation over the input). Double hashing approximates k independent hash functions using only two base hash functions: h_i(x) = (h1(x) + i × h2(x)) mod m. A 2002 analysis by Kirsch and Mitzenmacher proved this achieves the same asymptotic false positive rate as k truly independent hash functions. So we get near-optimal behaviour with just two hash computations and cheap arithmetic per probe — a major practical speedup.
