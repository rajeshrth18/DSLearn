Section 1 — The Analogy: The Postal Sorting Machine

Imagine a massive postal facility that receives millions of letters daily. Rather than searching through every mailbag for a specific recipient, a sorting machine reads the ZIP code and sends each letter directly to the correct bin — in one step, regardless of how many letters exist. The ZIP code is the address of the bin.

A hash table works on the same idea. Given any key (a name, an ID, a URL), a hash function computes a bucket index directly. Instead of searching O(n) elements, you jump to the exact slot in O(1). The hash function is the sorting machine; the array of buckets is the bin system.

?? The Core Promise: With a good hash function and a properly sized array, both insert and lookup run in O(1) average time — independent of how many elements are stored. This is the single most important performance guarantee in practical software engineering.
Section 2 — What is Hashing?

Hashing is the process of mapping a key of arbitrary type and size to a fixed-size integer index. The resulting integer — called the hash code — is then mapped into an array index using the modulo operator.

HASHING PIPELINE Key (any type) ? Hash Function ? Hash Code (int) ? index = code % capacity ? Array slot Example: key = "Alice" +-- hash("Alice") = 92748301 (some large integer) +-- 92748301 % 16 = 13 (capacity = 16) +-- Store value at array[13] Example: key = "Bob" +-- hash("Bob") = 47291 (different integer) +-- 47291 % 16 = 11 +-- Store value at array[11] Lookup("Alice"): +-- Recompute: hash("Alice") % 16 = 13 +-- Return array[13] — found in ONE step!

2.1 — Anatomy of a Hash Table

  • Backing Array: Fixed-size array of buckets (slots).
  • Hash Function: Converts a key to an integer hash code.
  • Index Mapping: index = Math.Abs(hashCode) % capacity
  • Collision Handler: Two keys can produce the same index — the strategy for handling this defines the hash table variant.
  • Load Factor: ? = n / capacity — ratio of stored elements to total buckets. Triggers resize when too high.

Section 4 — Common Hash Functions

4.1 — Division Method (Simple, Educational)

The simplest approach: h(k) = k % m where m is the table size. Works for integer keys. Choose m to be a prime number far from a power of 2 to avoid clustering.

4.2 — Polynomial Rolling Hash (Strings)

The standard approach for hashing strings. Treats the string as a polynomial evaluated at a chosen base:

h(s) = s[0]·p° + s[1]·pą + s[2]·p˛ + ... + s[n-1]·p^(n-1) mod m

Common choices: p = 31 (for lowercase letters), m = 10^9 + 7 (large prime).

// Polynomial rolling hash for strings (C#)
static long PolyHash(string s)
{
    const long p = 31, m = 1_000_000_007;
    long hash = 0, power = 1;
    foreach (char c in s)
    {
        hash  = (hash + (c - 'a' + 1) * power) % m;
        power = (power * p) % m;
    }
    return hash;
}
// "abc" ? deterministic long in [0, 10^9+6]

4.3 — DJB2 (Practical String Hashing)

Created by Dan Bernstein, DJB2 is one of the most cited non-cryptographic hash functions. It uses bit-shifting to mix bits aggressively:

// DJB2 hash function
static uint DJB2(string s)
{
    uint hash = 5381;
    foreach (char c in s)
        hash = ((hash << 5) + hash) + c;  // hash * 33 + c
    return hash;
}
// Fast, avalanche effect is good, widely used in practice

4.4 — FNV-1a (Fowler-Noll-Vo)

FNV-1a XORs before multiplying, which improves avalanche effect over DJB2:

// FNV-1a 32-bit hash
static uint FNV1a(string s)
{
    const uint FNV_prime  = 16777619;
    const uint FNV_offset = 2166136261;
    uint hash = FNV_offset;
    foreach (char c in s)
    {
        hash ^= c;            // XOR first (FNV-1a key difference)
        hash *= FNV_prime;
    }
    return hash;
}
?? In Practice — .NET's GetHashCode(): You almost never write your own hash function in production C#. object.GetHashCode() is well-implemented for all built-in types. The important thing is to understand why it works, not to replace it.
Section 5 — Collision Resolution: Separate Chaining

A collision occurs when two different keys hash to the same index. Separate chaining resolves this by storing all colliding entries in a linked list (or other structure) at that bucket.

COLLISION with "Bob"! Bucket Array: [0] ? null [1] ? null [2] ? ["Alice":99] ? ["Charlie":42] ? null (chain of length 2) [3] ? null [4] ? null [5] ? ["Bob":77] ? ["Dave":55] ? null (chain of length 2) [6] ? null Lookup("Charlie"): 1. hash("Charlie") % 7 = 2 2. Walk chain at [2]: skip "Alice", find "Charlie" ? return 42

5.1 — Chaining Performance

  • Average case: O(1 + ?) where ? = load factor = n/capacity
  • Worst case: O(n) if all keys hash to the same bucket
  • Best case: O(1) with perfect distribution
  • Works well at high load factors (? > 0.7) — just longer chains
?? Java's HashMap uses separate chaining. When a bucket's chain exceeds 8 elements, it converts the chain to a Red-Black Tree, bringing worst-case per-bucket lookup to O(log n) instead of O(n).
Section 6 — Collision Resolution: Open Addressing

In open addressing, all elements live directly in the backing array — no linked lists. When a collision occurs, the algorithm probes (searches) for the next empty slot according to a defined probing sequence.

6.1 — Linear Probing

On collision at index i, try i+1, i+2, i+3, ... (wrapping around). Simple, cache-friendly, but suffers from primary clustering — long runs of occupied slots build up.

LINEAR PROBING — Insert sequence: 10, 22, 31, 4, 15 (capacity = 7) h(10) = 10 % 7 = 3 ? [3] empty ? store at [3] h(22) = 22 % 7 = 1 ? [1] empty ? store at [1] h(31) = 31 % 7 = 3 ? [3] TAKEN ? probe [4] empty ? store at [4] h(4) = 4 % 7 = 4 ? [4] TAKEN ? probe [5] empty ? store at [5] h(15) = 15 % 7 = 1 ? [1] TAKEN ? probe [2] empty ? store at [2] Array: [ _ 22 15 10 31 4 _ ] 0 1 2 3 4 5 6 Note: 15 and 22 are adjacent even though they didn't collide with each other. This is PRIMARY CLUSTERING — probes build on each other.

6.2 — Quadratic Probing

On collision at index i, try i+1˛, i+2˛, i+3˛, .... Reduces primary clustering but can cause secondary clustering (all keys with the same initial hash follow the same probe sequence).

// Quadratic probing formula
// probe(i) = (hash + i*i) % capacity
int Probe(int hash, int i, int capacity)
    => (hash + i * i) % capacity;

6.3 — Double Hashing

Uses a second hash function to compute the step size. Eliminates both primary and secondary clustering — the best open addressing strategy in practice:

// Double hashing — probe(i) = (h1(key) + i * h2(key)) % capacity
// h2 must be non-zero and coprime to capacity (use prime capacity)
int h1 = Math.Abs(key.GetHashCode()) % capacity;
int h2 = 1 + (Math.Abs(key.GetHashCode()) % (capacity - 1));
// probe sequence for collision i: (h1 + i * h2) % capacity

Section 7 — Load Factor & Rehashing

The load factor ? = n / capacity measures how full the table is. As ? increases, collision probability rises and performance degrades toward O(n).

LOAD FACTOR EFFECTS ? = 0.1 ? Very few collisions, but wastes memory (90% empty) ? = 0.5 ? Good balance — low collision probability ? = 0.75 ? .NET Dictionary's default rehash threshold ? = 1.0 ? Open addressing fails (no empty slots), chaining degrades to O(n) ? > 1.0 ? Only possible with separate chaining REHASHING PROCESS: 1. Allocate new array with ~2× capacity (next prime) 2. Re-hash EVERY existing key into new array 3. Update capacity, reset count Cost: O(n) amortized — but happens rarely enough that insertion stays O(1) amortized
?? .NET Dictionary: Default load factor = 0.75. Grows by finding the next prime number = 2 × current capacity. The prime size helps the modulo operation distribute keys evenly.
Section 8 — Complete C# Implementation

A hash table using separate chaining with linked lists. Supports generic keys and values, auto-rehash at load factor 0.75.

C# 
using System;
using System.Collections.Generic;

public class HashTable<TKey, TValue> where TKey : notnull
{
    private List<(TKey key, TValue val)>[] _buckets;
    private int _count;
    private const double _loadThreshold = 0.75;

    public int Count => _count;

    public HashTable(int initialCapacity = 16)
    {
        _buckets = new List<(TKey, TValue)>[initialCapacity];
    }

    // -- Core: compute bucket index -----------------------
    private int BucketIndex(TKey key)
        => Math.Abs(key.GetHashCode()) % _buckets.Length;

    // -- Put (insert or update) ---------------------------
    public void Put(TKey key, TValue value)
    {
        if ((double)(_count + 1) / _buckets.Length > _loadThreshold)
            Rehash();

        int idx = BucketIndex(key);
        _buckets[idx] ??= new List<(TKey, TValue)>();

        for (int i = 0; i < _buckets[idx].Count; i++)
        {
            if (EqualityComparer<TKey>.Default.Equals(_buckets[idx][i].key, key))
            {
                _buckets[idx][i] = (key, value);  // update existing
                return;
            }
        }
        _buckets[idx].Add((key, value));  // new entry
        _count++;
    }

    // -- Get ----------------------------------------------
    public bool TryGet(TKey key, out TValue? value)
    {
        int idx = BucketIndex(key);
        if (_buckets[idx] != null)
            foreach (var (k, v) in _buckets[idx])
                if (EqualityComparer<TKey>.Default.Equals(k, key))
                    { value = v; return true; }
        value = default;
        return false;
    }

    // -- Indexer ------------------------------------------
    public TValue this[TKey key]
    {
        get
        {
            if (TryGet(key, out var v)) return v!;
            throw new KeyNotFoundException($"Key '{key}' not found");
        }
        set => Put(key, value);
    }

    // -- Remove -------------------------------------------
    public bool Remove(TKey key)
    {
        int idx = BucketIndex(key);
        if (_buckets[idx] == null) return false;
        for (int i = 0; i < _buckets[idx].Count; i++)
        {
            if (EqualityComparer<TKey>.Default.Equals(_buckets[idx][i].key, key))
            {
                _buckets[idx].RemoveAt(i);
                _count--;
                return true;
            }
        }
        return false;
    }

    // -- ContainsKey --------------------------------------
    public bool ContainsKey(TKey key) => TryGet(key, out _);

    // -- Rehash (double capacity) -------------------------
    private void Rehash()
    {
        var old = _buckets;
        _buckets = new List<(TKey, TValue)>[old.Length * 2];
        _count = 0;
        foreach (var bucket in old)
            if (bucket != null)
                foreach (var (k, v) in bucket)
                    Put(k, v);
    }
}

// -- Demo -------------------------------------------------
class Program
{
    static void Main()
    {
        var ht = new HashTable<string, int>();
        ht.Put("Alice", 92);
        ht.Put("Bob",   85);
        ht.Put("Carol", 91);
        ht["Alice"] = 95;  // update via indexer

        Console.WriteLine(ht["Alice"]);   // 95
        Console.WriteLine(ht.ContainsKey("Dave"));  // False
        ht.Remove("Bob");
        Console.WriteLine(ht.Count);  // 2

        /* -- Output --
        95
        False
        2
        */
    }
}
Section 9 — Complete Python Implementation
Python 
class HashTable:
    def __init__(self, initial_capacity=16):
        self._capacity = initial_capacity
        self._buckets  = [None] * initial_capacity  # list of lists
        self._count    = 0
        self._load_threshold = 0.75

    def _index(self, key):
        return abs(hash(key)) % self._capacity

    def put(self, key, value):
        if (self._count + 1) / self._capacity > self._load_threshold:
            self._rehash()
        idx = self._index(key)
        if self._buckets[idx] is None:
            self._buckets[idx] = []
        for i, (k, v) in enumerate(self._buckets[idx]):
            if k == key:
                self._buckets[idx][i] = (key, value)  # update
                return
        self._buckets[idx].append((key, value))
        self._count += 1

    def get(self, key, default=None):
        idx = self._index(key)
        if self._buckets[idx]:
            for k, v in self._buckets[idx]:
                if k == key:
                    return v
        return default

    def remove(self, key):
        idx = self._index(key)
        if self._buckets[idx]:
            for i, (k, v) in enumerate(self._buckets[idx]):
                if k == key:
                    self._buckets[idx].pop(i)
                    self._count -= 1
                    return True
        return False

    def contains(self, key):
        return self.get(key) is not None

    def _rehash(self):
        old = self._buckets
        self._capacity *= 2
        self._buckets = [None] * self._capacity
        self._count = 0
        for bucket in old:
            if bucket:
                for k, v in bucket:
                    self.put(k, v)

    def __setitem__(self, key, value): self.put(key, value)
    def __getitem__(self, key):
        v = self.get(key)
        if v is None: raise KeyError(key)
        return v
    def __len__(self): return self._count


if __name__ == "__main__":
    ht = HashTable()
    ht.put("Alice", 92)
    ht.put("Bob",   85)
    ht["Alice"] = 95   # update via __setitem__

    print(ht["Alice"])         # 95
    print(ht.contains("Bob"))  # True
    ht.remove("Bob")
    print(len(ht))               # 1

    # Output:
    # 95
    # True
    # 1

Section 11 — Check Your Understanding
Q1. Two keys A and B have the same hash code. What term describes this event, and which property of hash functions cannot prevent it?
Hint: Think about the pigeonhole principle.
This is a collision. Because the key space is infinite (any string) and the hash code space is finite (32-bit int = ~4 billion values), collisions are mathematically unavoidable by the pigeonhole principle. Even a perfect hash function for a fixed set cannot avoid collisions for arbitrary inputs. The injectivity property is the one that cannot be guaranteed for arbitrary inputs.
Q2. You have a hash table with capacity 10 and 7 elements. What is the load factor, and should you rehash?
Hint: ? = n / capacity. Default threshold = 0.75.
? = 7/10 = 0.70. This is just below the 0.75 threshold, so no rehash yet. But adding one more element would bring ? to 0.8, triggering a rehash to capacity ~20.
Q3. Why is open addressing typically faster in practice despite separate chaining being simpler to implement?
Hint: Think about memory and CPU caches.
Open addressing stores all data in one contiguous array, so CPU cache prefetching works effectively — once you load one memory page, nearby elements are already in the cache. Separate chaining requires following pointers to heap-allocated nodes, each at random memory addresses, causing cache misses on every probe. This makes open addressing ~2–5× faster in practice at low load factors, even though the algorithm is conceptually the same O(1).
Q4. In double hashing, why must h2(key) be non-zero and coprime to the table capacity?
Hint: What happens if h2 = 2 and capacity = 8?
If h2(key) = 2 and capacity = 8, the probe sequence visits indices: i, i+2, i+4, i+6, i+0 (wraps), i+2... — only even indices! Half the table is unreachable, and the algorithm may falsely report "table full" when there are empty slots at odd indices. By requiring h2 to be coprime to capacity (gcd = 1), we guarantee the probe sequence visits all n slots before repeating, ensuring complete coverage.
Q5. Why is GetHashCode() in C# not guaranteed to be consistent across different program runs (since .NET 4.5)?
Hint: Think about security and HashDoS attacks.
To prevent Hash DoS (denial of service) attacks. An attacker who knows a deterministic hash function can craft a large set of strings that all hash to the same bucket, degrading a web server's Dictionary lookups from O(1) to O(n) and causing CPU exhaustion. Since .NET 4.5, a random seed is injected per-process-run, so the attacker cannot predict which strings will collide. This means you must never persist hash codes across process boundaries (e.g., don't store them in databases or files).
