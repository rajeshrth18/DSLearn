Section 1 — What Is DFS?

Imagine exploring a maze by always taking the first unexplored path you see, going as deep as possible. When you hit a dead end, you backtrack to the last junction and try the next path. You fully explore one branch before touching another. That is Depth-First Search.

DFS uses a stack — either the call stack (recursive) or an explicit stack (iterative). It ventures as far as possible from the start vertex along a single chain of edges, only backtracking when it can go no further.

?? BFS vs DFS at a glance:
BFS — Queue (FIFO) — Level by level — Shortest path in unweighted graphs
DFS — Stack (LIFO) — Go deep first — Cycle detection, topological sort, SCC

B ? D ? E ? C ? F BFS ORDER: A ? B ? C ? D ? E ? F (for comparison) KEY INSIGHT: DFS goes DEEP into B's subtree (D, E) before visiting C. BFS visits C (same level as B) before D and E.
Section 3 — DFS Variants

3.1 — Recursive DFS (uses call stack)

The natural, elegant form. The system's call stack acts as the DFS stack. Simple to write, but risks stack overflow for very deep graphs (depth > ~5,000 in Python, ~millions in C#).

3.2 — Iterative DFS (explicit stack)

Uses a Stack<T> / Python list explicitly. Avoids recursion limits. Note: iterative DFS with a stack does not always visit nodes in the same order as recursive DFS because the stack reverses neighbor order. For DFS applications that depend on pre/post order (like topological sort), use recursive DFS or handle carefully.

RECURSIVE vs ITERATIVE DFS — Order difference Graph: A ? [B, C], B ? [D, E], C ? [F] Recursive DFS from A: Visit A, then first neighbor B (go deep), then D, E, then C, F Order: A B D E C F Iterative DFS (push neighbors in order) from A: Push A. Pop A, push C then B (reversed so B processes first) Pop B, push E then D. Pop D (leaf). Pop E (leaf). Pop C, push F. Pop F. Order: A B D E C F Note: If neighbors are pushed in REVERSE order, iterative DFS gives same order as recursive. If pushed forward, order is reversed. IMPORTANT: Neither DFS order is "correct" — DFS just needs to go deep. The specific order depends on neighbor ordering in the graph.
Section 4 — DFS Timestamps (Pre/Post Order)

During DFS, we can assign each vertex two timestamps:

  • pre[v] (discovery time) — clock value when v is first visited
  • post[v] (finish time) — clock value when DFS returns from v (all descendants done)

These timestamps reveal the DFS tree structure and are essential for topological sort and SCC algorithms.

DFS TIMESTAMPS Graph: A ? B ? D, A ? C ? D, B ? C (directed graph) DFS from A, clock starts at 0: Enter A: pre[A]=1 Enter B: pre[B]=2 Enter D: pre[D]=3 (no unvisited neighbors) Exit D: post[D]=4 Enter C: pre[C]=5 D already visited (skip) Exit C: post[C]=6 Exit B: post[B]=7 C already visited (skip) Exit A: post[A]=8 Timestamps: pre post A 1 8 B 2 7 C 5 6 D 3 4 RULE: if pre[u] < pre[v] < post[v] < post[u] then v is a DESCENDANT of u in the DFS tree. TOPOLOGICAL SORT: vertices in DECREASING post order = valid topological order For above: A(8), B(7), C(6), D(4) ? Topo order: A B C D ? (A must come before B and C; B before C and D; C before D)
Section 5 — Cycle Detection with DFS

5.1 — Undirected Graph: Back Edge = Cycle

During DFS on an undirected graph, if we find an edge to a vertex that is already visited AND is not our direct parent, it's a back edge ? cycle detected.

CYCLE DETECTION — UNDIRECTED GRAPH Graph: A–B–C–A (triangle — has cycle) DFS from A: Visit A, go to B (parent=A) Visit B, go to C (parent=B) Visit C, check neighbors: A is visited, A ? parent(B) ? CYCLE DETECTED! Graph: A–B–C (path — no cycle) Visit A, go to B, go to C C's only neighbor B = parent ? not a cycle ? no cycle found

5.2 — Directed Graph: Three-Color DFS

For directed graphs, an undirected "already visited" check is not enough — we need to distinguish between back edges (cycle) and cross/forward edges (no cycle).

  • WHITE — not yet visited
  • GRAY — currently in the DFS recursion stack (being processed)
  • BLACK — fully processed (all descendants done)

A back edge goes from a GRAY vertex to another GRAY vertex ? cycle detected.

THREE-COLOR DFS — DIRECTED CYCLE DETECTION Directed Graph: A ? B ? C ? A (cycle!) DFS from A: Color A = GRAY (in stack) DFS on B: Color B = GRAY (in stack) DFS on C: Color C = GRAY (in stack) C's neighbor A: A is GRAY ? BACK EDGE ? CYCLE DETECTED! Color C = BLACK Color B = BLACK Color A = BLACK Directed Graph: A ? B ? C (DAG — no cycle) DFS from A: A=GRAY, B=GRAY, C=GRAY?BLACK, B?BLACK, A?BLACK No edge ever goes to a GRAY vertex ? no cycle
Section 6 — Topological Sort using DFS

A topological sort of a Directed Acyclic Graph (DAG) is a linear ordering of vertices such that for every directed edge u?v, u appears before v. This represents dependency ordering — compile order, task scheduling, prerequisite chains.

DFS-based algorithm: Run DFS. When a vertex is fully processed (post-order), add it to the front of the result list. The resulting list is a valid topological order.

TOPOLOGICAL SORT — DFS POST-ORDER DAG: Courses with prerequisites A = Intro CS B = Data Structures (requires A) C = Algorithms (requires A, B) D = Systems (requires A) E = Compilers (requires C, D) Directed edges: A?B, A?D, B?C, C?E, D?E DFS from A: Enter A Enter B Enter C Enter E (no unvisited outgoing edges) Exit E ? prepend E: [E] Exit C ? prepend C: [C, E] Exit B ? prepend B: [B, C, E] Enter D E already visited — skip Exit D ? prepend D: [D, B, C, E] Exit A ? prepend A: [A, D, B, C, E] Topological order: A ? D ? B ? C ? E (Check: A before B ?, A before D ?, B before C ?, C before E ?, D before E ?)
Section 7 — DFS on Grids: Flood Fill & Islands

DFS on a 2D grid explores all connected cells from a starting cell. This is the "flood fill" algorithm (used in paint buckets) and powers the classic "number of islands" problem.

Count = # times DFS starts. Scan (0,0)=1: DFS! Mark connected land: (0,0)(0,1)(1,0)(1,1) ? all marked visited Islands found so far: 1 Continue scan... (0,2)=0, (0,3)=0, (0,4)=0... (2,2)=1: DFS! Mark (2,2) ? visited Islands found so far: 2 (3,3)=1: DFS! using System;
using System.Collections.Generic;

// ---------------------------------------------------------------
// DFS — Traversal, Cycle Detection, Topological Sort, Islands
// ---------------------------------------------------------------

public static class DFS
{
    // -- 1. Recursive DFS traversal ------------------------------
    public static List<string> Traverse(
        Dictionary<string, List<string>> graph, string start)
    {
        var visited = new HashSet<string>();
        var result  = new List<string>();
        DfsHelper(graph, start, visited, result);
        return result;
    }

    private static void DfsHelper(
        Dictionary<string, List<string>> graph,
        string vertex, HashSet<string> visited, List<string> result)
    {
        visited.Add(vertex);
        result.Add(vertex);
        if (!graph.ContainsKey(vertex)) return;
        foreach (var neighbor in graph[vertex])
            if (!visited.Contains(neighbor))
                DfsHelper(graph, neighbor, visited, result);
    }

    // -- 2. Cycle detection — undirected graph -------------------
    public static bool HasCycleUndirected(
        Dictionary<string, List<string>> graph)
    {
        var visited = new HashSet<string>();
        foreach (var v in graph.Keys)
            if (!visited.Contains(v))
                if (CycleUndirectedDFS(graph, v, "", visited))
                    return true;
        return false;
    }

    private static bool CycleUndirectedDFS(
        Dictionary<string, List<string>> graph,
        string v, string parent, HashSet<string> visited)
    {
        visited.Add(v);
        if (!graph.ContainsKey(v)) return false;
        foreach (var neighbor in graph[v])
        {
            if (!visited.Contains(neighbor))  // unvisited — recurse
            {
                if (CycleUndirectedDFS(graph, neighbor, v, visited))
                    return true;
            }
            else if (neighbor != parent)        // visited + not parent ? back edge ? CYCLE
                return true;
        }
        return false;
    }

    // -- 3. Cycle detection — directed graph (three-color) -------
    public static bool HasCycleDirected(
        Dictionary<string, List<string>> graph)
    {
        var color = new Dictionary<string, int>();  // 0=white, 1=gray, 2=black
        foreach (var v in graph.Keys)
            color[v] = 0;
        foreach (var v in graph.Keys)
            if (color[v] == 0)
                if (CycleDirectedDFS(graph, v, color))
                    return true;
        return false;
    }

    private static bool CycleDirectedDFS(
        Dictionary<string, List<string>> graph,
        string v, Dictionary<string, int> color)
    {
        color[v] = 1;  // gray = in recursion stack
        if (graph.ContainsKey(v))
        {
            foreach (var neighbor in graph[v])
            {
                if (!color.ContainsKey(neighbor)) color[neighbor] = 0;
                if (color[neighbor] == 1) return true;  // gray = back edge = cycle!
                if (color[neighbor] == 0)
                    if (CycleDirectedDFS(graph, neighbor, color))
                        return true;
            }
        }
        color[v] = 2;  // black = fully processed
        return false;
    }

    // -- 4. Topological Sort — DFS post-order --------------------
    public static List<string> TopologicalSort(
        Dictionary<string, List<string>> graph)
    {
        var visited = new HashSet<string>();
        var result  = new LinkedList<string>();  // prepend = O(1)
        foreach (var v in graph.Keys)
            if (!visited.Contains(v))
                TopoHelper(graph, v, visited, result);
        return new List<string>(result);
    }

    private static void TopoHelper(
        Dictionary<string, List<string>> graph,
        string v, HashSet<string> visited, LinkedList<string> result)
    {
        visited.Add(v);
        if (graph.ContainsKey(v))
            foreach (var neighbor in graph[v])
                if (!visited.Contains(neighbor))
                    TopoHelper(graph, neighbor, visited, result);
        result.AddFirst(v);  // POST-ORDER: add to front after all descendants
    }

    // -- 5. Number of Islands — DFS flood fill on grid -----------
    public static int NumIslands(char[][] grid)
    {
        int islands = 0;
        for (int r = 0; r < grid.Length; r++)
        {
            for (int c = 0; c < grid[r].Length; c++)
            {
                if (grid[r][c] == '1')
                {
                    islands++;
                    FloodFill(grid, r, c);  // mark entire island as visited
                }
            }
        }
        return islands;
    }

    private static void FloodFill(char[][] grid, int r, int c)
    {
        if (r < 0 || r >= grid.Length || c < 0 || c >= grid[0].Length) return;
        if (grid[r][c] != '1') return;  // water ('0') or already marked ('2')
        grid[r][c] = '2';               // mark visited by changing value
        FloodFill(grid, r - 1, c);  // up
        FloodFill(grid, r + 1, c);  // down
        FloodFill(grid, r, c - 1);  // left
        FloodFill(grid, r, c + 1);  // right
    }
}

        var dag = new Dictionary<string, List<string>>

import sys
from typing import Dict, List, Optional, Set
sys.setrecursionlimit(10000)  # increase limit for deep graphs

# ---------------------------------------------------------------
# DFS — Full Python Implementation
# ---------------------------------------------------------------

Graph = Dict[str, List[str]]

def dfs_traverse(graph: Graph, start: str) -> List[str]:
    """Recursive DFS traversal."""
    visited, result = set(), []

    def dfs(v: str):
        visited.add(v)
        result.append(v)
        for nb in graph.get(v, []):
            if nb not in visited:
                dfs(nb)

    dfs(start)
    return result


def has_cycle_undirected(graph: Graph) -> bool:
    """Cycle detection in undirected graph via DFS back-edge check."""
    visited = set()

    def dfs(v: str, parent: Optional[str]) -> bool:
        visited.add(v)
        for nb in graph.get(v, []):
            if nb not in visited:
                if dfs(nb, v):
                    return True
            elif nb != parent:  # visited AND not direct parent ? back edge ? cycle
                return True
        return False

    for v in graph:
        if v not in visited:
            if dfs(v, None):
                return True
    return False


def has_cycle_directed(graph: Graph) -> bool:
    """Cycle detection in directed graph — three-color DFS."""
    WHITE, GRAY, BLACK = 0, 1, 2
    color = {v: WHITE for v in graph}

    def dfs(v: str) -> bool:
        color[v] = GRAY  # mark "in progress"
        for nb in graph.get(v, []):
            if nb not in color:
                color[nb] = WHITE
            if color[nb] == GRAY:   # neighbor currently in stack = cycle!
                return True
            if color[nb] == WHITE:
                if dfs(nb):
                    return True
        color[v] = BLACK   # mark "done"
        return False

    return any(dfs(v) for v in graph if color[v] == WHITE)


def topological_sort(graph: Graph) -> List[str]:
    """Topological sort via DFS post-order (DAGs only)."""
    visited = set()
    result  = []

    def dfs(v: str):
        visited.add(v)
        for nb in graph.get(v, []):
            if nb not in visited:
                dfs(nb)
        result.append(v)  # POST-ORDER: append after all descendants

    for v in graph:
        if v not in visited:
            dfs(v)
    return result[::-1]  # reverse for correct topo order


def num_islands(grid: List[List[str]]) -> int:
    """Count islands using DFS flood fill. Modifies grid in-place."""
    if not grid:
        return 0
    rows, cols = len(grid), len(grid[0])
    islands = 0

    def flood_fill(r: int, c: int):
        if r < 0 or r >= rows or c < 0 or c >= cols:
            return
        if grid[r][c] != '1':
            return
        grid[r][c] = '2'          # mark visited
        flood_fill(r-1, c)
        flood_fill(r+1, c)
        flood_fill(r, c-1)
        flood_fill(r, c+1)

    for r in range(rows):
        for c in range(cols):
            if grid[r][c] == '1':
                islands += 1
                flood_fill(r, c)
    return islands

Time complexity O(V + E) Each vertex visited once, each edge examined once.
Space complexity (recursive) O(V) Call stack depth = max recursion depth = O(V) in worst case (linear graph).

Section 11 — Real-World Usage
?? Git — DAG & Topological Sort
Git's commit history is a Directed Acyclic Graph. Each commit points to its parent(s). DFS topological sort is used to: determine merge order, find common ancestors (for git merge), compute reachable commits (for git log), and detect cycles (there should never be one in a valid repo). Git's git log --graph output uses DFS traversal to render the commit tree.

?? Compilers — Dependency Resolution
When compiling C++/C# solutions with many header files, the compiler builds a dependency DAG (file A depends on B depends on C). DFS topological sort gives the correct compilation order. Cycle detection catches circular imports (#include cycles), which are compilation errors. Same pattern used in npm/pip/Maven for package resolution.

?? Maze Generation Algorithms
Many maze generators (Depth-First Maze, Recursive Backtracker) use DFS. Starting from a cell, randomly pick an unvisited neighbor, carve a path, repeat. When stuck, backtrack. The result is a perfect maze (exactly one path between any two cells). Used in game development (Dungeons, Rogue-likes) and procedural content generation.

?? Network Routing — STP (Spanning Tree Protocol)
Network switches use STP to prevent loops in Ethernet networks. The algorithm builds a spanning tree of the network topology using a process conceptually similar to DFS — marking edges as "tree edges" or "blocking edges". DFS-based SCC (Tarjan's algorithm) detects independent subnetworks.
Section 12 — Check Your Understanding

Q1 — You run recursive DFS on a graph with 10,000 vertices arranged in a straight line (v1?v2?v3?...?v10000). What happens in Python?

?? Python has a default recursion limit. What is it?
Answer: Python's default recursion limit is 1,000 (set by sys.getrecursionlimit()). A linear chain of 10,000 vertices requires recursion depth 10,000 — DFS will hit a RecursionError: maximum recursion depth exceeded at around vertex 1,000. Solutions: (1) sys.setrecursionlimit(20000) — increases the limit, risky as it can cause a C-level stack overflow; (2) Convert to iterative DFS using an explicit stack — eliminates recursion limit entirely; (3) BFS — uses a queue, no recursion depth issue. For competitive programming or production code on deep graphs, always use iterative DFS or set the recursion limit explicitly.
Rule: For graphs deeper than ~500 nodes in Python, prefer iterative DFS or increase the limit with sys.setrecursionlimit(). C# has a much larger call stack (~1 million frames) so recursion is safer there.

Q2 — Explain why DFS post-order gives a valid topological sort. Why must we add vertices to the result AFTER visiting all descendants?

?? Think about the dependency relationship and what "post-order" means.
Answer: If edge u?v exists, v must appear AFTER u in topological order. DFS post-order adds v to the result before u (because DFS finishes v before backtracking to u). When we reverse the post-order list, u comes before v — which is exactly what we need. More precisely: if u?v is an edge, then DFS must fully finish v (enter and exit v's subtree) before it can finish u, so post[v] < post[u]. In reversed post-order, u comes before v. This holds transitively for all edges, giving a valid topological order for DAGs. If the graph has a cycle (u?v?...?u), this breaks because post[u] would need to be both less than and greater than itself — hence topological sort only works on DAGs.
Mnemonic: "Post-order reversed = topological order". Add to result on the way OUT of DFS (after all neighbors processed), then reverse the list.

Q3 — Why is the "visited AND not parent" check necessary for cycle detection in undirected graphs? What goes wrong without it?

?? Consider the edge from B back to A in an undirected graph: A–B.
Answer: In an undirected graph, every edge A–B is represented as two directed edges: A?B and B?A. During DFS from A: we visit A, then visit B (parent=A). Now from B we check neighbors — A is visited. Without the parent check, we'd say "A is visited and we didn't come from there... wait, we DID come from A!" — so we'd incorrectly report a cycle. The parent check prevents this false positive: if the visited neighbor IS our parent, that's just the undirected edge we traversed to get here (not a back edge). Only flag a cycle if the visited neighbor is NOT our parent — meaning there's another path back to it (a true cycle).
Edge case: In graphs with multi-edges (parallel edges between same pair of vertices), parent check alone isn't sufficient — need parent edge tracking. For simple graphs (single edge between any pair), parent vertex check is correct.

Q5 — In the three-color DFS cycle detection for directed graphs, why is BLACK (fully processed) different from GRAY (in stack)? Can a BLACK neighbor indicate a cycle?

?? Think about what "BLACK" means — all descendants have been explored.
Answer: A BLACK vertex means all its descendants have been fully explored — there is no path from any BLACK vertex back to any vertex currently in the DFS stack (that would have been detected already). If we reach a BLACK vertex v from the current DFS path, it means: "we've explored v's entire subtree already, and found no cycles there." This is a cross-edge or forward-edge — NOT a cycle indicator. Only a GRAY vertex (currently on the recursion stack) as a neighbor indicates a back edge and therefore a cycle — because GRAY means "we're in the middle of exploring this vertex's path right now, and we found a way back to it." Gray ? Gray = cycle. Any ? Black = safe (already fully explored, no cycles found in that direction).
Analogy: GRAY = "currently walking this road." BLACK = "road fully explored, nothing bad found." Seeing a BLACK neighbor is like finding a road you've fully mapped — no new discoveries. Seeing a GRAY neighbor is like finding a road you're currently on — that's a loop!
