Section 1 — Why BFS Fails for Weighted Graphs

In Module 9, BFS found the shortest path perfectly — but only because every edge had the same cost (weight = 1). In weighted graphs, edges have different costs, and BFS completely ignores those weights.

PROBLEM — BFS picks the wrong "shortest" path in weighted graphs A --(1)-- B --(1)-- C ¦ ? +-----(10)----------+ BFS from A to C: BFS finds: A ? B ? C (2 hops) BFS ignores weight: cost = 1 + 1 = 2 ? BFS says this is shortest Actual shortest cost: A ? C direct = 10 ? WRONG, BFS path (cost 2) IS shorter here Another example where BFS fails: A --(1)-- B --(100)-- C ¦ ? +--------(3)------------+ BFS from A to C: BFS finds: A ? B ? C (2 hops = "fewer edges") Cost: 1 + 100 = 101 Actual shortest: A ? C direct = 3 ? BFS missed this! BFS counts HOPS. For weighted graphs, we need to count COST.

The solution is Dijkstra's algorithm, published by Edsger W. Dijkstra in 1959. Instead of processing vertices in order of discovery (like BFS), Dijkstra processes them in order of their current best distance from the source — using a min-heap priority queue.

?? The Core Insight: Dijkstra is "BFS with a priority queue instead of a regular queue." At each step, rather than exploring the vertex discovered earliest, we explore the vertex with the smallest known distance from source. This greedy choice is provably optimal — once a vertex's distance is finalized, no shorter path can exist.
Section 2 — The Algorithm: Step-by-Step

We maintain a distance array (or dictionary) where dist[v] = the shortest known distance from source to v. Initially all distances are infinity except dist[source] = 0. We also maintain a min-heap priority queue containing (distance, vertex) pairs.

2.1 — Dijkstra's Algorithm Steps

  1. Initialize: dist[source] = 0, all others = 8. Enqueue (0, source).
  2. Dequeue the vertex u with the minimum distance.
  3. If u is already finalized (we've seen it before with a lower distance), skip it.
  4. Mark u as finalized. For each neighbor v of u with edge weight w:
  5.    If dist[u] + w < dist[v], update dist[v] = dist[u] + w and enqueue (dist[v], v).
  6. Repeat from step 2 until the queue is empty.

?? Stale Entries in the Priority Queue: When we improve a distance (e.g., dist[B] goes from 4 to 3), we push a new entry (3,B) into the heap but do NOT remove the old (4,B). These "stale" entries remain. The fix is simple: when dequeuing, if the vertex is already finalized, skip it. This is the lazy deletion approach — standard in production Dijkstra implementations.
Section 3 — Path Reconstruction

The distance algorithm tells us the cost of the shortest path, but not the path itself. To recover the actual route, we maintain a parent map (same idea as in BFS) — for each vertex, record which vertex provided its best distance.

PARENT MAP DURING DIJKSTRA (same graph as above): After finalization: parent: { B?C, C?A, D?B, E?D } (B's best distance came through C) PATH RECONSTRUCTION — trace backward from destination: To find path A?E: Start at E ? parent[E] = D D ? parent[D] = B B ? parent[B] = C C ? parent[C] = A (source reached!) Path reversed: A ? C ? B ? D ? E
Section 4 — C# Implementation
C# — Full Implementation 
using System;
using System.Collections.Generic;

public class Dijkstra
{
    // Adjacency list: vertex ? list of (neighbor, weight)
    private readonly Dictionary<int, List<(int to, int w)>> adj = new();

    public void AddEdge(int from, int to, int weight, bool undirected = false)
    {
        if (!adj.ContainsKey(from)) adj[from] = new();
        adj[from].Add((to, weight));
        if (undirected)
        {
            if (!adj.ContainsKey(to)) adj[to] = new();
            adj[to].Add((from, weight));
        }
    }

    // --- Single-Source Shortest Distances ---------------------------
    // Returns dist[] where dist[v] = shortest cost from source to v
    public Dictionary<int, int> ShortestDistances(int source)
    {
        // dist[v] = best known distance from source to v
        var dist = new Dictionary<int, int> { [source] = 0 };

        // .NET 6+ PriorityQueue<TElement, TPriority>
        // Pair = (vertex, cost), priority = cost (min-heap by default)
        var pq = new PriorityQueue<int, int>();
        pq.Enqueue(source, 0);

        while (pq.Count > 0)
        {
            pq.TryDequeue(out int u, out int d);

            // Skip stale entries — vertex already finalized with lower dist
            if (dist.TryGetValue(u, out int best) && d > best) continue;

            foreach (var (v, w) in adj.GetValueOrDefault(u, new()))
            {
                int nd = d + w;
                if (!dist.TryGetValue(v, out int dv) || nd < dv)
                {
                    dist[v] = nd;            // relax the edge
                    pq.Enqueue(v, nd);     // push (potentially stale entries remain)
                }
            }
        }
        return dist;
    }

    // --- Shortest Path with Route Reconstruction --------------------
    public (int cost, List<int> path) ShortestPath(int source, int target)
    {
        var dist   = new Dictionary<int, int>  { [source] = 0  };
        var parent = new Dictionary<int, int>  { [source] = -1 };
        var pq     = new PriorityQueue<int, int>();
        pq.Enqueue(source, 0);

        while (pq.Count > 0)
        {
            pq.TryDequeue(out int u, out int d);
            if (dist.TryGetValue(u, out int b) && d > b) continue;

            if (u == target) break;  // early exit — target finalized

            foreach (var (v, w) in adj.GetValueOrDefault(u, new()))
            {
                int nd = d + w;
                if (!dist.TryGetValue(v, out int dv) || nd < dv)
                {
                    dist[v] = nd;
                    parent[v] = u;  // record the predecessor
                    pq.Enqueue(v, nd);
                }
            }
        }

        if (!dist.ContainsKey(target))
            return (-1, new List<int>());  // unreachable

        // Reconstruct path by tracing parent pointers backward
        var path = new List<int>();
        for (int cur = target; cur != -1; cur = parent[cur])
            path.Insert(0, cur);
        return (dist[target], path);
    }

    public static void Main()
    {
        var g = new Dijkstra();
        g.AddEdge(0, 1, 4);
        g.AddEdge(0, 2, 2);
        g.AddEdge(1, 2, 1);
        g.AddEdge(1, 3, 5);
        g.AddEdge(2, 1, 1);
        g.AddEdge(2, 3, 8);
        g.AddEdge(2, 4, 10);
        g.AddEdge(3, 4, 2);

        var dists = g.ShortestDistances(0);
        foreach (var (v, d) in dists)
            Console.WriteLine($"0?{v}: {d}");
        // 0?0: 0   0?1: 3   0?2: 2   0?3: 8   0?4: 10

        var (cost, path) = g.ShortestPath(0, 4);
        Console.WriteLine($"Path: {string.Join("?", path)}  Cost: {cost}");
        // Path: 0?2?1?3?4  Cost: 10
    }
}

Section 5 — Python Implementation
Python — Full Implementation 
import heapq
from collections import defaultdict
from typing import Dict, List, Tuple, Optional

class Dijkstra:
    def __init__(self):
        # adjacency list: vertex ? [(neighbor, weight), ...]
        self.adj = defaultdict(list)

    def add_edge(self, frm: int, to: int, w: int, undirected: bool = False):
        self.adj[frm].append((to, w))
        if undirected:
            self.adj[to].append((frm, w))

    # --- Single-Source Shortest Distances ---------------------------
    def shortest_distances(self, source: int) -> Dict[int, int]:
        # dist[v] = shortest known cost from source to v
        dist = {source: 0}
        # min-heap: (cost, vertex)  — heapq is a min-heap by default
        pq   = [(0, source)]

        while pq:
            d, u = heapq.heappop(pq)

            # Skip stale entries
            if d > dist.get(u, float('inf')):
                continue

            for v, w in self.adj[u]:
                nd = d + w
                if nd < dist.get(v, float('inf')):
                    dist[v] = nd          # relax edge
                    heapq.heappush(pq, (nd, v))

        return dist

    # --- Shortest Path with Route Reconstruction --------------------
    def shortest_path(self, source: int, target: int) -> Tuple[int, List[int]]:
        dist   = {source: 0}
        parent = {source: None}
        pq     = [(0, source)]

        while pq:
            d, u = heapq.heappop(pq)
            if d > dist.get(u, float('inf')): continue
            if u == target: break             # early exit

            for v, w in self.adj[u]:
                nd = d + w
                if nd < dist.get(v, float('inf')):
                    dist[v]   = nd
                    parent[v] = u       # record predecessor
                    heapq.heappush(pq, (nd, v))

        if target not in dist:
            return -1, []       # unreachable

        # Reconstruct by tracing parent pointers
        path, cur = [], target
        while cur is not None:
            path.append(cur)
            cur = parent[cur]
        return dist[target], path[::-1]


if __name__ == "__main__":
    g = Dijkstra()
    g.add_edge(0, 1, 4); g.add_edge(0, 2, 2)
    g.add_edge(1, 2, 1); g.add_edge(1, 3, 5)
    g.add_edge(2, 1, 1); g.add_edge(2, 3, 8)
    g.add_edge(2, 4, 10); g.add_edge(3, 4, 2)

    dists = g.shortest_distances(0)
    for v, d in sorted(dists.items()):
        print(f"0?{v}: {d}")
    # 0?0: 0   0?1: 3   0?2: 2   0?3: 8   0?4: 10

    cost, path = g.shortest_path(0, 4)
    print(f"Path: {'?'.join(map(str, path))}  Cost: {cost}")
    # Path: 0?2?1?3?4  Cost: 10
Section 6 — Dijkstra on Grids

Many problems present a weighted grid where each cell has a cost to enter. Dijkstra works perfectly here — treat each cell as a vertex and each valid move as a weighted edge.

6.1 — Cheapest Path Through a Grid

GRID — cost to enter each cell: 1 3 1 1 5 1 4 2 1 Find the cheapest path from top-left (0,0) to bottom-right (2,2). Each step costs the value of the cell you're ENTERING. Standard BFS would be wrong (ignores cell costs). Dijkstra gives the true minimum-cost path: (0,0)?(1,0)?(2,0)? costs: 1+4=5... (0,0)?(0,1)?(0,2)?(1,2)?(2,2) costs: 3+1+1+1=6 (0,0)?(1,0)?(1,1)?(1,2)?(2,2) costs: 1+5+1+1=8 (0,0)?(0,1)?(0,2)?(1,2)?(2,2) costs: 3+1+1+1=6 Cheapest: 1?1?1?1 = 4... try (0,0)?(1,0)?(2,0)?(2,1)?(2,2) costs: 1+4+2+1=8 Nope. (0,0)?(0,1)?(0,2)?(1,2)?(2,2) = 3+1+1+1 = 6. Answer = 6.
C# — Grid Dijkstra 
public static int MinCostGrid(int[][] grid)
{
    int rows = grid.Length, cols = grid[0].Length;
    var dist = new int[rows, cols];
    for (int i = 0; i < rows; i++)
        for (int j = 0; j < cols; j++)
            dist[i, j] = int.MaxValue;
    dist[0, 0] = grid[0][0];  // starting cost

    var pq = new PriorityQueue<(int r, int c), int>();
    pq.Enqueue((0, 0), dist[0, 0]);
    (int, int)[] dirs = { (-1,0), (1,0), (0,-1), (0,1) };

    while (pq.Count > 0)
    {
        pq.TryDequeue(out var pos, out int d);
        var (r, c) = pos;
        if (d > dist[r, c]) continue;  // stale

        foreach (var (dr, dc) in dirs)
        {
            int nr = r+dr, nc = c+dc;
            if (nr < 0 || nr >= rows || nc < 0 || nc >= cols) continue;
            int nd = d + grid[nr][nc];  // cost of entering (nr,nc)
            if (nd < dist[nr, nc])
            {
                dist[nr, nc] = nd;
                pq.Enqueue((nr, nc), nd);
            }
        }
    }
    return dist[rows-1, cols-1];
}
Python 
import heapq

def min_cost_grid(grid: list) -> int:
    rows, cols = len(grid), len(grid[0])
    dist = [[float('inf')] * cols for _ in range(rows)]
    dist[0][0] = grid[0][0]
    pq = [(dist[0][0], 0, 0)]
    dirs = [(-1,0),(1,0),(0,-1),(0,1)]
    while pq:
        d, r, c = heapq.heappop(pq)
        if d > dist[r][c]: continue   # stale
        for dr, dc in dirs:
            nr, nc = r + dr, c + dc
            if 0 <= nr < rows and 0 <= nc < cols:
                nd = d + grid[nr][nc]   # cost of entering (nr, nc)
                if nd < dist[nr][nc]:
                    dist[nr][nc] = nd
                    heapq.heappush(pq, (nd, nr, nc))
    return dist[rows-1][cols-1]

7.1 — Why O((V + E) log V)?

  • Each vertex is dequeued at most once: O(V) dequeue operations × O(log V) heap time = O(V log V)
  • Each edge causes at most one enqueue: O(E) enqueue operations × O(log V) = O(E log V)
  • Total: O((V + E) log V) = O(E log V) for connected graphs (E = V-1)
?? Dense vs Sparse: For sparse graphs (E ˜ V), binary heap Dijkstra is O(V log V). For dense graphs (E ˜ V²), the O(V²) simple array approach can actually be faster since O(V²) < O(V² log V). In practice, most real-world graphs (road networks, social graphs) are sparse — use the heap.
Section 8 — Limitations & When NOT to Use Dijkstra
?? Critical Limitation — No Negative Edge Weights: Dijkstra's greedy assumption is "once a vertex is finalized, its distance is optimal." This breaks with negative edges — a later-discovered path with a negative edge could give a shorter total distance than an already-finalized one. Use Bellman-Ford for graphs with negative edges (covered in Module 10B).
WHY NEGATIVE EDGES BREAK DIJKSTRA: A --(5)--? B ¦ ? (2) (-8) ? ¦ C --(1)--? D Dijkstra from A: 1. Dequeue A(0): dist[B]=5, dist[C]=2 2. Dequeue C(2): dist[D]=3 (2+1) 3. Dequeue D(3): tries B ? dist[B]=3+(-8)=-5 ? SHOULD update B! 4. But step 3 happens AFTER step 2 which finalized B=5 ? Dijkstra already output dist[B]=5 as FINAL ? WRONG! Correct answer: A?C?D?B = 2+1+(-8) = -5 Dijkstra says: dist[B] = 5 ? incorrect!

Section 9 — Real-World Applications
?? GPS & Mapping — Google Maps / Apple Maps / Waze
Every time you ask for driving directions, Dijkstra (or A*) runs on a road network graph where vertices = intersections, edges = roads, and weights = travel time or distance. Google Maps processes billions of route requests daily using highly optimized variants of this algorithm. The graph has hundreds of millions of vertices for the entire global road network.
?? Network Routing — OSPF (Open Shortest Path First)
The OSPF routing protocol, used in enterprise and ISP networks, explicitly uses Dijkstra's algorithm. Each router maintains a map of the entire network and runs Dijkstra to compute the shortest path to every other router (measured in link cost). When a link fails or a new router is added, Dijkstra runs again to update routing tables.
?? Game Development — Pathfinding in RTS Games
In real-time strategy games (StarCraft, Age of Empires), units must navigate through terrain with different movement costs (roads = fast, swamps = slow, mountains = blocked). The game world is a weighted grid, and Dijkstra (often enhanced with A* heuristic) computes optimal movement paths. Games like StarCraft II run pathfinding for hundreds of units simultaneously using highly optimized versions.
?? Flight Search Engines — Kayak / Google Flights
Finding the cheapest flight itinerary is a shortest path problem where vertices = airports, edges = flights, and weights = ticket price (or duration). Flight search engines use Dijkstra variants to find cheapest/fastest routes, often considering layovers and multi-hop journeys across hundreds of airports.
?? Social Networks — LinkedIn "Degrees of Separation"
LinkedIn's "You are 2nd degree connection" feature computes shortest path in a social graph. While the graph is unweighted (use BFS), LinkedIn's "InMail reach" features use weighted variants where edge weights represent relationship strength or response likelihood.
Section 10 — Common Mistakes
?? Mistake 1 — Forgetting to skip stale entries: When a vertex's distance is improved, the old entry remains in the heap. If you process it again later without checking, you'll re-process already-finalized edges, producing wrong results or infinite loops. Always check: if d > dist[u]: continue.
?? Mistake 2 — Using a list as a queue (wrong data structure): Dijkstra requires a min-heap priority queue. Using a regular queue (like BFS) ignores edge weights. Using list.pop(0) in Python is O(n) instead of O(log n) — use heapq.
?? Mistake 3 — Applying Dijkstra to negative-weight graphs: This is the most common conceptual error. Always verify your graph has no negative edges before using Dijkstra. If in doubt, use Bellman-Ford — it's slower but always correct.
?? Mistake 4 — Initializing dist[source] = 8 instead of 0: The source starts with distance 0 from itself. All other vertices start at infinity (or Integer.MaxValue). Setting source to 8 means it will never be processed.
?? Mistake 5 — Integer overflow when adding to MaxValue: In C#, int.MaxValue + weight overflows to a negative number! Guard with: if (d == int.MaxValue) continue; or use long instead of int for distances.
Section 11 — Check Your Understanding
Q1 — Why does Dijkstra use a min-heap priority queue instead of a regular FIFO queue?
A) To support negative edge weights
B) To always process the vertex with the current smallest known distance next
C) To reduce memory usage compared to BFS
D) To avoid revisiting vertices
Answer: B. The min-heap ensures we always process the vertex with the smallest currently known distance first. This greedy ordering is the key insight — by always choosing the closest unfinalized vertex, we guarantee that when a vertex is dequeued and processed, its distance is already optimal. A regular FIFO queue (BFS) processes vertices in discovery order, which ignores edge weights entirely.
Key property: Once a vertex is popped from the min-heap in Dijkstra, its distance is finalized — no shorter path will ever be found. This is not true with a regular queue.
Q2 — You run Dijkstra on a graph and find that the heap has many "stale" (vertex, distance) entries. What causes them and how do we handle them?
Cause: When a shorter path to vertex v is found, we push a new (new_dist, v) entry into the heap but do NOT remove the old (old_dist, v) entry (standard heaps don't support decrease-key efficiently). This leaves stale entries with outdated, larger distances. Fix: When dequeuing entry (d, u), check if d > dist[u] (i.e., we've already found a better path). If so, skip this entry — it's stale. This "lazy deletion" approach is the standard in all practical implementations.
Impact on complexity: Stale entries can increase heap size to O(E) entries instead of O(V), making total time O(E log E) ˜ O(E log V) — same asymptotic complexity as a "clean" implementation.
Q3 — A graph has 5 vertices and the following edges (all undirected): A–B(2), A–C(6), B–C(3), B–D(4), C–D(1), D–E(3). What is the shortest distance from A to E?
Answer: 9. Run Dijkstra from A:
• dist[A]=0 ? relax: dist[B]=2, dist[C]=6
• Dequeue B(2) ? relax: dist[C]=min(6, 2+3)=5, dist[D]=min(8, 2+4)=6
• Dequeue C(5) ? relax: dist[D]=min(6, 5+1)=6 (no improvement)
• Dequeue D(6) ? relax: dist[E]=min(8, 6+3)=9
• Dequeue E(9) — done!
Shortest path: A?B?C?D?E, cost = 2+3+1+3 = 9
Path: A?B?C?D?E (cost 9). Note: A?B?D?E = 2+4+3 = 9 (tied!). Both are valid shortest paths.
Q4 — You have a weighted graph where some edge weights are 0. Can you still use Dijkstra? What about edge weights of -1?
Zero weights: Yes, Dijkstra works. Zero-weight edges just mean traversing them doesn't increase cost. The algorithm handles them correctly — a vertex reached via a zero-weight edge can have the same distance as its predecessor. Negative weights (e.g., -1): No, Dijkstra is incorrect. Negative weights violate the greedy assumption. Once a vertex is "finalized," a later path using a negative edge could produce a shorter distance, but Dijkstra won't revisit finalized vertices. Use Bellman-Ford for graphs with negative edge weights.
Special case: If your graph has only 0 and 1 weights, use a deque (0-1 BFS) which is O(V+E) — faster than Dijkstra's O((V+E) log V).
Q5 — You need to find the shortest path between ALL pairs of vertices in a graph of 500 vertices. Would you run Dijkstra 500 times, or use a different approach?
It depends on graph density. Running Dijkstra from each vertex (V times) gives O(V × (V+E) log V) total. For a sparse graph (E ˜ V), this is O(V² log V) — very reasonable. For a dense graph (E ˜ V²), this becomes O(V³ log V) — expensive. Floyd-Warshall is O(V³) and simpler to implement for all-pairs, but it also handles negative edge weights (just not negative cycles). For V=500 with a sparse graph, running Dijkstra 500 times is perfectly acceptable. For V=500 dense graph or if negative edges exist, use Floyd-Warshall (covered in Module 10B).
Rule of thumb: V × Dijkstra for sparse graphs; Floyd-Warshall for dense graphs or graphs with negative edges.
