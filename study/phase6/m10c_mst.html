Section 1 — What is a Minimum Spanning Tree?

A Spanning Tree of a connected undirected graph is a subgraph that includes all V vertices and exactly V-1 edges — just enough edges to keep everything connected without any cycles. A Minimum Spanning Tree (MST) is the spanning tree with the minimum total edge weight.

SPANNING TREE vs MINIMUM SPANNING TREE Graph (undirected, weighted): 2 A ------- B ¦ \ / ¦ 5¦ 4X4 ¦3 ¦ / \ ¦ C ------- D 6 Three possible spanning trees (each has 4 vertices, 3 edges): Tree 1: A-B(2), B-D(3), A-C(5) ? total weight = 10 Tree 2: A-B(2), B-D(3), C-D(6) ? total weight = 11 Tree 3: A-B(2), A-D(4), A-C(5) ? total weight = 11 Tree 4: A-B(2), B-D(3), A-D(4) ? NOT valid (cycle A-B-D-A) Minimum Spanning Tree = Tree 1: edges A-B(2), B-D(3), A-C(5), total = 10 Key properties: • Always has exactly V-1 edges • Connects all V vertices • No cycles • MST is not unique if there are equal-weight edges
?? MST vs Shortest Path: MST minimizes the total weight of the tree, not the distance between specific vertices. The path from A to D in the MST might not be the shortest path in the original graph. These are fundamentally different problems — MST connects everything cheaply, shortest path finds the fastest route between two points.
?? Real-World: Network Infrastructure
MST is the foundation of many real-world problems: laying fiber-optic cables to connect cities with minimum cable length; designing electrical grids; setting up computer network infrastructure where the goal is to connect all machines with minimum wiring cost. MST is also used in cluster analysis, image segmentation, and approximation algorithms for NP-hard problems like TSP (Travelling Salesman).
Section 2 — Prim's Algorithm

Prim's algorithm (Robert Prim, 1957) builds the MST by growing a single tree from a starting vertex. It always adds the minimum-weight edge that connects the current tree to a vertex not yet in the tree. This is a greedy algorithm — always pick the cheapest available connection.

?? Prim's = Dijkstra's cousin: Prim's structure is nearly identical to Dijkstra's. Both use a min-heap priority queue. The difference: Dijkstra tracks dist[v] = shortest path cost from source; Prim tracks key[v] = weight of the cheapest edge connecting v to the current tree. Dijkstra accumulates costs; Prim only cares about the edge cost to add the next vertex.

Section 3 — Prim's C# & Python
C# — Prim's Algorithm 
using System;
using System.Collections.Generic;

public class Prims
{
    private readonly int n;
    private readonly List<(int to, int w)>[] adj;

    public Prims(int vertices)
    {
        n   = vertices;
        adj = new List<(int, int)>[n];
        for (int i = 0; i < n; i++) adj[i] = new();
    }

    public void AddEdge(int u, int v, int w)
    {
        adj[u].Add((v, w));
        adj[v].Add((u, w));  // undirected
    }

    // Returns (total MST weight, list of MST edges as (u, v, weight))
    public (int totalWeight, List<(int u, int v, int w)> edges) ComputeMST(int start = 0)
    {
        int[] key    = new int[n];    // key[v] = min edge weight to reach v
        int[] parent = new int[n];    // parent[v] = which vertex added v to MST
        bool[] inMST  = new bool[n];

        Array.Fill(key, int.MaxValue / 2);
        Array.Fill(parent, -1);
        key[start] = 0;

        // Priority queue: (key value, vertex)
        var pq = new PriorityQueue<int, int>();
        pq.Enqueue(start, 0);

        int totalWeight = 0;
        var mstEdges    = new List<(int, int, int)>();

        while (pq.Count > 0)
        {
            pq.TryDequeue(out int u, out int k);
            if (inMST[u]) continue;  // stale entry
            inMST[u] = true;

            if (parent[u] != -1)
            {
                mstEdges.Add((parent[u], u, key[u]));
                totalWeight += key[u];
            }

            foreach (var (v, w) in adj[u])
            {
                if (!inMST[v] && w < key[v])
                {
                    key[v]    = w;     // update: cheaper edge found
                    parent[v] = u;
                    pq.Enqueue(v, w);
                }
            }
        }
        return (totalWeight, mstEdges);
    }

    public static void Main()
    {
        var g = new Prims(5);
        g.AddEdge(0,1,2); g.AddEdge(0,3,6);
        g.AddEdge(1,2,3); g.AddEdge(1,3,8); g.AddEdge(1,4,5);
        g.AddEdge(2,4,7); g.AddEdge(3,4,9);

        var (weight, edges) = g.ComputeMST();
        Console.WriteLine($"MST Total Weight: {weight}");  // 16
        foreach (var (u, v, w) in edges)
            Console.WriteLine($"  {u}-{v} ({w})");
        // 0-1 (2), 1-2 (3), 1-4 (5), 0-3 (6)
    }
}
Python — Prim's Algorithm 
import heapq
from collections import defaultdict

class Prims:
    def __init__(self):
        self.adj = defaultdict(list)

    def add_edge(self, u, v, w):
        self.adj[u].append((w, v))  # (weight, vertex) for heapq ordering
        self.adj[v].append((w, u))

    def compute_mst(self, start=0):
        in_mst = set()
        key    = {}   # key[v] = min edge weight to v
        parent = {}   # parent[v] = which vertex added v
        pq     = [(0, start, -1)]  # (weight, vertex, parent)
        total_weight, mst_edges = 0, []

        while pq:
            w, u, par = heapq.heappop(pq)
            if u in in_mst: continue
            in_mst.add(u)
            if par != -1:
                mst_edges.append((par, u, w))
                total_weight += w

            for ew, v in self.adj[u]:
                if v not in in_mst:
                    heapq.heappush(pq, (ew, v, u))

        return total_weight, mst_edges


if __name__ == "__main__":
    g = Prims()
    g.add_edge(0,1,2); g.add_edge(0,3,6)
    g.add_edge(1,2,3); g.add_edge(1,3,8); g.add_edge(1,4,5)
    g.add_edge(2,4,7); g.add_edge(3,4,9)

    weight, edges = g.compute_mst()
    print(f"Total: {weight}")   # 16
    for u, v, w in edges:
        print(f"  {u}-{v} ({w})")
    # 0-1 (2), 1-2 (3), 1-4 (5), 0-3 (6)
Section 4 — Kruskal's Algorithm & Union-Find

Kruskal's algorithm (Joseph Kruskal, 1956) takes a completely different approach: sort all edges by weight and add them one by one, skipping any edge that would create a cycle. The key challenge is detecting cycles efficiently — this is solved by the Union-Find (Disjoint Set Union) data structure.

?? Kruskal's = Sort + Union-Find: Sort all edges ascending. For each edge (u, v, w), use Union-Find to check if u and v are already connected (same component). If they are, adding this edge creates a cycle — skip it. If they're in different components, add the edge to the MST and union the two components. After V-1 edges are added, the MST is complete.

4.1 — Union-Find Data Structure

Union-Find maintains a collection of disjoint sets. Two operations are needed:

  • Find(x): Returns the representative (root) of x's set. If two vertices have the same root, they're in the same component.
  • Union(x, y): Merges the sets containing x and y.

Two optimizations make Union-Find nearly O(1) amortized: Path Compression (flatten the tree on Find) and Union by Rank (always attach the smaller tree under the larger).

UNION-FIND — Path Compression + Union by Rank Initially: each vertex is its own set parent: [0, 1, 2, 3, 4] (each is own root) rank: [0, 0, 0, 0, 0] Union(0, 1): Find(0)=0, Find(1)=1, different ? merge rank[0]==rank[1], attach 1 under 0, rank[0]++ parent: [0, 0, 2, 3, 4] rank: [1, 0, 0, 0, 0] Union(0, 2): Find(0)=0, Find(2)=2 ? merge rank[0]=1 > rank[2]=0 ? attach 2 under 0 parent: [0, 0, 0, 3, 4] Union(3, 4): similarly parent: [0, 0, 0, 3, 3] Union(0, 3): Find(0)=0, Find(3)=3 ? merge parent: [0, 0, 0, 0, 3] (3?0) Find(4): 4?3?0 (2 hops) Path compression: set parent[4]=0 directly Now: parent[4]=0 (1 hop next time)

Section 5 — Kruskal's C# & Python
C# — Kruskal's with Union-Find 
using System;
using System.Collections.Generic;

// --- Union-Find (Disjoint Set Union) -----------------------------
public class UnionFind
{
    private int[] parent, rank;

    public UnionFind(int n)
    {
        parent = new int[n];
        rank   = new int[n];
        for (int i = 0; i < n; i++) parent[i] = i;
    }

    // Find with path compression
    public int Find(int x)
    {
        if (parent[x] != x)
            parent[x] = Find(parent[x]);  // path compression
        return parent[x];
    }

    // Union by rank — returns false if already in same set (cycle!)
    public bool Union(int x, int y)
    {
        int rx = Find(x), ry = Find(y);
        if (rx == ry) return false;  // same component — adding would make cycle

        // Union by rank: attach smaller tree under larger
        if      (rank[rx] < rank[ry]) parent[rx] = ry;
        else if (rank[rx] > rank[ry]) parent[ry] = rx;
        else                         { parent[ry] = rx; rank[rx]++; }
        return true;
    }

    public bool Connected(int x, int y) => Find(x) == Find(y);
}

// --- Kruskal's MST ------------------------------------------------
public class Kruskals
{
    private readonly List<(int w, int u, int v)> edges = new();
    private readonly int n;

    public Kruskals(int vertices) => n = vertices;

    public void AddEdge(int u, int v, int w) => edges.Add((w, u, v));

    public (int totalWeight, List<(int u, int v, int w)> mstEdges) ComputeMST()
    {
        // Step 1: Sort all edges by weight (ascending)
        edges.Sort();  // sorts by first tuple element (weight)

        var uf        = new UnionFind(n);
        int total     = 0;
        var mstEdges  = new List<(int, int, int)>();

        // Step 2: Process edges in order, add if no cycle
        foreach (var (w, u, v) in edges)
        {
            if (uf.Union(u, v))  // returns true = different components = add
            {
                mstEdges.Add((u, v, w));
                total += w;
                if (mstEdges.Count == n - 1) break;  // MST complete — V-1 edges
            }
        }
        return (total, mstEdges);
    }

    public static void Main()
    {
        var g = new Kruskals(5);
        g.AddEdge(0,1,2); g.AddEdge(0,3,6);
        g.AddEdge(1,2,3); g.AddEdge(1,3,8); g.AddEdge(1,4,5);
        g.AddEdge(2,4,7); g.AddEdge(3,4,9);

        var (weight, edges) = g.ComputeMST();
        Console.WriteLine($"MST Weight: {weight}");  // 16
        foreach (var (u, v, w) in edges)
            Console.WriteLine($"  {u}-{v} ({w})");
        // 0-1 (2), 1-2 (3), 1-4 (5), 0-3 (6)
    }
}
Python — Kruskal's with Union-Find 
class UnionFind:
    def __init__(self, n):
        self.parent = list(range(n))
        self.rank   = [0] * n

    def find(self, x):
        if self.parent[x] != x:
            self.parent[x] = self.find(self.parent[x])  # path compression
        return self.parent[x]

    def union(self, x, y) -> bool:
        rx, ry = self.find(x), self.find(y)
        if rx == ry: return False         # cycle detected
        if   self.rank[rx] < self.rank[ry]: self.parent[rx] = ry
        elif self.rank[rx] > self.rank[ry]: self.parent[ry] = rx
        else: self.parent[ry] = rx; self.rank[rx] += 1
        return True


def kruskals(n, edge_list):
    """edge_list: list of (u, v, w)"""
    edges    = sorted(edge_list, key=lambda e: e[2])  # sort by weight
    uf       = UnionFind(n)
    mst, total = [], 0
    for u, v, w in edges:
        if uf.union(u, v):
            mst.append((u, v, w))
            total += w
            if len(mst) == n - 1: break  # V-1 edges = MST complete
    return total, mst


if __name__ == "__main__":
    edges = [(0,1,2),(0,3,6),(1,2,3),(1,3,8),
             (1,4,5),(2,4,7),(3,4,9)]
    total, mst = kruskals(5, edges)
    print(f"Total: {total}")   # 16
    for u, v, w in mst:
        print(f"  {u}-{v} ({w})")

Section 7 — Real-World Applications
?? Network Design — Laying Cables / Pipes
Telecommunications companies use MST to plan fiber optic cable routes connecting cities, minimizing total cable length. The same problem applies to water/gas pipeline networks, electrical grid design, and highway planning. Real-world variants add constraints (avoid certain areas, capacity limits) making them NP-hard, but MST serves as the starting approximation.
?? Cluster Analysis — Machine Learning
Single-linkage hierarchical clustering uses MST: compute the MST of a point cloud (where edge weight = distance between points), then cut the heaviest k-1 edges to get k clusters. This is used in bioinformatics (gene clustering), image segmentation, and anomaly detection. The "cut the max edge" step in MST-based clustering directly gives natural cluster boundaries.
?? Approximation Algorithms — TSP
The Travelling Salesman Problem (TSP) — find the shortest tour visiting all cities — is NP-hard. A 2-approximation algorithm uses MST: compute MST, do a DFS to get an Eulerian circuit, then shortcut repeated vertices. This gives a tour of cost = 2 × optimal. This is the foundation of many practical TSP solvers used in logistics (UPS route optimization, delivery services).
?? Union-Find Beyond MST — Dynamic Connectivity
Union-Find (used in Kruskal's) is a standalone data structure used in many other contexts: detecting connected components in social networks as friendships form; Kruskal's for network reliability analysis; checking if a graph becomes connected after adding edges (used in network monitoring at companies like Cloudflare); percolation problems in physics simulations.
Section 8 — Common Mistakes
?? Using MST for a directed graph: MST is only defined for undirected graphs. For directed graphs, the analogous problem is the Minimum Spanning Arborescence, solved by Edmonds' algorithm (more complex). Always verify the graph is undirected before applying Prim's or Kruskal's.
?? Union-Find: Forgetting path compression or union by rank: Without both optimizations, Union-Find degrades to O(log n) per operation in average, and O(n) worst case. With both, it achieves nearly O(1) amortized (inverse Ackermann function — effectively O(1) for any practical input size). Always implement both.
?? Prim's: Applying to disconnected graphs: Prim's starting from one vertex only builds the MST for the connected component containing that vertex. If the graph has multiple components, no spanning tree exists (you need a minimum spanning forest). Always verify connectivity first, or loop to start Prim's on each unvisited vertex.
?? Confusing MST with shortest path: MST minimizes total edge weight of the spanning tree. Shortest path minimizes the distance between two specific vertices. The path from A to B in the MST is NOT necessarily the shortest path in the original graph. These are completely different problems with different algorithms.
Section 9 — Check Your Understanding
Q1 — A connected graph with V vertices has how many edges in its MST?
Answer: V-1 edges. A spanning tree of V vertices is a connected acyclic graph — it has exactly V-1 edges. Fewer edges and the graph becomes disconnected; more edges and you have a cycle. Both Kruskal's and Prim's stop after adding exactly V-1 edges. If you're solving an MST problem and your answer has a different number of edges, something is wrong.
Key invariant: tree with V vertices = connected + acyclic + exactly V-1 edges. Any two of these three properties imply the third.
Q2 — You run Union-Find on 5 elements. After Union(0,1), Union(1,2), and Union(3,4), what does Find(2) return? After Union(2,3), what does Find(4) return?
After Union(0,1), Union(1,2), Union(3,4): Find(2) returns 0 (0 is the root of the {0,1,2} component). After Union(2,3): This merges {0,1,2} (root=0, rank=1) with {3,4} (root=3, rank=1). Equal rank ? attach 3 under 0, rank[0]++. Now Find(4): 4?3?0, so Find(4) returns 0. Path compression then sets parent[4]=0 directly.
Components after all unions: {0,1,2,3,4} — all in one component with root 0.
Q3 — Kruskal's says: sort edges, add if no cycle. Why is it guaranteed to produce an MST and not just any spanning tree?
The Cut Property (MST Theorem): For any cut (partition) of the graph's vertices into two sets S and V-S, the minimum-weight edge crossing the cut is always part of some MST. Kruskal's works because each time it considers an edge (u,v,w) that connects two different components, that edge is the minimum-weight crossing edge for the cut separating those components (all lighter edges were processed first and are already in the MST, or they also connected the same components and were skipped). This "Cut Property" proof guarantees every edge Kruskal's adds is a valid MST edge.
Also: Cycle Property: The maximum-weight edge in any cycle is never in any MST. Kruskal's skips edges that form cycles — exactly these "cycle maximum" edges.
Q4 — If a graph has all unique edge weights, is the MST unique?
Yes — if all edge weights are distinct, the MST is unique. Proof by contradiction: suppose two different MSTs T1 and T2 exist. T1 has some edge e1 not in T2. Adding e1 to T2 creates exactly one cycle C. C must contain some edge e2 in T2 but not T1 (since T1 has no cycle). If w(e1) ? w(e2), we can swap to get a lighter spanning tree — contradicting one of them being the MST. With all distinct weights, this contradiction always holds, so only one MST exists.
When multiple edges have the same weight, there can be multiple valid MSTs with the same total weight but different edge sets.
Interview tip: "Is the MST unique?" is a common question. Answer: Yes if all edge weights are distinct; No (but all have the same total weight) if some edges have equal weights.
Q5 — How would you modify Kruskal's to find a MAXIMUM Spanning Tree (connect all vertices with maximum total weight)?
One change only: sort edges in descending order instead of ascending. The rest of the algorithm — same Union-Find cycle detection, same V-1 edge stopping condition — stays identical. By processing the heaviest edges first, you greedily include the largest-weight edges that don't form a cycle. Alternatively, negate all edge weights and run standard minimum spanning tree (MST of negated weights = maximum spanning tree of original). Maximum spanning trees are used in maximum bandwidth path problems, maximum bottleneck path, and clustering to find the most-connected spanning structure.
Same complexity: O(E log E) — the sort direction is the only difference.
