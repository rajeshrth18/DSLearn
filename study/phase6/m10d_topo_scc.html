Part 1 — Topological Sort
Section 1 — What is Topological Sort?

Given a Directed Acyclic Graph (DAG), a topological ordering is a linear ordering of vertices such that for every directed edge u?v, vertex u comes before v in the ordering. Intuitively, it answers: "Given dependencies, in what order should tasks be completed?"

TOPOLOGICAL SORT — Course Prerequisite Example Graph (directed edges = "must take before"): Math101 --? Math201 --? Math301 ¦ ? +--? CS101 --? CS201 ---+ ¦ +--? CS301 Valid topological orderings (multiple valid orderings exist): [Math101, Math201, CS101, CS201, Math301, CS301] [Math101, CS101, Math201, CS201, CS301, Math301] [CS101, Math101, CS201, Math201, CS301, Math301] ? Wait: CS201 requires CS101 AND Math101 So: Math101 and CS101 must both come before CS201. And CS201 must come before CS301. Correct ordering example: Math101 ? CS101 ? Math201 ? CS201 ? Math301 ? CS301 KEY REQUIREMENT: Only valid on a DAG (Directed Acyclic Graph). A cycle means "A requires B, B requires C, C requires A" ? impossible! Topological sort DETECTS cycles: if not all vertices can be ordered = cycle exists.
?? Two approaches to Topological Sort:
1. DFS-based (Tarjan): Run DFS; push each vertex to a stack after all its descendants are visited. Reverse the stack ? topological order.
2. BFS-based (Kahn's algorithm): Repeatedly remove vertices with in-degree 0 (no prerequisites). This is more intuitive and also detects cycles cleanly. We'll focus on Kahn's.
Section 2 — Kahn's Algorithm (BFS-based)

Kahn's algorithm (Arthur Kahn, 1962) uses the concept of in-degree — the number of incoming edges. A vertex with in-degree 0 has no prerequisites and can be processed first. After processing it, we "remove" it from the graph by decrementing the in-degree of its neighbors. Neighbors whose in-degree becomes 0 are enqueued next.

KAHN'S ALGORITHM WALKTHROUGH Graph: 5?2, 5?0, 4?0, 4?1, 2?3, 3?1 In-degrees: 0: 2 (from 5, from 4) 1: 2 (from 4, from 3) 2: 1 (from 5) 3: 1 (from 2) 4: 0 ? start here 5: 0 ? and here Queue: [4, 5] (in-degree 0 vertices) Result: [] STEP 1 — Process 4: Result: [4] 4?0: indegree[0] = 2-1 = 1 4?1: indegree[1] = 2-1 = 1 Queue: [5] STEP 2 — Process 5: Result: [4, 5] 5?2: indegree[2] = 1-1 = 0 ? enqueue 2 5?0: indegree[0] = 1-1 = 0 ? enqueue 0 Queue: [2, 0] STEP 3 — Process 2: Result: [4, 5, 2] 2?3: indegree[3] = 1-1 = 0 ? enqueue 3 Queue: [0, 3] STEP 4 — Process 0: Result: [4, 5, 2, 0] (0 has no outgoing edges) Queue: [3] STEP 5 — Process 3: Result: [4, 5, 2, 0, 3] 3?1: indegree[1] = 1-1 = 0 ? enqueue 1 Queue: [1] STEP 6 — Process 1: Result: [4, 5, 2, 0, 3, 1] All 6 vertices processed = no cycle. If fewer than V vertices processed = cycle detected!
Section 3 — Topological Sort C# & Python
C# — Kahn's Algorithm (BFS-based Topological Sort) 
using System.Collections.Generic;

public class TopologicalSort
{
    private readonly int n;
    private readonly List<int>[] adj;

    public TopologicalSort(int vertices)
    {
        n   = vertices;
        adj = new List<int>[n];
        for (int i = 0; i < n; i++) adj[i] = new();
    }

    public void AddEdge(int u, int v) => adj[u].Add(v);  // u must come before v

    // Returns (topological order, hasCycle)
    // hasCycle = true if graph is not a DAG
    public (List<int> order, bool hasCycle) KahnsSort()
    {
        // Step 1: Compute in-degrees
        int[] inDeg = new int[n];
        for (int u = 0; u < n; u++)
            foreach (int v in adj[u])
                inDeg[v]++;

        // Step 2: Enqueue all zero in-degree vertices
        var queue = new Queue<int>();
        for (int i = 0; i < n; i++)
            if (inDeg[i] == 0) queue.Enqueue(i);

        var order = new List<int>();

        // Step 3: BFS — process zero in-degree, decrement neighbors
        while (queue.Count > 0)
        {
            int u = queue.Dequeue();
            order.Add(u);
            foreach (int v in adj[u])
            {
                inDeg[v]--;
                if (inDeg[v] == 0) queue.Enqueue(v);
            }
        }

        // If fewer vertices processed than n ? cycle exists
        return (order, order.Count != n);
    }

    // DFS-based topological sort (alternative approach)
    public List<int> DfsSort()
    {
        bool[] visited = new bool[n];
        var stack = new Stack<int>();

        void Dfs(int u) {
            visited[u] = true;
            foreach (int v in adj[u])
                if (!visited[v]) Dfs(v);
            stack.Push(u);  // push AFTER all descendants
        }

        for (int i = 0; i < n; i++)
            if (!visited[i]) Dfs(i);

        var result = new List<int>();
        while (stack.Count > 0) result.Add(stack.Pop());
        return result;
    }

    public static void Main()
    {
        var g = new TopologicalSort(6);
        g.AddEdge(5,2); g.AddEdge(5,0);
        g.AddEdge(4,0); g.AddEdge(4,1);
        g.AddEdge(2,3); g.AddEdge(3,1);

        var (order, cycle) = g.KahnsSort();
        System.Console.WriteLine($"Has cycle: {cycle}");         // False
        System.Console.WriteLine(string.Join(" ? ", order));       // 4 5 2 0 3 1
    }
}
Python — Kahn's Algorithm + DFS Topological Sort 
from collections import defaultdict, deque

class TopologicalSort:
    def __init__(self, n):
        self.n   = n
        self.adj = defaultdict(list)

    def add_edge(self, u, v):
        self.adj[u].append(v)

    def kahns_sort(self):
        # Compute in-degrees
        in_deg = [0] * self.n
        for u in range(self.n):
            for v in self.adj[u]: in_deg[v] += 1

        queue  = deque(i for i in range(self.n) if in_deg[i] == 0)
        order  = []
        while queue:
            u = queue.popleft()
            order.append(u)
            for v in self.adj[u]:
                in_deg[v] -= 1
                if in_deg[v] == 0: queue.append(v)

        has_cycle = len(order) != self.n
        return order, has_cycle

    def dfs_sort(self):
        visited, stack = set(), []
        def dfs(u):
            visited.add(u)
            for v in self.adj[u]:
                if v not in visited: dfs(v)
            stack.append(u)          # push AFTER all descendants
        for i in range(self.n):
            if i not in visited: dfs(i)
        return stack[::-1]         # reverse = topological order


if __name__ == "__main__":
    g = TopologicalSort(6)
    g.add_edge(5,2); g.add_edge(5,0)
    g.add_edge(4,0); g.add_edge(4,1)
    g.add_edge(2,3); g.add_edge(3,1)
    order, cycle = g.kahns_sort()
    print(f"Has cycle: {cycle}")    # False
    print(" ? ".join(map(str, order)))  # 4 ? 5 ? 2 ? 0 ? 3 ? 1
Part 2 — Strongly Connected Components (SCC)
Section 4 — What are SCCs?

In a directed graph, a Strongly Connected Component (SCC) is a maximal set of vertices where every vertex is reachable from every other vertex in the set. In other words, for any u and v in the same SCC, there exists a path u?…?v AND a path v?…?u.

STRONGLY CONNECTED COMPONENTS — Example Directed graph: 1 --? 2 --? 3 --? 4 ? ¦ ? ¦ ¦ ? ¦ ? 0 ?-- 5 6 --? 7 ¦ ? +--? 6 -----+ Wait, let's use cleaner example: Edges: 0?1, 1?2, 2?0, ? forms a cycle: SCC = {0,1,2} 1?3, 3?4, 4?5, 5?3 ? forms a cycle: SCC = {3,4,5} 2?6 ? just a node: SCC = {6} SCCs: {0,1,2} {3,4,5} {6} Within {0,1,2}: 0?1?2?0 (can reach anyone from anyone) ? Within {3,4,5}: 3?4?5?3 (can reach anyone from anyone) ? {6}: can't go anywhere else, nothing comes back ? Condensation graph (replace each SCC with a node): {0,1,2} --? {3,4,5} (via 1?3) {0,1,2} --? {6} (via 2?6) This condensation is ALWAYS a DAG!
?? Why SCCs matter: SCCs reveal the "strongly connected clusters" of a directed graph — groups that can mutually communicate. The condensation (DAG of SCCs) shows the high-level structure. SCCs are used to: detect circular dependencies in build systems, analyze web link structure (which pages form tight communities), find clusters in social networks (who influences whom), and solve 2-SAT boolean satisfiability problems.
Section 5 — Kosaraju's Algorithm (Two-Pass DFS)

Kosaraju's algorithm finds all SCCs in O(V+E) using two DFS passes:

  1. Pass 1: Run DFS on the original graph. Push each vertex to a stack in the order DFS finishes (post-order). Vertices that finish later are "higher" in the condensation.
  2. Pass 2: Create the transposed graph (reverse all edges). Process vertices from the stack (highest finish time first). Each DFS in the transposed graph starting from an unvisited vertex discovers exactly one SCC.
KOSARAJU'S ALGORITHM WALKTHROUGH Original graph edges: 0?1, 1?2, 2?0, 1?3, 3?4, 4?5, 5?3, 2?6 === PASS 1: DFS on original graph, record finish order === Start DFS from 0: 0?1?2?0 (visited), 2?6: 6 finishes first ? stack: [6] 2 finishes ? stack: [6, 2] 1?3?4?5?3 (visited), 5 finishes ? stack: [6, 2, 5] 4 finishes ? stack: [6, 2, 5, 4] 3 finishes ? stack: [6, 2, 5, 4, 3] 1 finishes ? stack: [6, 2, 5, 4, 3, 1] 0 finishes ? stack: [6, 2, 5, 4, 3, 1, 0] Stack (top = last finished): [6, 2, 5, 4, 3, 1, 0] Process order (pop from stack): 0, 1, 3, 4, 5, 2, 6 === PASS 2: DFS on TRANSPOSED graph === Transposed edges: 1?0, 2?1, 0?2, 3?1, 4?3, 5?4, 3?5, 6?2 Process 0 (unvisited in transposed graph): 0?2?1?0 (visited), DFS collects: {0, 2, 1} ? SCC 1 Mark 0,1,2 as visited. Process 1 (already visited) ? skip Process 3 (unvisited): 3?5?4?3 (visited), DFS collects: {3, 5, 4} ? SCC 2 Process 4 (already visited) ? skip Process 5 (already visited) ? skip Process 2 (already visited) ? skip Process 6 (unvisited): 6?2 (visited), DFS collects: {6} ? SCC 3 === RESULT: SCCs = {0,1,2}, {3,4,5}, {6} ===
Section 6 — SCC C# & Python
C# — Kosaraju's SCC Algorithm 
using System;
using System.Collections.Generic;

public class KosarajuSCC
{
    private readonly int n;
    private readonly List<int>[] adj;    // original graph
    private readonly List<int>[] radj;   // transposed (reversed) graph

    public KosarajuSCC(int vertices)
    {
        n    = vertices;
        adj  = new List<int>[n];
        radj = new List<int>[n];
        for (int i = 0; i < n; i++) { adj[i] = new(); radj[i] = new(); }
    }

    public void AddEdge(int u, int v)
    {
        adj[u].Add(v);    // original
        radj[v].Add(u);  // transposed
    }

    public List<List<int>> FindSCCs()
    {
        bool[] visited = new bool[n];
        var finishStack = new Stack<int>();

        // -- Pass 1: DFS on original graph, record finish order --
        void DFS1(int u) {
            visited[u] = true;
            foreach (int v in adj[u])
                if (!visited[v]) DFS1(v);
            finishStack.Push(u);  // push on finish
        }

        for (int i = 0; i < n; i++)
            if (!visited[i]) DFS1(i);

        // -- Pass 2: DFS on transposed graph in reverse finish order --
        Array.Fill(visited, false);
        var sccs = new List<List<int>>();

        void DFS2(int u, List<int> scc) {
            visited[u] = true;
            scc.Add(u);
            foreach (int v in radj[u])
                if (!visited[v]) DFS2(v, scc);
        }

        while (finishStack.Count > 0)
        {
            int u = finishStack.Pop();
            if (!visited[u])
            {
                var scc = new List<int>();
                DFS2(u, scc);
                sccs.Add(scc);
            }
        }
        return sccs;
    }

    public static void Main()
    {
        var g = new KosarajuSCC(7);
        g.AddEdge(0,1); g.AddEdge(1,2); g.AddEdge(2,0);  // SCC: {0,1,2}
        g.AddEdge(1,3); g.AddEdge(3,4); g.AddEdge(4,5); g.AddEdge(5,3);  // SCC: {3,4,5}
        g.AddEdge(2,6);                                              // SCC: {6}

        var sccs = g.FindSCCs();
        Console.WriteLine($"{sccs.Count} SCCs found:");
        foreach (var scc in sccs)
            Console.WriteLine($"  [{string.Join(", ", scc)}]");
        // [0, 2, 1], [3, 5, 4], [6]  (order may vary)
    }
}
Python — Kosaraju's SCC 
from collections import defaultdict

class KosarajuSCC:
    def __init__(self, n):
        self.n    = n
        self.adj  = defaultdict(list)
        self.radj = defaultdict(list)

    def add_edge(self, u, v):
        self.adj[u].append(v)
        self.radj[v].append(u)

    def find_sccs(self):
        visited, finish_stack = set(), []

        # Pass 1: DFS on original, record finish times
        def dfs1(u):
            visited.add(u)
            for v in self.adj[u]:
                if v not in visited: dfs1(v)
            finish_stack.append(u)

        for i in range(self.n):
            if i not in visited: dfs1(i)

        # Pass 2: DFS on transposed graph in reverse finish order
        visited2, sccs = set(), []

        def dfs2(u, scc):
            visited2.add(u); scc.append(u)
            for v in self.radj[u]:
                if v not in visited2: dfs2(v, scc)

        while finish_stack:
            u = finish_stack.pop()
            if u not in visited2:
                scc = []
                dfs2(u, scc)
                sccs.append(scc)
        return sccs


if __name__ == "__main__":
    g = KosarajuSCC(7)
    g.add_edge(0,1); g.add_edge(1,2); g.add_edge(2,0)
    g.add_edge(1,3); g.add_edge(3,4); g.add_edge(4,5); g.add_edge(5,3)
    g.add_edge(2,6)
    sccs = g.find_sccs()
    print(f"{len(sccs)} SCCs:")
    for scc in sccs:
        print(f"  {sorted(scc)}")
    # [0, 1, 2], [3, 4, 5], [6]

Section 8 — Real-World Applications
?? Topological Sort — Build Systems & Package Managers
Make, CMake, Gradle, npm, pip — all use topological sort to determine the correct build/install order. Package A depends on B, B depends on C: install C first, then B, then A. If a circular dependency exists (A depends on B, B depends on A), the topological sort fails — detecting the circular dependency. This is exactly Kahn's algorithm: if not all packages can be ordered, a circular dependency must exist.
?? Topological Sort — Task Scheduling
Google's MapReduce, Apache Spark, and Airflow (workflow orchestration) all use DAGs to represent computation pipelines. Topological sort determines which tasks can run in parallel (same "level" in the DAG) and which must wait. Netflix uses this for data pipeline scheduling; Facebook uses it for their internal workflow engine.
?? SCCs — Web Crawling & SEO
Google's PageRank algorithm models the web as a directed graph. SCCs represent communities of tightly-linked pages. Pages within the same SCC have strong mutual influence. Finding SCCs helps identify "link farms" (sites that artificially create cycles to boost PageRank) and helps Google understand the web's community structure for better ranking.
?? SCCs — 2-SAT Problem
Boolean satisfiability with clauses of exactly 2 literals (2-SAT) is solvable in linear time using SCCs. Model each variable x as two nodes (x and ¬x). Build an implication graph. The formula is unsatisfiable if and only if some variable x and its negation ¬x are in the same SCC. Otherwise, the topological order of SCCs gives the assignment. Used in circuit design verification and constraint satisfaction.
Section 9 — Common Mistakes
?? Applying topological sort to a graph with cycles: If the graph has cycles, a topological ordering is impossible. Always check whether your input is guaranteed to be a DAG, or use Kahn's algorithm which detects cycles (process count < V means a cycle exists).
?? Kosaraju's Pass 2: forgetting to use the transposed graph: A common mistake is running both DFS passes on the original graph. Pass 2 must use the transposed (reversed-edge) graph. The intuition: in the original graph, edges go "into" SCCs from outside. In the transposed graph, those edges are reversed — so each DFS from an SCC's "representative vertex" stays confined to that SCC.
?? Stack overflow on large graphs with recursive DFS: Both DFS-based topological sort and Kosaraju's use recursive DFS. For graphs with V=100,000 vertices, this can cause stack overflow. Convert to iterative DFS using an explicit stack, or increase the call stack size (in Python: use sys.setrecursionlimit carefully, or rewrite iteratively). Kahn's BFS-based approach avoids this issue entirely.
?? Topological sort result is not unique: Multiple valid topological orderings can exist for the same DAG (whenever multiple vertices have in-degree 0 simultaneously). Don't assume there's one "correct" answer — any valid ordering is correct. If you need a specific ordering (e.g., lexicographically smallest), use a priority queue instead of a regular queue in Kahn's algorithm.
Section 10 — Check Your Understanding
Q1 — In Kahn's algorithm, what does it mean if the final result list has fewer than V vertices?
Answer: The graph contains a cycle (it's not a DAG). Kahn's starts with vertices of in-degree 0. After processing each vertex, it decrements the in-degrees of its neighbors. In a DAG, eventually all vertices reach in-degree 0 and are processed. In a cyclic graph, the vertices forming the cycle can never reach in-degree 0 — each vertex in the cycle always has at least one incoming edge from another cycle vertex. So if the result has k < V vertices, exactly V-k vertices are stuck in cycles. This is how LeetCode "Course Schedule" and "Course Schedule II" problems detect impossible schedules.
Cycle detection: if result.Count == n: no cycle else: cycle exists
Q2 — Why does Kosaraju's algorithm need to reverse all edges for Pass 2? What would happen if you used the original graph for both passes?
If you used the original graph for both passes, you'd get wrong results. The intuition: Pass 1 ends by identifying vertices with highest "reach" in the graph (they finish last in DFS because they can reach many others). In Pass 2, we want to find exactly the vertices that can reach back to them — i.e., vertices in the same SCC. In the original graph, DFS from a vertex could escape to other SCCs. In the transposed graph, edges pointing out of an SCC in the original now point into it — so DFS in the transposed graph starting from the SCC "leader" stays confined to that SCC, collecting exactly its members.
Key insight: In original graph: "leader" of SCC can reach many things. In transposed graph: many things can reach the "leader" — but only those within the same SCC.
Q3 — You have tasks with dependencies. Some tasks form a cycle. Can you use topological sort to find all tasks that CAN be completed (are not part of, or downstream of, any cycle)?
Yes — Kahn's algorithm naturally identifies these. Run Kahn's algorithm. All vertices that appear in the output list are completable (they're not in any cycle AND they don't depend on any cycle). Vertices NOT in the output list are either: (a) in a cycle, or (b) downstream of a cycle (their prerequisites include cyclic tasks). Since cyclic tasks can never complete, their dependent tasks also can never complete. Kahn's algorithm efficiently separates completable from incompletable tasks in one O(V+E) pass. This is exactly the LeetCode "Course Schedule II" solution: return the Kahn's output if it includes all courses, otherwise return [] (impossible).
Time: O(V + E). The output is exactly the set of completable tasks in dependency order.
Q4 — How many SCCs does a DAG (directed acyclic graph) have?
A DAG has exactly V SCCs — one per vertex. In a DAG, there are no cycles. For two distinct vertices u and v to be in the same SCC, there must exist paths u?v AND v?u — but a cycle u?…?v?…?u would contradict the DAG property. Therefore, every vertex forms its own singleton SCC. Conversely, condensing any directed graph to its SCCs always produces a DAG (the condensation graph). So "DAG" and "graph where every SCC is a singleton" are equivalent.
Implication: Topological sort on a DAG is the same as ordering the V singleton SCCs in topological order.
Q5 — LeetCode #207: Course Schedule. Given numCourses and a list of prerequisites [a, b] meaning "to take course a, you must first take course b", return true if it's possible to finish all courses.
This is cycle detection in a directed graph — equivalent to: "is this a DAG?" Use Kahn's algorithm: build the graph (b?a for each [a,b] pair), compute in-degrees, run Kahn's. If the result contains all numCourses vertices, return true (no cycle). Otherwise return false (cycle = impossible). Alternatively, use DFS with 3 states: 0=unvisited, 1=in current DFS path, 2=fully processed. If DFS visits a vertex with state 1, a cycle is found.
C#: var (order, hasCycle) = KahnsSort(); return !hasCycle;
Python: order, has_cycle = kahns_sort(); return not has_cycle
Time: O(V+E), Space: O(V+E)
