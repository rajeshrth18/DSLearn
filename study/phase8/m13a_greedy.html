Section 1 — What Makes a Greedy Algorithm Correct?

A greedy algorithm makes the locally optimal choice at each step, hoping to reach the global optimum without backtracking. It works when the problem has:

  • Greedy Choice Property: A globally optimal solution can be obtained by making a locally greedy choice at each step — we never need to "undo" a greedy decision.
  • Optimal Substructure: An optimal solution to the problem contains optimal solutions to subproblems (same as DP).
GREEDY vs DP: Both need OPTIMAL SUBSTRUCTURE. DP also needs OVERLAPPING SUBPROBLEMS. Greedy additionally needs GREEDY CHOICE PROPERTY. The greedy choice property is what makes greedy simpler than DP. It says: "I can always take the locally best option without regretting it later." Examples where greedy WORKS: ? Fractional Knapsack (sort by value/weight ratio) ? Activity Selection (sort by end time) ? Dijkstra's shortest path (always expand closest unvisited node) ? Prim's / Kruskal's MST ? Huffman coding (always merge two smallest frequency nodes) Examples where greedy FAILS (DP needed): ? 0/1 Knapsack (can't take fractions ? greedy ratio fails) ? Coin Change (e.g., coins=[1,3,4], target=6: greedy takes 4+1+1=3 coins vs DP 3+3=2) ? Longest Common Subsequence ? Edit Distance
Section 2 — The Exchange Argument

The standard technique to prove a greedy algorithm is correct. You assume an optimal solution differs from the greedy solution, then show you can "exchange" elements to transform it into the greedy solution without getting worse.

EXCHANGE ARGUMENT TEMPLATE: 1. Let OPT = any optimal solution. Let G = greedy solution. 2. If G == OPT, done. Otherwise, find the FIRST position where they differ. 3. Show that in OPT, you can swap the element at that position with the greedy choice, WITHOUT INCREASING cost (or without decreasing value). 4. This gives OPT' that agrees with G one step further. 5. Repeat until OPT is transformed into G — proving cost(G) = cost(OPT). Since OPT is optimal, cost(G) = cost(OPT), so cost(G) = cost(OPT). ? ACTIVITY SELECTION PROOF (Sort by end time): Claim: Among all activities that start after the last selected, pick the one ending first. Suppose OPT chose activity B instead of A, where end(A) = end(B). Swap B for A in OPT ? still valid (A ends no later than B, doesn't conflict). The number of activities selected is the same. Now OPT' agrees with greedy at this step. ?
Section 3 — Activity Selection Problem

Given n activities with start[i] and end[i], select the maximum number of non-overlapping activities.

GREEDY: Sort by END TIME. Always pick the activity with earliest end time that doesn't conflict with the last selected. EXAMPLE: Activities (start, end): A=(1,4), B=(3,5), C=(0,6), D=(5,7), E=(3,9), F=(5,9), G=(6,10), H=(8,11) Sort by end: A(1,4), B(3,5), C(0,6), D(5,7), E(3,9), F(5,9), G(6,10), H(8,11) Select A (end=4). last_end=4. B(3,5): start=3 < last_end=4 ? SKIP (conflicts) C(0,6): start=0 < last_end=4 ? SKIP D(5,7): start=5 >= last_end=4 ? SELECT. last_end=7 E(3,9): start=3 < 7 ? SKIP F(5,9): start=5 < 7 ? SKIP G(6,10): start=6 < 7 ? SKIP H(8,11): start=8 >= 7 ? SELECT. last_end=11 Result: {A, D, H} — 3 activities ? (maximum) WHY end time sort works: Finishing early "frees up" the most future time.
C# 
public int MaxActivities(int[] start, int[] end)
{
    int n = start.Length;
    var order = Enumerable.Range(0, n).OrderBy(i => end[i]).ToArray();
    int count = 1, lastEnd = end[order[0]];
    for (int i = 1; i < n; i++)
        if (start[order[i]] >= lastEnd)
        {
            count++;
            lastEnd = end[order[i]];
        }
    return count;
}
Python 
def max_activities(activities):
    # activities = [(start, end), ...]
    activities.sort(key=lambda x: x[1])
    count, last_end = 1, activities[0][1]
    for s, e in activities[1:]:
        if s >= last_end:
            count += 1; last_end = e
    return count
Section 4 — Jump Game II (LeetCode #45)

Given jump lengths at each position, find the minimum jumps to reach the last index.

GREEDY: Track the FARTHEST REACHABLE position. When you reach the END of the current jump range, you MUST jump. At each mandatory jump, advance to the farthest reachable. KEY INSIGHT: We don't care WHERE in the current range we jump from — only the farthest we can reach in one more jump. EXAMPLE: nums = [2, 3, 1, 1, 4] i=0: range=[0,0], farthest=0+2=2, end=0 ? reach end, jumps=1, end=2 i=1: farthest=max(2,1+3)=4 i=2: farthest=max(4,2+1)=4 reach end (i=2=end=2) ? jumps=2, end=4 i=3: (beyond last index, stop) Answer: 2 jumps ? (0?1?4) COMPLEXITY: O(n) time, O(1) space
C# 
// LeetCode #45 — Jump Game II
public int Jump(int[] nums)
{
    int jumps = 0, curEnd = 0, farthest = 0;
    for (int i = 0; i < nums.Length - 1; i++)
    {
        farthest = Math.Max(farthest, i + nums[i]);
        if (i == curEnd) { jumps++; curEnd = farthest; }
    }
    return jumps;
}
Python 
def jump(nums):
    jumps = cur_end = farthest = 0
    for i in range(len(nums) - 1):
        farthest = max(farthest, i + nums[i])
        if i == cur_end:
            jumps += 1; cur_end = farthest
    return jumps
Section 5 — Task Scheduler (LeetCode #621)

Given tasks and cooldown n, find the minimum time to execute all tasks where the same task must wait n intervals before repeating.

GREEDY: The most frequent task is the bottleneck. Let f = max frequency. Let k = count of tasks with max frequency. FORMULA: max(len(tasks), (f-1)*(n+1) + k) Intuition: Arrange the most frequent tasks in "frames" of size (n+1): EXAMPLE: tasks=[A,A,A,B,B,C], n=2 Most frequent: A(3). f=3, k=1 (only A has freq 3), B(2), C(1) Frame structure (size n+1 = 3): A _ _ | A _ _ | A Fill with B: A B _ | A B _ | A Fill with C: A B C | A B _ | A Total = (f-1)*(n+1)+k = 2*3+1 = 7 len(tasks)=6, so answer=max(7,6)=7 Another example: tasks=[A,A,B,B,C,C], n=2 f=2, k=3 (A,B,C all freq=2) (f-1)*(n+1)+k = 1*3+3 = 6 = len(tasks) ? max=6, no idle needed
C# 
// LeetCode #621 — Task Scheduler
public int LeastInterval(char[] tasks, int n)
{
    var freq = new int[26];
    foreach (var t in tasks) freq[t - 'A']++;
    int f = freq.Max();
    int k = freq.Count(x => x == f);
    return Math.Max(tasks.Length, (f - 1) * (n + 1) + k);
}
Python 
from collections import Counter
def leastInterval(tasks, n):
    freq = Counter(tasks)
    f = max(freq.values())
    k = sum(1 for v in freq.values() if v == f)
    return max(len(tasks), (f - 1) * (n + 1) + k)

Section 7 — Q&A Review
Q1: Why does sorting by end time work for Activity Selection, but sorting by start time or duration does NOT?
Sort by start time counterexample: One long activity starting at time 0 would be selected first, blocking many short activities. Sort by duration: A short activity with early start might conflict with many others; one long activity might end early enough to allow many more. Sort by end time is correct because selecting the activity that finishes earliest is always optimal — it leaves the most "remaining time" for subsequent activities. Exchange argument proof: If optimal solution chose activity B instead of greedy's A, and end(A) = end(B), then swapping B for A in the optimal solution doesn't reduce the count and doesn't introduce conflicts (A ends no later than B). The solution remains valid, now agreeing with greedy's choice at this step.
Q2: What is the greedy choice property, and how is it different from optimal substructure?
Optimal substructure: after making any decision (greedy or DP), the remaining subproblem has an optimal solution that is independent and can be solved optimally. Both DP and greedy need this. Greedy choice property: there exists an optimal solution that starts with the greedy choice. This means we never need to "undo" our first choice — we can commit to the locally best option and the remaining subproblem is still well-defined. DP doesn't have this: DP must consider ALL choices at each step (filling the whole table), because the optimal choice at step i depends on what's available at step i+1. Greedy saves work by committing to one choice per step.
Q3: In Jump Game II, why does the greedy approach work — why can't a "sub-optimal" early jump lead to a better outcome?
The greedy maximizes the farthest reachable position with each mandatory jump. Key insight: within the current reachable range, we WILL eventually visit every position anyway (before the next mandatory jump), so we might as well consider ALL positions in the range for reaching the farthest point. The greedy doesn't choose WHERE in the current range to jump — it considers all positions simultaneously and picks the maximum reach. It only "makes a jump" when it HAS to (when i reaches curEnd). This is similar to breadth-first search: the greedy processes all positions at the same "jump distance" before advancing. No early jump strategy can reach farther than the maximum reachable from all positions in the current range.
Q4: Why does the Task Scheduler formula max(len(tasks), (f-1)*(n+1)+k) work?
(f-1)*(n+1)+k represents: we arrange the f copies of the most frequent task in a frame structure. Between each pair of consecutive executions of the most frequent task, we need n idle or other-task slots. So each "frame" has (n+1) slots. We have (f-1) complete frames (the last copy doesn't need a trailing cooldown). At the end of the last frame, we append k (the number of tasks with maximum frequency, all of which fit at the end). If other tasks are abundant enough to fill all slots, the total time is just len(tasks) — no idle time needed. The max of the two handles both cases: more tasks than frames (no idle), or fewer tasks than frames (some idle needed).
Q5: Give a brief outline of how to prove a new greedy algorithm correct using the exchange argument.
1. Define the greedy choice precisely (what "locally optimal" means). 2. Assume an optimal solution OPT exists. 3. If OPT = greedy, done. Otherwise, find the first step where OPT differs from greedy. 4. Consider what OPT chose at that step (call it B) vs greedy's choice (call it A). 5. Show that swapping B for A in OPT produces OPT' that: (a) is still a valid/feasible solution, and (b) has cost no worse than OPT (equal or better for minimization; equal or better for maximization). 6. OPT' now agrees with greedy one step further. 7. Repeat steps 3–6 until OPT is fully transformed into greedy's solution. 8. Conclude greedy solution has cost = OPT, proving optimality. The key creative step is step 5: showing the swap doesn't hurt.
