Section 1 — The Divide & Conquer Framework

Divide & Conquer (D&C) works by recursively breaking a problem into smaller subproblems, solving each independently, then combining results. The key is that subproblems are independent (unlike DP, which has overlapping subproblems).

D&C FRAMEWORK: 1. DIVIDE: Split problem into k subproblems of size n/b. 2. CONQUER: Recursively solve each subproblem. 3. COMBINE: Merge/combine the solutions. BASE CASE: When problem is small enough to solve directly. RECURRENCE FORM: T(n) = aT(n/b) + f(n) a = number of subproblems b = factor by which problem size shrinks f(n) = cost of divide + combine steps MASTER THEOREM (simplified): Compare f(n) vs n^(log_b(a)): Case 1: f(n) = O(n^(log_b(a) - e)) ? T(n) = T(n^log_b(a)) Case 2: f(n) = T(n^log_b(a)) ? T(n) = T(n^log_b(a) · log n) Case 3: f(n) = O(n^(log_b(a) + e)) ? T(n) = T(f(n)) EXAMPLES: Merge Sort: T(n) = 2T(n/2) + O(n) ? Case 2 ? O(n log n) ? Binary Search: T(n) = T(n/2) + O(1) ? Case 2 (log_2(1)=0) ? O(log n) ? Strassen: T(n) = 7T(n/2) + O(n²) ? Case 1 (log_2(7)˜2.81>2) ? O(n^2.81) ?
Section 2 — Merge Sort & Counting Inversions

Merge Sort is the quintessential D&C algorithm. It also solves the Count Inversions problem (LeetCode #315 / similar) with no extra complexity.

MERGE SORT: Divide: split array in half Conquer: recursively sort each half Combine: merge two sorted halves in O(n) INVERSION COUNTING: An inversion is a pair (i,j) where i<j but arr[i]>arr[j]. Key insight: during merge, when we take an element from the RIGHT array before exhausting the LEFT array, all remaining left elements form inversions with it. inversions += mid - left_ptr + 1 (for each right element taken out of order) EXAMPLE: [3, 1, 2] Inversions: (3,1), (3,2) ? 2 inversions Merge [3] and [1,2]: Compare 3 vs 1: take 1 from right. inversions += 1 (only 3 remains left). Compare 3 vs 2: take 2 from right. inversions += 1. Take 3. Total = 2 ?
C# 
// Merge Sort + Count Inversions
public int MergeSortCount(int[] arr, int lo, int hi)
{
    if (hi - lo <= 1) return 0;
    int mid = (lo + hi) / 2;
    int inv = MergeSortCount(arr, lo, mid) + MergeSortCount(arr, mid, hi);
    var temp = new int[hi - lo];
    int i = lo, j = mid, k = 0;
    while (i < mid && j < hi)
    {
        if (arr[i] <= arr[j]) temp[k++] = arr[i++];
        else { inv += mid - i; temp[k++] = arr[j++]; } // right taken before all lefts
    }
    while (i < mid) temp[k++] = arr[i++];
    while (j < hi)  temp[k++] = arr[j++];
    Array.Copy(temp, 0, arr, lo, hi - lo);
    return inv;
}
Python 
def merge_sort_count(arr):
    if len(arr) <= 1: return arr, 0
    mid = len(arr) // 2
    L, l_inv = merge_sort_count(arr[:mid])
    R, r_inv = merge_sort_count(arr[mid:])
    merged, inv = [], l_inv + r_inv
    i = j = 0
    while i < len(L) and j < len(R):
        if L[i] <= R[j]: merged.append(L[i]); i += 1
        else: merged.append(R[j]); inv += len(L) - i; j += 1
    return merged + L[i:] + R[j:], inv
Section 3 — Quick Select: Kth Largest Element

Quick Select finds the kth element in O(n) average time, O(n²) worst case. It's the backbone of LeetCode #215 — Kth Largest Element in an Array.

QUICK SELECT: Like quicksort, but only recurse into ONE side after partitioning. Partition around pivot p: - Elements < p go left - Elements > p go right - Pivot ends at index q If q == target_index ? found! If q < target_index ? recurse right If q > target_index ? recurse left COMPLEXITY: Average: O(n) — each recursion reduces size by ~half Worst: O(n²) — always pick smallest/largest as pivot ? Use random pivot or median-of-3 to avoid worst case EXAMPLE: [3,2,1,5,6,4], k=2 (2nd largest = index 4 from left in 0-indexed sorted) Target index = n - k = 4 Partition [3,2,1,5,6,4]: pivot=4, result = [3,2,1,4,5,6]? No. Actually pivot=arr[right], partition puts it at correct position. ? O(n) average per call, total O(n) amortized.
C# 
// LeetCode #215 — Kth Largest Element
public int FindKthLargest(int[] nums, int k)
{
    int target = nums.Length - k;
    return QuickSelect(nums, 0, nums.Length - 1, target);
}
private int QuickSelect(int[] a, int lo, int hi, int target)
{
    if (lo == hi) return a[lo];
    int p = Partition(a, lo, hi);
    if (p == target) return a[p];
    return p < target ? QuickSelect(a, p + 1, hi, target)
                      : QuickSelect(a, lo, p - 1, target);
}
private int Partition(int[] a, int lo, int hi)
{
    int pivot = a[hi], i = lo;
    for (int j = lo; j < hi; j++)
        if (a[j] <= pivot) { (a[i], a[j]) = (a[j], a[i]); i++; }
    (a[i], a[hi]) = (a[hi], a[i]);
    return i;
}
Python 
import random
def findKthLargest(nums, k):
    target = len(nums) - k
    def quick_select(lo, hi):
        # Random pivot to avoid O(n²) worst case
        rand = random.randint(lo, hi)
        nums[rand], nums[hi] = nums[hi], nums[rand]
        pivot, i = nums[hi], lo
        for j in range(lo, hi):
            if nums[j] <= pivot:
                nums[i], nums[j] = nums[j], nums[i]; i += 1
        nums[i], nums[hi] = nums[hi], nums[i]
        if i == target: return nums[i]
        return quick_select(i+1, hi) if i < target else quick_select(lo, i-1)
    return quick_select(0, len(nums)-1)
Section 4 — Binary Search Deep Dive

Binary search is D&C with one subproblem (one-sided). The key challenge is getting the boundaries right. There are two templates that cover all variants.

BINARY SEARCH TEMPLATES: Template 1 — Exact match (or simple condition): lo, hi = 0, n-1 while lo <= hi: mid = lo + (hi-lo)//2 ? prevents integer overflow if arr[mid] == target: return mid elif arr[mid] < target: lo = mid + 1 else: hi = mid - 1 return -1 Template 2 — "Find leftmost/rightmost index satisfying condition": lo, hi = 0, n ? hi = n (open right bound) or n-1 (closed) while lo < hi: ? strict < mid = lo + (hi-lo)//2 if condition(mid): hi = mid ? for leftmost true else: lo = mid+1 return lo ? lo == hi, the answer CHOOSING TEMPLATE: - "Is target in array?" ? Template 1 - "First/last position of value" ? Template 2 (leftmost/rightmost) - "First index where f(x) = true" ? Template 2 (binary search on answer) - "Minimum value satisfying condition" ? Template 2 on answer space CLASSIC PATTERNS: • Lower bound = first index where arr[i] >= target • Upper bound = first index where arr[i] > target • Rotated sorted array ? check which half is sorted, narrow accordingly • 2D matrix search ? treat as 1D array (row = mid/cols, col = mid%cols)

Binary Search on Answer — Template 2

Many problems require finding the minimum value of x such that f(x) is true — the answer space itself is searched with binary search.

BINARY SEARCH ON ANSWER pattern: If "is f(x) satisfiable with threshold x?" can be checked in O(check_time), and the answer is monotone (f(x) true implies f(x+1) true for minimization), then binary search on x in O(log(answer_range) × check_time). EXAMPLE: LeetCode #875 — Koko Eating Bananas "Can Koko eat all piles within h hours at speed k?" ? greedy check: sum(ceil(pile/k)) <= h ? O(n) Binary search k in [1, max(piles)] ? O(n log(max)) EXAMPLE: LeetCode #1011 — Ship Packages Within D Days "Can we ship all packages within d days at capacity cap?" ? greedy check: greedily pack days, each up to cap ? O(n) Binary search cap in [max(weights), sum(weights)] ? O(n log(sum))
C# 
// LeetCode #875 — Koko Eating Bananas (Binary search on answer)
public int MinEatingSpeed(int[] piles, int h)
{
    int lo = 1, hi = piles.Max();
    while (lo < hi)
    {
        int mid = lo + (hi - lo) / 2;
        if (CanFinish(piles, mid, h)) hi = mid;
        else lo = mid + 1;
    }
    return lo;
}
private bool CanFinish(int[] piles, int k, int h)
    => piles.Sum(p => (p + k - 1) / k) <= h;

// LeetCode #33 — Search in Rotated Sorted Array
public int Search(int[] nums, int target) {
    int lo = 0, hi = nums.Length - 1;
    while (lo <= hi) {
        int mid = lo + (hi - lo) / 2;
        if (nums[mid] == target) return mid;
        if (nums[lo] <= nums[mid]) { // left half sorted
            if (nums[lo] <= target && target < nums[mid]) hi = mid - 1;
            else lo = mid + 1;
        } else { // right half sorted
            if (nums[mid] < target && target <= nums[hi]) lo = mid + 1;
            else hi = mid - 1;
        }
    }
    return -1;
}

import math
def minEatingSpeed(piles, h):
    lo, hi = 1, max(piles)
    while lo < hi:
        mid = (lo + hi) // 2
        if sum(math.ceil(p/mid) for p in piles) <= h: hi = mid
        else: lo = mid + 1
    return lo

def search_rotated(nums, target):
    lo, hi = 0, len(nums) - 1
    while lo <= hi:
        mid = (lo + hi) // 2
        if nums[mid] == target: return mid
        if nums[lo] <= nums[mid]:  # left half sorted
            if nums[lo] <= target < nums[mid]: hi = mid - 1
            else: lo = mid + 1
        else:  # right half sorted
            if nums[mid] < target <= nums[hi]: lo = mid + 1
            else: hi = mid - 1
    return -1

Section 6 — Q&A Review
Q1: What is the key difference between D&C and Dynamic Programming?
Both divide problems into subproblems. The key difference: D&C subproblems are INDEPENDENT — solving one doesn't help solve another. DP subproblems OVERLAP — the same subproblem recurs, so caching (memoization/tabulation) saves work. Merge sort's subproblems are completely independent (left half and right half never share elements). Fibonacci subproblems overlap: fib(5) needs fib(4) AND fib(3), and fib(4) also needs fib(3). If you applied memoization to merge sort, you'd get no benefit; if you didn't apply it to Fibonacci, you'd get exponential time.
Q2: Why use mid = lo + (hi - lo) / 2 instead of (lo + hi) / 2 in binary search?
Integer overflow prevention. If lo and hi are both large (e.g., near INT_MAX), lo + hi overflows 32-bit integer, producing a negative number. lo + (hi - lo)/2 computes the midpoint without overflow because hi - lo = hi which doesn't overflow when lo = 0. In Python, integers have arbitrary precision so overflow can't happen, but in C#/Java/C++ with int, this is a critical bug. The formula is equivalent: lo + (hi-lo)/2 = (2lo + hi - lo)/2 = (lo + hi)/2.
Q3: Quick Select is O(n) average but O(n²) worst case. How do you prevent the worst case?
Worst case occurs when the pivot is always the smallest or largest element (e.g., sorted array with last element as pivot). Solutions: (1) Random pivot: randomly select pivot before partitioning. This makes worst case occur with probability 1/n^k for k partitions, essentially impossible in practice. (2) Median-of-3: take median of first, middle, last elements as pivot. Reduces worst case for nearly-sorted inputs. (3) Introselect: hybrid algorithm that switches to deterministic O(n) "median-of-medians" algorithm after too many recursions. C++ std::nth_element typically uses introselect internally.
Q4: How do you apply the Master Theorem to T(n) = 2T(n/2) + O(n log n)?
Here a=2, b=2, so n^(log_b(a)) = n^(log_2(2)) = n^1 = n. The f(n) = n log n. Compare f(n) = n log n vs n^1 = n: n log n is slightly larger than n, but NOT by a polynomial factor. This falls OUTSIDE the standard Master Theorem (none of the three cases apply cleanly). This is the "Case 2 extension": when f(n) = n^(log_b a) · log^k(n), T(n) = T(n^(log_b a) · log^(k+1) n). Here k=1, so T(n) = T(n log² n). This applies to things like merge sort on linked lists where merging costs O(n log n).
Q5: Describe the binary search approach to find the median of two sorted arrays in O(log(min(m,n))).
Binary search on a partition index in the smaller array. We want to find i (in array A) and j = (m+n+1)/2 - i (in array B) such that: (1) i elements from A and j from B are on the "left" side of the partition, and (2) A[i-1] = B[j] and B[j-1] = A[i] (the partition is correct). Binary search i in [0, m] (m = len(A), m = n). If A[i-1] > B[j]: i is too large, move left. If B[j-1] > A[i]: i is too small, move right. When partition is correct: for odd total, median = max(A[i-1], B[j-1]); for even total, median = (max(A[i-1],B[j-1]) + min(A[i],B[j])) / 2. This is LeetCode #4, difficulty Hard.
