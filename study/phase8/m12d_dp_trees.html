Section 1 — DP on Trees: The Postorder Pattern

In tree DP, the state is defined per node. The recurrence combines results from children. Since children must be computed before the parent, tree DP uses postorder DFS (process children first, then use their results for the parent).

GENERAL TEMPLATE FOR TREE DP: dfs(node): if node is null: return base_value left_result = dfs(node.left) // compute left subtree first right_result = dfs(node.right) // compute right subtree // Combine: use node.val + left_result + right_result node_result = combine(node.val, left_result, right_result) // Optionally update a global answer global_ans = max(global_ans, some_function(node_result)) return node_result // return to parent KEY: What does dfs() RETURN vs what goes into global_ans? - Return value: the "best subproblem result rooted at this node" (restricted: must extend upward through this node) - global_ans: the "best overall answer" which may span both subtrees and NOT necessarily extend upward This distinction is crucial for problems like Tree Diameter and Max Path Sum.
Section 2 — Binary Tree Maximum Path Sum (LeetCode #124)

Find the maximum path sum in a binary tree. A path is any sequence of nodes from one node to another — it doesn't have to go through the root. Can go down and up.

TREE: -10 / \ 9 20 / \ 15 7 INSIGHT: At each node, we can either: 1. Extend one child's path upward (return max(0, left, right) + node.val) 2. "Use" this node as the TOP of a path spanning both subtrees (not returnable) ? left + node.val + right (stays local, updates global answer) dfs(-10): left=dfs(9): leaf ? gain=9, return 9 right=dfs(20): left=dfs(15): leaf ? gain=15, return 15 right=dfs(7): leaf ? gain=7, return 7 ? global: 15+20+7=42; return 20+max(15,7)=35 ? global: max(42, 9+(-10)+35=34, ...): stays 42 return: -10 + max(9, 35) = 25 Answer: global_max = 42 ?
C# 
public int MaxPathSum(TreeNode root)
{
    int ans = int.MinValue;
    DFS(root, ref ans);
    return ans;
}

private int DFS(TreeNode node, ref int ans)
{
    if (node == null) return 0;
    int L = Math.Max(0, DFS(node.left,  ref ans));  // ignore negative paths
    int R = Math.Max(0, DFS(node.right, ref ans));
    ans = Math.Max(ans, L + node.val + R);              // path through node
    return node.val + Math.Max(L, R);                    // best arm upward
}
Python 
def maxPathSum(root):
    ans = [float('-inf')]
    def dfs(node):
        if not node: return 0
        L = max(0, dfs(node.left))
        R = max(0, dfs(node.right))
        ans[0] = max(ans[0], L + node.val + R)
        return node.val + max(L, R)
    dfs(root)
    return ans[0]
Section 3 — Maximum Independent Set on Tree

In an unweighted tree, find the maximum set of nodes with no two nodes adjacent (connected by an edge). This is equivalent to LeetCode #337 House Robber III.

DP STATES per node: dp[node][0] = max nodes in MIS of subtree rooted at node, NOT including node dp[node][1] = max nodes in MIS of subtree rooted at node, INCLUDING node RECURRENCE: dp[node][0] = sum over children c: max(dp[c][0], dp[c][1]) (don't take node ? children can be taken or not) dp[node][1] = 1 + sum over children c: dp[c][0] (take node ? children CANNOT be taken) ANSWER: max(dp[root][0], dp[root][1]) EXAMPLE TREE: 1 / \ 2 3 / \ 4 5 dp[4] = (0, 1) (leaf) dp[5] = (0, 1) (leaf) dp[2][0] = max(0,1) + max(0,1) = 2 (take 4 and 5) dp[2][1] = 1 + 0 + 0 = 1 (take 2, can't take children) dp[3] = (0, 1) (leaf) dp[1][0] = max(2,1) + max(0,1) = 2+1 = 3 (take 4,5,3) dp[1][1] = 1 + dp[2][0] + dp[3][0] = 1+2+0 = 3 (take 1,4,5) Answer: max(3,3) = 3 ? ({4,5,3} or {1,4,5})
C# 
// LeetCode #337 — House Robber III
public int Rob(TreeNode root)
{
    var (noRob, rob) = DFS(root);
    return Math.Max(noRob, rob);
}

private (int noRob, int rob) DFS(TreeNode node)
{
    if (node == null) return (0, 0);
    var (lNo, lRob) = DFS(node.left);
    var (rNo, rRob) = DFS(node.right);
    int noRob = Math.Max(lNo, lRob) + Math.Max(rNo, rRob);
    int rob   = node.val + lNo + rNo;
    return (noRob, rob);
}
Python 
def rob337(root):

        no_rob = max(l_no, l_rob) + max(r_no, r_rob)
        rob    = node.val + l_no + r_no
        return no_rob, rob
    return max(dfs(root))
Section 4 — Diameter of Binary Tree (LeetCode #543)

The diameter is the length (in edges) of the longest path between any two nodes.

INSIGHT: The longest path either: a) Passes through a node: left_depth + right_depth (spans both subtrees) b) Stays entirely in a subtree dfs(node) returns the DEPTH (max edges from node to any leaf). CORRECT: at node 2, left=dfs(4)=0, right=dfs(5)=0 ans = max(0, 0+0)=0, return 1+max(0,0)=1 ? depth from 2 is 1 At node 1: left=1 (from 2), right=0 (from 3) ans = max(0, 1+0)=1... but correct answer is 3! C# 
public int DiameterOfBinaryTree(TreeNode root)
{
    int ans = 0;
    Depth(root, ref ans);
    return ans;
}
private int Depth(TreeNode node, ref int ans)
{
    if (node == null) return 0;
    int L = Depth(node.left,  ref ans);
    int R = Depth(node.right, ref ans);
    ans = Math.Max(ans, L + R);          // path through node (in edges)
    return 1 + Math.Max(L, R);           // depth (in nodes) for parent
}
Python 
def diameterOfBinaryTree(root):
    ans = [0]
    def depth(node):
        if not node: return 0
        L, R = depth(node.left), depth(node.right)
        ans[0] = max(ans[0], L + R)
        return 1 + max(L, R)
    depth(root)
    return ans[0]
Section 5 — DP on DAGs: Topological Order

On a directed acyclic graph (DAG), DP follows topological order. Common problems: longest path, shortest path, number of paths.

LONGEST PATH IN DAG: dp[u] = longest path starting at node u Process nodes in REVERSE topological order (sinks first): dp[u] = max(dp[v] + weight(u?v)) for all (u,v) edges EXAMPLE DAG: 0?1 (w=5), 0?2 (w=3), 1?3 (w=2), 2?3 (w=1), 3?4 (w=7) Topo order: [0, 1, 2, 3, 4] dp[4] = 0 (sink) dp[3] = max(dp[4]+7) = 7 dp[1] = max(dp[3]+2) = 9 dp[2] = max(dp[3]+1) = 8 dp[0] = max(dp[1]+5, dp[2]+3) = max(14, 11) = 14 Answer: max(dp) = 14 ? (path 0?1?3?4 with weight 5+2+7=14) KEY INSIGHT: On DAGs, both shortest and longest paths can be solved with DP in O(V+E). On graphs with cycles, Bellman-Ford is needed (and longest path is NP-hard!).
C# 
// Longest path in DAG using topological sort (Kahn's algorithm)
public int LongestPathDAG(int n, int[][] edges)
{
    var adj   = new List<(int to, int w)>[n];
    var indeg = new int[n];
    for (int i = 0; i < n; i++) adj[i] = new();
    foreach (var e in edges) { adj[e[0]].Add((e[1], e[2])); indeg[e[1]]++; }

    var q = new Queue<int>();
    var dp = new int[n];
    for (int i = 0; i < n; i++) if (indeg[i] == 0) q.Enqueue(i);
    while (q.Count > 0)
    {
        int u = q.Dequeue();
        foreach (var (v, w) in adj[u])
        {
            dp[v] = Math.Max(dp[v], dp[u] + w);
            if (--indeg[v] == 0) q.Enqueue(v);
        }
    }
    return dp.Max();
}
Python 
from collections import defaultdict, deque

def longest_path_dag(n, edges):
    adj   = defaultdict(list)
    indeg = [0] * n
    for u, v, w in edges:
        adj[u].append((v, w)); indeg[v] += 1
    q  = deque(i for i in range(n) if indeg[i] == 0)
    dp = [0] * n
    while q:
        u = q.popleft()
        for v, w in adj[u]:
            dp[v] = max(dp[v], dp[u] + w)
            indeg[v] -= 1
            if indeg[v] == 0: q.append(v)
    return max(dp)
Section 6 — Bellman-Ford as DP

Bellman-Ford computes shortest paths with negative edge weights. It's fundamentally a DP: dp[k][v] = shortest path to v using at most k edges.

BELLMAN-FORD DP FORMULATION: dp[k][v] = min dist to v using exactly = k edges dp[0][source] = 0, dp[0][v?source] = 8 RECURRENCE: dp[k][v] = min(dp[k-1][v], min over edges (u?v): dp[k-1][u] + w(u,v)) ANSWER: dp[n-1][dest] after n-1 iterations (max edges in shortest path = n-1) NEGATIVE CYCLE DETECTION: Run one more iteration (k=n). If dp[n][v] < dp[n-1][v] for any v, there's a negative cycle on the path to v. LeetCode #787 — Cheapest Flights with at most K Stops: dp[k][v] = min cost to reach v using at most k+1 flights Initialize dp[0][src]=0, rest=8 For each of K stops: For each edge (u,v,price): dp[k+1][v] = min(dp[k+1][v], dp[k][u] + price) Answer: dp[K][dst]
C# 
// LeetCode #787 — Cheapest Flights Within K Stops
public int FindCheapestPrice(int n, int[][] flights, int src, int dst, int k)
{
    int[] dp = Enumerable.Repeat(int.MaxValue/2, n).ToArray();
    dp[src] = 0;
    for (int i = 0; i <= k; i++)
    {
        var tmp = (int[])dp.Clone();  // snapshot to prevent using new values
        foreach (var f in flights)
            if (dp[f[0]] != int.MaxValue/2)
                tmp[f[1]] = Math.Min(tmp[f[1]], dp[f[0]] + f[2]);
        dp = tmp;
    }
    return dp[dst] == int.MaxValue/2 ? -1 : dp[dst];
}
Python 
def findCheapestPrice(n, flights, src, dst, k):
    INF = float('inf')
    dp = [INF] * n
    dp[src] = 0
    for _ in range(k + 1):
        tmp = dp[:]  # snapshot!
        for u, v, w in flights:
            if dp[u] != INF:
                tmp[v] = min(tmp[v], dp[u] + w)
        dp = tmp
    return dp[dst] if dp[dst] != INF else -1
Section 7 — Q&A Review
Q1: Why is the "return value" in tree DP different from the "global answer" in problems like Max Path Sum?
The DFS return value represents the best result that can be "extended upward" through the current node — it's constrained to form a valid path that continues to the parent. In Max Path Sum, a node can extend either its left OR right subtree to the parent (not both), so the return is node.val + max(L, R). The global answer, however, can be any path in the tree — including ones that "arch" through a node using both left and right children. Such a path cannot be extended further (it uses the node as a turning point), so it updates global_ans but is NOT returned. This "local computation vs global update" pattern appears in Diameter, Max Path Sum, Longest Univalue Path, and many others.
Q2: In the House Robber III (Max Independent Set on Tree), why is it correct to return a pair (noRob, rob) from each DFS call?
We need to track two states per node because the parent's decision depends on whether the current node is robbed. If parent is robbed, it needs dp[child][noRob]. If parent is not robbed, it needs max(dp[child][noRob], dp[child][rob]). By returning both values, the parent has full information to make its decision. If we only returned the "best" answer from each subtree (collapsing the two states), we'd lose the information about whether the root of that subtree was included or not. Returning a pair avoids the need for memoization or additional traversals.
Q3: Why can't we use DP to find the longest path in a general graph (with cycles)?
DP requires a well-defined computation order (topological order). On a general graph with cycles (e.g., A?B?C?A), there's no valid topological order — the subproblems are mutually dependent. dp[A] depends on dp[B], dp[B] depends on dp[C], dp[C] depends on dp[A]. This circular dependency breaks DP. Furthermore, on graphs with positive cycles, the "longest path" is infinite (just cycle forever). Longest path on general graphs is NP-hard (it generalizes the Hamiltonian Path problem). On DAGs, no cycles means valid topological ordering exists, and DP runs in O(V+E). This is why Dijkstra (shortest path, non-negative weights) and Bellman-Ford (handles negative edges but not cycles in shortest path) are needed for general graphs.
Q4: In Bellman-Ford / Cheapest Flights (#787), why must we "snapshot" the dp array before updating?
The DP state is dp[k][v] = min cost using at most k edges. When computing iteration k+1, we must use values from EXACTLY iteration k (at most k edges). If we update in-place and then use the updated values, we might compute a path that uses k+2 or more edges within a single iteration. For example: if we process edge (A?B) and update dp[B], then process edge (B?C) and use the newly updated dp[B], we've implicitly used k+2 edges in "one iteration" of the loop. The snapshot ensures each iteration strictly extends paths by exactly one more edge. This is the same reason as right-to-left traversal in 0/1 knapsack — prevent using "current iteration" values prematurely.
Q5: What is the time complexity of tree DP, and why is it O(n) and not O(n˛)?
Tree DP is O(n) where n is the number of nodes. Each node is visited exactly once in the postorder DFS. The key: unlike array DP where dp[i] might depend on all previous values dp[j] for j < i (leading to O(n˛)), tree DP has each node depend only on its direct children. The DFS visits each node once and processes each edge once (when returning from child to parent). Total work = O(nodes + edges) = O(n + n-1) = O(n) for a tree. There's no nested loop over all pairs of nodes. Each DFS call does O(1) work (constants per child). This is a key advantage of tree DP over general 2D DP.
