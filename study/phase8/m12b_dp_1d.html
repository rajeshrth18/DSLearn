Section 1 — Coin Change (LeetCode #322)

Given coins of certain denominations and an amount, find the minimum number of coins to make that amount. Return -1 if impossible.

STATE DEFINITION: dp[i] = minimum coins needed to make amount i RECURRENCE: For each coin c in coins: if i >= c: dp[i] = min(dp[i], dp[i - c] + 1) (try using each coin; if we can make i-c, then adding coin c makes i) BASE: dp[0] = 0 (0 coins needed for amount 0) dp[i] = 8 for i > 0 initially (impossible by default) EXAMPLE: coins=[1,5,6,9], amount=11 dp[0] = 0 dp[1] = min(inf, dp[0]+1) = 1 [use 1] dp[2] = min(inf, dp[1]+1) = 2 [use 1+1] dp[5] = min(4, dp[0]+1) = 1 [use 5] dp[6] = min(5, dp[1]+1, dp[0]+1) = 1 [use 6] dp[9] = 1 [use 9] dp[10]= min(dp[9]+1, dp[5]+1, dp[4]+1) = 2 [use 9+1 or 5+5] dp[11]= min(dp[10]+1, dp[6]+1, dp[5]+1, dp[2]+1) = 2 [use 6+5] Answer: 2 ?
C# 
public int CoinChange(int[] coins, int amount)
{
    int[] dp = Enumerable.Repeat(int.MaxValue, amount + 1).ToArray();
    dp[0] = 0;
    for (int i = 1; i <= amount; i++)
        foreach (int c in coins)
            if (c <= i && dp[i - c] != int.MaxValue)
                dp[i] = Math.Min(dp[i], dp[i - c] + 1);
    return dp[amount] == int.MaxValue ? -1 : dp[amount];
}
Python 
def coinChange(coins, amount):
    dp = [float('inf')] * (amount + 1)
    dp[0] = 0
    for i in range(1, amount + 1):
        for c in coins:
            if c <= i and dp[i - c] != float('inf'):

    return dp[amount] if dp[amount] != float('inf') else -1
?? Complexity: O(amount × |coins|) time, O(amount) space. Coin Change is the canonical "unbounded knapsack" — each coin can be used unlimited times.
Section 2 — Coin Change II (LeetCode #518)

Count the number of combinations (not permutations) of coins that sum to the amount.

KEY DIFFERENCE FROM #322: - #322: minimum coins (order doesn't matter, minimize count) - #518: count combinations (combinations, NOT permutations — [1,2] same as [2,1]) NAIVE (wrong) approach — count permutations: dp[i] += dp[i - c] for each coin c This counts [1,2] and [2,1] separately! CORRECT — enumerate coins in outer loop: For each coin c (outer loop ? locks in order): For each amount i from c to amount: dp[i] += dp[i - c] By iterating coins as the outer loop, we enforce that coin[0] always comes before coin[1] in any combination ? avoids duplicates! EXAMPLE: coins=[1,2,5], amount=5 Start: dp = [1,0,0,0,0,0] Process coin=1: dp[1] += dp[0] ? dp[1]=1 (one way: [1]) dp[2] += dp[1] ? dp[2]=1 ([1,1]) ... dp[5]=1 ([1,1,1,1,1]) Process coin=2: dp[2] += dp[0] ? dp[2]=2 ([1,1] or [2]) dp[3] += dp[1] ? dp[3]=2 ([1,1,1] or [2,1]) dp[4] += dp[2] ? dp[4]=3 ([1,1,1,1],[2,1,1],[2,2]) dp[5] += dp[3] ? dp[5]=4 Process coin=5: dp[5] += dp[0] ? dp[5]=5 ([5]) Answer: dp[5] = 4 ? ([1,1,1,1,1],[1,1,1,2],[1,2,2],[5])
C# 
public int Change(int amount, int[] coins)
{
    int[] dp = new int[amount + 1];
    dp[0] = 1;
    foreach (int c in coins)         // outer: coin
        for (int i = c; i <= amount; i++)  // inner: amount
            dp[i] += dp[i - c];
    return dp[amount];
}
Python 
def change(amount, coins):
    dp = [0] * (amount + 1)
    dp[0] = 1
    for c in coins:
        for i in range(c, amount + 1):
            dp[i] += dp[i - c]
    return dp[amount]
Section 3 — Longest Increasing Subsequence (LeetCode #300)

Find the length of the longest strictly increasing subsequence in an array.

O(n˛) DP Approach

STATE: dp[i] = length of LIS ending at index i RECURRENCE: dp[i] = 1 + max(dp[j]) for all j < i where nums[j] < nums[i] (extend the best increasing subsequence ending before i, if nums[j] is smaller) BASE: dp[i] = 1 (each element alone is a subsequence of length 1) ANSWER: max(dp[i]) for all i EXAMPLE: nums = [10, 9, 2, 5, 3, 7, 101, 18] dp[0]=1 (10) dp[1]=1 (9) — no j < 1 with nums[j] < 9 dp[2]=1 (2) — no j < 2 with nums[j] < 2 dp[3]=2 (2,5) — j=2: nums[2]=2 < 5, dp[2]+1=2 dp[4]=2 (2,3) — j=2: nums[2]=2 < 3, dp[2]+1=2 dp[5]=3 (2,5,7 or 2,3,7) — j=3: dp[3]+1=3; j=4: dp[4]+1=3 dp[6]=4 (2,5,7,101) — j=5: dp[5]+1=4 dp[7]=4 (2,5,7,18) — j=5: dp[5]+1=4 Answer: max(dp) = 4 ?
C# 
public int LengthOfLIS_N2(int[] nums)
{
    int n = nums.Length;
    int[] dp = Enumerable.Repeat(1, n).ToArray();
    for (int i = 1; i < n; i++)
        for (int j = 0; j < i; j++)
            if (nums[j] < nums[i])
                dp[i] = Math.Max(dp[i], dp[j] + 1);
    return dp.Max();
}
Python 
def length_of_lis_n2(nums):
    n  = len(nums)
    dp = [1] * n
    for i in range(1, n):
        for j in range(i):
            if nums[j] < nums[i]:
                dp[i] = max(dp[i], dp[j] + 1)
    return max(dp)

O(n log n) Binary Search Approach — Patience Sorting

PATIENCE SORTING — Build "piles" (tails array): tails[i] = smallest tail element of all increasing subsequences of length i+1 Process each number x: - Binary search tails for the first element >= x - If found at position j: replace tails[j] = x (keep the smallest tail) - If not found: append x (extends the longest subsequence) nums = [10, 9, 2, 5, 3, 7, 101, 18] x=10: tails=[] ? append ? tails=[10] x=9: bs([10],9)=0 ? replace ? tails=[9] x=2: bs([9],2)=0 ? replace ? tails=[2] x=5: bs([2],5)=1 ? append ? tails=[2,5] x=3: bs([2,5],3)=1? replace ? tails=[2,3] x=7: bs([2,3],7)=2? append ? tails=[2,3,7] x=101:bs([2,3,7],101)=3?append?tails=[2,3,7,101] x=18: bs([2,3,7,101],18)=3?replace?tails=[2,3,7,18] Length = len(tails) = 4 ? Note: tails is NOT the actual LIS! It's just used to track the length.
C# 
public int LengthOfLIS(int[] nums)
{
    var tails = new List<int>();
    foreach (int x in nums)
    {
        int lo = 0, hi = tails.Count;
        while (lo < hi)
        {
            int mid = lo + (hi - lo) / 2;
            if (tails[mid] < x) lo = mid + 1;
            else hi = mid;
        }
        if (lo == tails.Count) tails.Add(x);
        else tails[lo] = x;
    }
    return tails.Count;
}
Python 
import bisect

def lengthOfLIS(nums):
    tails = []
    for x in nums:
        pos = bisect.bisect_left(tails, x)
        if pos == len(tails): tails.append(x)
        else: tails[pos] = x
    return len(tails)  # O(n log n)
Section 4 — Word Break (LeetCode #139)

Given a string s and a dictionary wordDict, determine if s can be segmented into dictionary words.

STATE: dp[i] = True if s[0..i-1] can be segmented using wordDict RECURRENCE: dp[i] = any(dp[j] and s[j..i-1] in wordDict) for all j < i BASE: dp[0] = True (empty string — zero words) EXAMPLE: s = "leetcode", wordDict = ["leet", "code"] dp[0] = True dp[1]: j=0: dp[0] and s[0..0]="l" in dict? No dp[2]: j=0: "le"? No; j=1: dp[1]=F ... dp[4]: j=0: dp[0] and s[0..3]="leet" in dict? YES ? dp[4]=True dp[5]: j=0: "leetc"? No; j=4: dp[4] and s[4..4]="c"? No ... dp[8]: j=4: dp[4] and s[4..7]="code" in dict? YES ? dp[8]=True Answer: dp[8] = True ?
C# 
public bool WordBreak(string s, IList<string> wordDict)
{
    var dict = new HashSet<string>(wordDict);
    int n = s.Length;
    bool[] dp = new bool[n + 1];
    dp[0] = true;
    for (int i = 1; i <= n; i++)
        for (int j = 0; j < i; j++)
            if (dp[j] && dict.Contains(s.Substring(j, i - j)))
            { dp[i] = true; break; }
    return dp[n];
}
Python 
def wordBreak(s, wordDict):
    words = set(wordDict)
    n = len(s)
    dp = [False] * (n + 1)
    dp[0] = True
    for i in range(1, n + 1):
        for j in range(i):
            if dp[j] and s[j:i] in words:
                dp[i] = True; break
    return dp[n]
Section 5 — Jump Game II (LeetCode #45)

Find the minimum number of jumps to reach the last index. nums[i] = max jump length from i.

DP APPROACH — O(n˛): dp[i] = min jumps to reach index i dp[0] = 0 dp[i] = min(dp[j] + 1) for all j < i where j + nums[j] >= i GREEDY APPROACH — O(n): (much better!) Track: jumps, current_end (farthest we can reach with current jumps), farthest At each step i: farthest = max(farthest, i + nums[i]) When we reach current_end (used up the current jump budget): jumps += 1 current_end = farthest EXAMPLE: nums = [2, 3, 1, 1, 4] i=0: farthest=max(0,2)=2; end=0 ? reached end: jumps=1, end=2 i=1: farthest=max(2,4)=4 i=2: farthest=max(4,3)=4; end=2 ? reached end: jumps=2, end=4 Reached last index! Answer: 2 ? (0?1?4)
C# 
// Greedy O(n) — optimal
public int Jump(int[] nums)
{
    int jumps = 0, curEnd = 0, farthest = 0;
    for (int i = 0; i < nums.Length - 1; i++)
    {
        farthest = Math.Max(farthest, i + nums[i]);
        if (i == curEnd)
        {
            jumps++;
            curEnd = farthest;
        }
    }
    return jumps;
}
Python 
def jump(nums):
    jumps = cur_end = farthest = 0
    for i in range(len(nums) - 1):
        farthest = max(farthest, i + nums[i])
        if i == cur_end:
            jumps += 1
            cur_end = farthest
    return jumps
Section 6 — Decode Ways (LeetCode #91)

'A'?1, 'B'?2, ..., 'Z'?26. Count the number of ways to decode a string of digits.

STATE: dp[i] = number of ways to decode s[0..i-1] RECURRENCE: One-digit decode: if s[i-1] != '0': dp[i] += dp[i-1] Two-digit decode: if s[i-2..i-1] is in [10,26]: dp[i] += dp[i-2] BASE: dp[0] = 1, dp[1] = 1 if s[0] != '0' else 0 EXAMPLE: s = "226" dp[0] = 1 dp[1] = 1 (s[0]='2' != '0') dp[2]: one-digit s[1]='2': dp[2] += dp[1] = 1 two-digit s[0..1]="22" in [10,26]: dp[2] += dp[0] = 1 ? dp[2]=2 dp[3]: one-digit s[2]='6': dp[3] += dp[2] = 2 two-digit s[1..2]="26" in [10,26]: dp[3] += dp[1] = 1 ? dp[3]=3 Answer: 3 ? (2-2-6, 22-6, 2-26)
C# 
public int NumDecodings(string s)
{
    int n = s.Length;
    int[] dp = new int[n + 1];
    dp[0] = 1;
    dp[1] = s[0] != '0' ? 1 : 0;
    for (int i = 2; i <= n; i++)
    {
        if (s[i-1] != '0') dp[i] += dp[i-1];
        int two = (s[i-2] - '0') * 10 + (s[i-1] - '0');
        if (two >= 10 && two <= 26) dp[i] += dp[i-2];
    }
    return dp[n];
}
Python 
def numDecodings(s):
    n = len(s)
    dp = [0] * (n + 1)
    dp[0] = 1
    dp[1] = 1 if s[0] != '0' else 0
    for i in range(2, n + 1):
        if s[i-1] != '0': dp[i] += dp[i-1]
        two = int(s[i-2:i])
        if 10 <= two <= 26: dp[i] += dp[i-2]
    return dp[n]
Section 7 — Q&A Review
Q1: Coin Change vs Coin Change II — what's the key structural difference that forces the loop-order swap?
Coin Change (#322) asks for MINIMUM COINS — order of enumeration doesn't matter for correctness (we're taking the minimum of independent choices). Coin Change II (#518) asks for COMBINATIONS (not permutations) — we must avoid counting [1,2] and [2,1] as different. The trick: iterate coins in the outer loop and amounts in the inner loop. By fixing the coin order (processing coin c before looking at larger amounts), we guarantee that any combination is counted exactly once with coins in non-decreasing order. If you swap loops (amount outer, coin inner), you count permutations — e.g., for amount=3, coins=[1,2], you'd count [1,2] and [2,1] separately.
Q2: Why does the O(n log n) LIS algorithm (patience sorting / tails array) work? Is tails the actual LIS?
The tails array maintains the invariant: tails[i] = the smallest tail element among all increasing subsequences of length i+1 seen so far. The length of tails equals the LIS length. Why it works: we want to keep tails as small as possible at each position, to allow more numbers to extend later. When we encounter x and replace tails[j] with x (where tails[j] was the first element = x), we maintain this invariant. No: tails is NOT the actual LIS — its contents may be inconsistent (e.g., tails=[2,3,7,18] for the example, but the actual LIS might be [2,5,7,101] or [2,3,7,18]). To reconstruct the actual LIS, you need to maintain a parent array alongside.
Q3: In Word Break, what's the time complexity of the naive O(n˛) substring approach, and how can it be improved?
The naive approach: for each i (n iterations), for each j < i (n iterations), check if s[j..i] is in the dictionary (O(word_len) for hashing, or O(word_len˛) for comparison). Total: O(n˛ * L) where L = max word length. Improvement 1: Limit j to i - maxWordLen (skip substrings longer than any dict word). This gives O(n * maxWordLen). Improvement 2: Use a Trie built from wordDict — traverse backwards from position i, checking Trie for matches in O(maxWordLen) per position. Overall O(n * maxWordLen) with Trie. The O(n˛) formulation is typically accepted in interviews for n = 300.
Q4: Jump Game II — why does the greedy approach work? Why doesn't it miss the optimal solution?
The greedy invariant: at each jump, we always extend to the farthest reachable position. Proof by exchange argument: suppose the greedy makes jumps at positions p1, p2, p3, ... and an optimal solution uses jumps at q1, q2, q3, .... The greedy's first jump covers at least as far as any optimal first jump (it takes the maximum). So after one jump, the greedy is at least as far as optimal. By induction, after k jumps, greedy is at least as far as optimal. Therefore greedy uses at most as many jumps as optimal — it's optimal. The key insight: there's never a reason to jump to a shorter position — it can only hurt you by giving you a worse starting point for the next jump.
Q5: In Decode Ways, how do you handle leading zeros and the edge case of "30"?
Leading zeros are invalid: '0' by itself can't be decoded (there's no letter 'A' = 0). One-digit rule: if s[i-1] == '0', don't add dp[i-1] to dp[i] (cannot decode '0' alone). Two-digit rule: only add dp[i-2] if the two-digit number is in [10, 26]. For "30": two = 30, which is NOT in [10, 26], so we can't use two-digit decode. One-digit: s[1] = '0', so we can't decode '0' alone. Result: dp[2] = 0 (impossible). For "06": dp[1] = 0 (s[0]='0'), dp[2] = 0 + 0 = 0 (06=6 is NOT in [10,26], and single '6' can't extend dp[1]=0). Answer: 0. Always initialize dp[0] = 1 (empty string) and carefully handle '0' characters.
