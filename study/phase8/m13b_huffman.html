Section 1 — Huffman Coding: Optimal Prefix Codes

Given character frequencies, build a binary tree where each character is a leaf. The code for each character is the path from root to leaf (0=left, 1=right). Prefix-free means no code is a prefix of another — guaranteed by leaf placement. Huffman coding minimizes the total encoded length.

GOAL: Minimize total bits = S freq[c] × depth[c] GREEDY INSIGHT: Characters with HIGHER frequency should have SHORTER codes. ? Assign shorter paths (shallower tree depth) to frequent characters. ALGORITHM: 1. Insert all characters as leaf nodes into a min-heap (key=frequency). 2. While heap has more than 1 node: a. Extract two nodes with SMALLEST frequency: left, right. b. Create a new internal node with frequency = left.freq + right.freq. c. Insert internal node back into heap. 3. The remaining node is the root. EXAMPLE: a(45), b(13), c(12), d(16), e(9), f(5) Frequencies: f=5, e=9, b=13, c=12, d=16, a=45 Heap: [5,9,12,13,16,45] Step 1: Extract 5(f), 9(e) ? create 14(f,e). Heap: [12,13,14,16,45] Step 2: Extract 12(c), 13(b) ? create 25(c,b). Heap: [14,16,25,45] Step 3: Extract 14(f,e), 16(d) ? create 30. Heap: [25,30,45] Step 4: Extract 25(c,b), 30 ? create 55. Heap: [45,55] Step 5: Extract 45(a), 55 ? create 100 (root). Tree structure: 100 / \ 45(a) 55 / \ 25 30 / \ / \ 12(c) 13(b) 14 16(d) / \ 5(f) 9(e) Codes: a=0, c=100, b=101, f=1100, e=1101, d=111 Total bits = 45*1 + 13*3 + 12*3 + 16*3 + 9*4 + 5*4 = 45 + 39 + 36 + 48 + 36 + 20 = 224 bits
C# 
public class HuffmanNode : IComparable<HuffmanNode>
{
    public char? Ch; public int Freq;
    public HuffmanNode Left, Right;
    public int CompareTo(HuffmanNode other) => Freq - other.Freq;
}

public Dictionary<char, string> BuildHuffmanCodes(Dictionary<char, int> freqs)
{
    var heap = new SortedSet<HuffmanNode>(
        freqs.Select(kv => new HuffmanNode { Ch = kv.Key, Freq = kv.Value }));
    // Using PriorityQueue (.NET 6+) for correct min-heap:
    var pq = new PriorityQueue<HuffmanNode, int>();
    foreach (var kv in freqs) pq.Enqueue(new HuffmanNode { Ch=kv.Key, Freq=kv.Value }, kv.Value);
    while (pq.Count > 1)
    {
        var L = pq.Dequeue(); var R = pq.Dequeue();
        var parent = new HuffmanNode { Freq = L.Freq + R.Freq, Left = L, Right = R };
        pq.Enqueue(parent, parent.Freq);
    }
    var root = pq.Dequeue();
    var codes = new Dictionary<char, string>();
    Traverse(root, "", codes);
    return codes;
}
private void Traverse(HuffmanNode node, string path, Dictionary<char, string> codes)
{
    if (node == null) return;
    if (node.Ch.HasValue) { codes[node.Ch.Value] = path; return; }
    Traverse(node.Left, path + "0", codes);
    Traverse(node.Right, path + "1", codes);
}
Python 
import heapq

def huffman_codes(freqs):
    # heap entry: (freq, id, char_or_None, left, right)
    heap = [(f, i, c, None, None) for i, (c, f) in enumerate(freqs.items())]
    heapq.heapify(heap)
    counter = len(heap)
    while len(heap) > 1:
        f1, _, c1, l1, r1 = heapq.heappop(heap)
        f2, _, c2, l2, r2 = heapq.heappop(heap)
        heapq.heappush(heap, (f1+f2, counter, None, (f1,c1,l1,r1), (f2,c2,l2,r2)))
        counter += 1
    root = heap[0]
    codes = {}
    def traverse(node, path):
        f, _, c, left, right = node
        if c is not None: codes[c] = path or '0'; return
        if left:  traverse(left,  path + '0')
        if right: traverse(right, path + '1')
    traverse(root, '')
    return codes
Section 2 — Meeting Rooms II (LeetCode #253)

Given meeting intervals, find the minimum number of conference rooms needed.

GREEDY: Sort by start time. Use a min-heap tracking end times of ongoing meetings. - If earliest-ending meeting ends before new meeting starts ? reuse that room. - Otherwise ? allocate a new room. EXAMPLE: [(0,30),(5,10),(15,20)] Sort by start: same order. Start meeting (0,30): heap=[30], rooms=1 Start meeting (5,10): 30>5 ? new room. heap=[10,30], rooms=2 Start meeting (15,20): 10<15 ? reuse! Pop 10, push 20. heap=[20,30], rooms=2 Answer: 2 rooms ? ALTERNATIVE: Two sorted arrays (starts and ends) Count overlap at any given moment = max concurrent active meetings. Use two-pointer: for each start, if start[i] >= end[j], meeting ended, j++. Otherwise, open new room. COMPLEXITY: O(n log n)
C# 
// LeetCode #253 — Meeting Rooms II
public int MinMeetingRooms(int[][] intervals)
{
    var starts = intervals.Select(i => i[0]).OrderBy(x => x).ToArray();
    var ends   = intervals.Select(i => i[1]).OrderBy(x => x).ToArray();
    int rooms = 0, j = 0;
    for (int i = 0; i < intervals.Length; i++)
    {
        if (starts[i] >= ends[j]) j++;  // meeting ended, reuse room
        else rooms++;                   // need new room
    }
    return rooms;
}
Python 
def minMeetingRooms(intervals):
    starts = sorted(i[0] for i in intervals)
    ends   = sorted(i[1] for i in intervals)
    rooms = j = 0
    for s in starts:
        if s >= ends[j]: j += 1  # reuse a room
        else: rooms += 1          # new room needed
    return rooms
Section 3 — Non-overlapping Intervals (LeetCode #435)

Given intervals, find the minimum number of intervals to remove to make the rest non-overlapping.

KEY INSIGHT: This is EQUIVALENT to finding the maximum number of non-overlapping intervals (Activity Selection Problem!), then: answer = total_intervals - max_non_overlapping Sort by END TIME. Greedily keep intervals that don't overlap with last kept. Count kept intervals. Remove = n - kept. EXAMPLE: intervals = [[1,2],[2,3],[3,4],[1,3]] Sort by end: [1,2], [2,3], [1,3], [3,4] Keep [1,2]: last_end=2 Keep [2,3]: 2>=2 ? keep. last_end=3 [1,3]: 1<3 ? SKIP (overlap) Keep [3,4]: 3>=3 ? keep. last_end=4 Kept=3, Remove=4-3=1 ? NOTE: Sort by end (not start) is the standard. Ties in end time handled correctly.
C# 
// LeetCode #435 — Non-overlapping Intervals
public int EraseOverlapIntervals(int[][] intervals)
{
    Array.Sort(intervals, (a, b) => a[1] - b[1]);
    int kept = 1, lastEnd = intervals[0][1];
    for (int i = 1; i < intervals.Length; i++)
        if (intervals[i][0] >= lastEnd) { kept++; lastEnd = intervals[i][1]; }
    return intervals.Length - kept;
}
Python 
def eraseOverlapIntervals(intervals):
    intervals.sort(key=lambda x: x[1])
    kept, last_end = 1, intervals[0][1]
    for s, e in intervals[1:]:
        if s >= last_end: kept += 1; last_end = e
    return len(intervals) - kept

Section 5 — Q&A Review
Q1: Why is Huffman coding optimal? What property ensures no other prefix code can achieve fewer total bits?
Huffman coding produces an optimal prefix-free code. The greedy choice is correct because: (1) In any optimal tree, the two symbols with lowest frequency must be leaves at the same depth (otherwise we could swap a lower-frequency symbol with a deeper symbol to reduce total cost). (2) We can assume these two minimum-frequency symbols are siblings (share the same parent). The exchange argument: if in an optimal tree the two rarest symbols are NOT siblings, swap them with the actual deepest sibling pair — this doesn't increase total cost. (3) After merging the two rarest symbols into a "combined" symbol, the subproblem (n-1 symbols) is also optimally solved by recursion. This inductive argument proves Huffman's greedy construction produces the optimal code. The proof uses both the exchange argument and optimal substructure.
Q2: Meeting Rooms II — why does the two-pointer approach (sorted starts and ends) correctly count minimum rooms?
We process starts in sorted order. For each new meeting starting at starts[i], we check if the earliest-ending meeting (ends[j]) has already finished. If starts[i] >= ends[j]: that room is free — we "reuse" it (advance j). If starts[i] < ends[j]: all currently open meetings are still running — we need a new room. The number of rooms at any point equals the number of concurrent overlapping meetings. The maximum rooms count equals i - j + 1 at the peak, but since we only increment rooms when adding a new meeting and never decrement, rooms at the end = total rooms needed = peak concurrent meetings. The sorted arrays allow efficient O(1) comparison per event.
Q3: Why does Huffman coding assign shorter codes to higher-frequency characters?
Because total encoded length = S freq[c] × depth[c]. To minimize this sum, we want high-frequency characters to have small depth (short codes) and low-frequency characters to have large depth (long codes). This is exactly the weighted path-length minimization problem. The Huffman tree builds up from leaves: the two least frequent characters are merged first, making them deepest in the tree (longest codes). The most frequent character ends up closest to the root. This is essentially a greedy inversion of the priority: the min-heap ensures we always merge the two cheapest nodes, and the resulting tree automatically assigns code lengths inversely proportional to frequency.
Q4: When does the interval scheduling problem require DP instead of pure greedy?
When intervals have different values/weights (Weighted Job Scheduling). The greedy "sort by end time and pick non-overlapping" only maximizes the COUNT of non-overlapping intervals. If each interval has a profit, the problem is: maximize total profit of selected non-overlapping intervals. Greedy fails here because selecting a high-value interval might block many medium-value intervals whose combined value exceeds it. The DP approach: sort by end time. dp[i] = max profit using some subset of intervals 1..i. For each interval i: either exclude it (dp[i]=dp[i-1]) or include it (dp[i]=profit[i]+dp[j] where j is the last interval ending before interval i starts). Use binary search to find j in O(log n). Total O(n log n). This is LeetCode #1235.
Q5: What's the relationship between Non-overlapping Intervals (#435) and Activity Selection?
They are exactly equivalent: min_removals = n - max_non_overlapping_count. Activity Selection finds the maximum number of non-overlapping intervals (greedily by end time). If we keep max_non_overlapping intervals, we must remove the rest: n - max_non_overlapping. So solving #435 reduces to: (1) Sort by end time, (2) Greedily count non-overlapping intervals (standard Activity Selection), (3) Answer = n - count. The implementations look the same — the only difference is what you return: Activity Selection returns the selected intervals; #435 returns n minus the count of selected intervals.
