Section 1 — What is Dynamic Programming?

Dynamic Programming (DP) is an algorithmic technique that solves complex problems by breaking them down into simpler subproblems, solving each subproblem only once, and storing their solutions to avoid redundant work.

DP applies when a problem has two properties:

  1. Optimal Substructure: The optimal solution to the whole problem can be constructed from optimal solutions to subproblems.
  2. Overlapping Subproblems: The same subproblems are solved multiple times in a naive recursive approach.
📌 DP vs Divide & Conquer: Both break problems into subproblems. The difference: D&C subproblems are independent (merge sort: left and right halves don't share work). DP subproblems overlap — the same sub-computation appears many times. That's when caching pays off.
TechniqueSubproblemsExample
Divide & ConquerIndependent, non-overlappingMerge Sort, Binary Search
Dynamic ProgrammingOverlapping, sharedFibonacci, Shortest Path
GreedyNo subproblems — local choiceDijkstra, Interval Scheduling
Section 2 — The Fibonacci Problem: Naive vs DP

Fibonacci: F(n) = F(n-1) + F(n-2), with F(0)=0, F(1)=1.

NAIVE RECURSION — Exponential O(2^n): fib(5) ├── fib(4) │ ├── fib(3) │ │ ├── fib(2) │ │ │ ├── fib(1) ← leaf │ │ │ └── fib(0) ← leaf │ │ └── fib(1) ← DUPLICATE │ └── fib(2) ← DUPLICATE │ ├── fib(1) │ └── fib(0) └── fib(3) ← DUPLICATE ├── fib(2) │ ├── fib(1) │ └── fib(0) └── fib(1) fib(3) computed 2 times, fib(2) computed 3 times, fib(1) computed 5 times... Total calls for fib(n): O(2^n)
Section 3 — Top-Down: Memoization

Keep the recursive structure, but cache results in a dictionary/array. Before computing, check the cache. After computing, store in cache.

MEMOIZED RECURSION — O(n) time, O(n) space: First call to fib(5): fib(5) → cache miss → compute fib(4) + fib(3) fib(4) → cache miss → compute fib(3) + fib(2) fib(3) → cache miss → compute fib(2) + fib(1) fib(2) → cache miss → compute fib(1) + fib(0) = 1 cache[2] = 1 fib(3) = 1 + 1 = 2, cache[3] = 2 fib(2) → CACHE HIT → return 1 fib(4) = 2 + 1 = 3, cache[4] = 3 fib(3) → CACHE HIT → return 2 fib(5) = 3 + 2 = 5, cache[5] = 5 Total unique computations: n (one per distinct subproblem)
C# 
public class FibMemo
{
    private Dictionary<int, long> memo = new();

    public long Fib(int n)
    {
        if (n <= 1) return n;
        if (memo.TryGetValue(n, out long cached)) return cached;
        return memo[n] = Fib(n-1) + Fib(n-2);
    }
}

// C# 8+ with array memo (faster than Dictionary):
public long FibMemoArray(int n, long[] dp = null)
{
    dp ??= Enumerable.Repeat(-1L, n+1).ToArray();
    if (n <= 1) return n;
    if (dp[n] != -1) return dp[n];
    return dp[n] = FibMemoArray(n-1, dp) + FibMemoArray(n-2, dp);
}

from functools import lru_cache

# Method 1: @lru_cache — Python's built-in memoization decorator
@lru_cache(maxsize=None)
def fib(n):
    if n <= 1: return n
    return fib(n-1) + fib(n-2)

# Method 2: Manual memo dict
def fib_memo(n, memo={}):
    if n <= 1: return n
    if n not in memo:
        memo[n] = fib_memo(n-1, memo) + fib_memo(n-2, memo)
    return memo[n]

Section 4 — Bottom-Up: Tabulation

Instead of recursing top-down and caching, build the table bottom-up iteratively. Start from the base cases, fill the table in the correct order until you reach the desired answer.

TABULATION — O(n) time, O(n) space: n = 6 dp[0] = 0 (base case) dp[1] = 1 (base case) dp[2] = dp[1] + dp[0] = 1 dp[3] = dp[2] + dp[1] = 2 dp[4] = dp[3] + dp[2] = 3 dp[5] = dp[4] + dp[3] = 5 dp[6] = dp[5] + dp[4] = 8 Table: index: 0 1 2 3 4 5 6 dp[]: 0 1 1 2 3 5 8 ↑ ↑ ← computed from previous two
C# 
public long FibTabulation(int n)
{
    if (n <= 1) return n;
    long[] dp = new long[n + 1];
    dp[0] = 0; dp[1] = 1;
    for (int i = 2; i <= n; i++)
        dp[i] = dp[i-1] + dp[i-2];
    return dp[n];
}
Python 
def fib_tab(n):
    if n <= 1: return n
    dp = [0] * (n + 1)
    dp[1] = 1
    for i in range(2, n + 1):
        dp[i] = dp[i-1] + dp[i-2]
    return dp[n]
Section 5 — Space Optimization

Many DP problems only need the last few values — not the entire table. For Fibonacci, dp[i] only depends on dp[i-1] and dp[i-2], so we only need 2 variables.

C# 
public long FibSpaceOpt(int n)
{
    if (n <= 1) return n;
    long prev2 = 0, prev1 = 1;
    for (int i = 2; i <= n; i++)
    {
        long cur = prev1 + prev2;
        prev2 = prev1; prev1 = cur;
    }
    return prev1;  // O(1) space!
}
Python 
def fib_opt(n):
    if n <= 1: return n
    a, b = 0, 1
    for _ in range(2, n + 1):
        a, b = b, a + b
    return b  # O(1) space

Section 6 — DP Problem Recognition

Ask yourself these questions to identify if a problem is solvable with DP:

DP RECOGNITION CHECKLIST ✅ "Optimization" language: maximum, minimum, longest, shortest, count ways, best ✅ "Can I solve this by making a sequence of choices?" ✅ "Does the problem decompose into smaller versions of itself?" ✅ "Are there only polynomially many distinct subproblems?" (not exponential) ✅ "Does solving subproblems help? (are they reused?)" 🚫 NOT DP if: - Subproblems are completely independent (use D&C or simple recursion) - Problem is a simple greedy (one pass, no lookahead) - Problem requires tracking exponentially many states COMMON DP PATTERNS: 1. Linear sequences: dp[i] = f(dp[i-1], dp[i-2], ...) → Fibonacci, Climbing Stairs, House Robber, Jump Game 2. Subsequences: dp[i][j] = f(dp[i-1][j], dp[i][j-1], dp[i-1][j-1]) → LCS, Edit Distance, LIS, Knapsack 3. Partition: dp[i] = max/min over all splits dp[j] + dp[i-j+1] → Matrix Chain Multiplication, Burst Balloons 4. Trees/graphs: dp on subtrees, states per node → Max Independent Set on Tree, Diameter 5. Bitmask DP: dp[mask] = state for subset "mask" → Traveling Salesman, Matching
Section 7 — Framework: 4 Steps to Any DP Problem

Every DP solution follows this four-step framework:

THE 4-STEP DP FRAMEWORK Step 1: DEFINE THE STATE What does dp[i] represent? (Or dp[i][j] for 2D) Be precise: dp[i] = "the answer for the subproblem on input i" Example: dp[i] = "max profit from houses 0..i, given we can't rob adjacent houses" Step 2: WRITE THE RECURRENCE Express dp[i] in terms of previously computed values. Example: dp[i] = max(dp[i-1], dp[i-2] + nums[i]) → Either skip house i (take dp[i-1]) or rob it (dp[i-2] + nums[i]) Step 3: IDENTIFY BASE CASES What are the smallest subproblems we can answer directly (without recursion)? Example: dp[0] = nums[0], dp[1] = max(nums[0], nums[1]) Step 4: DETERMINE COMPUTATION ORDER Bottom-up: which order to fill the table? (Ensure that when we compute dp[i], all values dp[i] depends on are already computed.) Example: left to right i=2..n
💡 Interview Strategy: Always state the 4 steps explicitly before coding. Say: "Let dp[i] be... The recurrence is... Base cases are... I'll fill the table left to right." This shows structured thinking even before writing a single line of code.
Section 8 — LeetCode Walkthroughs

LeetCode #70 — Climbing Stairs (Easy)

You can climb 1 or 2 steps at a time. How many distinct ways to reach step n?

APPLY THE 4-STEP FRAMEWORK: State: dp[i] = number of ways to reach step i Recurrence: dp[i] = dp[i-1] + dp[i-2] (from step i-1, take 1 step) OR (from step i-2, take 2 steps) Base: dp[1] = 1, dp[2] = 2 Order: i = 3 to n n=5: dp = [_, 1, 2, 3, 5, 8] dp[3] = dp[2]+dp[1] = 3 dp[4] = dp[3]+dp[2] = 5 dp[5] = dp[4]+dp[3] = 8 ways ✓ This is EXACTLY Fibonacci! (with different base cases)
C# 
public int ClimbStairs(int n)
{
    if (n <= 2) return n;
    int prev2 = 1, prev1 = 2;
    for (int i = 3; i <= n; i++)
    {
        int cur = prev1 + prev2;
        prev2 = prev1; prev1 = cur;
    }
    return prev1;
}
Python 
def climbStairs(n):
    if n <= 2: return n
    a, b = 1, 2
    for _ in range(3, n+1): a, b = b, a+b
    return b

LeetCode #198 — House Robber (Medium)

Rob houses to maximize money. Cannot rob two adjacent houses.

APPLY THE 4-STEP FRAMEWORK: State: dp[i] = max money from houses 0..i Recurrence: dp[i] = max(dp[i-1], dp[i-2] + nums[i]) - dp[i-1]: skip house i, best from 0..i-1 - dp[i-2]+nums[i]: rob house i, can't use house i-1, best from 0..i-2 Base: dp[0] = nums[0], dp[1] = max(nums[0], nums[1]) Order: i = 2 to n-1 nums = [2, 7, 9, 3, 1] dp[0] = 2 dp[1] = max(2, 7) = 7 dp[2] = max(dp[1], dp[0]+9) = max(7, 11) = 11 dp[3] = max(dp[2], dp[1]+3) = max(11, 10) = 11 dp[4] = max(dp[3], dp[2]+1) = max(11, 12) = 12 ✓ (rob houses 0,2,4: 2+9+1=12)
C# 
public int Rob(int[] nums)
{
    if (nums.Length == 1) return nums[0];
    int prev2 = nums[0], prev1 = Math.Max(nums[0], nums[1]);
    for (int i = 2; i < nums.Length; i++)
    {
        int cur = Math.Max(prev1, prev2 + nums[i]);
        prev2 = prev1; prev1 = cur;
    }
    return prev1;
}
Python 
def rob(nums):
    if not nums: return 0
    if len(nums) == 1: return nums[0]
    a, b = nums[0], max(nums[0], nums[1])
    for i in range(2, len(nums)):
        a, b = b, max(b, a + nums[i])
    return b
Section 9 — Q&A Review
Q1: What are the two essential properties a problem must have for DP to apply?
1. Optimal Substructure: The optimal solution of the problem can be built from optimal solutions to its subproblems. Example: The shortest path from A to C via B = shortest path A→B + shortest path B→C. 2. Overlapping Subproblems: The same subproblems are re-encountered multiple times during recursion. Example: fib(3) is needed to compute both fib(4) and fib(5). Without overlap, caching doesn't help — that's why Merge Sort doesn't benefit from memoization (its subproblems don't overlap).
Q2: What are the pros and cons of memoization (top-down) vs tabulation (bottom-up)?
Memoization (top-down): Pros: natural recursive structure, only computes subproblems that are actually needed (lazy evaluation), easier to implement for some problems (trees). Cons: recursive call stack overhead, risk of stack overflow for large n (Python default recursion limit ~1000), slightly slower due to function call overhead + dictionary lookups. Tabulation (bottom-up): Pros: no recursion overhead, no stack overflow risk, easier space optimization (sliding window over table), typically faster in practice. Cons: must carefully determine fill order, always computes ALL subproblems even if not needed. In interviews: tabulation is generally preferred for linear/grid problems; memoization for tree/graph problems.
Q3: In House Robber, why is dp[i-2] + nums[i] the correct recurrence (not dp[i-1] + nums[i])?
If we rob house i, we cannot rob house i-1 (adjacent restriction). So we must look at the best solution ending at house i-2 or earlier. dp[i-2] already represents the maximum we can earn from houses 0..i-2 (which is the best we can do without using house i-1). Using dp[i-1] would mean we might have robbed house i-1 in that solution, which would make robbing house i illegal. Note: dp[i-2] includes the possibility of NOT robbing house i-2 either — it's the maximum over all valid choices in the range 0..i-2.
Q4: How do you determine the correct "fill order" for a bottom-up DP table?
Look at the recurrence: dp[i] depends on which previous values? Those values must be filled before dp[i]. For 1D DP: if dp[i] depends on dp[i-1], dp[i-2], fill left to right (i=2 to n). For 2D DP (e.g., dp[i][j] = f(dp[i-1][j], dp[i][j-1])): fill row by row, left to right. For interval DP (dp[i][j] depends on dp[i][k] and dp[k+1][j] for i≤k<j): fill by increasing length of interval. The key rule: draw a dependency arrow from dp[i] to each value it depends on. Fill in topological order of those dependencies.
Q5: When should you use space optimization in DP, and what's the general technique?
Use space optimization when dp[i] depends only on the last k rows/values (k is small and constant), and you don't need to reconstruct the actual path/solution. The technique: instead of keeping the full dp[0..n] array, keep only dp[i-1] (and dp[i-2] if needed) as separate variables. For 2D DP where dp[i][j] depends on dp[i-1][*], you only need to keep two rows (current and previous), reducing space from O(n*m) to O(m). Caveat: if you need to reconstruct the path (e.g., actual subsequence, not just length), you CANNOT space-optimize — you need the full table for backtracking.
