Section 1 — The Range Query Problem

Suppose you have an array of numbers and you need to repeatedly:

  • Query: "What is the sum (or min, or max) of elements from index L to R?"
  • Update: "Change the value at index i to v."

How efficiently can you answer these two operations?

ApproachBuildQueryUpdateSpace
Brute force (iterate)O(1)O(n)O(1)O(1)
Prefix sum arrayO(n)O(1)O(n) rebuildO(n)
Segment TreeO(n)O(log n)O(log n)O(n)
Fenwick Tree (BIT)O(n)O(log n)O(log n)O(n)
Sparse TableO(n log n)O(1)Not supportedO(n log n)
💡 When to use Segment Tree: You need BOTH efficient range queries AND point (or range) updates. If updates are rare, use prefix sums. If no updates at all, use Sparse Table for O(1) queries. If only prefix sums matter, use a Fenwick Tree (simpler code).
🌍 Real-World Applications:
  • Database engines — Range aggregate queries (SUM, MIN, MAX) with row updates
  • Competitive programming — The single most-used advanced data structure in ICPC
  • Gaming — Tracking score ranges, health bars across ranges of game entities
  • Finance — Stock price range queries (highest/lowest in a date range) with daily updates
Section 2 — Segment Tree Structure

A Segment Tree is a binary tree where each node stores the aggregate (sum/min/max) of a contiguous subarray (a "segment"). The root covers the entire array [0, n-1]. Each internal node splits its range in half.

ARRAY: [2, 4, 3, 1, 6, 7, 2, 5] (indices 0–7) SEGMENT TREE (SUM): [0,7]: 30 / \ [0,3]: 10 [4,7]: 20 / \ / \ [0,1]:6 [2,3]:4 [4,5]:13 [6,7]:7 / \ / \ / \ / \ [0]:2 [1]:4 [2]:3 [3]:1 [4]:6 [5]:7 [6]:2 [7]:5 STORAGE (1-indexed array): Node 1 = root = [0,7] Node 2 = left child of 1 = [0,3] Node 3 = right child of 1 = [4,7] ... Node k: left child = 2k, right child = 2k+1, parent = k/2 Array representation (1-indexed, size 4n): tree[1]=30 tree[2]=10 tree[3]=20 tree[4]=6 tree[5]=4 tree[6]=13 tree[7]=7 tree[8]=2 tree[9]=4 tree[10]=3 tree[11]=1 tree[12]=6 tree[13]=7 tree[14]=2 tree[15]=5
📐 Array Size: The tree is stored in an array of size 4×n. This ensures enough space even for non-power-of-2 sizes. Alternatively, size 2×(next power of 2) works, but 4n is the safe universal choice for any n.
Section 3 — Build: O(n)

Build the segment tree by recursively computing aggregate values from leaves (individual elements) upward to the root. Each internal node = combine(left child, right child).

BUILD WALKTHROUGH — Array [2, 4, 3, 1] build(node=1, start=0, end=3): build(2, 0, 1): build(4, 0, 0): tree[4] = arr[0] = 2 (leaf) build(5, 1, 1): tree[5] = arr[1] = 4 (leaf) tree[2] = tree[4] + tree[5] = 6 build(3, 2, 3): build(6, 2, 2): tree[6] = arr[2] = 3 (leaf) build(7, 3, 3): tree[7] = arr[3] = 1 (leaf) tree[3] = tree[6] + tree[7] = 4 tree[1] = tree[2] + tree[3] = 10 Final tree array: [_, 10, 6, 4, 2, 4, 3, 1] (index 0 unused)
C# 
void Build(int node, int start, int end)
{
    if (start == end)
    {
        tree[node] = arr[start];    // leaf node = array element
        return;
    }
    int mid = (start + end) / 2;
    Build(2 * node,     start, mid);   // build left subtree
    Build(2 * node + 1, mid+1, end); // build right subtree
    tree[node] = tree[2*node] + tree[2*node+1];  // combine children
}

Section 4 — Range Query: O(log n)

To query sum/min/max of range [L, R], recursively check each node:

  • No overlap: node's range is completely outside [L,R] → return identity (0 for sum, ∞ for min, -∞ for max)
  • Total overlap: node's range is completely inside [L,R] → return this node's value
  • Partial overlap: split and query both children, combine results
QUERY [1, 5] on array [2, 4, 3, 1, 6, 7, 2, 5] (sum = 4+3+1+6+7 = 21) query(1, [0,7], L=1, R=5): PARTIAL → recurse query(2, [0,3], L=1, R=5): PARTIAL → recurse query(4, [0,1], L=1, R=5): PARTIAL → recurse query(8, [0,0], L=1, R=5): NO OVERLAP → return 0 (0 < 1) query(9, [1,1], L=1, R=5): TOTAL → return 4 return 0 + 4 = 4 query(5, [2,3], L=1, R=5): TOTAL → return 4 return 4 + 4 = 8 query(3, [4,7], L=1, R=5): PARTIAL → recurse query(6, [4,5], L=1, R=5): TOTAL → return 13 query(7, [6,7], L=1, R=5): NO OVERLAP → return 0 (6 > 5) return 13 + 0 = 13 return 8 + 13 = 21 ✓
C# 
int Query(int node, int start, int end, int l, int r)
{
    if (r < start || end < l) return 0;          // no overlap → identity for sum
    if (l <= start && end <= r) return tree[node];   // total overlap
    int mid = (start + end) / 2;
    int left  = Query(2*node,   start, mid,  l, r);
    int right = Query(2*node+1, mid+1, end, l, r);
    return left + right;   // combine: use Math.Min for min-query, Math.Max for max-query
}
Python 
def query(node, start, end, l, r):
    if r < start or end < l: return 0           # no overlap → identity for sum
    if l <= start and end <= r: return tree[node]  # total overlap
    mid   = (start + end) // 2
    left  = query(2*node,   start, mid,  l, r)
    right = query(2*node+1, mid+1, end, l, r)
    return left + right   # use min/max for other query types
Section 5 — Point Update: O(log n)

When arr[i] changes by delta, all ancestor nodes in the tree must be updated. Walk down to the leaf, update it, and propagate the change back up.

POINT UPDATE — Change arr[3] from 1 to 5 (delta = +4) update(1, [0,7], idx=3, delta=4): mid=3 → go right: update(3, [4,7], idx=3, delta=4)? No, 3 < 4, go left mid=1 → go right: update(2, [0,3], idx=3, delta=4) mid=1 → go right: update(5, [2,3], idx=3, delta=4) mid=2 → go right: update(11, [3,3], idx=3, delta=4) LEAF: tree[11] += 4 (1 → 5) tree[5] = tree[10] + tree[11] = 3 + 5 = 8 tree[2] = tree[4] + tree[5] = 6 + 8 = 14 tree[1] = tree[2] + tree[3] = 14 + 20 = 34
C# 
void Update(int node, int start, int end, int idx, int delta)
{
    if (start == end)
    {
        arr[idx]   += delta;         // update original array
        tree[node] += delta;         // update leaf
        return;
    }
    int mid = (start + end) / 2;
    if (idx <= mid) Update(2*node,   start, mid,  idx, delta);
    else            Update(2*node+1, mid+1, end, idx, delta);
    tree[node] = tree[2*node] + tree[2*node+1];  // recompute from children
}

Section 6 — Range Update with Lazy Propagation

A naive range update (e.g., "add 3 to all elements from index 2 to 7") would require O(n) point updates. Lazy propagation defers updates — store the pending update in a lazy[] array and apply it only when a node is actually visited.

LAZY PROPAGATION CONCEPT lazy[node] = "pending value to add to ALL elements in this node's range" RANGE UPDATE: add val to all elements in [l, r] If total overlap: tree[node] += val * (end - start + 1) (sum: add to each) lazy[node] += val (defer to children) If partial: PUSH DOWN lazy first, then recurse PUSH DOWN (propagate lazy to children): If lazy[node] != 0: Apply lazy[node] to both children (update tree[] and lazy[]) Reset lazy[node] = 0 Result: each operation is still O(log n) amortized — the "lazy" deferred work is bounded because each node is pushed down at most once per query.
C# 
// Lazy segment tree — range add + range sum
void PushDown(int node, int start, int end)
{
    if (lazy[node] == 0) return;
    int mid = (start + end) / 2;
    int lc = 2*node, rc = 2*node+1;
    // Apply pending update to both children
    tree[lc] += lazy[node] * (mid - start + 1);
    lazy[lc] += lazy[node];
    tree[rc] += lazy[node] * (end - mid);
    lazy[rc] += lazy[node];
    lazy[node] = 0;    // clear
}

void RangeUpdate(int node, int start, int end, int l, int r, int val)
{
    if (r < start || end < l) return;
    if (l <= start && end <= r)
    {
        tree[node] += val * (end - start + 1);   // total overlap
        lazy[node] += val;
        return;
    }
    PushDown(node, start, end);          // must push before recursing
    int mid = (start + end) / 2;
    RangeUpdate(2*node,   start, mid,  l, r, val);
    RangeUpdate(2*node+1, mid+1, end, l, r, val);
    tree[node] = tree[2*node] + tree[2*node+1];
}

long LazyQuery(int node, int start, int end, int l, int r)
{
    if (r < start || end < l) return 0;
    if (l <= start && end <= r) return tree[node];
    PushDown(node, start, end);          // push before reading children
    int mid = (start + end) / 2;
    return LazyQuery(2*node, start, mid, l, r) + LazyQuery(2*node+1, mid+1, end, l, r);
}
Python 
# Lazy segment tree — range add + range sum
def push_down(node, start, end):
    if lazy[node] == 0: return
    mid = (start + end) // 2
    lc, rc = 2*node, 2*node+1
    tree[lc] += lazy[node] * (mid - start + 1); lazy[lc] += lazy[node]

def range_update(node, start, end, l, r, val):
    if r < start or end < l: return
    if l <= start and end <= r:
        tree[node] += val * (end - start + 1)   # total overlap
        lazy[node] += val; return
    push_down(node, start, end)   # must push before recursing
    mid = (start + end) // 2
    range_update(2*node,   start, mid,  l, r, val)
    range_update(2*node+1, mid+1, end, l, r, val)
    tree[node] = tree[2*node] + tree[2*node+1]

def lazy_query(node, start, end, l, r):
    if r < start or end < l: return 0
    if l <= start and end <= r: return tree[node]
    push_down(node, start, end)   # push before reading children
    mid = (start + end) // 2
    return lazy_query(2*node, start, mid, l, r) + lazy_query(2*node+1, mid+1, end, l, r)

7.1 — Sum Segment Tree (Point Update + Range Query)

C# 
public class SegmentTree
{
    private readonly int n;
    private readonly int[] tree;   // 1-indexed, size 4n
    private readonly int[] arr;

    public SegmentTree(int[] nums)
    {
        n    = nums.Length;
        arr  = (int[])nums.Clone();
        tree = new int[4 * n];
        Build(1, 0, n - 1);
    }

    private void Build(int node, int s, int e)
    {
        if (s == e) { tree[node] = arr[s]; return; }
        int m = (s + e) / 2;
        Build(2*node, s, m);
        Build(2*node+1, m+1, e);
        tree[node] = tree[2*node] + tree[2*node+1];
    }

    /// Update arr[idx] to newVal — O(log n)
    public void Update(int idx, int newVal)
    {
        int delta = newVal - arr[idx];
        arr[idx] = newVal;
        UpdateHelper(1, 0, n-1, idx, delta);
    }

    private void UpdateHelper(int node, int s, int e, int idx, int delta)
    {
        if (s == e) { tree[node] += delta; return; }
        int m = (s + e) / 2;
        if (idx <= m) UpdateHelper(2*node,   s,   m, idx, delta);
        else          UpdateHelper(2*node+1, m+1, e, idx, delta);
        tree[node] = tree[2*node] + tree[2*node+1];
    }

    /// Sum of arr[l..r] — O(log n)
    public int RangeSum(int l, int r) => QueryHelper(1, 0, n-1, l, r);

    private int QueryHelper(int node, int s, int e, int l, int r)
    {
        if (r < s || e < l) return 0;
        if (l <= s && e <= r) return tree[node];
        int m = (s + e) / 2;
        return QueryHelper(2*node, s, m, l, r)
             + QueryHelper(2*node+1, m+1, e, l, r);
    }
}

// Usage
var st = new SegmentTree(new int[] {2,4,3,1,6,7,2,5});
Console.WriteLine(st.RangeSum(1, 5));   // 21  (4+3+1+6+7)
st.Update(3, 5);                         // change index 3 from 1 to 5
Console.WriteLine(st.RangeSum(1, 5));   // 25  (4+3+5+6+7)
Python 
class SegmentTree:
    def __init__(self, nums):
        self.n = len(nums)
        self.arr = nums[:]
        self.tree = [0] * (4 * self.n)
        self._build(1, 0, self.n - 1)

    def _build(self, node, s, e):
        if s == e: self.tree[node] = self.arr[s]; return
        m = (s + e) // 2
        self._build(2*node, s, m)
        self._build(2*node+1, m+1, e)
        self.tree[node] = self.tree[2*node] + self.tree[2*node+1]

    def update(self, idx, new_val):
        delta = new_val - self.arr[idx]; self.arr[idx] = new_val
        self._update(1, 0, self.n-1, idx, delta)

    def _update(self, node, s, e, idx, delta):
        if s == e: self.tree[node] += delta; return
        m = (s + e) // 2
        if idx <= m: self._update(2*node, s, m, idx, delta)
        else:        self._update(2*node+1, m+1, e, idx, delta)
        self.tree[node] = self.tree[2*node] + self.tree[2*node+1]

    def range_sum(self, l, r): return self._query(1, 0, self.n-1, l, r)

    def _query(self, node, s, e, l, r):
        if r < s or e < l: return 0
        if l <= s and e <= r: return self.tree[node]
        m = (s + e) // 2
        return self._query(2*node, s, m, l, r) + self._query(2*node+1, m+1, e, l, r)

class MinSegTree:
    def __init__(self, nums):
        self.n = len(nums)
        self.tree = [float('inf')] * (4 * self.n)
        self._build(1, 0, self.n - 1, nums)

        self._build(2*nd, s, m, a); self._build(2*nd+1, m+1, e, a)

    def _q(self, nd, s, e, l, r):
        if r < s or e < l: return float('inf')   # identity for min
        if l <= s and e <= r: return self.tree[nd]
        m = (s + e) // 2
        return min(self._q(2*nd, s, m, l, r), self._q(2*nd+1, m+1, e, l, r))

LeetCode #218 — The Skyline Problem (Hard)

Given the locations and heights of all buildings, return the skyline outline.

📝 Approach: Coordinate compress the x-values. Use a segment tree to track the maximum height over any x-range, sweeping from left to right with building start/end events. At each event, query max height, compare with previous — if different, it's a key point.

This problem showcases the segment tree's power for "max over a range" queries with dynamic updates, but the full implementation is complex (50+ lines). The structure of the solution is: events → sort → sweep with max-segtree → detect height changes → output key points.

LeetCode #315 — Count of Smaller Numbers After Self (Hard)

For each element, count how many elements to its right are smaller.

📝 Approach: Coordinate compress values to range [0, n-1]. Process array right to left. For each element at position i with compressed value v: query segment tree for sum in [0, v-1] (count of smaller values seen so far), then update position v. Time: O(n log n). This is a classic "merge sort tree" or "BIT" problem, but segment tree on compressed coordinates also works.
C# 
public IList<int> CountSmaller(int[] nums)
{
    // Coordinate compress
    var sorted = nums.Distinct().OrderBy(x => x).ToArray();
    var rank = new Dictionary<int,int>();
    for(int i=0;i<sorted.Length;i++) rank[sorted[i]]=i+1;  // 1-indexed
    int m = sorted.Length;
    int[] tree = new int[m+1];  // BIT (Fenwick) is cleaner here

    void Add(int i) { for;(int j=i;j<=m;j+=j&(-j)) tree[j]++; }
    int  Sum(int i) { int s=0; for(int j=i;j>0;j-=j&(-j)) s+=tree[j]; return s; }

    var res = new int[nums.Length];
    for(int i=nums.Length-1;i>=0;i--)
    {
        res[i] = Sum(rank[nums[i]] - 1);  // count smaller to right
        Add(rank[nums[i]]);
    }
    return res;
}
// Note: Using Fenwick Tree here — simpler for this specific pattern.
// Time: O(n log n)  Space: O(n)
Python 
def count_smaller(nums: list) -> list:
    # Coordinate compress
    sorted_u = sorted(set(nums))
    rank = {v: i+1 for i, v in enumerate(sorted_u)}  # 1-indexed
    m = len(sorted_u)
    tree = [0] * (m + 1)

    def add(i):
        while i <= m: tree[i] += 1; i += i & (-i)

    def prefix_sum(i):
        s = 0
        while i > 0: s += tree[i]; i -= i & (-i)
        return s

    res = [0] * len(nums)
    for i in range(len(nums) - 1, -1, -1):
        res[i] = prefix_sum(rank[nums[i]] - 1)  # count smaller to right
        add(rank[nums[i]])
    return res

Section 9 — Q&A Review
Q1: Why is the segment tree array sized 4n and not 2n or 2×(next power of 2)?
For a power-of-2 sized array, 2n nodes suffice. But for arbitrary n, the tree can be "unbalanced" at the last level — some branches go one level deeper than others. In the worst case this requires up to 4n nodes. For example, n=3: the left subtree has range [0,1] (2 leaves) and the right has [2,2] (1 leaf), so the tree has 7 nodes = ~2.3n, but for n=5 you can get close to 4n. Using 4n is the universal safe upper bound. Using 2×(next power of 2) is equivalent but requires computing the next power of 2.
Q2: What's the key insight of lazy propagation? Why is it still O(log n) per operation?
Without lazy propagation, a range update on [L,R] could require updating O(n) leaves individually, making it O(n). Lazy propagation says: "when I have total overlap, record the pending update in the lazy array and update only this node's aggregate, DON'T recurse to children yet." The children will be updated lazily when they are actually visited. Since each node is visited at most O(log n) times per query (due to the O(log n) segments that cover any range), the total work per operation remains O(log n) amortized.
Q3: How would you adapt the segment tree for a MINIMUM query instead of SUM?
Two changes: (1) In build/update: tree[node] = Math.Min(tree[2*node], tree[2*node+1]) instead of addition. (2) In query: return int.MaxValue (identity for min) for the no-overlap case instead of 0 (identity for sum). The structural logic of build/query/update is identical — only the "combine" operation and identity value change. This is why segment trees are so versatile — they implement a "monoid" over the combine operation.
Q4: When would you choose a Fenwick Tree (BIT) over a Segment Tree?
Choose Fenwick Tree when: (1) You only need prefix queries (not arbitrary [L,R] ranges — though you can compute range sum as prefixSum(R) - prefixSum(L-1)), (2) The operation must be invertible (sum is invertible via subtraction; min is NOT), (3) You want simpler, shorter code with better cache performance. Choose Segment Tree when: (1) You need range min/max (not invertible, can't use Fenwick), (2) You need range updates with lazy propagation, (3) You need to store complex information per node (e.g., count + sum both), (4) You need segment-level queries that aren't decomposable into prefix queries.
Q5: What does "coordinate compression" mean and why is it needed with segment trees?
Coordinate compression maps a sparse set of values to a dense range [0, m-1] where m is the number of distinct values. For example, [100, 50000, 3, 99999] → [1, 2, 0, 3] after compression. It's needed when values can be up to 10^9 but there are only n ≤ 10^5 distinct values — you can't create a segment tree over index 0 to 10^9 (too much memory). Instead, compress to [0, n-1] and build a tree of size 4n. The compressed index is just the rank of each value in the sorted unique list. Binary search is used to look up the compressed index in O(log n).
