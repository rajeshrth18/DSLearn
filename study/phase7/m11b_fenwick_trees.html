Section 1 — The Binary Indexed Tree Idea

A Fenwick Tree (also called a Binary Indexed Tree or BIT) is a data structure that supports prefix sum queries and point updates in O(log n) time using only O(n) space — and with remarkably simple code.

The key insight: instead of storing individual values, each index i in the BIT array stores the sum of a specific range of original array elements. The range size is determined by the lowest set bit of i.

FENWICK TREE STRUCTURE — Array [1, 3, 5, 7, 9, 11, 13, 15] (1-indexed) Index (binary): 1(001) 2(010) 3(011) 4(100) 5(101) 6(110) 7(111) 8(1000) Low-bit (i&-i): 1 2 1 4 1 2 1 8 Range covered: [1,1] [1,2] [3,3] [1,4] [5,5] [5,6] [7,7] [1,8] BIT values: bit[1] = arr[1] = 1 bit[2] = arr[1]+arr[2] = 4 bit[3] = arr[3] = 5 bit[4] = arr[1]+..+arr[4] = 16 bit[5] = arr[5] = 9 bit[6] = arr[5]+arr[6] = 20 bit[7] = arr[7] = 13 bit[8] = arr[1]+..+arr[8] = 64 INDEX: 1 2 3 4 5 6 7 8 BIT[]: 1 4 5 16 9 20 13 64
Section 2 — The Low-Bit Trick: i & (-i)

The entire Fenwick Tree is built around one operation: lowbit(i) = i & (-i)

In two's complement, -i flips all bits of i and adds 1. The result of i & (-i) isolates the lowest set bit of i.

LOW-BIT EXAMPLES (how i & (-i) works) i = 6 ? binary: 0110 -i = -6 ? binary: 1010 (flip all bits = 1001, add 1 = 1010) i & (-i) = 0110 & 1010 = 0010 = 2 ? lowest set bit of 6 i = 12 ? binary: 01100 -i = -12 ? binary: 10100 i & (-i) = 00100 = 4 ? lowest set bit of 12 WHAT IT MEANS for the BIT: bit[i] stores the sum of the last lowbit(i) elements: arr[i-lowbit(i)+1 .. i] bit[6] ? lowbit=2 ? stores arr[5..6] bit[4] ? lowbit=4 ? stores arr[1..4] bit[8] ? lowbit=8 ? stores arr[1..8] QUERY navigation (subtract lowbit to go to previous "bucket"): prefix(7): 7?6?4 (7-1=6, 6-2=4, 4-4=0 stop) bit[7] + bit[6] + bit[4] = 13 + 20 + 16 = 49 ? sum of arr[1..7] ? UPDATE navigation (add lowbit to update all covering buckets): update(3): 3?4?8 (3+1=4, 4+4=8, 8+8=16 out of range) Update bit[3], bit[4], bit[8]
?? Why is this elegant? The lowbit operation encodes a complete tree structure implicitly in the index bits — no explicit tree pointers needed. The query and update paths visit at most O(log n) indices because each step either clears the lowest bit (query) or sets the next bit (update), and there are only log2(n) bits.
Section 3 — Prefix Sum Query: O(log n)

To compute sum(1..i): walk backward by repeatedly subtracting the low bit.

QUERY prefix(6) — sum of arr[1..6] i=6: add bit[6]=20 ? i -= (6&-6)=2 ? i=4 i=4: add bit[4]=16 ? i -= (4&-4)=4 ? i=0 Stop (i=0). prefix(6) = 20 + 16 = 36 ? (1+3+5+7+9+11 = 36)
C# 
// Query: prefix sum of arr[1..i] — O(log n)
// BIT is 1-indexed; bit[0] is unused
int Query(int i)
{
    int sum = 0;
    for (; i > 0; i -= i & (-i))
        sum += bit[i];
    return sum;
}
Python 
# Query: prefix sum of arr[1..i] — O(log n)
# BIT is 1-indexed; bit[0] is unused
def query(i):
    s = 0
    while i > 0:
        s += bit[i]; i -= i & (-i)
    return s
Section 4 — Point Update: O(log n)

To add delta to arr[i]: walk forward by repeatedly adding the low bit, updating all BIT cells responsible for covering index i.

UPDATE index 3 (+5) — update all cells covering position 3 i=3: bit[3] += 5 ? i += (3&-3)=1 ? i=4 i=4: bit[4] += 5 ? i += (4&-4)=4 ? i=8 i=8: bit[8] += 5 ? i += (8&-8)=8 ? i=16 (out of range, stop) All cells that "include" position 3 in their sum have been updated ?
C# 
// Update: add delta to arr[i] — O(log n)
void Update(int i, int delta)
{
    for (; i <= n; i += i & (-i))
        bit[i] += delta;
}
Python 
# Update: add delta to arr[i] — O(log n)
def update(i, delta):
    while i <= n:
        bit[i] += delta; i += i & (-i)
Section 5 — Range Sum Query

A Fenwick Tree naturally gives prefix sums. For an arbitrary range [L, R]:

RangeSum(L, R) = prefix(R) - prefix(L - 1)

?? 1-indexed requirement: Fenwick Trees are almost always 1-indexed. If your array is 0-indexed, either shift all indices by +1 when calling BIT operations, or use a wrapper that does the shifting internally. Calling Query(0) or Update(0, delta) will cause an infinite loop (0 & -0 = 0).
C# 
// Range sum of arr[l..r] (1-indexed)
int RangeSum(int l, int r) => Query(r) - Query(l - 1);
Python 
# Range sum of arr[l..r] (1-indexed)
def range_sum(l, r): return query(r) - query(l - 1)
Section 6 — Build in O(n)

Calling Update(i, arr[i]) n times builds the BIT in O(n log n). But there's an O(n) method:

O(n) BUILD METHOD For each index i from 1 to n: bit[i] += arr[i] ? add the original value j = i + (i & -i) ? find the parent if j <= n: bit[j] += bit[i] ? propagate up to parent This visits each cell at most twice and runs in O(n).
C# 
void BuildLinear(int[] nums)   // O(n) build
{
    for (int i = 1; i <= n; i++)
    {
        bit[i] += nums[i - 1];         // nums is 0-indexed, bit is 1-indexed
        int j = i + (i & (-i));
        if (j <= n) bit[j] += bit[i];
    }
}
Python 
def build_linear(nums):   # O(n) build
    for i in range(1, n + 1):
        bit[i] += nums[i - 1]       # nums is 0-indexed, bit is 1-indexed
        j = i + (i & (-i))
        if j <= n: bit[j] += bit[i]
Section 7 — Full C# & Python Implementations
C# 
public class FenwickTree
{
    private readonly int n;
    private readonly int[] bit;   // 1-indexed, size n+1

    public FenwickTree(int n)
    {
        this.n = n;
        bit = new int[n + 1];
    }

    public FenwickTree(int[] nums) : this(nums.Length)
    {
        for (int i = 1; i <= n; i++)
        {
            bit[i] += nums[i-1];
            int j = i + (i & -i);
            if (j <= n) bit[j] += bit[i];
        }
    }

    /// Add delta to position i (1-indexed) — O(log n)
    public void Add(int i, int delta)
    {
        for (; i <= n; i += i & -i) bit[i] += delta;
    }

    /// Prefix sum [1..i] — O(log n)
    public int Prefix(int i)
    {
        int s = 0;
        for (; i > 0; i -= i & -i) s += bit[i];
        return s;
    }

    /// Range sum [l..r] (1-indexed) — O(log n)
    public int RangeSum(int l, int r) => Prefix(r) - Prefix(l - 1);
}

// Usage
var ft = new FenwickTree(new int[] {1,3,5,7,9,11,13,15});
Console.WriteLine(ft.Prefix(4));       // 16  (1+3+5+7)
Console.WriteLine(ft.RangeSum(3,6)); // 32  (5+7+9+11)
ft.Add(3, 4);                         // arr[3] += 4  (5 ? 9)
Console.WriteLine(ft.RangeSum(3,6)); // 36  (9+7+9+11)
Python 
class FenwickTree:
    def __init__(self, nums):
        self.n = len(nums)
        self.bit = [0] * (self.n + 1)
        for i in range(1, self.n + 1):  # O(n) build
            self.bit[i] += nums[i-1]
            j = i + (i & -i)
            if j <= self.n: self.bit[j] += self.bit[i]

    def add(self, i, delta):  # 1-indexed, O(log n)
        while i <= self.n:
            self.bit[i] += delta
            i += i & -i

    def prefix(self, i):  # sum [1..i], O(log n)
        s = 0
        while i > 0:
            s += self.bit[i]
            i -= i & -i
        return s

    def range_sum(self, l, r):  # sum [l..r] (1-indexed)
        return self.prefix(r) - self.prefix(l - 1)

7.1 — Range Update Variant (using two BITs)

A standard BIT supports only point updates. To support range updates (add val to all elements in [L, R]) while still answering prefix sum queries, use two BITs with a mathematical trick:

RANGE UPDATE TRICK with two BITs (B1, B2) Goal: RangeUpdate(L, R, val) and Query(i) = prefix sum of modified array. Observation: after adding val to arr[L..R], prefix(i) = sum of original arr[1..i] + val * (i - L + 1) if L <= i <= R + val * (R - L + 1) if i > R This can be expressed as: prefix(i) = B1.prefix(i) * i - B2.prefix(i) Range update: B1.add(L, val); B1.add(R+1, -val) B2.add(L, val*(L-1)); B2.add(R+1, -val*R)
C# 
public class RangeUpdateFenwick
{
    private readonly FenwickTree b1, b2;
    private readonly int n;

    public RangeUpdateFenwick(int n) {
        this.n = n;
        b1 = new FenwickTree(n);
        b2 = new FenwickTree(n);
    }

    /// Add val to arr[l..r] — O(log n)
    public void RangeAdd(int l, int r, int val)
    {
        b1.Add(l,   val);
        b1.Add(r+1, -val);
        b2.Add(l,   val * (l - 1));
        b2.Add(r+1, -val * r);
    }

    /// Prefix sum [1..i] — O(log n)
    public long Prefix(int i) => (long)b1.Prefix(i) * i - b2.Prefix(i);
    public long RangeSum(int l, int r) => Prefix(r) - Prefix(l - 1);
}
Python 
class RangeUpdateFenwick:
    def __init__(self, n):
        self.n  = n
        self.b1 = FenwickTree(n)
        self.b2 = FenwickTree(n)

    def range_add(self, l, r, val):
        """Add val to arr[l..r] — O(log n)"""
        self.b1.add(l,   val);        self.b1.add(r+1, -val)
        self.b2.add(l,   val * (l - 1)); self.b2.add(r+1, -val * r)

    def prefix(self, i) -> int:
        """Prefix sum [1..i] — O(log n)"""
        return self.b1.prefix(i) * i - self.b2.prefix(i)

    def range_sum(self, l, r) -> int:
        return self.prefix(r) - self.prefix(l - 1)
Section 8 — 2D Fenwick Tree

Extending to 2D: each cell bit[i][j] stores the sum over a 2D "rectangle" of elements. Updates and queries use nested loops applying the lowbit trick in both dimensions.

C# 
public class FenwickTree2D
{
    private readonly int[,] bit;
    private readonly int rows, cols;

    public FenwickTree2D(int rows, int cols)
    {
        this.rows = rows; this.cols = cols;
        bit = new int[rows+1, cols+1];
    }

    public void Update(int r, int c, int delta)
    {
        for (int i = r; i <= rows; i += i & -i)
            for (int j = c; j <= cols; j += j & -j)
                bit[i,j] += delta;
    }

    public int Prefix(int r, int c)
    {
        int s = 0;
        for (int i = r; i > 0; i -= i & -i)
            for (int j = c; j > 0; j -= j & -j)
                s += bit[i,j];
        return s;
    }

    /// Sum of rectangle with top-left (r1,c1) and bottom-right (r2,c2)
    public int RectSum(int r1, int c1, int r2, int c2)
        => Prefix(r2,c2) - Prefix(r1-1,c2) - Prefix(r2,c1-1) + Prefix(r1-1,c1-1);
}
// Time: O(log(rows) * log(cols)) per operation
Python 
class FenwickTree2D:
    def __init__(self, rows, cols):
        self.rows = rows; self.cols = cols
        self.bit  = [[0] * (cols + 1) for _ in range(rows + 1)]

    def update(self, r, c, delta):
        i = r
        while i <= self.rows:
            j = c
            while j <= self.cols:
                self.bit[i][j] += delta
                j += j & (-j)
            i += i & (-i)

    def prefix(self, r, c) -> int:
        s, i = 0, r
        while i > 0:
            j = c
            while j > 0: s += self.bit[i][j]; j -= j & (-j)
            i -= i & (-i)
        return s

    def rect_sum(self, r1, c1, r2, c2) -> int:
        """Sum of rectangle top-left (r1,c1) to bottom-right (r2,c2)"""
        return (self.prefix(r2, c2) - self.prefix(r1-1, c2)
                - self.prefix(r2, c1-1) + self.prefix(r1-1, c1-1))
# Time: O(log(rows) * log(cols)) per operation

public class NumArray
{
    private readonly FenwickTree ft;
    private readonly int[] nums;

    public NumArray(int[] nums)
    {
        this.nums = (int[])nums.Clone();
        ft = new FenwickTree(nums);
    }

    public void Update(int index, int val)
    {
        ft.Add(index + 1, val - nums[index]);   // BIT is 1-indexed
        nums[index] = val;
    }

    public int SumRange(int l, int r)
        => ft.RangeSum(l + 1, r + 1);   // convert 0-indexed to 1-indexed
}

LeetCode #493 — Reverse Pairs (Hard)

Count pairs (i,j) where i < j and nums[i] > 2 * nums[j].

?? Approach: Merge Sort or BIT with coordinate compression. BIT approach: process array left to right. For each element x, query the BIT for count of elements already inserted that are in range [2x+1, 8] (elements already seen that are less than x/2). Coordinate compress all values and 2*values together. Then insert x into the BIT, repeating for all elements. Time: O(n log n). This showcases how BIT can count "inversions" and "reverse pairs."

LeetCode #1649 — Create Sorted Array through Instructions (Hard)

While inserting elements into a sorted array, count the cost (min of insertions before and after current element). Return total cost mod 10?+7.

C# 
public int CreateSortedArray(int[] instructions)
{
    const int MOD = 1_000_000_007;
    int maxVal = instructions.Max() + 1;
    int[] bit = new int[maxVal + 1];
    long cost = 0;
    int total = 0;

    int Query(int i) {
        int s = 0;
        for (; i > 0; i -= i&-i) s += bit[i];
        return s;
    }
    void Update(int i) {
        for (; i <= maxVal; i += i&-i) bit[i]++;
    }

    foreach (int x in instructions)
    {
        int less    = Query(x - 1);                // count of elements < x
        int greater = total - Query(x);            // count of elements > x
        cost = (cost + Math.Min(less, greater)) % MOD;
        Update(x);
        total++;
    }
    return (int)cost;
}
Python 
def create_sorted_array(instructions: list) -> int:
    MOD     = 1_000_000_007
    max_val = max(instructions) + 1
    bit     = [0] * (max_val + 1)
    cost = total = 0

    def query(i):
        s = 0
        while i > 0: s += bit[i]; i -= i & (-i)
        return s

    def update(i):
        while i <= max_val: bit[i] += 1; i += i & (-i)

    for x in instructions:
        less    = query(x - 1)           # count of elements < x
        greater = total - query(x)       # count of elements > x
        cost    = (cost + min(less, greater)) % MOD
        update(x); total += 1
    return cost
Section 10 — Q&A Review
Q1: Why is the Fenwick Tree always 1-indexed? What happens if you call Update(0)?
The algorithm relies on i & (-i) to find the low bit. When i=0: 0 & (-0) = 0 & 0 = 0, so i += 0 — an infinite loop. The 1-indexed convention avoids this edge case. Index 0 is left unused (or used as a sentinel). When working with 0-indexed arrays, always pass index + 1 to BIT operations and store the BIT array with size n + 1.
Q2: The Fenwick Tree is sometimes called "the easiest O(log n) data structure." Why? What's the simplest way to remember the two core operations?
Each operation is literally a 3-line loop: Query walks backwards subtracting lowbit (i -= i&(-i)), Update walks forwards adding lowbit (i += i&(-i)). The only thing to remember: Query goes down/backward (smaller indices, toward prefix start), Update goes up/forward (larger indices, updating all nodes that "cover" the updated position). Memory trick: "Q minus, U plus."
Q3: Can a Fenwick Tree compute the minimum over a range? Why or why not?
No. Minimum is not an invertible operation — given prefix_min(R) and prefix_min(L-1), you cannot recover range_min(L,R) via subtraction or any other formula. Fenwick Trees work because prefix_sum(R) - prefix_sum(L-1) = range_sum(L,R) due to the invertibility of addition. Since min has no inverse, range minimum queries require a Segment Tree or Sparse Table. This is the key limitation of Fenwick Trees vs. Segment Trees.
Q4: How does the O(n) build compare to calling Update(i, arr[i]) n times?
Calling Update n times: each Update is O(log n), so total O(n log n). The O(n) linear build works by directly constructing the BIT bottom-up: for each i, copy the value to bit[i], then propagate to the immediate parent (bit[i + lowbit(i)]) — similar to how a segment tree builds bottom-up. This avoids the overhead of the while-loop in Update. For n = 10^5, the speedup is ~17x (log2(10^5) ˜ 17). In competitive programming this often makes the difference between TLE and AC.
Q5: Describe the "inversion count" problem and how a Fenwick Tree solves it.
Inversion count: given an array, count pairs (i,j) where i < j but arr[i] > arr[j]. Brute force: O(n²). BIT solution: coordinate compress values to [1..n]. Process elements left to right. For each element x, the number of inversions it creates = number of elements already inserted with value > x = (total elements inserted so far) - Query(x). Then insert x: Update(x, 1). Total time: O(n log n). This pattern (count elements in a range seen so far) appears in many problems: Count Smaller Numbers After Self, Reverse Pairs, etc.
