Section 1 — Interval Merging
MERGE INTERVALS PATTERN: Given a list of intervals, merge all overlapping ones. ALGORITHM: 1. Sort intervals by start time. 2. Maintain a "current" interval. Iterate: - If current.end >= next.start ? they overlap ? extend current.end - Otherwise ? no overlap ? push current to result, start fresh OVERLAP CHECK: two intervals [a,b] and [c,d] overlap iff a <= d AND c <= b AFTER SORTING: c >= a always, so just check: c <= b (next.start <= curr.end) VISUAL: Input: [[1,3],[2,6],[8,10],[15,18]] Sorted: [[1,3],[2,6],[8,10],[15,18]] (already sorted) curr=[1,3], next=[2,6]: 2<=3 ? merge ? curr=[1,6] curr=[1,6], next=[8,10]: 8>6 ? push [1,6], curr=[8,10] curr=[8,10], next=[15,18]: 15>10 ? push [8,10], curr=[15,18] Push [15,18]. Output: [[1,6],[8,10],[15,18]]
C# 
// #56 Merge Intervals — O(n log n)
public int[][] Merge(int[][] intervals)
{
    Array.Sort(intervals, (a, b) => a[0] - b[0]);
    var res = new List<int[]>();
    int[] curr = intervals[0];
    foreach (var iv in intervals) {
        if (iv[0] <= curr[1])  // overlap: extend
            curr[1] = Math.Max(curr[1], iv[1]);
        else {                  // no overlap: commit current, start new
            res.Add(curr);
            curr = iv;
        }
    }
    res.Add(curr);
    return res.ToArray();
}
Python 
def merge(intervals):
    intervals.sort()
    res = [intervals[0]]
    for s, e in intervals[1:]:
        if s <= res[-1][1]: res[-1][1] = max(res[-1][1], e)
        else: res.append([s, e])
    return res
Section 2 — Insert Interval & Meeting Rooms
INSERT INTERVAL (#57): Insert a new interval into a sorted non-overlapping list. Three phases: 1. Copy all intervals that end BEFORE newInterval starts (no overlap) 2. Merge all intervals that overlap with newInterval 3. Copy all remaining intervals (start after merged end) MEETING ROOMS II (#253 — Min conference rooms needed): METHOD: Sweep Line with sorted events (or min-heap) HEAP APPROACH: Sort meetings by start time. Use a min-heap of end times. For each meeting: - If heap.top <= meeting.start ? reuse that room (pop heap) - Always push meeting.end to heap Answer = heap.size (rooms in use simultaneously) VISUAL: [[0,30],[5,10],[15,20]] Sort by start: [[0,30],[5,10],[15,20]] Process [0,30]: heap=[] ? push 30 ? heap=[30] Process [5,10]: heap.top=30 > 5 ? new room, push 10 ? heap=[10,30] Process [15,20]: heap.top=10 = 15 ? reuse, pop 10, push 20 ? heap=[20,30] Answer = heap.size = 2
C# 
// #57 Insert Interval — O(n)
public int[][] Insert(int[][] intervals, int[] newInterval)
{
    var res = new List<int[]>();
    int i = 0, n = intervals.Length;
    // Phase 1: add all that end before new starts
    while (i < n && intervals[i][1] < newInterval[0]) res.Add(intervals[i++]);
    // Phase 2: merge overlapping
    while (i < n && intervals[i][0] <= newInterval[1]) {
        newInterval[0] = Math.Min(newInterval[0], intervals[i][0]);
        newInterval[1] = Math.Max(newInterval[1], intervals[i][1]);
        i++;
    }
    res.Add(newInterval);
    // Phase 3: rest
    while (i < n) res.Add(intervals[i++]);
    return res.ToArray();
}

// #253 Meeting Rooms II — min heap approach — O(n log n)
public int MinMeetingRooms(int[][] intervals)
{
    Array.Sort(intervals, (a, b) => a[0] - b[0]);
    var heap = new PriorityQueue<int, int>();  // min-heap of end times
    foreach (var iv in intervals) {
        if (heap.Count > 0 && heap.Peek() <= iv[0]) heap.Dequeue();
        heap.Enqueue(iv[1], iv[1]);
    }
    return heap.Count;
}
Python 
import heapq

def minMeetingRooms(intervals):
    intervals.sort()
    heap = []  # min-heap of end times
    for s, e in intervals:
        if heap and heap[0] <= s: heapq.heappop(heap)
        heapq.heappush(heap, e)
    return len(heap)
Section 3 — Non-Overlapping Intervals (Greedy)
NON-OVERLAPPING INTERVALS (#435): Find minimum number of intervals to REMOVE so remainder is non-overlapping. GREEDY — "Activity Selection" style: Sort by END time (not start!). Keep a "last end" pointer. Greedily keep the interval that ends earliest, since it leaves the most room for future intervals. If next interval overlaps with last kept (start < lastEnd) ? remove it (count++). Otherwise ? keep it, update lastEnd. WHY SORT BY END? Sorting by end time and always keeping the earliest-ending interval is provably optimal (exchange argument: any optimal solution can be transformed to this one without increasing the count). ALTERNATIVE FRAMING: Max non-overlapping = n - min_removals ? Max intervals you can KEEP without overlap. Same greedy: count kept intervals.
C# 
// #435 Non-Overlapping Intervals — O(n log n)
public int EraseOverlapIntervals(int[][] intervals)
{
    Array.Sort(intervals, (a, b) => a[1] - b[1]);  // sort by end
    int lastEnd = int.MinValue, remove = 0;
    foreach (var iv in intervals) {
        if (iv[0] < lastEnd) remove++;  // overlap ? remove
        else lastEnd = iv[1];             // keep ? update lastEnd
    }
    return remove;
}
Python 
def eraseOverlapIntervals(intervals):
    intervals.sort(key=lambda x: x[1])
    last_end = float('-inf'); remove = 0
    for s, e in intervals:
        if s < last_end: remove += 1
        else: last_end = e
    return remove
Section 4 — Top K Elements Pattern
TOP K ELEMENTS — Min-Heap Approach: GOAL: Find K largest elements from a stream/array. INSIGHT: Maintain a min-heap of exactly K elements. - For each new element: if heap.size < K ? push - Else if element > heap.min ? pop min, push new element - At the end, heap contains exactly the K largest elements. WHY MIN-HEAP FOR K LARGEST? The min-heap keeps track of the "minimum of the top K." If a new element beats this minimum, it replaces it. Counter-intuitive but correct: to find K largest, use min-heap. SIZE-K MIN-HEAP = O(n log K) total — better than O(n log n) sort when K << n. TOP K FREQUENT ELEMENTS (#347): 1. Count frequencies (O(n)). 2. Use min-heap of (freq, element) of size K. OR use BUCKET SORT: 3. Create buckets[freq] = [elements with that freq]. 4. Scan from highest bucket down, collect K elements. ? O(n) time (bucket sort approach).
C# 
// #215 Kth Largest Element in Array — O(n log k) using min-heap
public int FindKthLargest(int[] nums, int k)
{
    var minH = new PriorityQueue<int, int>();
    foreach (int n in nums) {
        minH.Enqueue(n, n);
        if (minH.Count > k) minH.Dequeue();
    }
    return minH.Peek();
}

// #347 Top K Frequent Elements — O(n) bucket sort approach
public int[] TopKFrequent(int[] nums, int k)
{
    var freq = new Dictionary<int, int>();
    foreach (int n in nums) freq[n] = freq.GetValueOrDefault(n) + 1;
    var buckets = new List<int>[nums.Length + 1];
    foreach (var kv in freq) {
        buckets[kv.Value] ??= new List<int>();
        buckets[kv.Value].Add(kv.Key);
    }
    var res = new List<int>();
    for (int i = buckets.Length - 1; i >= 0 && res.Count < k; i--)
        if (buckets[i] != null) res.AddRange(buckets[i]);
    return res.Take(k).ToArray();
}
Python 
import heapq
from collections import Counter

def findKthLargest(nums, k):
    return heapq.nlargest(k, nums)[-1]  # built-in, O(n log k)

def topKFrequent(nums, k):
    freq = Counter(nums)
    # Bucket sort O(n)
    buckets = [[] for _ in range(len(nums)+1)]
    for v, f in freq.items(): buckets[f].append(v)
    res = []
    for i in range(len(buckets)-1, -1, -1):
        res.extend(buckets[i])
        if len(res) >= k: return res[:k]
    return res
Section 5 — Quickselect (K-th Largest in O(n) Average)
QUICKSELECT — Hoare's partition-based selection: Based on QuickSort partitioning: after partitioning around pivot, pivot is in its final sorted position. If pivot_index == k-1 (0-indexed from largest): found answer. If pivot_index < k-1: recurse right partition. If pivot_index > k-1: recurse left partition. Average: O(n). Worst case: O(n˛) with bad pivot. Use random pivot to prevent worst case. WHEN TO USE QUICKSELECT vs MIN-HEAP: - Need exact K-th: Quickselect O(n) average (modifies array) - Streaming / online data: Min-heap O(n log k) (works incrementally) - K-th from multiple sorted arrays: Min-heap merge - K largest from sorted matrix: Min-heap
C# 
// Quickselect — K-th largest, O(n) average
public int FindKthLargestQS(int[] nums, int k)
{
    var rng = new Random();
    return QuickSelect(nums, 0, nums.Length - 1, nums.Length - k, rng);
}
private int QuickSelect(int[] a, int lo, int hi, int target, Random rng)
{
    if (lo == hi) return a[lo];
    int pivIdx = lo + rng.Next(hi - lo + 1);
    (a[pivIdx], a[hi]) = (a[hi], a[pivIdx]);
    int pivot = a[hi], store = lo;
    for (int i = lo; i < hi; i++)
        if (a[i] <= pivot) (a[i], a[store]) = (a[store], a[i++]), store++;  // wait — fix
    (a[store], a[hi]) = (a[hi], a[store]);
    if (store == target) return a[store];
    return store < target ? QuickSelect(a, store + 1, hi, target, rng)
                           : QuickSelect(a, lo, store - 1, target, rng);
}
Python 
import random

def findKthLargestQS(nums, k):
    target = len(nums) - k  # k-th largest = (n-k)-th smallest
    def qs(lo, hi):
        pivot = nums[random.randint(lo, hi)]
        # 3-way partition
        mid = lo
        i, j = lo, hi
        while mid <= j:
            if nums[mid] < pivot: nums[lo], nums[mid] = nums[mid], nums[lo]; lo+=1; mid+=1
            elif nums[mid] > pivot: nums[mid], nums[j] = nums[j], nums[mid]; j-=1
            else: mid+=1
        if i <= target <= j: return nums[target]
        elif target < i: return qs(i, j-1)  # oops — use lo/j
        else: return qs(j+1, hi)
    return qs(0, len(nums)-1)
Note: For interviews, Python's heapq.nlargest(k, nums)[-1] and C#'s PriorityQueue min-heap approach are the most readable. Quickselect is worth knowing for "O(n) expected" discussions but is error-prone under pressure.

Q1: Why do we sort by end time (not start) for the non-overlapping intervals greedy?
Sorting by end time implements the "earliest deadline first" strategy — a classic greedy optimality proof. If we always keep the interval that ends earliest, we maximize the remaining time available for future intervals. Formal proof by exchange argument: suppose an optimal solution keeps interval A over interval B (where B ends earlier). We can swap A for B — since B ends no later than A, it cannot cause more conflicts with future intervals. So sorting by end time and greedily taking non-conflicting intervals yields the maximum independent set of intervals, which minimizes removals.
Q2: Why use a min-heap for "K largest" elements? Isn't a max-heap more natural?
A max-heap would let you efficiently find the largest element, but it doesn't help maintain only K elements efficiently. With a min-heap of size K: the top is the smallest of the K largest elements found so far. When a new element arrives, compare it to this minimum. If it's larger, replace the minimum (pop + push). This is O(log K) per element. A max-heap of all elements would be O(n) to build then O(K log n) to extract K elements — O(n + K log n), worse than O(n log K) when K is small. Bottom line: min-heap acts as a "filter" that efficiently discards elements too small to be in the top K.
Q3: How does the merge interval algorithm handle [1,4],[2,3] (fully contained intervals)?
After sorting by start: [1,4],[2,3]. Current = [1,4]. Next = [2,3]: start(2) = end(4) ? overlap ? extend current.end = max(4,3) = 4. So the inner interval is absorbed: result is [1,4]. The key is using max(curr.end, next.end) during merge — not just next.end. If you only used next.end, [2,3] would incorrectly shrink the merged interval to end at 3. Always take the maximum of the two ends when merging.
Q4: What's the O(n) solution for Top K Frequent Elements and why doesn't everyone use it?
Bucket sort: create an array of n+1 buckets indexed by frequency (since max frequency = n). Count element frequencies. Place each element in the bucket at its frequency index. Then scan from bucket[n] down to bucket[1] collecting K elements. This achieves O(n) time and O(n) space. Why isn't it universal? (1) It requires knowing the maximum possible frequency upfront (needs array size n). (2) It uses O(n) space even if K is tiny. (3) For top-K from a stream where n is unknown or huge, the O(n log K) heap approach is more practical. (4) Some follow-ups ask for sort-stable results (same frequency ? alphabetical), which the heap handles naturally with custom comparators.
Q5: Meeting Rooms II — why does the sweep-line approach also work and how does it relate to the heap approach?
Sweep line: create 2n events — +1 at each start, -1 at each end. Sort events (ties: process end before start so ending and starting at same time is counted as reuse, not new room). Scan events, track current rooms in use, record maximum. This is O(n log n). It's equivalent to the heap approach because both track the "watermark" of simultaneous meetings. The heap approach is slightly cleaner for coding interviews (no tie-breaking edge cases). The sweep line is more general and extends to problems like "maximum overlap of intervals" and "number of events happening at any time," making it a more powerful technique despite being the same asymptotic complexity.
