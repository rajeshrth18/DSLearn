Section 6.1 — The String Immutability Problem

Strings are the most-used data type in every real program — user input, file paths, API responses, log messages, HTML, JSON. In C#, the string type has one critical rule that causes serious performance bugs if you don't understand it: strings are immutable.

📌 Immutable = Cannot Change After Creation
Once a string object exists in memory, its characters can never be modified. Every operation that looks like a modification — +, Replace, ToUpper, Trim — secretly creates a brand-new string object in memory. The old one is abandoned and eventually garbage-collected.
🌍 Real-World Analogy:
Imagine string concatenation like writing a book.
With immutability (+= each time): Every time you add a sentence, you must re-type the ENTIRE book from scratch on a new piece of paper. After 100 pages, adding one sentence requires copying 100 pages. After 1,000 pages... you see the problem.

With StringBuilder: You write on a whiteboard. Adding a sentence just means writing at the end. When the whole book is done, you print it once.

Why Was Immutability Designed This Way?

  • Thread safety: Multiple threads can read the same string simultaneously — no locks needed, because nobody can change it.
  • String interning: The CLR maintains a pool of string literals. "hello" used 1,000 times in your code shares one memory address.
  • Security: A file path, password, or URL passed to a function can't be changed behind your back by malicious code.
  • Hash stability: Strings can be safely used as Dictionary keys because their hash code never changes.

Seeing Immutability in Memory

IMMUTABILITY — What Really Happens in Memory string name = "Alice"; ───────────────────────────────────────────────── Heap: [ A | l | i | c | e ] ← address 0x4A20 Stack: name ──────────────────────────────→ 0x4A20 name = name.ToUpper(); ← looks like modification, but... ───────────────────────────────────────────────── Heap: [ A | l | i | c | e ] ← 0x4A20 (UNTOUCHED) [ A | L | I | C | E ] ← 0x4B30 (brand new!) Stack: name ──────────────────────────────→ 0x4B30 Old string 0x4A20 is now abandoned → garbage collector will eventually free that memory.
C# 
string city = "london";

// These do NOT modify city — they return new strings
string upper   = city.ToUpper();     // "LONDON"  — new object
string titled  = city.Replace('l', 'L'); // "London"  — new object

Console.WriteLine(city);    // "london"  — completely unchanged!
Console.WriteLine(upper);   // "LONDON"
Console.WriteLine(titled);  // "London"

// You CAN reassign the variable (changes the reference, not the object):
city = city.ToUpper(); // city now points to "LONDON" — old "london" is abandoned

// You CANNOT modify individual characters — no setter on string indexer:
// city[0] = 'l';  ← COMPILE ERROR: string indexer has no setter

The O(n²) Concatenation Disaster

The most common beginner performance bug is building a string inside a loop with +=:

WHY += IN A LOOP IS O(n²) — Counting character copies Building a string from 5 words: ["The","quick","brown","fox","jumps"] Pass 1: "" + "The" → copy 3 chars → new string "The" Pass 2: "The" + " quick" → copy 9 chars → new string "The quick" Pass 3: "The quick" + " brown" → copy 15 chars → new string "The quick brown" Pass 4: "The quick brown" + " fox" → copy 19 chars → ... Pass 5: ... + " jumps" → copy 25 chars Total: 3+9+15+19+25 = 71 copies for 5 words. For n words of average length k: Total copies = k×(1+2+3+...+n) = k × n(n+1)/2 = O(n²) For 10,000 words: billions of character copies = MINUTES to complete.
C# 
// ❌ NEVER DO THIS IN A LOOP — O(n²)
string[] words  = GetWords(); // thousands of words
string  result = "";
foreach (string word in words)
    result += word + " "; // Each += copies EVERYTHING again from scratch
// Fine for 5 words. Catastrophic for 50,000 words.
Python 
# ✗ NEVER DO THIS IN A LOOP — O(n²)
words = get_words()  # thousands of words
result = ""
for word in words:
    result += word + " "  # Each += copies EVERYTHING again from scratch
# Fine for 5 words. Catastrophic for 50,000 words.
🚨 Real Performance Benchmark:
Building a string from 50,000 words using +=: approximately 12 seconds.
The same operation with StringBuilder: approximately 2 milliseconds.
That is 6,000× slower. This has crashed production servers.
Section 6.2 — StringBuilder: The Correct Solution

StringBuilder (namespace: System.Text) maintains a mutable, resizable character buffer internally. Appending characters writes directly into this buffer without creating new string objects. The final string is constructed only once when you call .ToString().

How the Buffer Works

STRINGBUILDER INTERNAL BUFFER — Visual Default capacity = 16. After sb.Append("Hello"): Buffer: [ H ][ e ][ l ][ l ][ o ][ _ ][ _ ][ _ ][ _ ][ _ ][ _ ][ _ ][ _ ][ _ ][ _ ][ _ ] Length=5, Capacity=16, WritePos=5 After sb.Append(", World!"): Buffer: [ H ][ e ][ l ][ l ][ o ][ , ][ ][ W ][ o ][ r ][ l ][ d ][ ! ][ _ ][ _ ][ _ ] Length=13, Capacity=16, WritePos=13 Buffer almost full. Next large Append triggers RESIZE: → Allocate new buffer of 32 chars (capacity doubles) → Copy 16 existing chars to new buffer ← O(n) cost, happens rarely → Continue appending ← O(1) per char again TOTAL: n appends at O(1) each + rare O(n) doublings = O(n) amortized ✓ vs String +=: every append is O(total length so far) = O(n²) ✗

Basic Usage

C# 
using System.Text; // Required namespace

// ✅ CORRECT — O(n) — StringBuilder for loop concatenation
string[] words = { "Hello", "from", "C#", "DSA", "Course" };

StringBuilder sb = new StringBuilder();
foreach (string word in words)
{
    sb.Append(word);
    sb.Append(' '); // direct buffer write — no new object created
}
string result = sb.ToString(); // ONE string built here, at the end
Console.WriteLine(result); // "Hello from C# DSA Course "
Python 
# ✓ CORRECT — O(n) — collect into list, then join ONCE at the end
words = ["Hello", "from", "Python", "DSA", "Course"]
parts = []
for word in words:
    parts.append(word)
    parts.append(' ')  # direct list write — no new string created
result = ''.join(parts)  # ONE string built here, at the end
print(result)  # "Hello from Python DSA Course "

# Even cleaner — join with separator directly:
result2 = " ".join(words)  # "Hello from Python DSA Course"

int  cap  = sb5.Capacity;         // 16 (or whatever it was allocated)

// ── CLEAR & RESET ────────────────────────────────────────────────
sb5.Clear();                      // resets to "" — buffer is REUSED (fast)
sb5.Length = 0;                  // faster alternative — same effect

// ── CONVERT TO STRING ─────────────────────────────────────────────
string final = sb5.ToString();   // the ONLY time a string is created

Pre-Sizing for Maximum Performance

C# 
// If you know the approximate final size, pre-allocate to skip ALL buffer doublings
int estimatedSize = logLines.Length * 80; // ~80 chars per line
StringBuilder sb = new StringBuilder(capacity: estimatedSize);

foreach (string line in logLines)
{
    sb.Append(line);
    sb.Append('\n');
}
// Zero resizes = maximum throughput
Python 
# Python list.append() is already amortized O(1) — no pre-sizing needed
# But collecting into a list and joining once is the correct pattern:
log_lines = ["line1", "line2", "line3"]  # example
parts = []
for line in log_lines:
    parts.append(line)
    parts.append('\n')
result = ''.join(parts)  # Zero extra copies — maximum throughput

# For very large joins, io.StringIO is the fastest approach:
import io
buf = io.StringIO()
for line in log_lines:
    buf.write(line); buf.write('\n')
result = buf.getvalue()  # equivalent to StringBuilder.ToString()
💡 When to Use What:
  • String +: 1–3 concatenations, outside loops. "Hello " + name + "!" is fine.
  • StringBuilder: Any concatenation in a loop. Building HTML/CSV/JSON/SQL dynamically.
  • string.Join(): Joining a collection with a separator. Clean, readable, O(n).
  • $"" interpolation: Formatting a single string with values mixed in.
Section 6.3 — All Common String Methods

Every string method returns either a new string or a non-string result. The original is always unchanged. Always assign the result: s = s.Trim();

6.3.1 — Inspection (Read-Only, No New String)

C# 
string s = "  Hello, World!  ";

// Size
int  len    = s.Length;                          // 18  — O(1), stored property
char first  = s[0];                             // ' ' — O(1) index access

// Null checks
bool empty  = string.IsNullOrEmpty(s);            // false
bool ws     = string.IsNullOrWhiteSpace("  ");   // true

// Search
bool has    = s.Contains("World");               // true  — O(n)
int  idx    = s.IndexOf("World");                // 9     — first occurrence, -1 if absent
int  last   = s.LastIndexOf('l');               // 12    — last occurrence
bool starts = s.StartsWith("  He");             // true  — O(prefix length)
bool ends   = s.EndsWith("!  ");               // true

// Case-insensitive search (C# 10+):
bool hasCI  = s.Contains("world", StringComparison.OrdinalIgnoreCase); // true
Python

starts = s.startswith("  He")           # True  — O(prefix length)

Section 6.4 — Character-Level Operations

A string is fundamentally a read-only array of char values. Each char is a 16-bit Unicode character. Working at the character level is essential for DSA problems: palindrome checking, anagram detection, frequency counting, custom parsing.

ASCII Values — Memorize These Ranges!

ASCII RANGES TO MEMORIZE — Used constantly in DSA '0'–'9' → ASCII 48–57 (10 digit characters) 'A'–'Z' → ASCII 65–90 (26 uppercase letters) 'a'–'z' → ASCII 97–122 (26 lowercase letters) KEY ARITHMETIC: c - '0' = integer digit value ('7' - '0' = 7) c - 'a' = 0-based letter index ('e' - 'a' = 4) c - 'A' = 0-based letter index ('E' - 'A' = 4) 'a' - 'A' = 32 (case offset) (char)(c+32) = lowercase of c ('A'+32 = 'a') (char)(c-32) = uppercase of c ('a'-32 = 'A')

Essential DSA Pattern — Frequency Counting with char Array

C# 
// Count character frequencies — O(n) time, O(1) space (fixed 26-char array)
public static int[] LetterFrequency(string s)
{
    int[] freq = new int[26]; // index 0='a', 1='b', ..., 25='z'
    foreach (char c in s.ToLower())
        if (c >= 'a' && c <= 'z')
            freq[c - 'a']++;
    return freq;
}

// Check anagram — O(n)
public static bool IsAnagram(string a, string b)
{
    if (a.Length != b.Length) return false;
    int[] count = new int[26];
    for (int i = 0; i < a.Length; i++)
    {
        count[a[i] - 'a']++; // increment for 'a'
        count[b[i] - 'a']--; // decrement for 'b'
    }
    foreach (int n in count) if (n != 0) return false;
    return true;
}
Console.WriteLine(IsAnagram("listen", "silent")); // true
Console.WriteLine(IsAnagram("hello", "world"));   // false

// Check palindrome — O(n) time, O(1) space
public static bool IsPalindrome(string s)
{
    int l = 0, r = s.Length - 1;
    while (l < r)
    {
        if (s[l] != s[r]) return false;
        l++; r--;
    }
    return true;
}
Console.WriteLine(IsPalindrome("racecar"));  // true
Console.WriteLine(IsPalindrome("madam"));    // true
Console.WriteLine(IsPalindrome("hello"));    // false
Python 
# Count character frequencies — O(n) time, O(1) space (fixed 26-char array)
def letter_frequency(s):
    freq = [0] * 26  # index 0='a', 1='b', ..., 25='z'
    for c in s.lower():
        if 'a' <= c <= 'z':
            freq[ord(c) - ord('a')] += 1
    return freq

# Check anagram — O(n)
def is_anagram(a, b):
    if len(a) != len(b): return False
    count = [0] * 26
    for i in range(len(a)):
        count[ord(a[i]) - ord('a')] += 1  # increment for a
        count[ord(b[i]) - ord('a')] -= 1  # decrement for b
    return all(x == 0 for x in count)
print(is_anagram("listen", "silent"))  # True
print(is_anagram("hello", "world"))    # False

# Check palindrome — O(n) time, O(1) space
def is_palindrome(s):
    l, r = 0, len(s) - 1
    while l < r:
        if s[l] != s[r]: return False
        l += 1; r -= 1
    return True
print(is_palindrome("racecar"))  # True
print(is_palindrome("madam"))    # True
print(is_palindrome("hello"))    # False
Section 6.5 — String Comparison & Equality

String comparison is a common source of subtle bugs. C# gives you multiple comparison methods, each with different behavior around case sensitivity and cultural rules.

C# 
string a = "Hello", b = "hello", c = "Hello";

// ── == operator ──────────────────────────────────────────────────
Console.WriteLine(a == c);   // true  — same value
Console.WriteLine(a == b);   // false — case differs!

// ── Equals with StringComparison ─────────────────────────────────
Console.WriteLine(a.Equals(b, StringComparison.OrdinalIgnoreCase));  // true ✓
Console.WriteLine(a.Equals(b, StringComparison.Ordinal));           // false

// ── string.Compare — returns -1, 0, or +1 ────────────────────────
Console.WriteLine(string.Compare("apple", "banana"));            // -1 (a < b)
Console.WriteLine(string.Compare("HELLO", "hello", ignoreCase: true)); // 0

// ── Null-safe comparison ──────────────────────────────────────────
string? nullStr = null;
bool safe = string.Equals(nullStr, "hello", StringComparison.OrdinalIgnoreCase);
// false — no NullReferenceException (static Equals handles null)
// nullStr.Equals("hello") would THROW NullReferenceException!
Python 
a, b, c = "Hello", "hello", "Hello"

print(a == b)   # False — case differs!

🚨 The Turkish 'i' Problem — Why Culture Matters:
In Turkish locale, ToUpper('i') produces 'İ' (dotted I), not 'I'.
If you use .ToUpper() for technical comparisons, your code breaks on Turkish machines.
Rule: For any technical/non-display comparison, always use StringComparison.Ordinal or OrdinalIgnoreCase.
Section 6.6 — String Formatting

Tool 1 — String Interpolation $"..." (Preferred)

C# 
string name  = "Alice";
int    score = 9542;
double price = 1234.5678;

// Basic:
Console.WriteLine($"Player {name} scored {score} points");

// Format specifiers inside {}:
Console.WriteLine($"Price:   {price:F2}");   // "1234.57"  — 2 decimals
Console.WriteLine($"Score:   {score:N0}");   // "9,542"    — thousands separator
Console.WriteLine($"Hex:     {score:X}");    // "2545E"    — hexadecimal
Console.WriteLine($"Percent: {0.875:P0}");  // "88 %"     — percentage

// Column alignment (negative = left, positive = right):
Console.WriteLine($"{"Name",-12}{"Score",8}{"Grade",6}");
Console.WriteLine($"{name,-12}{score,8}{"A+",6}");
// Name           9542    A+

// Expressions are allowed inside {}:
Console.WriteLine($"Length > 3: {name.Length > 3}"); // "Length > 3: True"
Console.WriteLine($"Upper: {name.ToUpper()}");        // "Upper: ALICE"

Key Takeaways:
• Strings are immutable — every modification creates a NEW string object
• Never use += inside a loop — that's O(n²). Use StringBuilder for O(n)
• StringBuilder uses a resizable buffer — amortized O(1) append, O(n) for final ToString
• Use StringComparison.OrdinalIgnoreCase for safe case-insensitive comparison
• Char arithmetic ('e' - 'a' = 4) is essential for frequency counting in DSA problems
