Section 1 — What is Big O Notation?

Before you write a single algorithm, you need to answer one question: "How does my code perform as the amount of data grows?" This is exactly what Big O Notation answers. It is the universal language that every software engineer uses to describe and compare how efficient an algorithm is.

Big O gives you a way to describe the worst-case growth rate of an algorithm's time or memory usage, expressed in terms of the input size n.

?? Real-World Analogy — The Library Problem:
You walk into a library and need to find a specific book.
  • O(1) — Instant: A librarian hands you the book immediately — it's always on the front desk. Doesn't matter if the library has 10 or 10 million books.
  • O(n) — Shelf Search: You check every book one by one from the first shelf. More books = more time, linearly.
  • O(log n) — Catalogue System: You open the catalogue to the middle, decide which half the book is in, repeat. A library of 1 million books takes only ~20 lookups.
?? Key Rules of Big O:
  • Worst case: We always measure the worst possible scenario (e.g., item not found at all).
  • Drop constants: O(2n) simplifies to O(n). We care about growth rate, not exact count.
  • Drop lower terms: O(n² + n) simplifies to O(n²). Only the dominant term matters.
  • n = input size: n is whatever drives the work — array length, number of nodes, string length.

Section 2 — Time Complexity

Time Complexity measures how many operations an algorithm performs relative to the input size n. It answers: "If I double the data, does the time double? Quadruple? Stay the same?"

2.1 — O(1): Constant Time — The Gold Standard

No matter how large the input, the operation always takes the same number of steps. This is the best possible complexity.

?? Analogy: A sports scoreboard shows the current score. It doesn't matter if 10 or 100,000 people are in the stadium — reading the score is always one glance.
C# 
// O(1) — All of these take the same time regardless of array size
int[] scores = { 85, 92, 78, 96, 88 };

int first = scores[0];              // O(1) — index access
int last  = scores[scores.Length-1]; // O(1) — index access
int len   = scores.Length;           // O(1) — stored property
scores[2] = 100;                    // O(1) — index write

// These are also O(1):
Stack<int> stack = new Stack<int>();
stack.Push(5);   // O(1) — add to top
stack.Pop();     // O(1) — remove from top
stack.Peek();    // O(1) — look at top
Python

last  = scores[-1]             # O(1) — index access

stack = []
stack.append(5)   # O(1) — push to top

stack[-1]         # O(1) — peek at top (if not empty)

2.2 — O(log n): Logarithmic Time — Excellent

The algorithm halves the problem with each step. A key insight: log2(1,000,000) ˜ 20. So even with a million items, you only need ~20 steps!

?? Analogy: Guessing a number between 1 and 1024. Each guess, you say "higher or lower". With a perfect strategy (guess the middle each time), you always find it in at most 10 guesses — because 2¹° = 1024.
C# 
// O(log n) — Binary Search (requires a SORTED array)
public static int BinarySearch(int[] sortedArr, int target)
{
    int left = 0, right = sortedArr.Length - 1;

    while (left <= right)
    {
        int mid = left + (right - left) / 2; // avoids integer overflow

        if (sortedArr[mid] == target) return mid;   // Found!
        if (sortedArr[mid] < target)  left = mid + 1;  // Discard left half
        else                          right = mid - 1; // Discard right half
    }
    return -1; // Not found
}

int[] sorted = { 5, 12, 23, 35, 47, 58, 69, 74, 81, 99 };
Console.WriteLine(BinarySearch(sorted, 58)); // 5 (found at index 5)
Console.WriteLine(BinarySearch(sorted, 40)); // -1 (not found)
Python 
# O(log n) — Binary Search (requires a SORTED list)
def binary_search(sorted_arr, target):

        else:                           right = mid - 1  # Discard right half
    return -1  # Not found

2.3 — O(n): Linear Time — Acceptable

The algorithm visits each element once. If the input doubles, the time doubles. This is the most common complexity in everyday code.

?? Analogy: Counting how many people in a queue are wearing hats. You must look at each person once. 50 people = 50 looks. 500 people = 500 looks.
C# 
// O(n) — Finding maximum value in an unsorted array
public static int FindMax(int[] arr)
{
    int max = arr[0];
    for (int i = 1; i < arr.Length; i++) // visits every element once
        if (arr[i] > max) max = arr[i];
    return max;
}

// O(n) — Sum all elements
public static int Sum(int[] arr)
{
    int total = 0;
    foreach (int x in arr) total += x;
    return total;
}

// O(n) — Print all elements
int[] data = { 3, 1, 4, 1, 5, 9, 2, 6 };
Console.WriteLine($"Max: {FindMax(data)}"); // 9
Console.WriteLine($"Sum: {Sum(data)}");    // 31
Python

print(f"Sum: {sum(data)}")  # 31 — built-in sum() is O(n)

2.5 — O(n²): Quadratic Time — Avoid in Production

Usually caused by a loop inside a loop where both iterate over the same data. At n=10,000 items: 100 million operations. At n=100,000: 10 billion operations. Catastrophically slow.

?? Analogy: Every person in a room shakes hands with every other person. 5 people ? 10 handshakes. 100 people ? 4,950 handshakes. 1,000 people ? 499,500 handshakes. The work explodes!
C# 
// O(n²) — Bubble Sort (educational only — never use in production)
public static void BubbleSort(int[] arr)
{
    for (int i = 0; i < arr.Length - 1; i++)         // outer loop: n passes
    {
        for (int j = 0; j < arr.Length - i - 1; j++) // inner loop: n comparisons
        {
            if (arr[j] > arr[j + 1])
            {
                (arr[j], arr[j+1]) = (arr[j+1], arr[j]); // swap
            }
        }
    }
}
// n=10: ~45 comparisons. n=1000: ~500,000. n=10,000: ~50,000,000.
Python 
# O(n²) — Bubble Sort (educational only — never use in production)
def bubble_sort(arr):
    n = len(arr)
    for i in range(n - 1):           # outer loop: n passes
        for j in range(n - i - 1):  # inner loop: n comparisons
            if arr[j] > arr[j + 1]:
                arr[j], arr[j + 1] = arr[j + 1], arr[j]  # swap
# n=10: ~45 comparisons. n=1000: ~500,000. n=10,000: ~50,000,000.
?? Common O(n²) Pitfall — Nested Loops on the Same Data:
Any time you see a loop inside a loop where both are iterating over the same array, you likely have O(n²). Ask yourself: "Can I use a hash set or sort first to eliminate the inner loop?"

2.6 — O(2n): Exponential Time — Never in Production

Each new element doubles the work. With n=30, that's over a billion operations. With n=50, more operations than atoms in the universe.

C# 
// O(2n) — Naive Fibonacci — never use this version!
public static int FibNaive(int n)
{
    if (n <= 1) return n;
    return FibNaive(n - 1) + FibNaive(n - 2); // 2 calls per call = 2n total
}
// FibNaive(30)  ? ~1 billion operations
// FibNaive(50)  ? ~1 quadrillion operations (takes years!)

// O(n) — Correct Fibonacci using memoization (Dynamic Programming)
public static int FibFast(int n)
{
    if (n <= 1) return n;
    int prev = 0, curr = 1;
    for (int i = 2; i <= n; i++)
    {
        int next = prev + curr;
        prev = curr;
        curr = next;
    }
    return curr;
}
// FibFast(1000) ? 1000 operations — instant!
Python 
# O(2^n) — Naive Fibonacci — never use this version!
def fib_naive(n):
    if n <= 1: return n
    return fib_naive(n - 1) + fib_naive(n - 2)  # 2 calls per call = 2^n total
# fib_naive(30)  → ~1 billion operations
# fib_naive(50)  → ~1 quadrillion operations (takes years!)

# O(n) — Correct Fibonacci using iteration (Dynamic Programming)
def fib_fast(n):
    if n <= 1: return n
    prev, curr = 0, 1
    for _ in range(2, n + 1):
        prev, curr = curr, prev + curr
    return curr
# fib_fast(1000) → 1000 operations — instant!

Section 2.8 — Big Ω (Omega) & Big Θ (Theta) — The Full Picture

Big O tells you the upper bound — the worst case. But it doesn't tell you how the algorithm behaves when you're lucky, or whether best and worst case are the same. That's where Big O and Big T complete the story. Together, all three notations give you a full mathematical description of any algorithm's growth behaviour.

?? The Three Asymptotic Notations — Each Answers a Different Question:
  • O (Big O) — Upper bound. "Runs in AT MOST this much time." Worst-case ceiling.
  • O (Big Omega) — Lower bound. "Runs in AT LEAST this much time." Best-case floor.
  • T (Big Theta) — Tight bound. "Runs in EXACTLY this growth rate." Best = Worst = same class.
THE THREE BOUNDS — Linear Search on an array of n elements BEST CASE ? target is at index 0 ? 1 comparison ? O(1) WORST CASE ? target absent / at last position ? n comparisons ? O(n) Steps ? ¦· · · · · · · · · · O(n) ? upper bound: worst case ceiling ¦· · · · · · · · ¦· · · · · · ? no T exists here: best ? worst ¦· · · · ¦· O(1) ? lower bound: best case floor +----------------------? n For Merge Sort (always does the same work): Steps ? ¦· · · · · O(n log n) ? upper bound ¦· · · · ¦· · · T(n log n) ? TIGHT bound: both lines meet! ¦· · ¦· O(n log n) ? lower bound +----------------------? n When O = O (same growth class) ? T exists and equals both.

2.8a — Big Ω (Omega) — Lower Bound (Best Case)

Big O answers: "In the most optimistic scenario — the luckiest possible input — how much work must this algorithm still do?" It is a guaranteed minimum. Even magic cannot make the algorithm do less work than O.

?? Analogy: A restaurant guarantees your food takes at least 2 minutes (they still need to cook it!). Even if you're the only customer, you wait at minimum 2 minutes. That's Big O — the unavoidable floor.
C# 
// O(1) — Linear Search best case: target is the very first element
public static int LinearSearch(int[] arr, int target)
{
    for (int i = 0; i < arr.Length; i++)
        if (arr[i] == target) return i;  // could hit on very first check ? O(1)
    return -1;
}
// Best:  arr[0] == target ? 1 check  ? O(1)
// Worst: not found        ? n checks ? O(n)

// O(n log n) — Any comparison-based sort — proven mathematical minimum
// No matter how clever, a sort-by-comparison MUST do = n log n comparisons
// This is a mathematical proof — you cannot build an O(n) comparison sort

// O(n) — Printing all elements — you MUST visit each element at least once
foreach (int x in arr) Console.Write(x);  // O(n) and O(n) ? T(n)
Python 
# Ω(1) — Linear Search best case: target is the very first element
def linear_search(arr, target):
    for i, val in enumerate(arr):
        if val == target: return i  # could hit on very first check → Ω(1)
    return -1
# Best:  arr[0] == target → 1 check  → Ω(1)
# Worst: not found        → n checks → O(n)

# Ω(n log n) — Any comparison-based sort — proven mathematical minimum
# No matter how clever, a sort-by-comparison MUST do ≥ n log n comparisons
# This is a mathematical proof — you cannot build an O(n) comparison sort

# Ω(n) — Printing all elements — you MUST visit each element at least once
for x in arr: print(x, end=" ")  # Ω(n) and O(n) → Θ(n)
?? The Sorting Lower Bound — Why Merge Sort Is Optimal:
It is mathematically proven that any comparison-based sorting algorithm requires at least O(n log n) comparisons in the worst case. This means Merge Sort and Heap Sort (both T(n log n)) are asymptotically optimal — no comparison-based algorithm can ever do asymptotically better. Radix Sort achieves O(n·k) by avoiding comparisons entirely, which is why it can "beat" the lower bound.

2.8b — Big Θ (Theta) — Tight Bound (Exact Growth Rate)

Big T is the strongest statement you can make about an algorithm. It says: "Both the best and worst case grow at exactly this rate." T(f(n)) exists if and only if O = O = the same growth class. When you have T, there's no luck involved — every input causes the same asymptotic work.

?? Analogy: A factory production line processes exactly one item per second, always. Whether you give it 5 items or 5 million, each item takes one second. Best case = n seconds. Worst case = n seconds. No shortcuts. That's T(n) — the exact, tight bound.
C# 
// T(n) — Summing all elements — must visit EVERY element, always, no shortcuts
public static long Sum(int[] arr)
{
    long total = 0;
    foreach (int x in arr) total += x; // exactly n additions, no early exit
    return total;
}
// Best case:  n additions ? O(n)
// Worst case: n additions ? O(n)
// O = O ? T(n) ?  — tight bound confirmed

// T(n log n) — Merge Sort — same work regardless of input order
// Divide phase: always splits log n times
// Merge phase:  always merges n elements at each level
// Total: always exactly T(n log n) — sorted or reversed, doesn't matter

// T(1) — Array index access — one arithmetic operation, always
int val = arr[42]; // T(1) — same whether arr has 5 or 5 million elements
Python 
# Θ(n) — Summing all elements — must visit EVERY element, always, no shortcuts
def total_sum(arr):
    total = 0
    for x in arr: total += x  # exactly n additions, no early exit
    return total
# Best case:  n additions → Ω(n)
# Worst case: n additions → O(n)

?? The Industry Convention — Why Everyone Says "Big O":
In day-to-day engineering and most interview settings, people say "Big O" even when they technically mean Big T. When an engineer says "Merge Sort is O(n log n)", they almost always mean it's exactly n log n growth in all cases — i.e., T(n log n). This informal-but-universal convention exists because:
  • Big O is more conservative — it's never wrong to say O(n) even if T(n) is tighter
  • In practice, we care most about worst-case guarantees
  • Writing O is shorter than writing T every time
Know the difference for theoretical/academic contexts. Use Big O comfortably in engineering.
C# 
// Full annotation of the same function with all three notations:
public static int[] MergeSort(int[] arr)
{
    // Time:  O(n log n) — worst case upper bound
    //        O(n log n) — best case lower bound
    //        T(n log n) — tight bound: ALWAYS exactly n log n, any input
    // Space: O(n)       — auxiliary merge buffer needed
    // ...
    return arr; // placeholder
}

public static int LinearSearch(int[] arr, int target)
{
    // Time:  O(n) — worst case: target absent, check all n elements
    //        O(1) — best case: target is arr[0], return immediately
    //        No T — best and worst grow at DIFFERENT rates (1 vs n)
    for (int i = 0; i < arr.Length; i++)
        if (arr[i] == target) return i;
    return -1;
}

// Key rule: If you can prove O(f(n)) AND O(f(n)) for the same f(n),
// you have proven T(f(n)) — the strongest possible complexity claim.

Section 3 — Space Complexity

Space Complexity measures how much additional memory an algorithm uses as the input grows. Servers run thousands of requests simultaneously — memory matters just as much as speed.

?? Input Space vs Auxiliary Space:
We typically only count extra memory the algorithm allocates — not the input itself. An algorithm that sorts an array in-place uses O(1) extra space even though the array is O(n).
C# 
// O(1) Space — uses the same fixed memory regardless of input size
public static int MaxValue(int[] arr)
{
    int max = arr[0]; // ONE extra variable — always O(1)
    foreach (int x in arr)
        if (x > max) max = x;
    return max;
}

// O(n) Space — extra memory grows proportionally with input
public static int[] DoubleAll(int[] arr)
{
    int[] result = new int[arr.Length]; // New array same size as input = O(n)
    for (int i = 0; i < arr.Length; i++)
        result[i] = arr[i] * 2;
    return result;
}

// O(n) Space — Recursive Fibonacci uses O(n) call stack space
// Each recursive call sits on the call stack until it returns.
// FibNaive(n) puts n frames on the stack simultaneously.
Python 
# O(1) Space — uses the same fixed memory regardless of input size
def max_value(arr):
    maximum = arr[0]  # ONE extra variable — always O(1)
    for x in arr:
        if x > maximum: maximum = x
    return maximum

# O(n) Space — extra memory grows proportionally with input
def double_all(arr):
    result = [x * 2 for x in arr]  # New list same size as input = O(n)
    return result

# O(n) Space — Recursive Fibonacci uses O(n) call stack space
# Each recursive call sits on the call stack until it returns.
# fib_naive(n) puts n frames on the stack simultaneously.

Section 4 — Arrays: The Foundation of All Data Structures

The array is the single most important data structure in computer science. Everything else is built on top of it. Linked lists, stacks, queues, hash tables, heaps — all of them use arrays internally or are inspired by the array's memory model. You must understand arrays at the memory level before moving forward.

4.1 — How Arrays Work in Memory

When you create an array, the operating system allocates a single, contiguous block of memory. All elements sit side by side in RAM with no gaps. This layout is what makes arrays so powerful.

ARRAY MEMORY LAYOUT — int[] temps = { 23, 31, 18, 27, 35 } Memory Address: 1000 1004 1008 1012 1016 +------++------++------++------++------+ Stored Value: | 23 || 31 || 18 || 27 || 35 | +------++------++------++------++------+ Array Index: [0] [1] [2] [3] [4] ? Base address (1000) Each int = 4 bytes. Formula: address(index) = baseAddress + (index × elementSize) address(3) = 1000 + (3 × 4) = 1012 ? value = 27 ? This formula is ONE arithmetic operation = O(1), always instant!
?? Why Index Access is O(1):
The CPU computes address = base + index × size in a single instruction. This formula works the same whether the array has 5 elements or 5 million. That is why arr[i] is always O(1) — it's just arithmetic on memory addresses.

?? IndexOutOfRangeException — The Most Common Beginner Bug:
Arrays are zero-indexed. A 5-element array has valid indices 0, 1, 2, 3, 4.
Accessing index 5 throws IndexOutOfRangeException at runtime.

Safe pattern: always use arr.Length - 1 for the last element, never hardcode the max index.
Section 5 — Array Operations & Their Complexity

5.1 — Read (Access by Index) — O(1)

The superpower of arrays. Instant access to any element using the address formula.

C# 
string[] capitals = { "London", "Paris", "Berlin", "Tokyo", "Sydney" };

string second = capitals[1];              // "Paris"  — O(1)
string last   = capitals[capitals.Length-1]; // "Sydney" — O(1)

// Read is O(1) regardless of array size — the address formula never changes
Console.WriteLine($"Second capital: {second}");
Console.WriteLine($"Last capital: {last}");

capitals = ["London", "Paris", "Berlin", "Tokyo", "Sydney"]

5.2 — Search — O(n) unsorted, O(log n) sorted

When you don't know the index, you must search. Your strategy depends on whether the data is sorted.

LINEAR SEARCH — Find "Tokyo" in unsorted ["London","Paris","Berlin","Tokyo","Sydney"] Check [0]: "London" ? continue Check [1]: "Paris" ? continue Check [2]: "Berlin" ? continue Check [3]: "Tokyo" ? FOUND at index 3 Worst case: item is last or absent ? n checks ? O(n)
C# 
// Linear Search — works on ANY array, sorted or not — O(n)
public static int LinearSearch<T>(T[] arr, T target)
{
    for (int i = 0; i < arr.Length; i++)
        if (arr[i].Equals(target)) return i;
    return -1;
}

// Binary Search — requires SORTED array — O(log n)
// (shown in Section 2.2, and also built in: Array.BinarySearch)
int[] sorted = { 11, 22, 33, 44, 55, 66, 77, 88 };
int   idx    = Array.BinarySearch(sorted, 55); // 4
Python 
# Linear Search — works on ANY list, sorted or not — O(n)
def linear_search(arr, target):
    for i in range(len(arr)):
        if arr[i] == target: return i
    return -1

# Binary Search — requires SORTED list — O(log n)
# Built-in: bisect module (uses binary search internally)
import bisect
sorted_arr = [11, 22, 33, 44, 55, 66, 77, 88]
idx = bisect.bisect_left(sorted_arr, 55)   # finds insertion point
if idx < len(sorted_arr) and sorted_arr[idx] == 55:
    print(f"Found at index {idx}")  # Found at index 4

5.3 — Insert — O(n) worst case

Arrays have fixed size in C#. "Inserting" means creating a new larger array, copying elements, and placing the new value. The closer to the start you insert, the more shifting required.

INSERT 99 at index 2 into [10, 20, 30, 40, 50] Step 1: Allocate new array of size 6 Step 2: Copy elements 0..1: [ 10 | 20 | | | | ] Step 3: Place new value: [ 10 | 20 | 99 | | | ] Step 4: Copy rest shifted: [ 10 | 20 | 99 | 30 | 40 | 50 ] ? Elements shifted: 3 (indices 2,3,4) Worst case: insert at index 0 ? shift ALL n elements ? O(n)
C# 
public static int[] InsertAt(int[] orig, int index, int value)
{
    int[] newArr = new int[orig.Length + 1];
    for (int i = 0; i < index; i++)
        newArr[i] = orig[i];        // copy left portion
    newArr[index] = value;          // insert new value
    for (int i = index; i < orig.Length; i++)
        newArr[i + 1] = orig[i];   // shift right portion
    return newArr;
}
int[] arr = { 10, 20, 30, 40, 50 };
int[] res = InsertAt(arr, 2, 99); // [10,20,99,30,40,50]

// Append to end — O(1) since no shifting:
int[] extended = new int[arr.Length + 1];
Array.Copy(arr, extended, arr.Length);
extended[arr.Length] = 999; // [10,20,30,40,50,999]
Python 
def insert_at(arr, index, value):
    new_arr = [0] * (len(arr) + 1)
    for i in range(index):
        new_arr[i] = arr[i]           # copy left portion
    new_arr[index] = value            # insert new value
    for i in range(index, len(arr)):
        new_arr[i + 1] = arr[i]      # shift right portion
    return new_arr

arr = [10, 20, 30, 40, 50]
res = insert_at(arr, 2, 99)  # [10, 20, 99, 30, 40, 50]

# Pythonic: list.insert() does the same in one call:
arr.insert(2, 99)              # in-place: [10, 20, 99, 30, 40, 50]
# Append to end — O(1) amortized since no shifting:
arr.append(999)               # adds to end

5.4 — Delete — O(n) worst case

Deletion requires shifting all elements after the removed index left by one to close the gap.

DELETE index 1 from [10, 20, 30, 40, 50] Before: [ 10 | 20 | 30 | 40 | 50 ] ? remove index 1 (value 20) Shift left — all elements after index 1 move one position left: After: [ 10 | 30 | 40 | 50 ] ? new size = n-1 Worst case: delete first element ? shift ALL n-1 elements ? O(n)
C# 
public static int[] DeleteAt(int[] orig, int index)
{
    int[] newArr = new int[orig.Length - 1];
    for (int i = 0; i < index; i++)
        newArr[i] = orig[i];            // copy before
    for (int i = index + 1; i < orig.Length; i++)
        newArr[i - 1] = orig[i];       // shift left
    return newArr;
}
int[] arr = { 10, 20, 30, 40, 50 };
int[] res = DeleteAt(arr, 1); // [10,30,40,50]
Python 
def delete_at(arr, index):
    new_arr = [0] * (len(arr) - 1)
    for i in range(index):
        new_arr[i] = arr[i]           # copy before
    for i in range(index + 1, len(arr)):
        new_arr[i - 1] = arr[i]      # shift left
    return new_arr

arr = [10, 20, 30, 40, 50]
res = delete_at(arr, 1)   # [10, 30, 40, 50]

# Pythonic: list.pop(index) or del arr[index]:
arr.pop(1)    # removes index 1 in-place: [10, 30, 40, 50]
# del arr[1]  # equivalent

?? Arrays in Real Production Systems:
  • CPU Caches: Arrays are cache-friendly because adjacent memory = cache hits. Linked lists cause cache misses.
  • Image Processing: Every image is a 2D array of pixels (RGB values). Video frames are sequences of these arrays.
  • Machine Learning: Neural network weights are stored in arrays (tensors). Matrix multiplication is O(n³) array operations.
  • Game Development: Entity positions, tile maps, collision grids — all arrays for O(1) access performance.
