Section 1 — The Analogy: Why Trees?

Think of your family tree. You (the root) have parents, each parent has grandparents, and so on. Every person has exactly one biological parent (except you at the top), and the relationships branch outward. You can't have two parents who are also siblings — the structure is strictly hierarchical with no cycles.

Or think of a file system. Your C:\ drive contains folders, each folder contains sub-folders and files, and sub-folders contain more sub-folders. Navigate "down" to go deeper, "up" to go toward the root. That's a tree.

Or think of a decision tree: "Is the number > 50? Yes ? Is it > 75? No ? Is it > 62?" — each yes/no branches into two choices, creating a binary tree that cuts the search space in half at every level. This is why binary search (and BSTs) are so powerful.

?? The Core Insight: A tree is a recursive structure. Any node is the root of its own subtree. This means tree algorithms are naturally expressed recursively — "do something at this node, then recurse left and right" — and that's exactly how all traversals work.
Section 2 — The Problem Trees Solve

Arrays and linked lists are linear — data is arranged in a single sequence. They struggle with hierarchical relationships:

  • How do you store a file system where folders contain folders? An array would need to know the depth upfront.
  • How do you search 1 million sorted items in 20 steps? A linear scan takes 500,000 on average.
  • How do you evaluate (3 + 5) * (4 - 2)? The expression is inherently nested/hierarchical.
  • How does a compiler parse if (x > 0) { return x; } else { return -x; }? The syntax is a tree.

Trees solve all of these. The key property: at each node you make a binary decision (go left or right). With n items in a balanced tree of height h, h = log2(n). So 1,000,000 items ? height 20. You reach any node in at most 20 steps. That's O(log n) search instead of O(n).

?? Binary Tree vs Binary Search Tree: A binary tree just means each node has at most 2 children — no ordering constraint. A Binary Search Tree (BST) adds the rule "left < root < right". This module covers binary trees (structure + traversals). Module 6B covers BSTs (search, insert, delete).

3.2 — The Five Traversal Orders

A traversal visits every node exactly once in a defined order. The "when you visit the current node" relative to its children defines the traversal type.

Root ? Right): 4, 2, 5, 1, 3, 6 ? Visit left subtree, THEN current node, THEN right subtree ? In a BST: produces SORTED order ? this is why it's called "inorder" PREORDER (Root ? Left ? Right): 1, 2, 4, 5, 3, 6 ? Visit current node FIRST, then left, then right ? Used to: copy/serialize a tree, prefix expression evaluation POSTORDER (Left ? Right ? Root): 4, 5, 2, 6, 3, 1 ? Visit children FIRST, then current node ? Used to: delete a tree, evaluate postfix expressions, calculate sizes LEVEL-ORDER / BFS (level by level): 1, 2, 3, 4, 5, 6 ? Visit level 0, then level 1, then level 2... ? Uses a QUEUE (not recursion) ? Used to: find shortest path, serialize by level REVERSE INORDER (Right ? Root ? Left): 6, 3, 1, 5, 2, 4 ? In a BST: produces DESCENDING sorted order

Section 4 — Formal Definitions & Terminology

4.1 — Tree Types

TypeDefinitionExample
Binary TreeEach node has 0, 1, or 2 childrenGeneral purpose
Full Binary TreeEvery node has 0 or 2 children (never 1)Huffman trees
Complete Binary TreeAll levels filled except possibly the last; last level filled left-to-rightBinary heaps
Perfect Binary TreeAll internal nodes have 2 children AND all leaves are at the same levelTheoretical ideal
Balanced Binary TreeHeight difference between left and right subtrees = 1 for every nodeAVL trees, Red-Black
Degenerate TreeEvery node has only one child — essentially a linked listWorst-case BST

4.2 — Key Properties

  • Height of tree = number of edges on the longest path from root to any leaf. A single node has height 0.
  • Depth of node = number of edges from root to that node. Root has depth 0.
  • Level = depth + 1 (some texts define level starting at 1 instead of 0 — be aware).
  • Perfect binary tree with height h has exactly 2h+1 - 1 nodes.
  • Complete binary tree with n nodes has height ?log2 n?.
FULL vs COMPLETE vs PERFECT FULL (every node has 0 or 2 children): 1 / \ 2 3 / \ 4 5 ? Full (2 and 3 have 0 or 2 children each) COMPLETE (all levels full; last level left-to-right): 1 / \ 2 3 / \ / 4 5 6 ? Complete — 6 is leftmost available position PERFECT (all internals have 2 children, all leaves same level): 1 / \ 2 3 / \ / \ 4 5 6 7 ? Perfect — completely symmetric
Section 5 — Complete C# Implementation
C# 
using System;
using System.Collections.Generic;

// -- Node class --------------------------------------------------
public class TreeNode<T>
{
    public T Value;
    public TreeNode<T>? Left;    // Left child (or null)
    public TreeNode<T>? Right;   // Right child (or null)

    public TreeNode(T val) { Value = val; }
}

// -- BinaryTree class with all five traversals -------------------
public class BinaryTree<T>
{
    public TreeNode<T>? Root;

    // -- INORDER: Left ? Root ? Right ------------------------------
    // In a BST, produces sorted ascending order
    public void Inorder(TreeNode<T>? node, List<T> result)
    {
        if (node == null) return;          // Base case: empty subtree
        Inorder(node.Left, result);           // 1. Recurse left
        result.Add(node.Value);              // 2. Visit current node
        Inorder(node.Right, result);          // 3. Recurse right
    }

    // -- PREORDER: Root ? Left ? Right -----------------------------
    // Useful for serializing/copying a tree structure
    public void Preorder(TreeNode<T>? node, List<T> result)
    {
        if (node == null) return;
        result.Add(node.Value);              // 1. Visit FIRST
        Preorder(node.Left, result);          // 2. Recurse left
        Preorder(node.Right, result);         // 3. Recurse right
    }

    // -- POSTORDER: Left ? Right ? Root ----------------------------
    // Useful for deleting a tree or evaluating expression trees
    public void Postorder(TreeNode<T>? node, List<T> result)
    {
        if (node == null) return;
        Postorder(node.Left, result);         // 1. Recurse left
        Postorder(node.Right, result);        // 2. Recurse right
        result.Add(node.Value);              // 3. Visit LAST
    }

    // -- LEVEL-ORDER (BFS): level by level using a Queue -----------
    public List<List<T>> LevelOrder()
    {
        var result = new List<List<T>>();
        if (Root == null) return result;

        var queue = new Queue<TreeNode<T>>();
        queue.Enqueue(Root);                // Start with root

        while (queue.Count > 0)
        {
            int levelSize = queue.Count;     // All nodes at THIS level
            var level = new List<T>();

            for (int i = 0; i < levelSize; i++)
            {
                var node = queue.Dequeue();
                level.Add(node.Value);       // Visit this node

                if (node.Left != null) queue.Enqueue(node.Left);    // Queue children for NEXT level
                if (node.Right != null) queue.Enqueue(node.Right);
            }
            result.Add(level);              // Save this level's nodes
        }
        return result;
    }

    // -- HEIGHT: longest path from root to any leaf ----------------
    public int Height(TreeNode<T>? node)
    {
        if (node == null) return -1;          // -1 for empty, 0 for leaf
        int leftH  = Height(node.Left);
        int rightH = Height(node.Right);
        return 1 + Math.Max(leftH, rightH);   // 1 + taller subtree height
    }

    // -- COUNT: total number of nodes ------------------------------
    public int Count(TreeNode<T>? node)
    {
        if (node == null) return 0;
        return 1 + Count(node.Left) + Count(node.Right);
    }

    // -- INORDER ITERATIVE: same as recursive but explicit stack ---
    public List<T> InorderIterative()
    {
        var result = new List<T>();
        var stack  = new Stack<TreeNode<T>>();
        var current = Root;

        while (current != null || stack.Count > 0)
        {
            // Keep going left, pushing to stack
            while (current != null)
            {
                stack.Push(current);
                current = current.Left;
            }
            // Pop node, visit it, then go right
            current = stack.Pop();
            result.Add(current.Value);
            current = current.Right;
        }
        return result;
    }
}

// -- Main: Build the sample tree and run all traversals ----------
class Program
{
    static void Main()
    {
        // Build this tree:
        //         1
        //        / \
        //       2   3
        //      / \ /
        //     4  5 6
        var tree = new BinaryTree<int>();
        tree.Root = new TreeNode<int>(1);
        tree.Root.Left              = new TreeNode<int>(2);
        tree.Root.Right             = new TreeNode<int>(3);
        tree.Root.Left.Left         = new TreeNode<int>(4);
        tree.Root.Left.Right        = new TreeNode<int>(5);
        tree.Root.Right.Left        = new TreeNode<int>(6);

        // Run all traversals
        var inorder = new List<int>();
        tree.Inorder(tree.Root, inorder);
        Console.WriteLine("Inorder:    " + string.Join(", ", inorder));

        var preorder = new List<int>();
        tree.Preorder(tree.Root, preorder);
        Console.WriteLine("Preorder:   " + string.Join(", ", preorder));

        var postorder = new List<int>();
        tree.Postorder(tree.Root, postorder);
        Console.WriteLine("Postorder:  " + string.Join(", ", postorder));

        var levels = tree.LevelOrder();
        Console.Write("LevelOrder: ");
        foreach (var lvl in levels)
            Console.Write("[" + string.Join(",", lvl) + "] ");
        Console.WriteLine();

        var iterInorder = tree.InorderIterative();
        Console.WriteLine("Inorder(it):" + string.Join(", ", iterInorder));

        Console.WriteLine($"Height: {tree.Height(tree.Root)}");
        Console.WriteLine($"Count:  {tree.Count(tree.Root)}");
    }
}

Section 6 — Complete Python Implementation
Python 
from collections import deque
from typing import Optional, List

# -- Node class --------------------------------------------------
class TreeNode:
    def __init__(self, val: int):
        self.val = val
        self.left: Optional['TreeNode'] = None   # Left child
        self.right: Optional['TreeNode'] = None  # Right child

# -- BinaryTree with all traversals ------------------------------
class BinaryTree:
    def __init__(self):
        self.root: Optional[TreeNode] = None

    # -- INORDER: Left ? Root ? Right ------------------------------
    def inorder(self, node: Optional[TreeNode], result: List[int]) -> None:
        if node is None: return          # Base case: empty node
        self.inorder(node.left, result)    # 1. Go left
        result.append(node.val)           # 2. Visit current
        self.inorder(node.right, result)   # 3. Go right

    # -- PREORDER: Root ? Left ? Right -----------------------------
    def preorder(self, node: Optional[TreeNode], result: List[int]) -> None:
        if node is None: return
        result.append(node.val)           # 1. Visit FIRST
        self.preorder(node.left, result)
        self.preorder(node.right, result)

    # -- POSTORDER: Left ? Right ? Root ----------------------------
    def postorder(self, node: Optional[TreeNode], result: List[int]) -> None:
        if node is None: return
        self.postorder(node.left, result)
        self.postorder(node.right, result)
        result.append(node.val)           # 3. Visit LAST

    # -- LEVEL-ORDER (BFS): use deque as efficient queue -----------
    def level_order(self) -> List[List[int]]:
        if not self.root: return []
        result: List[List[int]] = []
        queue = deque([self.root])

        while queue:
            level_size = len(queue)          # Nodes at current level
            level: List[int] = []

            for _ in range(level_size):
                node = queue.popleft()       # O(1) with deque
                level.append(node.val)
                if node.left:  queue.append(node.left)
                if node.right: queue.append(node.right)

            result.append(level)
        return result

    # -- HEIGHT ----------------------------------------------------
    def height(self, node: Optional[TreeNode]) -> int:
        if node is None: return -1         # -1 for empty, 0 for leaf
        return 1 + max(self.height(node.left), self.height(node.right))

    # -- COUNT -----------------------------------------------------
    def count(self, node: Optional[TreeNode]) -> int:
        if node is None: return 0
        return 1 + self.count(node.left) + self.count(node.right)

    # -- INORDER ITERATIVE: explicit stack replaces call stack -----
    def inorder_iterative(self) -> List[int]:
        result, stack = [], []
        current = self.root

        while current or stack:
            while current:               # Push all left nodes
                stack.append(current)
                current = current.left
            current = stack.pop()       # Visit node
            result.append(current.val)
            current = current.right       # Move to right subtree
        return result


if __name__ == "__main__":
    # Build: 1 at root, 2 left, 3 right, 4&5 children of 2, 6 left child of 3
    tree = BinaryTree()
    tree.root           = TreeNode(1)
    tree.root.left      = TreeNode(2)
    tree.root.right     = TreeNode(3)
    tree.root.left.left  = TreeNode(4)
    tree.root.left.right = TreeNode(5)
    tree.root.right.left = TreeNode(6)

    in_res = []
    tree.inorder(tree.root, in_res)
    print("Inorder:    ", in_res)

    pre_res = []
    tree.preorder(tree.root, pre_res)
    print("Preorder:   ", pre_res)

    post_res = []
    tree.postorder(tree.root, post_res)
    print("Postorder:  ", post_res)

    print("LevelOrder: ", tree.level_order())
    print("Inorder(it):", tree.inorder_iterative())

Section 7 — Complexity Analysis
OperationTimeSpace (stack)Why
Inorder / Preorder / PostorderO(n)O(h)Visit every node once; stack depth = tree height h
Level-Order (BFS)O(n)O(w)Visit every node once; queue holds at most w nodes (max width of tree)
HeightO(n)O(h)Must visit all nodes to find the tallest path
CountO(n)O(h)Must visit every node
Inorder IterativeO(n)O(h)Same logic, explicit stack holds at most h nodes
?? Balanced vs Degenerate Space Cost:
  • Balanced tree: h = O(log n) ? recursion depth = O(log n) ? safe for n = 106 (depth ˜ 20)
  • Degenerate tree (linked-list shape): h = O(n) ? recursion depth = O(n) ? stack overflow for n > 10,000 on most runtimes
  • This is why iterative inorder matters in interviews — it avoids stack overflow
  • BFS space = O(w) = O(n) for a perfect tree's last level (holds n/2 nodes)
Section 8 — Real-World Usage
?? File System (Every OS) — Preorder for listing, Postorder for sizing
Every file system is a tree. C:\Windows\System32\kernel32.dll is a path from root ? Windows ? System32 ? kernel32.dll. When you run du -sh * to get folder sizes, the OS does a postorder traversal — calculate children sizes first, then the parent's total.
Windows Explorer's tree view is preorder — show the folder first, then recurse into its contents.
?? HTML DOM (Every Browser) — Level-order for layout, DFS for matching
A web page's HTML is a tree: <html> ? <body> ? <div> ? etc. Browsers use DFS (preorder) when applying CSS selectors — walk down the tree, check if this node matches div.container > p. They use BFS (level-order) for layout calculations — lay out parent dimensions before children.
?? Compilers (Roslyn / GCC) — Postorder for code generation
The expression (3 + 5) * (4 - 2) is stored as an Abstract Syntax Tree (AST). The compiler uses postorder traversal: evaluate children first (3+5=8, 4-2=2), then apply the operator at the parent (8*2=16). Roslyn (C#'s compiler) stores every C# program as a syntax tree and traverses it postorder for type checking and code emission.
Section 9 — Common Mistakes
?? Mistake 1 — Null check missing: Every recursive call must check if (node == null) return first. Missing this causes a NullReferenceException on leaf nodes when you try to access node.Left. This is the #1 tree interview bug.
?? Mistake 2 — Confusing height and depth: Height is measured from a node DOWN to the deepest leaf. Depth is measured from the root DOWN to the node. A leaf node at depth 5 has height 0. The tree's height equals the root's height.
?? Mistake 3 — Using list.sort() instead of inorder: On a BST, inorder traversal already gives you sorted output in O(n). Collecting all values then sorting is O(n log n) — twice as slow and shows you don't understand the structure.
?? Mistake 4 — Stack overflow on deep trees: Recursive traversal uses the call stack. A degenerate BST built from sorted input (1?2?3?...?n) has height n. Recursing into it with n = 100,000 will crash with StackOverflowException. Use iterative traversal for production code on untrusted inputs.
?? Mistake 5 — Using list as queue in Python: list.pop(0) is O(n). Always use collections.deque with popleft() for O(1) BFS dequeues. Using a list as a queue turns O(n) BFS into O(n²).
Section 10 — Check Your Understanding
Q1. Given a binary tree with n nodes, what is the space complexity of a recursive inorder traversal on a balanced tree?
?? Think about what the call stack holds at any point during recursion.
O(log n) — In a balanced tree, the height h = ?log2 n?. At any point, the recursive call stack only holds nodes along the current path from root to the deepest active node — at most h frames. So space = O(h) = O(log n) for a balanced tree. For a degenerate tree (linked list), it becomes O(n).
Q2. You need to serialize a binary tree to a string, then reconstruct it exactly. Which traversal should you use?
?? Which traversal tells you the structure before you know the children?
Preorder — Preorder visits the root before its children. When deserializing, you read the root value first (so you know what to reconstruct), then recursively rebuild left and right subtrees. Include null markers (e.g., "#") so you know where subtrees end. Inorder alone is not sufficient to uniquely reconstruct a tree.
Q3. For the tree below, what is the level-order output?
5 / \ 3 8 / \ \ 1 4 9
?? BFS visits one level at a time, left to right.
[5], [3, 8], [1, 4, 9]
Level 0: 5
Level 1: 3, 8 (left-to-right)
Level 2: 1, 4 (children of 3), 9 (right child of 8; 8 has no left child)
The queue processes: Enqueue 5 ? dequeue 5, enqueue 3,8 ? dequeue 3, enqueue 1,4 ? dequeue 8, enqueue 9 ? ...
Q4. A "perfect" binary tree has 4 levels (level 0 through 3). How many nodes does it have?
?? Use the formula: 2^(h+1) - 1 where h is the height.
15 nodes — Height = 3 (levels 0–3). Formula: 2^(3+1) - 1 = 24 - 1 = 16 - 1 = 15.
Alternatively: Level 0 = 1 node, Level 1 = 2, Level 2 = 4, Level 3 = 8. Sum = 1+2+4+8 = 15.
Q5. Which traversal would you use to delete all nodes in a tree without causing memory leaks?
?? Think about the order in which it's safe to free memory — you can't delete a parent before its children.
Postorder — You must delete children before their parent. If you deleted the root first, you'd lose your only references to the subtrees, causing memory leaks. Postorder visits left subtree ? right subtree ? current node, so children are always processed (and freed) before their parent. In garbage-collected languages (C#, Python) this matters less, but in C/C++ it's critical.
