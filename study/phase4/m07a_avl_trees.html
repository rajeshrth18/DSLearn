Section 1 — The Analogy: The Self-Adjusting Bookshelf

Imagine a balanced mobile — the kind that hangs from the ceiling, with ornaments on each arm. If you add a heavy ornament to one side, the mobile tilts. To rebalance, you either move weights around or restructure the arms. The mobile always restores balance after each change.

An AVL tree works the same way. After each insert, it measures how "tilted" each subtree is. If a node becomes too unbalanced (difference in subtree heights > 1), it performs a rotation — a structural rearrangement that preserves BST ordering while restoring balance. The tree automatically adjusts itself after every modification.

💡 Named after: Georgy Adelson-Velsky and Evgenii Landis, who published the first self-balancing BST in 1962. The AVL tree was the first data structure to guarantee O(log n) search, insert, and delete in the worst case.
Section 2 — The Problem AVL Trees Solve

A plain BST can degenerate into a linked list when elements are inserted in sorted order. For n = 1,000,000 elements inserted in order: height = 999,999, search = O(n). The solution? After every insert, check the balance and rotate if needed.

📌 AVL Tree Invariant: For every node N in an AVL tree, the difference between the heights of the left and right subtrees (called the balance factor) must be −1, 0, or +1.
balance_factor(N) = height(N.right) - height(N.left)
If |balance_factor| > 1 after an insert or delete, perform rotations to restore balance.
Section 3 — Visual Walkthrough: The 4 Rotation Cases

3.1 — Right Rotation (LL Case)

Triggered when: a node is left-heavy (bf = −2) AND its left child is also left-heavy (bf ≤ 0). The imbalance is in the Left-Left direction.

LL CASE — Insert 10, 5, 2 into empty AVL tree: After insert 2, node 10 becomes unbalanced (bf = -2, left-left): 10 ← bf = -2 (unbalanced!) / 5 ← bf = -1 / 2 RIGHT ROTATION on node 10: • The left child (5) becomes the new root of this subtree • 10 becomes the right child of 5 • 5's right child (if any) becomes 10's left child BEFORE: AFTER rotation: 10 5 / / \ 5 → 2 10 / 2 Result: balanced! bf of 5 = 0, bf of 2 = 0, bf of 10 = 0

3.2 — Left Rotation (RR Case)

Triggered when: a node is right-heavy (bf = +2) AND its right child is also right-heavy (bf ≥ 0). Mirror of LL case.

RR CASE — Insert 2, 5, 10: After insert 10, node 2 becomes unbalanced (bf = +2, right-right): 2 ← bf = +2 (unbalanced!) \ 5 ← bf = +1 \ 10 LEFT ROTATION on node 2: • The right child (5) becomes the new root • 2 becomes the left child of 5 • 5's left child (if any) becomes 2's right child BEFORE: AFTER rotation: 2 5 \ / \ 5 → 2 10 \ 10

3.3 — Left-Right (LR) Rotation

Triggered when: node is left-heavy (bf = −2) BUT left child is right-heavy (bf = +1). Requires two rotations: first a left rotation on the left child, then a right rotation on the unbalanced node.

LR CASE — Insert 10, 2, 5: After insert 5: 10 ← bf = -2 (unbalanced) / 2 ← bf = +1 (right-heavy child — this is the LR signal!) \ 5 STEP 1: Left rotate on the left child (2): 10 / 5 ← 5 moves up, 2 becomes its left child / 2 STEP 2: Right rotate on unbalanced node (10): 10 5 / → / \ 5 2 10 / 2 Result: balanced!

3.4 — Right-Left (RL) Rotation

Triggered when: node is right-heavy (bf = +2) BUT right child is left-heavy (bf = −1). Mirror of LR case.

RL CASE — Insert 2, 10, 5: After insert 5: 2 ← bf = +2 (unbalanced) \ 10 ← bf = -1 (left-heavy child — this is the RL signal!) / 5 STEP 1: Right rotate on right child (10): 2 \ 5 ← 5 moves up, 10 becomes its right child \ 10 STEP 2: Left rotate on unbalanced node (2): 2 5 \ → / \ 5 2 10 \ 10 Result: balanced!
📌 Summary of 4 Cases:
CaseConditionFix
LL (Left-Left)bf(node) = -2, bf(left child) ≤ 0Right rotate on node
RR (Right-Right)bf(node) = +2, bf(right child) ≥ 0Left rotate on node
LR (Left-Right)bf(node) = -2, bf(left child) = +1Left rotate on left child, then right rotate on node
RL (Right-Left)bf(node) = +2, bf(right child) = -1Right rotate on right child, then left rotate on node
Section 4 — Formal Definitions
  • Height of a node: longest path from that node to any leaf. Height of null = −1. Height of leaf = 0.
  • Balance factor: bf(N) = height(N.right) - height(N.left). Valid range: {−1, 0, +1}.
  • AVL property: every node in the tree has |bf| ≤ 1.
  • Height of AVL tree: at most 1.44 × log₂(n+2) − 0.328. In practice, very close to log₂(n).
💡 Critical implementation order: When updating heights after rotation, ALWAYS update the children's heights before the parent's height, because the parent's height depends on children's heights. In rotations: update the node that moved DOWN first, then update the new root.
Section 5 — Complete C# Implementation
C# 
using System;

// ── AVL Node ──────────────────────────────────────────────────────
public class AVLNode
{
    public int Value, Height;
    public AVLNode? Left, Right;
    public AVLNode(int val) { Value = val; Height = 0; }
}

// ── AVL Tree ──────────────────────────────────────────────────────
public class AVLTree
{
    public AVLNode? Root;

    // ── Height helper (handles null) ────────────────────────────
    private int H(AVLNode? n) => n == null ? -1 : n.Height;

    // ── Balance factor = height(right) - height(left) ───────────
    private int BF(AVLNode? n) => n == null ? 0 : H(n.Right) - H(n.Left);

    // ── Update height after structural changes ───────────────────
    private void UpdateHeight(AVLNode n) =>
        n.Height = 1 + Math.Max(H(n.Left), H(n.Right));

    // ── RIGHT ROTATION (fixes LL imbalance) ─────────────────────
    //         y                x
    //        / \             / \
    //       x   T3    →     T1   y
    //      / \                  / \
    //     T1  T2               T2  T3
    private AVLNode RotateRight(AVLNode y)
    {
        AVLNode x  = y.Left!;   // New root
        AVLNode? T2 = x.Right;   // x's right subtree moves to y's left

        x.Right = y;             // y becomes right child of x
        y.Left  = T2;            // T2 moves to y's left

        UpdateHeight(y);         // Update y FIRST (it's now lower)
        UpdateHeight(x);         // Then update x (it's the new parent)
        return x;               // Return new root of this subtree
    }

    // ── LEFT ROTATION (fixes RR imbalance) ──────────────────────
    //       x                  y
    //      / \                / \
    //     T1   y    →         x   T3
    //         / \            / \
    //        T2  T3          T1  T2
    private AVLNode RotateLeft(AVLNode x)
    {
        AVLNode y  = x.Right!;
        AVLNode? T2 = y.Left;

        y.Left  = x;
        x.Right = T2;

        UpdateHeight(x);
        UpdateHeight(y);
        return y;
    }

    // ── BALANCE: apply correct rotation(s) based on balance factor ─
    private AVLNode Balance(AVLNode node)
    {
        UpdateHeight(node);
        int bf = BF(node);

        // Left-heavy: bf = -2
        if (bf == -2)
        {
            if (BF(node.Left) > 0)                  // LR case: left child is right-heavy
                node.Left = RotateLeft(node.Left!);  // Step 1: left rotate left child
            return RotateRight(node);               // LL or LR step 2: right rotate node
        }

        // Right-heavy: bf = +2
        if (bf == 2)
        {
            if (BF(node.Right) < 0)                   // RL case: right child is left-heavy
                node.Right = RotateRight(node.Right!);  // Step 1: right rotate right child
            return RotateLeft(node);                   // RR or RL step 2: left rotate node
        }

        return node;  // Already balanced (-1, 0, or 1)
    }

    // ── INSERT ───────────────────────────────────────────────────
    public void Insert(int val) => Root = InsertRec(Root, val);

    private AVLNode InsertRec(AVLNode? node, int val)
    {
        // Standard BST insert
        if (node == null) return new AVLNode(val);
        if (val < node.Value)  node.Left  = InsertRec(node.Left,  val);
        else if (val > node.Value) node.Right = InsertRec(node.Right, val);
        else return node;   // Duplicate: ignore

        // Rebalance on the way back up (post-order)
        return Balance(node);
    }

    // ── CONTAINS ─────────────────────────────────────────────────
    public bool Contains(int val)
    {
        var n = Root;
        while (n != null)
        {
            if   (val == n.Value) return true;
            else if (val <  n.Value) n = n.Left;
            else                     n = n.Right;
        }
        return false;
    }

    // ── HEIGHT of root ────────────────────────────────────────────
    public int Height() => H(Root);
}

// ── Main ──────────────────────────────────────────────────────────
class Program
{
    static void Main()
    {
        var avl = new AVLTree();

        // Insert in SORTED order — would degenerate a plain BST!
        foreach (int v in new[] { 1, 2, 3, 4, 5, 6, 7 })
            avl.Insert(v);

        Console.WriteLine($"Height after inserting 1-7 in order: {avl.Height()}");
        // Plain BST would be height 6; AVL keeps it at 2

        Console.WriteLine($"Contains 4: {avl.Contains(4)}");
        Console.WriteLine($"Contains 9: {avl.Contains(9)}");

        // Test LR case
        var avl2 = new AVLTree();
        foreach (int v in new[] { 10, 2, 5 })  // LR case
            avl2.Insert(v);
        Console.WriteLine($"LR case height: {avl2.Height()}");  // Should be 1
    }
}

/* ── Output ──────────────────────────────────────────────────────
   Height after inserting 1-7 in order: 2   ← AVL keeps it balanced!
   Contains 4: True
   Contains 9: False
   LR case height: 1                         ← Correctly balanced
   ──────────────────────────────────────────────────────────────── */
Section 6 — Complete Python Implementation
Python 
from typing import Optional

# ── AVL Node ──────────────────────────────────────────────────────
class AVLNode:
    def __init__(self, val: int):
        self.val    = val
        self.height = 0
        self.left:  Optional['AVLNode'] = None
        self.right: Optional['AVLNode'] = None

# ── AVL Tree ──────────────────────────────────────────────────────
class AVLTree:
    def __init__(self):
        self.root: Optional[AVLNode] = None

    def _h(self, n: Optional[AVLNode]) -> int:
        return -1 if n is None else n.height

    def _bf(self, n: Optional[AVLNode]) -> int:
        return 0 if n is None else self._h(n.right) - self._h(n.left)

    def _update_height(self, n: AVLNode) -> None:
        n.height = 1 + max(self._h(n.left), self._h(n.right))

    # ── Rotations ────────────────────────────────────────────────
    def _rotate_right(self, y: AVLNode) -> AVLNode:
        x  = y.left
        T2 = x.right
        x.right = y
        y.left  = T2
        self._update_height(y)   # y first (now lower)
        self._update_height(x)   # x is new root
        return x

    def _rotate_left(self, x: AVLNode) -> AVLNode:
        y  = x.right
        T2 = y.left
        y.left  = x
        x.right = T2
        self._update_height(x)
        self._update_height(y)
        return y

    # ── Balance ───────────────────────────────────────────────────
    def _balance(self, node: AVLNode) -> AVLNode:
        self._update_height(node)
        bf = self._bf(node)

        if bf == -2:  # Left heavy
            if self._bf(node.left) > 0:          # LR case
                node.left = self._rotate_left(node.left)
            return self._rotate_right(node)

        if bf == 2:   # Right heavy
            if self._bf(node.right) < 0:         # RL case
                node.right = self._rotate_right(node.right)
            return self._rotate_left(node)

        return node

    # ── Insert ───────────────────────────────────────────────────
    def insert(self, val: int) -> None:
        self.root = self._insert(self.root, val)

    def _insert(self, node: Optional[AVLNode], val: int) -> AVLNode:
        if node is None: return AVLNode(val)
        if   val < node.val: node.left  = self._insert(node.left,  val)
        elif val > node.val: node.right = self._insert(node.right, val)
        else: return node   # Duplicate
        return self._balance(node)

    # ── Contains (same as plain BST) ─────────────────────────────
    def contains(self, val: int) -> bool:
        node = self.root
        while node:
            if   val == node.val: return True
            elif val <  node.val: node = node.left
            else:                 node = node.right
        return False

    def height(self) -> int:
        return self._h(self.root)


if __name__ == "__main__":
    avl = AVLTree()
    for v in [1, 2, 3, 4, 5, 6, 7]:
        avl.insert(v)

    print("Height (sorted insert):", avl.height())  # 2, not 6!
    print("Contains 4:", avl.contains(4))           # True
    print("Contains 9:", avl.contains(9))           # False

    avl2 = AVLTree()
    for v in [10, 2, 5]:  # LR case
        avl2.insert(v)
    print("LR case height:", avl2.height())         # 1

# Output:
# Height (sorted insert): 2
# Contains 4: True
# Contains 9: False
# LR case height: 1

Section 8 — Real-World Usage
🌍 Where AVL Trees Excel: AVL trees are preferred over Red-Black trees when search performance is more critical than insert performance. AVL trees are more strictly balanced (height ≤ 1.44 log n) than Red-Black trees (height ≤ 2 log n), meaning searches are faster. They're used in: in-memory databases, compiler symbol tables that are built once and queried frequently, and the Windows NT kernel object manager.
🌍 AVL vs. Red-Black: Red-Black trees (Module 7B) allow slightly looser balance but require fewer rotations on insert/delete (amortized O(1) rotations vs. O(log n) for AVL). This is why most modern language standard libraries (C++ std::map, Java TreeMap, C# SortedDictionary) use Red-Black trees — they optimize for the "lots of inserts/deletes" workload. Use AVL when your workload is "build once, query many times."
Section 9 — Common Mistakes
⚠️ Mistake 1 — Wrong height update order in rotations: Always update heights bottom-up. In a right rotation: first call UpdateHeight(y) (the node that moved down), then UpdateHeight(x) (the new root). Reversing this gives x an incorrect height because it depends on y's updated height.
⚠️ Mistake 2 — Not calling Balance() on the way back up: Balance must be called on EVERY ancestor of the inserted node as the recursion unwinds — not just the first unbalanced ancestor. The first rotation may fix one imbalance but higher ancestors may become unbalanced too.
⚠️ Mistake 3 — Confusing LR and RL detection: LR case is when the left child is right-heavy (bf = +1). RL case is when the right child is left-heavy (bf = −1). Check the balance factor of the child, not the current node.
⚠️ Mistake 4 — Height of null node: Always define height of null as −1 (not 0). A leaf node (height = 0) has two null children of height −1. This ensures the balance factor formula works consistently: a leaf has bf = (−1) − (−1) = 0.
Section 10 — Check Your Understanding
Q1. Insert 30, 20, 10 into an empty AVL tree. What rotation fires and what is the resulting root?
💡 Trace the balance factors after inserting 10. What case do you see?
LL case → Right rotation → root becomes 20.
After inserting 30, 20, 10: node 30 has bf = −2 (left-heavy). Its left child 20 has bf = −1 (also left-heavy). This is the LL case. Apply a right rotation on 30: 20 becomes the new root, 10 stays left of 20, 30 becomes right of 20. Final tree: 20 (root), left=10, right=30. All balance factors = 0.
Q2. What rotation(s) fire when you insert 10, 30, 20 into an empty AVL tree?
💡 After inserting 20, check the balance of node 10. Then check 30's balance factor.
RL case → Right rotation on 30, then Left rotation on 10.
After 10, 30, 20: node 10 has bf = +2 (right-heavy). Its right child 30 has bf = −1 (left-heavy). This is RL case. Step 1: right rotate 30 → 20 moves up, 30 becomes right of 20. Step 2: left rotate 10 → 20 becomes new root, 10 becomes left, 30 becomes right. Final: root = 20.
Q3. What is the maximum height of an AVL tree with 7 nodes?
💡 Use the formula h ≤ 1.44 × log₂(n+2), or just think about it structurally.
Height = 2.
With 7 nodes in a perfectly balanced complete binary tree: root at level 0, 2 nodes at level 1, 4 nodes at level 2. That's 1+2+4 = 7 nodes, height = 2. Formula: ⌊log₂(7)⌋ = 2. In general for 7 nodes, the minimum height is ⌈log₂(7+1)⌉ = 3, but since we have exactly 7 = 2³−1 (perfect tree), height is exactly 2.
Q4. After a right rotation on node Y with left child X: which node's height must be updated FIRST, and why?
💡 Think about which node is now lower in the tree after the rotation.
Y's height must be updated first.
After a right rotation, Y moves DOWN and becomes X's right child. Y's height depends on its (new) left and right children. X's height depends on Y's (updated) height. If you update X first, it sees Y's old (incorrect) height. Always update heights from the bottom up: first update the node that moved down (Y), then update the new root (X).
Q5. You need a data structure that supports 10 million inserts (in random order) followed by 500 million searches. Should you use an AVL tree or a Red-Black tree?
💡 Consider the relative search performance vs. insert performance of each tree type.
AVL tree.
The workload is dominated by searches (500M vs. 10M inserts). AVL trees maintain stricter balance (height ≤ 1.44 log n vs. 2 log n for Red-Black), giving faster searches. The extra rotation overhead during the 10M inserts is a one-time cost. If the workload had more inserts than searches, Red-Black would be preferable because its amortized O(1) rotation cost per insert is cheaper.
