Section 1 — The Analogy: Library Card Catalog

Before digital catalogs, libraries used physical card catalogs — drawers organized alphabetically. Each drawer held many cards (not just one). To find a book, you went to the right drawer, found the right divider tab, then scanned a few cards. The key insight: each drawer lookup is expensive (physical trip), but looking through the cards within a drawer is cheap (sequential scan).

A B-tree works exactly this way. Each "node" is like a drawer — it holds multiple keys and pointers. Instead of one comparison per node (like a BST), you do many comparisons within a node but make far fewer node-to-node jumps. Since each jump is a disk read (4KB–16KB page), fewer jumps means dramatically fewer I/O operations.

Section 2 — The Problem B-Trees Solve

When data doesn't fit in RAM, it lives on disk. Disk reads are 10,000× slower than RAM access (magnetic HDD: ~5ms seek; SSD: ~0.1ms; RAM: ~100ns). A BST or Red-Black tree with 1 billion records has height ~30. Each "node visit" is a random disk read → 30 disk reads per lookup. Unacceptable for a database serving thousands of queries per second.

A B-tree with branching factor t=1000 packs 999 keys per node. The same 1 billion records fit in a tree of height:

h = ⌈log₁₀₀₀(10⁹)⌉ = ⌈3⌉ = 3 levels → 3 disk reads

This is why every database uses B-trees (or B+ trees) for indexes.

📌 B-tree vs B+ tree:
B-tree: Keys AND data stored in all nodes (internal + leaves).
B+ tree: Keys in internal nodes serve only as routing guides. All actual data stored ONLY in leaf nodes. Leaves are linked as a doubly-linked list for efficient range scans.
MySQL InnoDB, PostgreSQL, SQLite all use B+ trees because the leaf-linked-list makes range queries (BETWEEN, ORDER BY) trivially efficient.
Section 3 — Visual Walkthrough: B-Tree of Minimum Degree t=2
B-TREE PROPERTIES (minimum degree t): • Every node has at most 2t - 1 keys (max 2t children) • Every non-root node has at least t - 1 keys (min t children) • The root has at least 1 key (min 2 children if non-leaf) • All leaves are at the same depth For t=2: each node holds 1 to 3 keys, 2 to 4 children. ═══════════════════════════════════════════════════════════ INSERTING [10, 20, 30, 40, 50] into an empty B-tree (t=2, max 3 keys/node): After inserting 10, 20, 30: [10 | 20 | 30] ← root is full (3 keys = 2t-1 = 3) Inserting 40 — root is full → SPLIT first: Step 1: Find median = 20 (middle key, index t-1 = 1) Step 2: Split into two nodes: [10] and [30, 40] Step 3: Push median 20 up to new root Result: [20] / \ [10] [30] Then insert 40: [20] / \ [10] [30 | 40] Insert 50 — right child has room (2 keys < 3 max): [20] / \ [10] [30 | 40 | 50] ← full again ═══════════════════════════════════════════════════════════ SEARCH for 35 in the above tree: Start at root [20]: 35 > 20 → go right child At node [30 | 40 | 50]: 30 ≤ 35 < 40 → 35 is between 30 and 40 → not found (leaf node, no more children) → KEY NOT FOUND

B+ Tree Leaf Linkage

B+ TREE difference: all data is in leaf nodes, linked left-to-right: [20 | 40] ← internal (routing only) / | \ [10|15] [20|30] [40|50] ← leaves store actual data ↔ ↔ ↔ ← doubly-linked for range scans RANGE QUERY: "Find all keys between 15 and 40" Step 1: Find leaf containing 15 → [10|15], O(log_t n) Step 2: Scan linked list rightward: 15, 20, 30, 40 → DONE No tree traversal needed for the range — just follow the list!
Section 4 — Formal Definitions
  • Minimum degree t (≥ 2): The fundamental parameter. Every non-root node has between t-1 and 2t-1 keys. Every internal node has between t and 2t children.
  • Order (sometimes called k or m): Different books define "order" differently. Some define order = max number of children (= 2t). Others define order = min number of children. Always clarify which definition is being used.
  • Height guarantee: For n keys, h ≤ log_t((n+1)/2). With t=1000 and n=10⁹: h ≤ log₁₀₀₀(500,000,000) ≈ 3.
  • Split on full node: When inserting, if a node reaches 2t-1 keys, it's split: the median key is pushed to the parent, the node splits into two nodes each with t-1 keys.
  • Proactive splitting: Split full nodes top-down on the way DOWN during insert (not bottom-up). This avoids needing to traverse back up.
Section 5 — Complete C# Implementation
C# 
using System;
using System.Collections.Generic;

public class BTreeNode
{
    public List<int>         Keys     = new();
    public List<BTreeNode>  Children = new();
    public bool              IsLeaf   = true;
}

public class BTree
{
    private readonly int T;   // Minimum degree
    private BTreeNode Root;

    public BTree(int t)
    {
        T    = t;
        Root = new BTreeNode();
    }

    // ── SEARCH ────────────────────────────────────────────────────
    public bool Search(int key)
        => SearchNode(Root, key);

    private bool SearchNode(BTreeNode node, int key)
    {
        int i = 0;
        // Find the first key >= search key
        while (i < node.Keys.Count && key > node.Keys[i]) i++;

        if (i < node.Keys.Count && key == node.Keys[i]) return true;  // Found!
        if (node.IsLeaf) return false;   // Leaf: not found
        return SearchNode(node.Children[i], key);  // Recurse into correct child
    }

    // ── INSERT ────────────────────────────────────────────────────
    public void Insert(int key)
    {
        if (Root.Keys.Count == 2 * T - 1)  // Root is full → grow the tree
        {
            var newRoot = new BTreeNode { IsLeaf = false };
            newRoot.Children.Add(Root);
            Root = newRoot;
            SplitChild(newRoot, 0);  // Split the old root
        }
        InsertNonFull(Root, key);
    }

    // Insert into a node that is guaranteed NOT full
    private void InsertNonFull(BTreeNode node, int key)
    {
        int i = node.Keys.Count - 1;

        if (node.IsLeaf)
        {
            // Insert into sorted position in leaf
            node.Keys.Add(0);  // Make room
            while (i >= 0 && key < node.Keys[i])
            {
                node.Keys[i + 1] = node.Keys[i];
                i--;
            }
            node.Keys[i + 1] = key;
        }
        else
        {
            // Find the correct child
            while (i >= 0 && key < node.Keys[i]) i--;
            i++;  // i = index of the child to recurse into

            if (node.Children[i].Keys.Count == 2 * T - 1)
            {
                SplitChild(node, i);   // Proactively split full child
                if (key > node.Keys[i]) i++;
            }
            InsertNonFull(node.Children[i], key);
        }
    }

    // Split child at index childIdx of parent
    private void SplitChild(BTreeNode parent, int childIdx)
    {
        BTreeNode full  = parent.Children[childIdx];
        BTreeNode right = new BTreeNode { IsLeaf = full.IsLeaf };

        int mid = T - 1;   // Index of median key in full node

        // Right half of full.Keys (after median) → new right node
        right.Keys.AddRange(full.Keys.GetRange(T, T - 1));
        if (!full.IsLeaf)
            right.Children.AddRange(full.Children.GetRange(T, T));

        // Promote median key to parent
        parent.Keys.Insert(childIdx, full.Keys[mid]);
        parent.Children.Insert(childIdx + 1, right);

        // Truncate full node to left half (keys before median)
        full.Keys.RemoveRange(mid, T);
        if (!full.IsLeaf)
            full.Children.RemoveRange(T, T);
    }

    // ── PRINT tree (BFS level by level) ──────────────────────────
    public void Print()
    {
        var queue = new Queue<(BTreeNode node, int level)>();
        queue.Enqueue((Root, 0));
        int currentLevel = 0;
        while (queue.Count > 0)
        {
            var (node, level) = queue.Dequeue();
            if (level > currentLevel) { Console.WriteLine(); currentLevel = level; }
            Console.Write("[" + string.Join("|", node.Keys) + "] ");
            foreach (var child in node.Children) queue.Enqueue((child, level + 1));
        }
        Console.WriteLine();
    }
}

class Program
{
    static void Main()
    {
        var bt = new BTree(2);  // t=2: each node holds 1..3 keys
        foreach (int k in new[] { 10, 20, 5, 6, 12, 30, 7, 17 })
            bt.Insert(k);

        Console.WriteLine("B-Tree (t=2) after inserting [10,20,5,6,12,30,7,17]:");
        bt.Print();

        Console.WriteLine("Search 6:  " + bt.Search(6));   // True
        Console.WriteLine("Search 15: " + bt.Search(15));  // False
    }
}

/* ── Output ──────────────────────────────────────────────────────
   B-Tree (t=2) after inserting [10,20,5,6,12,30,7,17]:
   [6|12]
   [5] [7|10] [17|20|30]
   Search 6:  True
   Search 15: False
   ──────────────────────────────────────────────────────────────── */
Section 6 — Complete Python Implementation
Python 
from typing import List, Optional
from collections import deque


class BTreeNode:
    def __init__(self, is_leaf: bool = True):
        self.keys:     List[int]           = []
        self.children: List['BTreeNode']   = []
        self.is_leaf:  bool                  = is_leaf


class BTree:
    def __init__(self, t: int):
        self.t    = t               # Minimum degree
        self.root = BTreeNode()

    def search(self, key: int) -> bool:
        return self._search(self.root, key)

    def _search(self, node: BTreeNode, key: int) -> bool:
        i = 0
        while i < len(node.keys) and key > node.keys[i]: i += 1
        if i < len(node.keys) and key == node.keys[i]: return True
        if node.is_leaf: return False
        return self._search(node.children[i], key)

    def insert(self, key: int) -> None:
        if len(self.root.keys) == 2 * self.t - 1:
            new_root = BTreeNode(is_leaf=False)
            new_root.children.append(self.root)
            self.root = new_root
            self._split_child(new_root, 0)
        self._insert_non_full(self.root, key)

    def _insert_non_full(self, node: BTreeNode, key: int) -> None:
        i = len(node.keys) - 1
        if node.is_leaf:
            node.keys.append(0)   # Make room
            while i >= 0 and key < node.keys[i]:
                node.keys[i + 1] = node.keys[i]
                i -= 1
            node.keys[i + 1] = key
        else:
            while i >= 0 and key < node.keys[i]: i -= 1
            i += 1
            if len(node.children[i].keys) == 2 * self.t - 1:
                self._split_child(node, i)
                if key > node.keys[i]: i += 1
            self._insert_non_full(node.children[i], key)

    def _split_child(self, parent: BTreeNode, idx: int) -> None:
        t    = self.t
        full  = parent.children[idx]
        right = BTreeNode(is_leaf=full.is_leaf)
        mid   = t - 1

        right.keys = full.keys[t:]
        if not full.is_leaf:
            right.children = full.children[t:]

        parent.keys.insert(idx, full.keys[mid])
        parent.children.insert(idx + 1, right)

        full.keys     = full.keys[:mid]
        if not full.is_leaf:
            full.children = full.children[:t]

    def print_tree(self) -> None:
        q = deque([(self.root, 0)])
        cur_level = 0
        while q:
            node, level = q.popleft()
            if level > cur_level: print(); cur_level = level
            print("[" + "|".join(map(str, node.keys)) + "]", end=" ")
            for ch in node.children: q.append((ch, level + 1))
        print()


if __name__ == "__main__":
    bt = BTree(t=2)
    for k in [10, 20, 5, 6, 12, 30, 7, 17]:
        bt.insert(k)
    print("B-Tree (t=2):")
    bt.print_tree()
    print("search 6:",  bt.search(6))    # True
    print("search 15:", bt.search(15))   # False

# Output:
# B-Tree (t=2):
# [6|12]
# [5] [7|10] [17|20|30]
# search 6: True
# search 15: False

Section 8 — Real-World Usage
🌍 MySQL InnoDB — B+ Tree Clustered Index
InnoDB's primary key index is a clustered B+ tree — leaf nodes contain the actual row data (not just pointers). Secondary indexes are non-clustered B+ trees whose leaves contain the primary key value. A typical MySQL B+ tree page is 16KB, holding ~1000 keys per page. A table with 1 billion rows needs only 3-4 levels → 3-4 page reads per lookup.
🌍 PostgreSQL — B-Tree Indexes
PostgreSQL uses a variant of B+ trees for all its standard indexes (CREATE INDEX). The default page size is 8KB. Unlike MySQL's clustered index, PostgreSQL's heap storage is separate from the index, so all indexes are non-clustered. The B+ tree stores (key, tuple pointer) pairs in leaf pages.
🌍 NTFS, HFS+ — File System Metadata
File systems use B-trees to store directory entries and file metadata. NTFS's Master File Table (MFT) uses B-tree indices for directory lookups. Finding a file in a directory with 100,000 entries takes O(log t 100000) ≈ a few I/Os, not O(100000) sequential scan.
Section 9 — Common Mistakes
⚠️ Mistake 1 — Confusing B-tree with B+ tree: A B-tree stores data in all nodes (internal + leaf). A B+ tree stores data ONLY in leaf nodes; internal nodes have copies of keys for routing. B+ trees support faster range queries because leaves are linked. When someone says "database B-tree index" they almost always mean B+ tree.
⚠️ Mistake 2 — Using wrong split index: The median key in a full node (2t-1 keys) is at index t-1 (0-based). It's pushed UP to the parent. The left node keeps keys[0..t-2] (t-1 keys) and the right node gets keys[t..2t-2] (t-1 keys). Off-by-one errors here cause nodes to have too many or too few keys.
⚠️ Mistake 3 — Forgetting to split children pointers during node split: When splitting a non-leaf node, you must also split the Children list: left keeps children[0..t-1] (t children) and right gets children[t..2t-1] (t children). Forgetting to split children leaves the tree structurally inconsistent.
⚠️ Mistake 4 — Treating B-tree as a 2-3 tree: A 2-3 tree is a B-tree with t=2 (nodes have 1 or 2 keys; 2 or 3 children). But B-trees generalize to any t ≥ 2. Don't assume all B-trees have the 2-3 structure — database B+ trees typically use t=500 or higher.
Section 10 — Check Your Understanding
Q1. A B-tree with minimum degree t=3 has what minimum and maximum number of keys per internal node?
💡 Rules: non-root nodes have at least t-1 and at most 2t-1 keys.
Minimum: 2 keys (t-1 = 3-1 = 2). Maximum: 5 keys (2t-1 = 2×3-1 = 5).
This means internal nodes have 3 to 6 children (between t and 2t). The root can have as few as 1 key (with 2 children) if the tree has more than one node.
Q2. Why does a B-tree with t=1000 outperform a Red-Black tree for a disk-based database index?
💡 Count the number of disk reads (node accesses) for 10⁹ records in each structure.
Red-Black tree: height ≤ 2 log₂(10⁹+1) ≈ 60. Each node access = 1 disk read → ~60 disk reads per lookup. At ~5ms per HDD read → 300ms per query.
B-tree (t=1000): height ≤ log₁₀₀₀(5×10⁸) ≈ 3. Each node = 1 disk page (16KB, ~1000 keys) → ~3 disk reads per lookup → ~15ms per query.
The B-tree is 20× faster because each disk read brings in 1000 keys (a full page), amortizing the I/O cost across 1000 comparisons done in fast RAM.
Q3. What is the key difference between a B-tree and a B+ tree? Which is better for range queries?
💡 Think about where data is stored in each, and what a range query needs to do.
B-tree: Data (actual records or pointers to records) stored in ALL nodes — both internal and leaf nodes.
B+ tree: Internal nodes store ONLY keys (routing guides). Actual data stored ONLY in leaf nodes. Leaf nodes are linked in a doubly-linked list.
B+ tree is better for range queries: Once you find the start of the range (one O(log_t n) search), you can scan forward through the leaf linked list without any tree traversal. A B-tree range query requires an in-order tree traversal, which is more complex and involves revisiting internal nodes.
Q4. When inserting into a B-tree, what causes a "split" and what happens during the split?
💡 What condition triggers a split? Which key moves up to the parent?
A split is triggered when a node is full — it already has 2t-1 keys and we need to insert another.
During split:
1. The median key (at index t-1) is promoted up to the parent node.
2. The full node becomes the left child with keys[0..t-2] (t-1 keys).
3. A new right child is created with keys[t..2t-2] (t-1 keys).
4. If the node is non-leaf, children[0..t-1] go to left, children[t..2t-1] go to right.
The parent gains one key and one additional child. If the parent was also full, it must be split too (proactive splitting avoids this by pre-splitting on the way down).
Q5. Why is proactive splitting (split on the way DOWN) preferred over bottom-up splitting (split on the way UP)?
💡 Think about how many passes through the tree each approach needs.
Bottom-up splitting (insert at leaf, then propagate splits upward if needed) requires keeping a stack of the path from root to leaf, then possibly traversing back up for cascading splits. This means re-reading nodes that were already read → extra disk I/Os.
Proactive (top-down) splitting: On the way DOWN, whenever you encounter a full node, split it preemptively. By the time you reach the leaf, you're guaranteed the leaf's parent is not full, so no upward cascade is needed. This is a single-pass operation — no stack needed, no re-reads. The implementation in this module uses proactive splitting.
